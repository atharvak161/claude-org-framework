# BUG-016 — Period Comparison: logExp returns negative when spending exceeds income in log
**Severity:** Medium
**File:** js/pages/analytics.js (line 633)
**Symptom:** In the Period Comparison table, the "Last Month" and "Same Month Last Year" values in the Expenses row can show negative numbers when the logged month had a deficit (spent more than net income). A negative expenses figure is meaningless and renders as e.g. `−£200` in the Expenses row.
**Root cause:** `logExp(entry) = (entry.netGBP || 0) - (entry.savedGBP || 0)`. This derives expenses as net pay minus savings. However `savedGBP` in the log schema represents actual savings/surplus, which can be negative if the user ran a deficit. In a deficit month, `savedGBP` is negative, so `logExp = netGBP - (negative) = netGBP + abs(deficit)`, which can exceed netGBP — or if the field is stored differently (positive surplus only), a month with zero surplus gives `logExp = netGBP`, which may overstate expenses. More critically, if `savedGBP` is stored as negative during a deficit, the formula produces expenses > netGBP, which is plausible, but if it is stored as 0 (clamped), expenses = netGBP and surplus is silently lost.
**Root cause (precise):** The formula assumes `savedGBP` is always non-negative and equals `netGBP - actualExpenses`. If the log stores `savedGBP` as the raw surplus (which can be negative), the subtraction double-counts the deficit. The log schema and this formula must agree on whether `savedGBP` is clamped.
**Reproduction:** Save a monthly log snapshot for a month where expenses exceed income. Verify what value is stored in `savedGBP`. Then check the Period Comparison Expenses cell for that month — it may show a negative or inaccurate number.
**Fix hint:** Either (a) store `expensesGBP` explicitly in the monthly log, or (b) clamp `savedGBP` to 0 when logging and document that `logExp` is only valid when `savedGBP >= 0`. Option (a) is more robust.
