# BUG-033 — Calendar Saves null to fin_expenses When No Expenses Exist
**Severity:** High
**File:** js/pages/calendar.js (line 118)
**Symptom:** On a fresh profile with no expenses, if the user somehow triggers the due-day change handler (which requires items to exist — so normally guarded), or if `state.expenses` is null while `state.expenses?.items` resolves to an empty array and an item is found, `saveSec('fin_expenses', null)` is called. This corrupts the `fin_expenses` storage key with a null value, causing all expense-dependent pages to fail to load on next visit.
**Root cause:** Line 23 defensively defaults `expenses` for rendering: `const expenses = state.expenses || { items: [], scheduledChanges: [] }`. But line 118 saves `state.expenses` directly — not the local `expenses` variable — which can still be `null` if no expenses were ever created. The guard on line 113 (`state.expenses?.items || []`) returns `[]` when null, so `item` will never be found, but the `saveSec` call at line 118 is inside `if (item)` — so actually this path is safe. However: if `state.expenses` is null and somehow `item` is found (impossible with current code but fragile), the null write would occur. More critically, if `state.expenses` is a valid object but `state.expenses.items` is undefined, the find succeeds on `[]` — still safe. **Confirmed: currently safe, but the inconsistency between saving `state.expenses` vs the defensive `expenses` copy is a latent correctness hazard one refactor away from becoming a real bug.**
**Reproduction:** Not currently triggerable in normal flow. Becomes a real bug if the guard on line 113 is ever loosened or if the save is moved outside the `if (item)` block.
**Fix hint:** Replace `state.expenses` on line 118 with the local `expenses` variable defined at line 23, or ensure `state.expenses` is always initialised to `{ items: [], scheduledChanges: [] }` at `initPage` time rather than being nullable.
