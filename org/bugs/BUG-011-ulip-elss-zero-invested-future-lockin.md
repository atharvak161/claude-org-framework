# BUG-011 — ULIP/ELSS: invested=0 when lock-in date is far in the future
**Severity:** High
**File:** js/pages/assets.js (lines 756–763, 801–806)
**Symptom:** For a ULIP or ELSS where the lock-in date is more than 5 years (ULIP) or 3 years (ELSS) away from today, the estimated invested amount is calculated as £0. The Gain/Loss cell then shows the full current value as a gain (e.g. +£5,000 when nothing meaningful has been computed), and ROAI % displays as effectively infinite/incorrect.
**Root cause:** `start` is computed as `lockInDate − 5yr` (ULIP) or `lockInDate − 3yr` (ELSS). If the lock-in date is far enough in the future, `start` itself lands in the future. `monthsBetween(start, today)` returns `Math.max(0, ...)` = 0, so `invested = premium * 0 = 0`. The `invested = 0` case is not guarded in `gainCell` — only `invested === null` is checked — so `gainCell` proceeds with `g = valueGBP - 0 = valueGBP`, rendering a spurious positive gain.
**Reproduction:** Add a ULIP with a `lockInDate` more than 5 years from today (e.g. today is 2026-06-13, set lockInDate = 2032-01-01). The start date becomes 2027-01-01, which is in the future. `monthsBetween` returns 0, `invested = 0`.
**Fix hint:** (1) Guard against `start > today` before computing invested — if `start` is in the future, set `invested = null` rather than computing 0. Alternatively, clamp `start = new Date(Math.min(start.getTime(), today.getTime()))`. (2) Also guard `gainCell` and `rowBadge` against `r.invested === 0` the same way they guard `null` — or unify the null/zero handling so 0 is treated as "unknown" the same way.
