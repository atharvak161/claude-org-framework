# 05.1 — Serverless Attack Surface

## What This Note Covers

What makes serverless security fundamentally different from traditional compute, the attack surface unique to event-driven architectures, and how the threat model shifts when there's no persistent OS to target.

---

## What Is Serverless?

Serverless computing lets you run code without managing servers. You write a function, deploy it, and the cloud provider handles:
- Provisioning compute
- Scaling (zero to thousands of instances)
- OS patching and maintenance
- Runtime environment

**You are responsible for:**
- Function code and its dependencies
- IAM execution role and permissions
- Configuration (environment variables, triggers, timeouts)
- Input validation and output handling

The three major platforms:

| Platform | AWS | Azure | GCP |
|---|---|---|---|
| **Product** | Lambda | Azure Functions | Cloud Functions / Cloud Run |
| **Triggers** | S3, SQS, SNS, API GW, DynamoDB, EventBridge | HTTP, Timer, Event Hub, Blob Storage, Queue | HTTP, Pub/Sub, GCS, Firestore, Scheduler |
| **Execution Role** | IAM Role | Managed Identity | Service Account |
| **Runtime** | Python, Node, Java, Go, Ruby, .NET, custom | C#, Java, JS, Python, PowerShell | Python, Node, Go, Java, Ruby, PHP |

---

## How Serverless Differs from Traditional Compute

### What You Lose as an Attacker

```
Traditional server attack:
  RCE → persistent shell → enumerate OS → lateral move via network → dump creds → pivot

Serverless attack:
  No persistent shell (function dies after execution)
  No traditional network pivot (functions are isolated per invocation)
  No OS to enumerate (runtime is managed, filesystem is ephemeral)
  No rootkit or backdoor persistence (no persistent process)
```

### What You Gain as an Attacker

```
Serverless attack surface:
  IAM execution role (often wildcard permissions)
  Environment variables (where secrets live instead of config files)
  Function code (application vulnerabilities in the business logic)
  Event sources (attacker-controlled input → injection attacks)
  Cold start timing (information leakage)
  Shared runtime environment (tenant isolation concerns)
```

> [!tip] The attack surface moves from OS to IAM + code
> In serverless, you can't exploit the OS. But the IAM role, the environment variables, and the function's business logic are all entirely the customer's responsibility — and frequently misconfigured.

## The Serverless Threat Model

### Threat 1: Over-Permissive Execution Role

This is the equivalent of a privileged container — a function with too many IAM permissions.

```
Lambda with roles/AdministratorAccess:
  Any code injected into the function → acts as admin → modifies IAM → persistence
  
Lambda with s3:* on *:
  Code injection → read every bucket in the account → exfiltrate all data
  
GCP Cloud Function with roles/editor on project:
  Execution role abuse → access every GCP service in the project
```

### Threat 2: Sensitive Data in Environment Variables

Environment variables are the configuration mechanism for serverless. Developers use them for:
- Database connection strings (with credentials)
- API keys for downstream services
- Encryption keys
- Feature flags with sensitive values

Anyone with `lambda:GetFunctionConfiguration` (AWS) or `cloudfunctions.functions.get` (GCP) can read them.

### Threat 3: Event Injection

Serverless functions process events from many sources. If the function trusts event data without validation, attackers who can influence the event source can inject malicious payloads.

**Sources of attacker-controlled input:**
- API Gateway HTTP requests (direct)
- SQS messages (if attacker can write to the queue)
- S3 events (if attacker can write objects to the bucket)
- SNS notifications (if attacker can publish to the topic)
- DynamoDB streams (if attacker can write to the table)

### Threat 4: Dependency Vulnerabilities

Serverless functions include third-party libraries in their deployment package. The cloud provider patches the runtime (Python interpreter, Node.js) but not your dependencies.

```bash
# A Lambda function's requirements.txt with a vulnerable library:
requests==2.25.0    # CVE-2023-32681 (fixed in 2.31.0)
cryptography==3.4   # multiple CVEs
flask==1.1.2        # CVE-2023-30861
```

### Threat 5: Unauthenticated Invocation

Functions exposed via HTTP can be configured with no authentication. If misconfigured as public, any internet user can invoke them directly.

### Threat 6: Side-Channel Information from Cold Starts

Cold starts (first invocation after a period of inactivity) take longer. If timing differences are observable, this can leak whether a function was recently invoked — relevant for timing attacks on OTP verification, rate limiting bypass checks, etc.

## Serverless Attack Chain — General Pattern

```
1. DISCOVER
   → Enumerate functions (lambda list-functions, gcloud functions list)
   → Find function trigger types and event sources
   → Identify public HTTP endpoints

2. RECON
   → Read function configuration (env vars, execution role, timeout, memory)
   → Download and analyse function code
   → Identify event sources you can influence

3. EXPLOIT
   → Abuse execution role (if overprivileged) via code injection or direct API
   → Extract environment variable secrets
   → Inject malicious event payloads
   → Exploit application vulnerabilities in function code

4. PERSIST / ESCALATE
   → Use execution role to create IAM backdoor
   → Use execution role to access Secrets Manager / Key Vault
   → Chain to other services with the function's permissions
   → Modify function code to add backdoor (if you have update permissions)
```

## Serverless vs Traditional — Side-by-Side Comparison

| Attack Objective | Traditional Compute | Serverless |
|---|---|---|
| Initial access | RCE via CVE, SSH brute force, web shell | SSRF, event injection, public function endpoint |
| Credential access | /etc files, memory dump, env vars | Environment variables, execution role via IMDS |
| Lateral movement | Network pivot, pass-the-hash | IAM role chaining, assume-role to other services |
| Persistence | Cron job, rootkit, SSH key | IAM backdoor, function code modification, trigger addition |
| Exfiltration | Network egress | S3 replication, SQS, API calls to external endpoints |
| Defence evasion | Log deletion, process hiding | Use services that don't generate CloudTrail data events |

## The Execution Role — Central to Everything

Every serverless function has an IAM execution role (or equivalent). When the function runs, it has the permissions of that role. This means:

```python
# Any code running inside a Lambda function can use boto3 with the execution role
import boto3

# No credentials needed — uses execution role automatically
s3 = boto3.client('s3')
response = s3.list_buckets()  # uses Lambda's execution role
print(response['Buckets'])

iam = boto3.client('iam')
iam.attach_user_policy(  # if role has iam:AttachUserPolicy
    UserName='attacker',
    PolicyArn='arn:aws:iam::aws:policy/AdministratorAccess'
)
```

> [!danger] Code injection in a Lambda function = acting as its IAM role
> If you can inject code into a Lambda function's execution flow (via event injection, dependency confusion, or function code modification), you operate with the function's IAM permissions. If those permissions are broad, the impact is account-wide.

## Quick Reference — Serverless Attack Surface Checklist

```
DISCOVERY:
[ ] List all functions across all regions
[ ] Identify trigger types (HTTP, S3, SQS, scheduled)
[ ] Find publicly accessible HTTP endpoints

CONFIGURATION REVIEW:
[ ] Read environment variables (lambda:GetFunctionConfiguration)
[ ] Check execution role permissions
[ ] Download and review function code
[ ] Check runtime version (EOL runtimes)

EXPLOITATION:
[ ] Test HTTP endpoints for injection (SQLi, SSTI, command injection)
[ ] Test event sources for payload injection
[ ] Check for missing input validation on event data
[ ] Verify execution role blast radius

CLOUD-SPECIFIC:
AWS:    aws lambda list-functions --region REGION
        aws lambda get-function-configuration --function-name NAME
        aws lambda get-function --function-name NAME --query Code.Location

GCP:    gcloud functions list --project PROJECT_ID
        gcloud functions describe NAME --region REGION
        
Azure:  az functionapp list --output table
        az functionapp config appsettings list --name NAME --resource-group RG
```
