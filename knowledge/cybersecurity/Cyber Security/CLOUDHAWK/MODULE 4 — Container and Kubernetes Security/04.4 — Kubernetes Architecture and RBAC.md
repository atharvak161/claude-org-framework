# 04.4 — Kubernetes Architecture and RBAC

## What This Note Covers

Kubernetes architecture from a security perspective — the control plane, worker nodes, and every component that can be attacked. Then Kubernetes RBAC — how it works, how it's misconfigured, and what those misconfigurations let an attacker do.

---

## Kubernetes Architecture

Kubernetes (K8s) is a container orchestration platform. It manages where and how containers run across a cluster of machines.

```
Kubernetes Cluster

Control Plane (master node):
├── kube-apiserver       ← all interactions go through here
├── etcd                 ← cluster state database (contains EVERYTHING)
├── kube-scheduler       ← decides which node runs which pod
├── kube-controller-manager ← maintains desired state
└── cloud-controller-manager ← cloud provider integration

Worker Nodes (run workloads):
├── kubelet              ← agent, talks to API server, runs pods
├── kube-proxy           ← network rules, service routing
└── Container Runtime    ← Docker, containerd, CRI-O
```

### From an Attacker's Perspective

| Component | Why It Matters |
|---|---|
| **kube-apiserver** | Every action in K8s is an API call here — main target for initial access |
| **etcd** | Contains ALL cluster state including secrets — direct access = game over |
| **kubelet** | Runs on every node — API on port 10250/10255 — often misconfigured |
| **service account tokens** | Every pod gets one — frequently over-permissioned |
| **kube-proxy** | Network routing — relevant for lateral movement |

## Kubernetes Objects — What You're Attacking

### Pods

A **pod** is the smallest deployable unit in Kubernetes — one or more containers running together.

```bash
# List pods
kubectl get pods
kubectl get pods --all-namespaces  # all namespaces
kubectl get pods -n kube-system    # system namespace

# Get pod details
kubectl describe pod POD_NAME -n NAMESPACE

# Execute inside a pod
kubectl exec -it POD_NAME -n NAMESPACE -- /bin/bash
kubectl exec -it POD_NAME -n NAMESPACE -c CONTAINER_NAME -- /bin/sh

# Get pod spec (YAML — shows security context, volumes, service account)
kubectl get pod POD_NAME -n NAMESPACE -o yaml
```

### Namespaces

Logical partitions within a cluster. RBAC can be scoped per-namespace.

```bash
kubectl get namespaces
kubectl config set-context --current --namespace=TARGET_NAMESPACE
```

### Services

Network endpoints that expose pods. Service types:
- `ClusterIP` — internal only
- `NodePort` — exposed on every node's IP at a static port
- `LoadBalancer` — cloud load balancer (external)

```bash
kubectl get services --all-namespaces
kubectl get service SERVICE_NAME -o yaml
```

### Secrets

Kubernetes Secrets store sensitive data — but are only **base64 encoded**, not encrypted (unless encryption at rest is configured).

```bash
# List secrets
kubectl get secrets --all-namespaces

# Decode a secret
kubectl get secret SECRET_NAME -n NAMESPACE -o yaml
# Find the data values and decode:
echo "BASE64_VALUE" | base64 --decode

# Common secrets to look for:
# - docker-registry credentials (image pull secrets)
# - TLS certificates
# - Service account tokens
# - Application credentials (database passwords, API keys)
```

> [!danger] Kubernetes secrets are not encrypted by default
> Without encryption at rest configured (and only ~30% of clusters have it), secrets in etcd are only base64 encoded. Anyone who can read etcd directly — or has `get/list` on secrets — gets plaintext credentials.

### ConfigMaps

ConfigMaps store non-sensitive configuration, but developers frequently put sensitive data in them by mistake.

```bash
kubectl get configmaps --all-namespaces
kubectl get configmap CONFIG_NAME -n NAMESPACE -o yaml
# Look for: database URLs with credentials, API endpoints with keys
```

### Service Accounts

Every pod runs as a Kubernetes service account. The token is automatically mounted at:
`/var/run/secrets/kubernetes.io/serviceaccount/token`

```bash
# From inside a pod — read the service account token
cat /var/run/secrets/kubernetes.io/serviceaccount/token

# Decode the JWT to see what it contains
echo "TOKEN" | cut -d. -f2 | base64 -d 2>/dev/null | python3 -m json.tool
# Shows: namespace, service account name, expiry

# Use the token to talk to the API server
APISERVER=https://kubernetes.default.svc
SERVICEACCOUNT=/var/run/secrets/kubernetes.io/serviceaccount
TOKEN=$(cat ${SERVICEACCOUNT}/token)
CACERT=${SERVICEACCOUNT}/ca.crt

# Query the API server as this service account
curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  $APISERVER/api/v1/namespaces/default/pods
```

## Kubernetes RBAC

RBAC controls who can do what in the cluster. Three components:

```
Subject (who)  →  RoleBinding  →  Role (what)  →  at Scope (where)
```

### Subjects

- `User` — external users (managed by the authentication mechanism, not K8s itself)
- `Group` — groups of users
- `ServiceAccount` — pod identities

### Roles and ClusterRoles

A **Role** grants permissions within a **single namespace**.
A **ClusterRole** grants permissions **cluster-wide** (or can be bound to a namespace via RoleBinding).

```yaml
# Example Role: allows listing and getting pods in the default namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
```

**Critical verbs:**

| Verb | What It Allows |
|---|---|
| `get` | Read a specific resource by name |
| `list` | List all resources of a type |
| `watch` | Stream updates to resources |
| `create` | Create new resources |
| `update` / `patch` | Modify existing resources |
| `delete` | Delete resources |
| `exec` | Execute commands in pods |
| `bind` | Create RoleBindings (escalation!) |
| `escalate` | Grant roles you don't have (escalation!) |
| `impersonate` | Act as another user/SA |
| `*` | All verbs — admin |

### RoleBindings and ClusterRoleBindings

```bash
# List all RoleBindings (namespace-scoped)
kubectl get rolebindings --all-namespaces -o wide

# List all ClusterRoleBindings (cluster-wide)
kubectl get clusterrolebindings -o wide

# Find who has cluster-admin
kubectl get clusterrolebindings \
  -o jsonpath='{range .items[?(@.roleRef.name=="cluster-admin")]}{.subjects}{"\n"}{end}'

# Check your own permissions
kubectl auth can-i --list                    # in current namespace
kubectl auth can-i --list --all-namespaces   # across all namespaces
kubectl auth can-i create pods
kubectl auth can-i exec pods
kubectl auth can-i get secrets
```

## RBAC Misconfigurations — Attack Surface

### Misconfiguration 1: Wildcard Permissions

```yaml
# Critical — full cluster admin via RBAC
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["*"]
# This is equivalent to cluster-admin
```

### Misconfiguration 2: Overprivileged Service Account

```yaml
# Pod spec with cluster-admin service account
spec:
  serviceAccountName: cluster-admin-sa
  # Now every pod using this SA has full cluster control
```

```bash
# Find service accounts with ClusterRoleBindings
kubectl get clusterrolebindings -o json | \
  python3 -c "
import json, sys
data = json.load(sys.stdin)
for item in data['items']:
    subjects = item.get('subjects', [])
    for s in subjects:
        if s.get('kind') == 'ServiceAccount':
            print(f\"{item['roleRef']['name']} → {s['namespace']}/{s['name']}\")
"
```

### Misconfiguration 3: `pods/exec` Permission

If a subject can `exec` into pods, they can:
- Access data inside running pods
- Steal service account tokens
- Use the pod as a pivot point

```bash
kubectl auth can-i exec pods
# "yes" → you can exec into any pod in this namespace/cluster
```

### Misconfiguration 4: `secrets` List/Get

```bash
kubectl auth can-i list secrets
kubectl auth can-i get secrets

# If yes — dump all secrets
kubectl get secrets --all-namespaces -o yaml
```

### Misconfiguration 5: `bind` Verb

The `bind` verb allows creating RoleBindings — granting yourself roles you don't have.

```bash
kubectl auth can-i bind clusterrole --subresource clusterrolebindings

# If yes → escalation:
kubectl create clusterrolebinding myself-admin \
  --clusterrole=cluster-admin \
  --user=YOUR_USERNAME
```

### Misconfiguration 6: `impersonate` Verb

Allows acting as another user, group, or service account.

```bash
kubectl auth can-i impersonate users

# If yes → impersonate a more privileged user
kubectl --as=admin get secrets --all-namespaces
kubectl --as=system:serviceaccount:kube-system:default get pods --all-namespaces
```

## The `default` Service Account — A Common Mistake

Every namespace has a `default` service account. Pods that don't specify a `serviceAccountName` use it automatically. In misconfigured clusters, this account has been granted elevated permissions — giving every pod in the namespace those permissions.

```bash
# Check what the default SA can do in each namespace
kubectl auth can-i --list \
  --as=system:serviceaccount:default:default
  
kubectl auth can-i --list \
  --as=system:serviceaccount:kube-system:default
```

## Quick Reference — Architecture and RBAC

```bash
# Cluster structure
kubectl get nodes
kubectl get namespaces
kubectl get pods --all-namespaces
kubectl get services --all-namespaces

# RBAC enumeration
kubectl get clusterroles
kubectl get clusterrolebindings -o wide
kubectl get rolebindings --all-namespaces -o wide

# Find cluster-admin subjects
kubectl get clusterrolebindings \
  -o jsonpath='{range .items[?(@.roleRef.name=="cluster-admin")]}{.subjects}{"\n"}{end}'

# Permission check
kubectl auth can-i --list
kubectl auth can-i exec pods
kubectl auth can-i get secrets
kubectl auth can-i list secrets

# Service account token (from inside pod)
cat /var/run/secrets/kubernetes.io/serviceaccount/token

# Secrets
kubectl get secrets --all-namespaces
kubectl get secret NAME -n NAMESPACE -o yaml | \
  grep -A1 "data:" | awk '{print $2}' | base64 -d
```
