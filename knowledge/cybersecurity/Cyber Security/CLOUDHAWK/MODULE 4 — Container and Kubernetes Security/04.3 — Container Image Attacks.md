# 04.3 — Container Image Attacks

## What This Note Covers

How to extract secrets from container image layers, analyse Dockerfiles for vulnerabilities, scan images for CVEs, attack container registries, and understand supply chain risks in container images.

---

## Why Container Images Are a Target

Container images are built in layers, and every layer is permanently stored in the image's history. Developers frequently:
- Add credentials in one layer and delete them in the next (the delete doesn't remove the data from the prior layer)
- Hardcode API keys, passwords, and tokens in environment variables (visible in image metadata)
- Base their images on outdated, CVE-laden base images
- Pull images from untrusted registries (supply chain risk)

## Extracting Secrets from Image Layers

### Method 1: Docker History

```bash
# Show all layers and the commands that created them
docker history IMAGE_NAME --no-trunc

# Look for:
# - RUN commands that set environment variables with secrets
# - ADD/COPY commands that copied credential files
# - ENV instructions with passwords
# Example of a bad layer:
# RUN curl -H "Authorization: Bearer abc123secret" https://api.example.com/setup
```

### Method 2: docker inspect (Environment Variables)

```bash
# Inspect image metadata including environment variables
docker inspect IMAGE_NAME

# Extract just env vars
docker inspect IMAGE_NAME \
  --format='{{range .Config.Env}}{{println .}}{{end}}'

# Secrets found here are baked into the image permanently
# Common findings:
# DATABASE_URL=postgres://admin:password@db:5432/prod
# AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI...
# GITHUB_TOKEN=ghp_...
```

### Method 3: Export and Analyse All Layers

```bash
# Save image to tar archive
docker save IMAGE_NAME -o image.tar

# Extract
mkdir /tmp/image-analysis
tar xf image.tar -C /tmp/image-analysis/
ls /tmp/image-analysis/
# Each layer is a directory containing a layer.tar

# Extract and search each layer
cd /tmp/image-analysis
for layer_dir in */; do
  if [ -f "${layer_dir}layer.tar" ]; then
    echo "=== Extracting: ${layer_dir} ==="
    mkdir -p "/tmp/layers/${layer_dir}"
    tar xf "${layer_dir}layer.tar" -C "/tmp/layers/${layer_dir}" 2>/dev/null
  fi
done

# Search all extracted layers for secrets
grep -r --include="*.env" \
  --include="*.py" \
  --include="*.js" \
  --include="*.json" \
  --include="*.yml" \
  --include="*.yaml" \
  --include="*.conf" \
  --include="*.ini" \
  -riE "password|secret|api_key|aws_access|private_key|token|credential" \
  /tmp/layers/ 2>/dev/null

# Search for AWS credential patterns specifically
grep -rP "AKIA[0-9A-Z]{16}" /tmp/layers/ 2>/dev/null

# Search for private key headers
grep -r "BEGIN RSA PRIVATE KEY\|BEGIN EC PRIVATE KEY\|BEGIN PRIVATE KEY" \
  /tmp/layers/ 2>/dev/null

# Search for service account JSON files (GCP)
grep -r '"type": "service_account"' /tmp/layers/ 2>/dev/null
```

### Method 4: truffleHog on Images

```bash
# truffleHog scans Docker images for secrets using regex + entropy detection
pip3 install trufflehog

# Scan a local image
trufflehog docker --image IMAGE_NAME

# Scan a remote image (pulls automatically)
trufflehog docker --image registry.example.com/team/app:latest

# Output: verified and unverified secrets found across all image layers
# Verified = secret confirmed active (truffleHog calls the API to check)
```

### Method 5: dive — Interactive Layer Explorer

```bash
# dive provides an interactive TUI to explore image layers
# Install
brew install dive           # macOS
apt-get install dive        # Linux (check releases)

# Run
dive IMAGE_NAME

# Interface:
# Left panel: layer list (shows what command created each layer)
# Right panel: filesystem changes in the selected layer
# Look for: modified/added files that contain credentials
# Use filter: type "secret" or "credential" to filter filenames
```

## Dockerfile Security Analysis

Dockerfiles contain the build instructions and are a rich source of security findings.

```bash
# Common Dockerfile security issues:

# 1. Hardcoded credentials
ENV DATABASE_PASSWORD=SuperSecret123  # CRITICAL — baked into image

# 2. Running as root
# No USER instruction = runs as root
# All containers without USER are root by default

# 3. Outdated base image
FROM ubuntu:18.04  # EOL — unpatched CVEs

# 4. Unnecessary packages / attack surface
RUN apt-get install -y curl wget netcat nmap  # debugging tools left in prod image

# 5. Secrets passed via build args (visible in history)
ARG DB_PASSWORD  # ARG values appear in docker history

# 6. Sensitive files copied into image
COPY .env /app/.env        # credentials copied into image
COPY ~/.ssh /root/.ssh     # SSH keys in image

# 7. No HEALTHCHECK — makes it harder to detect tampering

# 8. Mutable tags
FROM python:latest  # "latest" tag changes — supply chain risk

# Automated Dockerfile analysis:
# hadolint (Dockerfile linter)
docker run --rm -i hadolint/hadolint < Dockerfile
```

## CVE Scanning with Trivy

**Trivy** by Aqua Security is the industry-standard open-source container image vulnerability scanner.

```bash
# Install
brew install aquasecurity/trivy/trivy    # macOS
apt-get install trivy                    # Linux (add repo first)

# Or via Docker
docker run --rm aquasec/trivy image IMAGE_NAME

# Scan a local image
trivy image IMAGE_NAME

# Scan a remote image
trivy image nginx:1.21.0

# Output includes:
# - Library: package name
# - Vulnerability ID: CVE number
# - Severity: CRITICAL / HIGH / MEDIUM / LOW
# - Fixed version: version to upgrade to

# Filter by severity
trivy image --severity CRITICAL,HIGH IMAGE_NAME

# Output formats
trivy image --format json IMAGE_NAME -o results.json
trivy image --format table IMAGE_NAME

# Scan a tar archive (exported image)
trivy image --input image.tar

# Scan for secrets (not just CVEs)
trivy image --scanners secret IMAGE_NAME

# Scan a filesystem (Dockerfile directory)
trivy fs /path/to/project/

# Scan a git repository
trivy repo https://github.com/target/repo
```

### Interpreting Trivy Output

```
IMAGE_NAME (ubuntu 22.04)
=========================
Total: 23 (CRITICAL: 2, HIGH: 8, MEDIUM: 10, LOW: 3)

Library          │ Vulnerability   │ Severity │ Installed │ Fixed    │ Title
─────────────────┼─────────────────┼──────────┼───────────┼──────────┼───────────────────────────────
libssl1.1        │ CVE-2023-0286   │ CRITICAL │ 1.1.1f    │ 1.1.1n   │ X.400 address type confusion
libc-bin         │ CVE-2023-4911   │ CRITICAL │ 2.35-0    │ 2.35-3   │ Looney Tunables buffer overflow
```

**Prioritise:** CRITICAL with a fixed version available → immediate update required.

## Container Registry Attacks

### Unauthenticated Registry Access

```bash
# Check if registry API is open
curl https://REGISTRY/v2/
# 200 OK with no auth challenge → unauthenticated access

# List all repositories (requires catalog access)
curl https://REGISTRY/v2/_catalog

# List tags for a repository
curl https://REGISTRY/v2/REPO/tags/list

# Get manifest (image metadata)
curl https://REGISTRY/v2/REPO/manifests/TAG

# Pull the image using Docker
docker pull REGISTRY/REPO:TAG
```

### Exposed Docker Registry (port 5000)

Self-hosted registries often run on port 5000 without TLS or authentication:

```bash
# Discover via Shodan/Censys
# shodan query: port:5000 product:"Docker Registry"

# Direct access
curl http://TARGET:5000/v2/_catalog
# Lists all repositories

# Pull an image
docker pull TARGET:5000/REPO:TAG
# Docker may complain about insecure registry — add to daemon.json:
# "insecure-registries": ["TARGET:5000"]
```

### AWS ECR — Credential Requirements

```bash
# ECR requires temporary credentials obtained via AWS CLI
aws ecr get-login-password --region REGION | \
  docker login --username AWS --password-stdin ACCOUNT_ID.dkr.ecr.REGION.amazonaws.com

# List repositories
aws ecr describe-repositories --region REGION

# List images in a repository
aws ecr list-images --repository-name REPO_NAME --region REGION

# Pull (after login)
docker pull ACCOUNT_ID.dkr.ecr.REGION.amazonaws.com/REPO:TAG
```

## Supply Chain Attacks via Container Images

### Typosquatting on Docker Hub

Attackers publish images with names similar to popular ones:
- `ngixn` instead of `nginx`
- `ubuntu-latest` instead of `ubuntu:latest`
- `python-3.11` instead of `python:3.11`

These images often contain backdoors or crypto-miners.

```bash
# Always use official images with specific version tags
FROM python:3.11.6-slim-bookworm  # specific version, official, slim variant

# Verify image digests
docker pull nginx:1.25.3@sha256:DIGEST  # pin to exact digest
docker trust inspect --pretty nginx     # check Docker Content Trust signatures
```

### Compromised Base Images

If a base image is compromised (via a compromised registry account or supply chain attack), every image built on top of it is affected.

```bash
# Check image provenance
docker inspect IMAGE --format='{{.RepoDigests}}'
# Should match the official published digest

# Enable Docker Content Trust (enforces signed images)
export DOCKER_CONTENT_TRUST=1
docker pull nginx:latest  # fails if not signed
```
