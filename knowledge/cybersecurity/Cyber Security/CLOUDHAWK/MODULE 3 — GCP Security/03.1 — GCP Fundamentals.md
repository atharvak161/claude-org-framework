# 03.1 — GCP Fundamentals

## What This Note Covers

The foundational GCP concepts every pentester must know: the resource hierarchy, IAM model, service accounts, and how GCP's permission system differs meaningfully from AWS and Azure.

---

## GCP Resource Hierarchy

GCP uses a strict parent-child hierarchy where IAM policies are inherited downward:

```
Organisation (root — tied to Google Workspace or Cloud Identity domain)
└── Folder (optional grouping — departments, teams, environments)
    └── Project (primary resource and billing boundary)
        └── Resources (Compute Engine VMs, Cloud Storage buckets, Cloud Functions, etc.)
```

**IAM inheritance:** Permissions granted at a higher level are inherited by all resources below. A binding at the organisation level applies to all projects. A binding at the project level applies to all resources in that project.

> [!warning] Inheritance amplifies mistakes
> A service account granted `roles/editor` at the organisation level has editor access across every project in the organisation. Always check what scope a binding is applied at, not just what role it grants.

### Organisations

The root node. Tied to a Google Workspace or Cloud Identity domain (e.g., `company.com`). Org-level policies and IAM bindings apply everywhere below.

```bash
# List organisations you have access to
gcloud organizations list

# Get IAM policy at org level (requires resourcemanager.organizations.getIamPolicy)
gcloud organizations get-iam-policy ORG_ID
```

### Folders

Optional intermediate grouping. Large enterprises use folders for departments (Engineering, Finance, HR) or environments (Production, Development, Staging).

```bash
# List folders in an organisation
gcloud resource-manager folders list --organization ORG_ID

# Get IAM policy on a folder
gcloud resource-manager folders get-iam-policy FOLDER_ID
```

### Projects

The primary boundary in GCP. Everything lives in a project:
- Resources (VMs, buckets, databases, functions)
- Service accounts
- APIs enabled/disabled
- Billing account linked

```bash
# List all projects you have access to
gcloud projects list

# Switch to a project
gcloud config set project PROJECT_ID

# Get project details
gcloud projects describe PROJECT_ID

# Get IAM policy on a project (critical — shows all bindings)
gcloud projects get-iam-policy PROJECT_ID
```

> [!tip] `gcloud projects list` is your first enumeration command
> How many projects does the compromised identity have access to? Often more than expected. Each project is a separate attack surface. Enumerate them all.

## GCP IAM Model

GCP IAM uses a **binding** model: you bind a **member** (who) to a **role** (what) at a **resource** (where/scope).

```
IAM Policy = list of bindings
Binding = { role: "roles/storage.objectViewer", members: ["user:alice@company.com"] }
```

### Members (Who)

| Member Type | Syntax | Description |
|---|---|---|
| Google Account | `user:alice@company.com` | Individual Google/G Suite account |
| Service Account | `serviceAccount:sa@PROJECT.iam.gserviceaccount.com` | Non-human identity for applications |
| Google Group | `group:devs@company.com` | Collection of users |
| Google Workspace Domain | `domain:company.com` | All users in the domain |
| All Authenticated Users | `allAuthenticatedUsers` | Any authenticated Google account (public!) |
| All Users | `allUsers` | Completely public — no auth required |

> [!danger] allUsers and allAuthenticatedUsers are public
> `allUsers` on a Cloud Storage bucket = fully public. `allAuthenticatedUsers` = any Google account in the world can access. Both are common critical misconfigurations.

### Roles (What)

**Primitive Roles** (legacy, avoid — too broad):

| Role | Permissions |
|---|---|
| `roles/owner` | Full control + IAM management |
| `roles/editor` | Read/write on all resources, no IAM |
| `roles/viewer` | Read-only on all resources |

**Predefined Roles** (recommended — service-specific):

Examples:
- `roles/storage.objectViewer` — read GCS objects
- `roles/storage.objectAdmin` — full GCS object control
- `roles/compute.instanceAdmin.v1` — full VM management
- `roles/iam.serviceAccountUser` — impersonate service accounts
- `roles/iam.serviceAccountTokenCreator` — create tokens for any SA

**Custom Roles** — customer-defined sets of permissions.

```bash
# List all predefined roles
gcloud iam roles list --format="table(name, title)"

# List roles available in a project
gcloud iam roles list --project PROJECT_ID

# Get details of a specific role (what permissions it grants)
gcloud iam roles describe roles/storage.objectAdmin
```

## Service Accounts — The Core of GCP Identity

A **service account** is a non-human Google identity for applications and workloads. Service accounts are used by:
- Compute Engine VMs (attached at creation)
- Cloud Functions, Cloud Run, GKE pods
- Any application that needs to interact with GCP APIs

### Service Account Types

| Type | Description |
|---|---|
| **User-managed** | Created by you in a project |
| **Google-managed** | Created automatically by GCP for internal use |
| **Default service accounts** | Auto-created when certain APIs are enabled |

```bash
# List all service accounts in a project
gcloud iam service-accounts list --project PROJECT_ID

# Get details of a specific service account
gcloud iam service-accounts describe SA_EMAIL --project PROJECT_ID

# Get the IAM policy on a service account (who can use/impersonate it)
gcloud iam service-accounts get-iam-policy SA_EMAIL --project PROJECT_ID
```

### Service Account Keys

Service accounts can have **JSON key files** — long-lived credentials that allow authenticating as that SA from anywhere. These are the GCP equivalent of AWS access keys and just as dangerous when leaked.

```bash
# List keys for a service account
gcloud iam service-accounts keys list \
  --iam-account SA_EMAIL \
  --project PROJECT_ID

# Create a new key (generates a JSON file — requires iam.serviceAccountKeys.create)
gcloud iam service-accounts keys create key.json \
  --iam-account SA_EMAIL \
  --project PROJECT_ID

# Authenticate using a key file
gcloud auth activate-service-account \
  --key-file key.json

gcloud config set project PROJECT_ID
gcloud auth list  # confirm you're authenticated as the SA
```

> [!danger] Service account JSON keys = the highest-value credential in GCP
> Unlike temporary tokens from the metadata service, SA JSON keys are permanent until explicitly revoked. A leaked key file gives an attacker indefinite access as that service account. Finding one in a GitHub repo, S3 bucket, or config file is a critical finding.

### Default Service Accounts — A Critical Attack Surface

When you enable certain GCP APIs or create certain resources, GCP automatically creates service accounts:

```
Compute Engine default service account:
  PROJECT_NUMBER-compute@developer.gserviceaccount.com
  Default role: roles/editor (on the project!)

App Engine default service account:
  PROJECT_ID@appspot.gserviceaccount.com
  Default role: roles/editor (on the project!)
```

> [!danger] Default service accounts have Editor by default
> The Compute Engine default SA has `roles/editor` on the entire project. Every VM launched without a custom SA gets this identity. Compromise any such VM → steal metadata token → Editor access on the project.

## GCP IAM Roles for Attackers — Key Permissions to Find

| Permission | Why It Matters |
|---|---|
| `iam.serviceAccounts.actAs` | Impersonate any SA — essential for privilege escalation |
| `iam.serviceAccounts.getAccessToken` | Generate OAuth tokens for any SA |
| `iam.serviceAccountKeys.create` | Create permanent key for any SA |
| `resourcemanager.projects.setIamPolicy` | Modify project IAM — add any binding |
| `resourcemanager.organizations.setIamPolicy` | Modify org-level IAM — ultimate escalation |
| `compute.instances.setServiceAccount` | Change VM's SA — escalate via reassignment |
| `iam.roles.update` | Modify custom role permissions |
| `storage.buckets.setIamPolicy` | Make buckets public or grant access |

## GCP IAM Policy — JSON Structure

```json
{
  "bindings": [
    {
      "role": "roles/storage.objectAdmin",
      "members": [
        "serviceAccount:deploy@myproject.iam.gserviceaccount.com",
        "user:alice@company.com"
      ]
    },
    {
      "role": "roles/viewer",
      "members": [
        "allAuthenticatedUsers"
      ]
    }
  ],
  "etag": "BwXgS..."
}
```

```bash
# Get and parse IAM policy
gcloud projects get-iam-policy PROJECT_ID --format=json | python3 -m json.tool

# Check specific role bindings
gcloud projects get-iam-policy PROJECT_ID \
  --flatten="bindings[].members" \
  --format="table(bindings.role, bindings.members)" \
  --filter="bindings.role:roles/owner OR bindings.role:roles/editor"
```

## Org Policies

GCP **Org Policies** are constraints applied at organisation, folder, or project level that restrict what can be done — regardless of IAM permissions.

```bash
# List org policies on a project
gcloud resource-manager org-policies list --project PROJECT_ID

# Describe a specific policy
gcloud resource-manager org-policies describe \
  compute.requireOsLogin \
  --project PROJECT_ID

# Common security-relevant org policies:
# compute.requireOsLogin → forces OS Login instead of SSH keys
# compute.disableSerialPortAccess → disables serial console
# compute.vmExternalIpAccess → restricts VMs from having external IPs
# iam.disableServiceAccountKeyCreation → blocks SA key creation
# iam.disableServiceAccountCreation → blocks new SA creation
```

> [!tip] Org policy bypass is a finding
> If you can modify org policies (`orgpolicy.policies.create` / `delete`), you can remove restrictions. For example, removing `iam.disableServiceAccountKeyCreation` re-enables creating permanent SA keys.

## Quick Reference — GCP Fundamentals Commands

```bash
# Identity and auth
gcloud auth list
gcloud auth activate-service-account --key-file key.json
gcloud config list

# Hierarchy enumeration
gcloud organizations list
gcloud projects list
gcloud resource-manager folders list --organization ORG_ID

# IAM enumeration
gcloud projects get-iam-policy PROJECT_ID
gcloud iam service-accounts list --project PROJECT_ID
gcloud iam service-accounts get-iam-policy SA_EMAIL --project PROJECT_ID
gcloud iam service-accounts keys list --iam-account SA_EMAIL --project PROJECT_ID

# Org policies
gcloud resource-manager org-policies list --project PROJECT_ID

# Switch project
gcloud config set project PROJECT_ID
```
