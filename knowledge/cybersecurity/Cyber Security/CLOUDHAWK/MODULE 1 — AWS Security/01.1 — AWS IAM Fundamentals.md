# 01.1 — AWS IAM Fundamentals

## What This Note Covers

AWS Identity and Access Management (IAM) is the absolute foundation of AWS security. Every action in AWS is governed by IAM. This note covers what IAM is, how it works, every component type, and how each component maps to an attack surface.

---

## What Is AWS IAM?

IAM is AWS's system for controlling **who can do what** across all AWS services and resources. Every API call made to AWS — whether by a human, an application, or an AWS service — goes through IAM for authorisation.

IAM answers two questions:
1. **Authentication:** Are you who you say you are? (Valid credentials?)
2. **Authorisation:** Are you allowed to do what you're asking? (Permissions?)

IAM is **global** — it's not region-specific. An IAM user or role created in one region is available everywhere.

---

## IAM Components

### 1. IAM Users

A **user** is a permanent identity representing a person or application. It has:
- A username
- Optional **password** (for console access)
- Optional **access keys** (for CLI/API access) — up to 2 at a time
- Optional **MFA device**

```bash
# List all IAM users
aws iam list-users

# Get details on a specific user
aws iam get-user --user-name alice

# List access keys for a user
aws iam list-access-keys --user-name alice

# Check when access keys were last used
aws iam get-access-key-last-used --access-key-id AKIA...
```

> [!warning] Access keys are the primary attack surface
> Unlike passwords, access keys work programmatically — no MFA challenge, no browser, just the key pair. A leaked access key gives an attacker full programmatic access as that user, 24/7, from anywhere.

### 2. IAM Groups

A **group** is a collection of IAM users. You attach policies to groups rather than individual users. A user can belong to multiple groups and inherits all permissions from all groups.

```bash
# List all groups
aws iam list-groups

# List users in a group
aws iam get-group --group-name Developers

# List policies attached to a group
aws iam list-attached-group-policies --group-name Developers
```

**Attack relevance:** If you can add your compromised user to a group (via `iam:AddUserToGroup`), you inherit that group's permissions — a common privilege escalation path.

### 3. IAM Roles

A **role** is an identity with permissions, but **no permanent credentials**. It's assumed by:
- AWS services (EC2 instances, Lambda functions, ECS tasks)
- IAM users in the same or different accounts
- Web identity / OIDC federated users
- SAML federated users

When a role is assumed, AWS issues **temporary security credentials** (access key + secret key + session token) that expire (typically 1-12 hours).

```bash
# List all roles
aws iam list-roles

# Get role details including trust policy
aws iam get-role --role-name MyRole

# Assume a role (if you have permission)
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/TargetRole \
  --role-session-name pentest-session

# Use the returned temporary credentials
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
aws sts get-caller-identity  # confirm you're now the role
```

> [!tip] Roles are the gold standard target
> Roles are attached to services. Compromise an EC2 instance → steal its instance profile role credentials → act as that role. This is the most common cloud lateral movement path.

### 4. IAM Policies

A **policy** is a JSON document that defines permissions. It states what **actions** are allowed or denied on what **resources** under what **conditions**.

**Policy types:**

| Type | What It Is |
|---|---|
| **AWS Managed Policy** | Pre-built by AWS, e.g. `AdministratorAccess`, `ReadOnlyAccess` |
| **Customer Managed Policy** | Created by the customer, reusable |
| **Inline Policy** | Embedded directly in a user/group/role — not reusable |
| **Resource-Based Policy** | Attached to a resource (S3 bucket, SQS queue, Lambda) instead of an identity |
| **Permission Boundary** | Maximum permissions an identity can ever have, regardless of other policies |
| **Service Control Policy (SCP)** | Org-level guardrail — restricts what accounts in an AWS Org can do |
| **Session Policy** | Passed at `AssumeRole` time to further restrict the role's permissions for that session |

### 5. Policy Structure — JSON Anatomy

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Read",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {
        "StringEquals": {
          "aws:RequestedRegion": "eu-west-1"
        }
      }
    }
  ]
}
```

**Key fields:**
- `Effect`: `Allow` or `Deny`. Explicit Deny always wins.
- `Action`: The API action being permitted/denied. `*` = all actions.
- `Resource`: The ARN of the resource this applies to. `*` = all resources.
- `Condition`: Optional conditions (IP range, MFA required, time of day, etc.)

> [!danger] Wildcard policies are the root of all evil
> `"Action": "*"` on `"Resource": "*"` = full admin. This is AdministratorAccess. Finding it on any non-admin user or role is a critical finding.

## Identity Policies vs Resource Policies

**Identity policy:** Attached to a user, group, or role. Defines what that identity can do.

**Resource policy:** Attached to a resource (S3 bucket, SQS queue, Lambda, KMS key). Defines who can access that resource.

```json
// Example: S3 Bucket Policy (resource policy)
// This allows any AWS principal to read from the bucket — PUBLIC READ
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::example-bucket/*"
    }
  ]
}
```

**Cross-account access requires both sides:** For a principal in Account A to access a resource in Account B, both the identity policy (in Account A) AND the resource policy (in Account B) must allow it.

---

## Trust Policies

A **trust policy** is the special policy attached to an IAM role that defines **who is allowed to assume that role**. It's a resource-based policy on the role itself.

```json
// Example Trust Policy: allows EC2 service to assume this role
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "ec2.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

```json
// Example Trust Policy: allows another account to assume this role (cross-account)
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::ATTACKER_ACCOUNT_ID:root"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
```

> [!warning] Overly permissive trust policies = privilege escalation
> If a trust policy allows `"Principal": "*"` or allows an account you control, you can assume that role. If the role has powerful permissions, you've escalated. Modifying trust policies (`iam:UpdateAssumeRolePolicy`) is itself a privilege escalation technique.

---

## Permission Boundaries

A **permission boundary** is an advanced IAM feature. It sets the maximum permissions that an identity can have. Even if a policy grants `AdministratorAccess`, if the permission boundary only allows `s3:*`, the effective permissions are only S3.

```bash
# Check if a user has a permission boundary
aws iam get-user --user-name alice
# Look for: PermissionsBoundary field in the output

# Get the boundary policy ARN
aws iam get-user --user-name alice \
  --query 'User.PermissionsBoundary.PermissionsBoundaryArn'
```

**Attack relevance:** If you can remove or replace a permission boundary (`iam:PutUserPermissionsBoundary`, `iam:DeleteUserPermissionsBoundary`), you can escalate privileges even if boundaries were used as a security control.

---

## IAM Policy Evaluation Logic

When AWS evaluates whether to allow an action, it follows this order:

```
1. Is there an explicit DENY in any policy? → DENY (always wins, no exceptions)
2. Is there an SCP (Service Control Policy) that denies? → DENY
3. Is there a resource-based policy that allows? → ALLOW (for same-account)
4. Is there a permission boundary that denies? → DENY
5. Is there an identity-based policy that allows? → ALLOW
6. Default: → DENY (implicit deny)
```

> [!tip] The default is DENY
> Everything is denied unless explicitly allowed. This is the principle of least privilege baked into the system. The attacker's goal is to find identities where "explicitly allowed" includes far more than it should.

## The ARN (Amazon Resource Name)

Every AWS resource has an ARN — a unique identifier.

```
Format: arn:partition:service:region:account-id:resource-type/resource-id

Examples:
  arn:aws:iam::123456789012:user/alice
  arn:aws:iam::123456789012:role/LambdaExecutionRole
  arn:aws:s3:::my-bucket
  arn:aws:s3:::my-bucket/secret-file.txt
  arn:aws:ec2:us-east-1:123456789012:instance/i-0abc123
  arn:aws:lambda:us-east-1:123456789012:function:MyFunction
```

Note: IAM ARNs have no region (IAM is global). S3 bucket ARNs have no region or account.

## Quick Reference — IAM Enumeration Commands

```bash
# Identity
aws sts get-caller-identity

# Users
aws iam list-users
aws iam get-user --user-name USERNAME
aws iam list-access-keys --user-name USERNAME
aws iam list-mfa-devices --user-name USERNAME

# Groups
aws iam list-groups
aws iam list-groups-for-user --user-name USERNAME
aws iam get-group --group-name GROUPNAME

# Roles
aws iam list-roles
aws iam get-role --role-name ROLENAME
aws iam list-role-policies --role-name ROLENAME      # inline policies
aws iam list-attached-role-policies --role-name ROLENAME  # managed policies

# Policies on current user
aws iam list-user-policies --user-name USERNAME       # inline
aws iam list-attached-user-policies --user-name USERNAME  # managed

# Get policy content
aws iam get-policy --policy-arn arn:aws:iam::ACCOUNT:policy/POLICYNAME
aws iam get-policy-version \
  --policy-arn arn:aws:iam::ACCOUNT:policy/POLICYNAME \
  --version-id v1

# List all customer-managed policies
aws iam list-policies --scope Local

# Check what services the current identity can access
# (use enumerate-iam tool — see 01.2)
```
