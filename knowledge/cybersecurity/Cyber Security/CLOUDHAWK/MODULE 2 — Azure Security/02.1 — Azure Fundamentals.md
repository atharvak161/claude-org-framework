# 02.1 — Azure Fundamentals

## What This Note Covers

The foundational Azure concepts every pentester must internalise before touching any Azure environment: the hierarchy of tenants, subscriptions, resource groups, and the critical distinction between Azure AD (Entra ID) and Azure RBAC.

---

## The Azure Hierarchy

Azure organises everything in a strict parent-child hierarchy. Permissions flow downward — access granted at a higher level is inherited by everything below it.

```
Management Group (optional, org-level grouping)
└── Subscription (billing + resource boundary)
    └── Resource Group (logical container)
        └── Resources (VMs, storage accounts, databases, etc.)
```

### Management Groups

Optional grouping above subscriptions. Large enterprises use them to apply governance (policies, RBAC) across many subscriptions at once.

```bash
# List management groups (requires elevated permissions)
az account management-group list
```

### Subscriptions

A **subscription** is the primary billing and resource management boundary in Azure. Each subscription:
- Has a unique Subscription ID (GUID)
- Belongs to exactly one Azure AD tenant
- Can contain multiple resource groups
- Has its own RBAC assignments

```bash
# List subscriptions you have access to
az account list --output table

# Switch to a specific subscription
az account set --subscription "SUBSCRIPTION_NAME_OR_ID"

# Show current subscription
az account show
```

> [!tip] Multiple subscriptions = multiple attack surfaces
> Large organisations have dozens or hundreds of subscriptions (prod, dev, staging, per-team, per-project). Each is a separate resource boundary. Enumerate all subscriptions accessible with your current token — don't assume you're only in one.

### Resource Groups

A **resource group** is a logical container for related resources. Key properties:
- Every Azure resource must belong to exactly one resource group
- Resource groups have a region (but can contain resources from other regions)
- RBAC can be assigned at the resource group level — Owner of a resource group = full control of everything in it
- Deleting a resource group deletes everything in it

```bash
# List all resource groups in current subscription
az group list --output table

# List resources in a specific resource group
az resource list --resource-group TARGET_RG --output table
```

## Azure AD (Entra ID) — The Identity Plane

**Azure Active Directory** (recently rebranded **Microsoft Entra ID**) is Azure's identity and access management service. It is completely separate from Azure Resource Manager (ARM) — the service that manages Azure resources.

This is the most important architectural concept in Azure security:

```
Azure AD (Entra ID) = Identity plane
  - Users, Groups, Service Principals, Managed Identities
  - App Registrations, OAuth flows
  - Directory roles (Global Admin, etc.)

Azure Resource Manager (ARM) = Resource plane
  - VMs, Storage Accounts, Databases, Networks
  - RBAC roles (Owner, Contributor, Reader)
  - Subscriptions, Resource Groups
```

**They are linked but separate.** A Global Administrator in Azure AD has no automatic access to Azure resources (subscriptions). A subscription Owner has full control of Azure resources but no implicit Azure AD admin rights.

> [!warning] This separation is where attackers get caught out — and where defenders create gaps
> An attacker who compromises a subscription Owner can delete VMs but can't necessarily read Azure AD user passwords. An attacker who compromises a Global Admin can control the identity plane and elevate to any subscription through Azure AD settings.

## Azure AD — Core Identity Objects

### Users

Azure AD users are either:
- **Member users** — employees, internal accounts
- **Guest users** — external collaborators (B2B)

```bash
# List all users in the tenant
az ad user list --output table

# Get details on a specific user
az ad user show --id user@company.com

# List user's group memberships
az ad user get-member-groups --id user@company.com
```

### Groups

Security groups and Microsoft 365 groups. Used to assign permissions to collections of users.

```bash
# List all groups
az ad group list --output table

# List members of a group
az ad group member list --group "IT Admins" --output table

# Check if a user is a member of a group
az ad group member check --group "IT Admins" --member-id USER_OBJECT_ID
```

### Service Principals

A **service principal** is the identity representation of an application (or managed service) in Azure AD. When an app registration is created, a service principal is automatically created in the tenant. Service principals can be assigned:
- Azure AD directory roles (e.g., Global Reader)
- Azure RBAC roles (e.g., Contributor on a subscription)
- API permissions to Microsoft Graph or other services

```bash
# List all service principals
az ad sp list --output table

# Get details on a specific service principal
az ad sp show --id APP_ID_OR_OBJECT_ID

# List permissions (API permissions) granted to a service principal
az ad app permission list --id APP_ID
```

> [!danger] Overprivileged service principals are common and high-value
> Developers create app registrations for CI/CD pipelines, automation scripts, and integrations. These often accumulate excessive permissions and their credentials (client secrets or certificates) get leaked in code repos. A service principal with `Contributor` on the subscription = full resource control.

### App Registrations

An **app registration** is the application object in Azure AD. Think of it as the "blueprint" — the service principal is the "instance" in a tenant.

App registrations define:
- Which APIs the app can access (API permissions)
- OAuth redirect URIs
- Credentials (client secrets, certificates)

```bash
# List all app registrations
az ad app list --output table

# Get an app's credentials (lists secret IDs, not values — values only retrievable at creation)
az ad app credential list --id APP_ID

# Get API permissions granted to an app
az ad app permission list --id APP_ID
```

### Managed Identities

A **managed identity** is an Azure AD identity automatically managed by Azure for a resource (VM, App Service, Function App, etc.). No credentials to manage — Azure handles rotation. Two types:

- **System-assigned:** Tied to a specific resource. Deleted when the resource is deleted.
- **User-assigned:** Standalone resource. Can be assigned to multiple resources.

```bash
# List user-assigned managed identities in a resource group
az identity list --resource-group TARGET_RG

# Get the client ID and principal ID of a managed identity
az identity show --name IDENTITY_NAME --resource-group TARGET_RG
```

When a VM has a managed identity, it can get an access token from the Azure IMDS metadata endpoint — the same SSRF → token theft pattern as AWS.

```bash
# From inside an Azure VM (or via SSRF):
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://management.azure.com/" \
  | python3 -m json.tool

# Returns access_token — use to call Azure ARM APIs
```

## Azure RBAC

**Azure Role-Based Access Control (RBAC)** governs access to Azure **resources** (not Azure AD). RBAC assignments have three components:

1. **Security Principal** — who (user, group, service principal, managed identity)
2. **Role Definition** — what permissions (built-in or custom role)
3. **Scope** — where (management group, subscription, resource group, or individual resource)

### Built-in Roles (Most Important)

| Role | Permissions |
|---|---|
| **Owner** | Full access + can grant access to others |
| **Contributor** | Full access to manage resources, cannot grant access |
| **Reader** | Read-only access to everything |
| **User Access Administrator** | Manage access only (no resource control) |
| **Virtual Machine Contributor** | Manage VMs only |
| **Storage Blob Data Owner** | Full S3-equivalent access to blob storage |
| **Key Vault Administrator** | Full access to Key Vault secrets, keys, certs |

```bash
# List all RBAC role assignments in the current subscription
az role assignment list --all --output table

# List role assignments for a specific user
az role assignment list --assignee user@company.com --all

# List role assignments at subscription level
az role assignment list --scope /subscriptions/SUBSCRIPTION_ID

# Check what roles you have
az role assignment list --assignee $(az account show --query user.name -o tsv) --all
```

### Azure AD Directory Roles (Separate from RBAC)

Directory roles govern Azure AD itself — not Azure resources.

| Role | What It Controls |
|---|---|
| **Global Administrator** | Full control of Azure AD — can take over any subscription |
| **Privileged Role Administrator** | Can assign any Azure AD role to anyone |
| **Application Administrator** | Manage all app registrations and service principals |
| **Cloud Application Administrator** | Same minus on-prem app proxy |
| **Global Reader** | Read everything in Azure AD |
| **Security Administrator** | Manage security policies and read security data |

```bash
# List directory role assignments
az rest --method GET \
  --url "https://graph.microsoft.com/v1.0/directoryRoles" \
  --headers "Authorization=Bearer TOKEN"

# Using ROADtools (better for directory enumeration):
roadrecon gather
roadrecon gui  # browse at localhost:5000
```

> [!danger] Global Administrator = total compromise
> A Global Admin can elevate themselves to Owner on any Azure subscription via the "Access management for Azure resources" setting in Azure AD properties. This is the highest-value target in any Azure engagement.

## Quick Reference — Azure Fundamentals Commands

```bash
# Auth and identity
az login
az account show
az account list --output table
az account set --subscription SUBSCRIPTION_ID

# Hierarchy enumeration
az account management-group list
az group list --output table
az resource list --resource-group RG_NAME

# Azure AD enumeration
az ad user list --output table
az ad group list --output table
az ad sp list --output table
az ad app list --output table

# RBAC
az role assignment list --all --output table
az role definition list --output table  # list all available roles

# Current identity's role assignments
az role assignment list \
  --assignee $(az account show --query user.name -o tsv) \
  --all --output table
```
