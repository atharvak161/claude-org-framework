# 10.4 — Mock Assessment Methodology and Practice Guide

## What This Note Covers

How to conduct a mock cloud security assessment against your own lab environment to build professional-grade assessment skills — scenario design, execution discipline, documentation, and the full report output that demonstrates competency to employers.

---

## Why Mock Assessments Matter

Reading about cloud security and doing CTF challenges are different from conducting a structured professional engagement. The gap between "I can exploit CloudGoat scenarios" and "I can run a professional cloud security engagement" is bridged by deliberately practising the full process — scope, recon, automated scanning, manual testing, documentation, and reporting.

A completed mock assessment report in your portfolio is worth more than any certification to a technical interviewer.

---

## Mock Assessment Scenario Design

Design scenarios that mirror real engagement types. Three recommended mock scenarios:

### Scenario 1 — Startup AWS Account (Grey Box)

```
Setup:
  Create a dedicated AWS account (separate from your personal one)
  Deploy intentionally misconfigured resources using Terraform or manually:
    - S3 bucket with public read
    - EC2 instance with IMDSv1 enabled + open security group SSH 0.0.0.0/0
    - IAM user with excessive permissions + no MFA
    - Lambda function with AdministratorAccess execution role
    - CloudTrail disabled in half the regions
    - RDS instance with public access enabled
    - Secrets Manager with a secret that has no rotation

Starting position:
  Give yourself: one low-privilege IAM user with ReadOnlyAccess
  Objective: escalate to admin

Time-box: 8 hours (simulate a 1-day grey box pentest)
```

### Scenario 2 — Enterprise Azure Tenant (White Box)

```
Setup:
  Use your existing Azure free account
  Create:
    - A Service Principal with Contributor on subscription + leaked credentials
    - An App Registration with a client secret stored in a public GitHub gist
    - A Key Vault with overly permissive access policy (all users can get secrets)
    - A VM with a managed identity that has Storage Blob Data Contributor

Starting position:
  Full read access (Security Reader role on subscription)
  Objective: identify all misconfigurations and produce a findings report

Time-box: 6 hours (simulate a compliance-focused assessment)
```

### Scenario 3 — Full Simulation Against CloudGoat

```
Use CloudGoat's most complex scenarios:
  Deploy: ecs_takeover + lambda_privesc + detection_evasion simultaneously
  Treat it as a single connected environment
  Document every step as if writing a professional report

Objective: Produce a complete pentest report with:
  - Executive summary
  - 5+ findings with full technical detail
  - MITRE ATT&CK mapping for each
  - Remediation guidance with CLI commands
  - Attack narrative showing the kill chain
```

## Execution Discipline — How to Run a Mock Assessment

### Phase 1: Scoping (30 minutes)

Before touching any tool, write the scope document:

```markdown
# Mock Assessment Scope

Date: [DATE]
Environment: [AWS Account ID / Azure Tenant / GCP Project]
Type: Grey Box
Access Level: [IAM user ARN or equivalent]
Time box: 8 hours

In scope:
  - All services in us-east-1
  - IAM configuration
  - S3 buckets
  - EC2 instances
  - Lambda functions
  - RDS instances

Out of scope:
  - DoS testing
  - Data deletion
  - Permanent IAM modifications (demonstrate only)

Objective:
  Escalate from ReadOnlyAccess to admin, document the path,
  identify all misconfigurations encountered along the way.
```

### Phase 2: Automated Scanning (1-2 hours)

```bash
# Run all automated tools and save outputs
mkdir -p ~/mock-assessment/automated

# ScoutSuite
python3 scout.py aws --profile mock-assessment-profile \
  -o ~/mock-assessment/automated/scoutsuite

# Prowler
prowler aws --profile mock-assessment-profile \
  --status FAIL -M html \
  -o ~/mock-assessment/automated/prowler/

# CloudMapper (if EC2/VPC in scope)
python3 cloudmapper.py collect --account mock
python3 cloudmapper.py prepare --account mock

# Note: Document the time you started and ended each tool
echo "ScoutSuite started: $(date)" >> ~/mock-assessment/timeline.txt
```

### Phase 3: Manual Enumeration (1-2 hours)

Work through the checklist in [[08.3 — Manual Assessment Checklist by Service]].

```bash
# Document every command and its output
# Create a findings draft file as you go

cat >> ~/mock-assessment/findings-draft.md << 'EOF'
## [TIME] IAM Enumeration

aws sts get-caller-identity
Output: [paste output]

Finding candidate: User has no MFA configured
Evidence: aws iam list-virtual-mfa-devices returns empty for this user
Severity: High
EOF
```

### Phase 4: Exploitation / Privilege Escalation (2-3 hours)

```bash
# Document each escalation step
# For each technique attempted:
#   - What you tried
#   - The exact command
#   - The output (success or failure)
#   - Time taken

# Example documentation format:
cat >> ~/mock-assessment/attack-chain.md << 'EOF'
### Step 1: IAM Enumeration
Time: 09:45

Command:
  python3 enumerate-iam.py --access-key KEY --secret-key SECRET

Key finding:
  Current user has lambda:UpdateFunctionCode on *
  Current user has iam:PassRole on LambdaExecutionRole

### Step 2: Identify Target Role
Time: 10:15

Command:
  aws iam list-attached-role-policies --role-name LambdaExecutionRole

Finding:
  Role has AdministratorAccess attached

### Step 3: Exploit
Time: 10:30

Action: Updated DataProcessor function code with privilege escalation payload
Result: Successfully escalated to AdministratorAccess at 10:47

Evidence: aws sts get-caller-identity → [paste output showing admin role]
EOF
```

### Phase 5: Documentation (1-2 hours)

Convert your notes into formal findings using the template in [[08.5 — Reporting Cloud Findings]]:

```
For each finding:
  1. Give it a severity and title
  2. Write the technical description
  3. Paste the evidence (command + output)
  4. Write the business impact (plain English)
  5. Write specific remediation with CLI commands
  6. Add MITRE ATT&CK mapping
  7. Add reference links
```

### Phase 6: Report Writing (2 hours)

Produce a complete report in PDF or Word format:

```markdown
# Cloud Security Assessment Report
## Target: Mock AWS Environment
## Date: [DATE]
## Assessor: Atharva K.

---

## Executive Summary

This assessment identified [N] findings across the evaluated AWS environment,
including [X] Critical findings that require immediate remediation.

[2-3 sentences on key findings in plain language]

The most critical finding was [NAME], which allowed escalation from
a low-privilege IAM user to full account administrator in [TIME].

---

## Findings Summary

| ID  | Title                    | Severity | Service |
|-----|--------------------------|----------|---------|
| F01 | Lambda Admin Role        | Critical | Lambda  |
| F02 | Public S3 Bucket         | Critical | S3      |
| F03 | IMDSv1 Enabled           | High     | EC2     |
| F04 | No MFA on IAM User       | High     | IAM     |
| F05 | CloudTrail Gaps          | Medium   | Logging |

---

[Full findings follow — one per section using template from 08.5]

---

## Appendix A: Tools Used
## Appendix B: Full ScoutSuite Output
## Appendix C: Prowler Compliance Summary
```

## Portfolio Presentation

A completed mock assessment demonstrates more than theoretical knowledge. For job applications and interviews:

```
What to include in your portfolio:
  1. Redacted PDF report (remove account IDs, replace with [REDACTED])
  2. Brief description of the scenario (3-4 sentences)
  3. Key findings with severity justification
  4. Evidence screenshots (tool output, API responses)
  5. The attack chain as a diagram or narrative

Where to publish:
  - GitHub repository (private, share URL in interviews)
  - Personal blog post (sanitised findings, great for visibility)
  - LinkedIn article (high-level summary — good for recruiter attention)

What interviewers want to see:
  - Can you structure an assessment professionally?
  - Do you understand severity and business impact?
  - Can you write a finding that a CISO can understand?
  - Do you know what to fix and how to fix it?
```

## Bug Bounty as Real-World Practice

After mock assessments, move to real environments via bug bounty programmes:

```
Cloud-heavy bug bounty targets (public programmes):
  - HackerOne: look for programs with "cloud" or "AWS" in scope
  - Bugcrowd: filter by technology tags
  - Self-managed: GitLab, Shopify, Dropbox — all have cloud infra in scope

Starting approach:
  1. Read the programme's scope document carefully (cloud resources often excluded)
  2. Do passive recon first (CT logs, Shodan, GitHub)
  3. Test for public S3 buckets and subdomain takeover (safe, permitted, impactful)
  4. Escalate to authenticated testing only if explicitly permitted
  5. Submit clear, reproducible reports with severity justification

Cloud-specific bounties:
  - Public storage buckets: typically $100-500 (low) to $1000+ (if PII found)
  - Subdomain takeover via cloud CNAME: typically $200-1000
  - SSRF to cloud metadata: typically $500-5000 (high impact)
  - IAM misconfiguration requiring auth: typically $1000-10000+ (if demonstrable)
```

## Quick Reference — Mock Assessment Commands

```bash
# Setup
mkdir -p ~/mock-assessment/{automated,evidence,findings}
echo "Assessment started: $(date)" > ~/mock-assessment/timeline.txt

# Automated scanning
python3 scout.py aws --profile PROFILE -o ~/mock-assessment/automated/scoutsuite/
prowler aws --profile PROFILE --status FAIL -M html -o ~/mock-assessment/automated/prowler/

# Evidence capture (pipe all commands to log)
exec > >(tee -a ~/mock-assessment/evidence/commands.log) 2>&1
aws sts get-caller-identity
python3 enumerate-iam.py --access-key KEY --secret-key SECRET

# Finding documentation
cat >> ~/mock-assessment/findings/F01-lambda-admin-role.md << 'EOF'
[Paste finding template from 08.5]
EOF

# Cleanup (post-assessment)
# Work through checklist from 09.5
```
