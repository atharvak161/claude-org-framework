# 08.4 — Cloud Security Benchmarks and MITRE ATT&CK for Cloud

## What This Note Covers

The CIS Benchmarks for AWS, Azure, and GCP — what they cover and how to use them as an assessment framework. Then MITRE ATT&CK for Cloud — how to map your findings and attack techniques to the official taxonomy.

---

## CIS Benchmarks — Overview

The Center for Internet Security (CIS) publishes benchmarks for major cloud platforms. These are configuration standards developed by consensus among security practitioners, vendors, and governments. Two levels:

- **Level 1:** Foundational security controls, low operational impact. Every organisation should meet these.
- **Level 2:** Stricter controls for high-security environments. May have operational trade-offs.

CIS Benchmarks are the most widely referenced cloud security standard and are directly mapped by Prowler, ScoutSuite, and Security Hub.

## CIS AWS Foundations Benchmark

Current version: **CIS AWS Foundations Benchmark v3.0.0**

### Key Control Areas

**Section 1 — Identity and Access Management**

| Control | Description | Level |
|---|---|---|
| 1.1 | Root account MFA enabled | L1 |
| 1.2 | Root account no access keys | L1 |
| 1.4 | IAM password policy: uppercase | L1 |
| 1.5 | IAM password policy: lowercase | L1 |
| 1.9 | IAM password policy: reuse prevention | L1 |
| 1.10 | MFA enabled for all IAM users | L1 |
| 1.14 | Access keys rotated every 90 days | L1 |
| 1.16 | IAM policies attached only to groups/roles | L1 |
| 1.17 | Support role created | L1 |
| 1.20 | No wildcard admin policies | L1 |

**Section 2 — Storage (S3)**

| Control | Description | Level |
|---|---|---|
| 2.1.1 | S3 Block Public Access enabled | L1 |
| 2.1.2 | MFA delete on S3 versioning | L2 |
| 2.1.3 | S3 server-side encryption enabled | L1 |
| 2.1.4 | S3 access logging enabled | L1 |

**Section 3 — Logging**

| Control | Description | Level |
|---|---|---|
| 3.1 | CloudTrail enabled in all regions | L1 |
| 3.2 | CloudTrail log file validation | L1 |
| 3.3 | S3 bucket not publicly accessible | L1 |
| 3.4 | CloudTrail integrated with CloudWatch | L1 |
| 3.5–3.14 | CloudWatch metric filters for security events | L1 |
| 3.10 | VPC Flow Logs enabled | L1 |

**Section 4 — Monitoring (CloudWatch Alarms)**

Specific alarms required:
- Unauthorised API calls
- Console sign-in without MFA
- Root account usage
- IAM policy changes
- CloudTrail configuration changes
- S3 bucket policy changes
- VPC changes
- Failed console authentication

**Section 5 — Networking**

| Control | Description | Level |
|---|---|---|
| 5.1 | No NACL allows ingress 0.0.0.0/0 to RDP | L1 |
| 5.2 | No NACL allows ingress 0.0.0.0/0 to SSH | L1 |
| 5.3 | Default security group restricts all traffic | L1 |
| 5.4 | VPC peering routing principle of least access | L2 |

```bash
# Run Prowler against CIS AWS L1
prowler aws --compliance cis_1.5_aws --status FAIL -M html -o ./cis-report/

# Run against CIS AWS v3.0
prowler aws --compliance cis_3.0_aws --status FAIL -M html
```

## CIS Azure Foundations Benchmark

Current version: **CIS Microsoft Azure Foundations Benchmark v2.0.0**

Key sections:

**1 — Identity and Access Management**
- 1.1: Custom subscription owner roles minimised
- 1.2: Security defaults enabled (if no CA policies)
- 1.3: Guest invite restrictions configured
- 1.4: Guest user access restriction
- 1.12: MFA required for all users

**2 — Microsoft Defender for Cloud**
- 2.1: Defender for Cloud enabled
- 2.2: Auto-provisioning of monitoring agents

**3 — Storage Accounts**
- 3.1: Secure transfer required
- 3.2: Storage account public access disabled
- 3.3: Storage account shared key access disabled
- 3.4: Default action deny for storage network rules

**4 — Azure SQL**
- 4.1: Auditing enabled
- 4.2: Data encryption enabled
- 4.5: Transparent Data Encryption enabled

**7 — Logging and Monitoring**
- 7.1: Activity log diagnostic settings configured
- 7.2: Activity log alert for create/update/delete policy assignment

```bash
# Prowler CIS Azure
prowler azure --compliance cis_2.0_azure --az-cli-auth --status FAIL -M html
```

## CIS GCP Foundations Benchmark

Current version: **CIS Google Cloud Platform Foundation Benchmark v2.0.0**

Key sections:

**1 — Identity and Access Management**
- 1.1: Corporate login credentials used
- 1.2: Multi-factor authentication enabled
- 1.3: No service account keys older than 90 days
- 1.4: SA keys not managed by users (use Google-managed)
- 1.5: SA not assigned primitive roles
- 1.6: No user-managed SA keys for default SAs

**2 — Logging and Monitoring**
- 2.1: Cloud Audit Logs enabled
- 2.2: Sinks configured for all log entries
- 2.3: Retention policy for log buckets

**3 — Networking**
- 3.1: Default VPC network not used
- 3.2: SSH access restricted (no 0.0.0.0/0)
- 3.3: RDP access restricted

**4 — VMs**
- 4.1: Instances not use service account with Admin privileges
- 4.3: Instances not configured to connect serially

```bash
# Prowler CIS GCP
prowler gcp --compliance cis_2.0_gcp --project-id PROJECT_ID --status FAIL -M html
```

## MITRE ATT&CK for Cloud

MITRE ATT&CK for Cloud (part of the Enterprise matrix) maps adversary techniques used against cloud environments to a standardised taxonomy. Use this to:
1. Map your pentest findings to recognised TTPs
2. Demonstrate comprehensive coverage in reports
3. Communicate with blue teams using shared language
4. Prioritise defences against real-world threats

### Cloud-Relevant Tactics

| Tactic | ID | What It Covers |
|---|---|---|
| Initial Access | TA0001 | How attackers get in |
| Execution | TA0002 | Running attacker code |
| Persistence | TA0003 | Maintaining access |
| Privilege Escalation | TA0004 | Gaining higher permissions |
| Defence Evasion | TA0005 | Avoiding detection |
| Credential Access | TA0006 | Stealing credentials |
| Discovery | TA0007 | Enumerating the environment |
| Lateral Movement | TA0008 | Moving to other systems/services |
| Collection | TA0009 | Gathering target data |
| Exfiltration | TA0010 | Getting data out |
| Impact | TA0040 | Destroying or disrupting |

### Key Cloud Techniques — AWS

| Technique | ID | Description |
|---|---|---|
| Valid Accounts: Cloud Accounts | T1078.004 | Use stolen cloud credentials |
| Exploit Public-Facing Application | T1190 | RCE on cloud-hosted app |
| Cloud Infrastructure Discovery | T1580 | Enumerate cloud resources |
| Steal Application Token | T1528 | Steal OAuth/API tokens |
| Cloud Instance Metadata API | T1552.005 | SSRF to metadata service |
| Additional Cloud Credentials | T1098.001 | Create new IAM keys |
| Cloud Accounts | T1136.003 | Create new cloud accounts |
| Modify Cloud Compute Configurations | T1578 | Change VM/function configs |
| Exfiltration to Cloud Storage | T1567.002 | Copy data to S3/GCS/Blob |
| Transfer Data to Cloud Account | T1537 | Move data to attacker account |
| Abuse Elevation Control Mechanism | T1548 | IAM privilege escalation |
| Unused/Unsupported Cloud Regions | T1535 | Use under-monitored regions |

### Mapping Findings to ATT&CK

When writing your pentest report, include ATT&CK mappings for each finding:

```
FINDING: Lambda execution role has iam:AttachUserPolicy on Resource: *

MITRE ATT&CK MAPPING:
  Tactic:    Privilege Escalation (TA0004)
  Technique: Abuse Elevation Control Mechanism: Cloud Administration Command (T1548)
  
  Tactic:    Persistence (TA0003)
  Technique: Additional Cloud Credentials (T1098.001)

ATTACK SCENARIO:
  An attacker who gains code execution in the Lambda function
  (e.g., via event injection — T1059) can use the execution role
  to attach AdministratorAccess to an IAM user they control,
  gaining persistent admin access to the AWS account.
```

## AWS Security Hub — Automated Benchmark Checks

AWS Security Hub aggregates findings from GuardDuty, Inspector, Macie, and partner tools, and runs continuous compliance checks against standards.

```bash
# Enable Security Hub
aws securityhub enable-security-hub \
  --enable-default-standards \
  --region us-east-1

# List enabled standards
aws securityhub describe-standards --region us-east-1

# Get all failed findings
aws securityhub get-findings \
  --filters '{"ComplianceStatus":[{"Value":"FAILED","Comparison":"EQUALS"}]}' \
  --query 'Findings[].{Title:Title,Severity:Severity.Label,Standard:ProductName}' \
  --region us-east-1 \
  --output table

# Standards available:
# - AWS Foundational Security Best Practices
# - CIS AWS Foundations Benchmark
# - PCI DSS
# - NIST SP 800-53
```
