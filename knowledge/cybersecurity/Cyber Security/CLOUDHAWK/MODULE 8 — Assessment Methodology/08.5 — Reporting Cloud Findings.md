# 08.5 — Reporting Cloud Findings

## What This Note Covers

How to write cloud security findings that communicate risk clearly — severity rating in cloud context, finding structure, remediation guidance, cloud-specific severity considerations, and the full report structure.

---

## Why Cloud Finding Reporting Is Different

Cloud findings often don't map cleanly to traditional CVSS scoring:

- **A public S3 bucket containing customer PII** — no CVE, no exploit, but Critical severity because the data is already exposed.
- **Lambda execution role with `AdministratorAccess`** — no exploit needed, but Critical because any code execution in that function = full account compromise.
- **IMDSv1 enabled with an SSRF-vulnerable application** — the combination is Critical; either alone is medium.

The key principle: **rate cloud findings by business impact and exploitability of the chain, not just the individual misconfiguration.**

---

## Cloud-Adjusted Severity Framework

Use a modified severity scale that accounts for cloud-specific context:

### Critical

Conditions that meet **any** of the following:
- Publicly accessible data (storage, database, API) with no authentication
- IAM configuration that provides direct path to account admin with no additional steps
- Exposed credentials (access keys, SA keys, SAS tokens) with confirmed active permissions
- Unauthenticated remote code execution on cloud-hosted infrastructure
- Cross-account access allowing attacker to pivot to production from compromised dev

```
Examples:
- Public S3 bucket containing PII or credentials
- Lambda execution role with AdministratorAccess
- EC2 metadata SSRF chain to IAM credential theft → account takeover
- Terraform state file publicly accessible containing DB passwords
- Management account OrganizationAccountAccessRole exploitable
```

### High

- IAM privilege escalation path with 2-3 steps available to low-privilege user
- Sensitive data accessible to any authenticated cloud user (not just org members)
- Missing MFA on privileged accounts (admin, root, Global Admin)
- Publicly exposed admin panel (Grafana, Kibana, Jenkins) with weak/default auth
- RDS instance publicly accessible with known-weak credentials
- CloudTrail disabled in regions with production workloads

```
Examples:
- IAM user can assume role → role has iam:AttachUserPolicy
- AWS root account has no MFA
- RDS instance publicly accessible, port 3306 open
- CloudTrail disabled in eu-west-1 where production runs
```

### Medium

- IAM misconfiguration that could lead to escalation but requires additional factors
- Overly permissive security groups that increase attack surface without direct exploitation
- Logging gaps (CloudTrail data events not enabled, VPC Flow Logs off)
- Missing encryption on data at rest
- Unrotated secrets/access keys (90+ days)
- Default VPC in use for non-trivial workloads

```
Examples:
- Security group allows SSH from 0.0.0.0/0 to instances not hosting sensitive data
- S3 bucket without server-side encryption (internal data only)
- Access key not rotated in 180 days
- Lambda runtime reaching end-of-life
```

### Low / Informational

- Best practice deviations with no direct security impact
- Missing tags, cost allocation configuration
- Sub-optimal configurations that don't increase risk in the current environment
- Hardening improvements recommended for defence in depth

## Finding Structure Template

Every finding should include these sections:

```markdown
## [FINDING TITLE]

**Severity:** Critical / High / Medium / Low
**Service:** AWS IAM / S3 / Lambda / etc.
**MITRE ATT&CK:** T1078.004 — Valid Accounts: Cloud Accounts (or relevant TTP)
**Affected Resources:** 
  - arn:aws:iam::123456789012:role/LambdaProcessingRole
  - arn:aws:lambda:us-east-1:123456789012:function:DataProcessor

---

### Description

[2-4 sentences explaining what the misconfiguration is, why it exists, 
and why it matters in plain English. No jargon. Client stakeholder must
understand this without technical background.]

The Lambda function `DataProcessor` is configured to execute with the 
IAM role `LambdaProcessingRole`. This role has the `AdministratorAccess` 
managed policy attached, granting it unrestricted access to all AWS services 
and resources across the account. If an attacker were to gain code execution 
within this function — for example, through an event injection vulnerability 
in the function's input handling — they would immediately operate with full 
administrative control of the AWS account.

---

### Technical Details

[Exact commands run, output observed, step-by-step reproduction.]

The following commands confirm the execution role and its permissions:

```bash
# Get function execution role
aws lambda get-function-configuration \
  --function-name DataProcessor \
  --query Role
# Output: arn:aws:iam::123456789012:role/LambdaProcessingRole

# Confirm AdministratorAccess is attached
aws iam list-attached-role-policies \
  --role-name LambdaProcessingRole
# Output: AdministratorAccess (arn:aws:iam::aws:policy/AdministratorAccess)
```

The following demonstrates the blast radius — a function invocation 
with injected payload that lists all S3 buckets:

[SCREENSHOT OR OUTPUT]

---

### Impact

[What can an attacker do? Be specific about business impact: data types 
accessible, accounts reachable, potential for destruction or fraud.]

Successful exploitation of this misconfiguration would allow an attacker to:
- Read, modify, or delete all data across all S3 buckets in the account
- Create or destroy EC2 instances, RDS databases, and all other resources
- Create permanent IAM backdoor users or access keys for persistent access
- Pivot to other AWS accounts in the AWS Organisation via cross-account roles
- Access all secrets stored in AWS Secrets Manager across the account

---

### Remediation

[Specific, actionable fix. Include code/CLI where possible.]

Apply the principle of least privilege to the Lambda execution role. 
Review the function's actual requirements and create a custom IAM policy 
that grants only the permissions required:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::data-processing-bucket/*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "*"
    }
  ]
}
```

Remove the AdministratorAccess policy:
```bash
aws iam detach-role-policy \
  --role-name LambdaProcessingRole \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
```

## Report Structure

A complete cloud security assessment report should contain:

```
1. EXECUTIVE SUMMARY
   - Engagement overview (scope, dates, access level)
   - Key findings summary (Critical: X, High: X, Medium: X, Low: X)
   - Top 3-5 most impactful findings in plain English
   - Positive findings (what was done well)
   - Overall risk rating

2. SCOPE AND METHODOLOGY
   - Cloud accounts/subscriptions/projects tested
   - Services in scope
   - Access level provided
   - Tools used (ScoutSuite, Prowler, manual)
   - Testing approach

3. FINDINGS
   - Ordered by severity (Critical → Low)
   - Each finding: Title, Severity, Service, Description, Technical Details, 
     Impact, Remediation, References

4. FINDINGS SUMMARY TABLE
   ID | Title | Severity | Service | Status
   F01 | Public S3 Bucket | Critical | S3 | Open
   F02 | Lambda Admin Role | Critical | Lambda | Open

5. APPENDIX
   - Tool output (ScoutSuite report, Prowler output)
   - Full list of resources reviewed
   - Glossary of cloud security terms for non-technical readers
```

## Cloud-Specific Reporting Language

Use precise cloud terminology in findings. Clients and their cloud teams will read these:

```
AVOID:                              USE INSTEAD:
"The server is publicly accessible" "The S3 bucket has a resource policy granting s3:GetObject to Principal:*"
"Admin access was possible"         "The IAM role LambdaRole has iam:AttachUserPolicy on Resource:* enabling privilege escalation to AdministratorAccess"
"Credentials were found"            "An AWS access key (AKIA...) with IAM user ReadOnlyAccess permissions was found hardcoded in the GitHub repository TARGET_REPO at commit HASH"
"Encryption was missing"            "The RDS instance db-prod-01 has StorageEncrypted: false, storing data unencrypted on disk"
"Logging was disabled"              "CloudTrail is not enabled as a multi-region trail. Regions eu-west-2 and ap-southeast-1 have no active trail, leaving API activity in those regions unlogged."
```

## Quick Reference — Severity Decision Tree

```
Is data currently exposed publicly with no auth? → CRITICAL
Is there a direct path to account admin (1-2 steps)? → CRITICAL
Are credentials exposed and confirmed active? → CRITICAL

Is RCE achievable on cloud infrastructure? → CRITICAL/HIGH
Is privilege escalation achievable (3+ steps)? → HIGH
Is MFA missing on admin/privileged accounts? → HIGH
Are production workloads logging-blind? → HIGH

Are security groups overly permissive without direct exploitation path? → MEDIUM
Is encryption missing on non-sensitive data? → MEDIUM
Are keys/secrets unrotated? → MEDIUM

Is it a best-practice deviation without risk increase? → LOW/INFO
```
