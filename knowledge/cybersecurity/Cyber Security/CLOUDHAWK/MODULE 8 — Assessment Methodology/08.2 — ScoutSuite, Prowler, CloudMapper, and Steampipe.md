# 08.2 — ScoutSuite, Prowler, CloudMapper, and Steampipe

## What This Note Covers

The automated cloud security scanning tools used in every assessment — full command references, how to interpret output, what each tool finds best, and how to combine them for comprehensive coverage.

---

## The Role of Automated Scanning

Automated tools do the broad sweep — they flag every misconfiguration against a ruleset so you don't miss anything obvious. Manual testing then confirms exploitability, finds chained paths, and tests business logic that tools can't evaluate.

**Rule:** Always run automated tools first. Never file a finding that a tool should have caught if you skipped the tool.

---

## ScoutSuite — Multi-Cloud Misconfiguration Scanner

ScoutSuite by NCC Group is the most comprehensive multi-cloud automated scanner. It pulls configuration data from cloud APIs and generates an HTML report with findings categorised by severity.

### Install and Setup

```bash
git clone https://github.com/nccgroup/ScoutSuite
cd ScoutSuite
pip3 install virtualenv
virtualenv -p python3 venv
source venv/bin/activate
pip install -r requirements.txt
```

### AWS Scanning

```bash
# Using current AWS CLI profile
python3 scout.py aws

# Specific profile
python3 scout.py aws --profile pentest-profile

# Specific regions only
python3 scout.py aws --regions us-east-1,eu-west-1

# Specific services only (faster)
python3 scout.py aws --services iam,s3,ec2,lambda,rds,cloudtrail

# Exclude services
python3 scout.py aws --exclude-services glacier,snowball

# Report saved to:
# scoutsuite-report/scoutsuite-results/scoutsuite_results_aws-*.js
# Open: scoutsuite-report/index.html in browser
```

### Azure Scanning

```bash
# Interactive login
python3 scout.py azure --cli

# Service principal
python3 scout.py azure --service-principal \
  --client-id CLIENT_ID \
  --client-secret CLIENT_SECRET \
  --tenant-id TENANT_ID

# Specific subscription
python3 scout.py azure --cli --subscription-id SUB_ID

# All subscriptions
python3 scout.py azure --cli --all-subscriptions
```

### GCP Scanning

```bash
# User account
python3 scout.py gcp --user-account --project PROJECT_ID

# Service account key
python3 scout.py gcp \
  --service-account /path/to/key.json \
  --project PROJECT_ID

# All accessible projects
python3 scout.py gcp --user-account --all-projects
```

### Reading ScoutSuite Output

The HTML report organises findings by service. For each service:
- **Danger** (red): Immediate action required — actively exploitable or severely misconfigured
- **Warning** (orange): Misconfiguration that increases risk but may have compensating controls
- **Good** (green): Correctly configured

**Priority review order in the ScoutSuite report:**

```
1. IAM
   → Users without MFA
   → Access keys older than 90 days
   → Root account with access keys
   → Policies with admin wildcards
   → Roles with overly permissive trust policies

2. S3
   → Public buckets
   → Buckets without encryption
   → Buckets without access logging
   → Buckets without versioning

3. EC2
   → Security groups with 0.0.0.0/0 on SSH/RDP
   → Instances without monitoring
   → AMIs shared publicly
   → Security groups with all ports open

4. CloudTrail
   → Trails not enabled in all regions
   → Log file validation disabled
   → Trails not sending to CloudWatch

5. RDS
   → Instances publicly accessible
   → Instances without encryption
   → Automated backups disabled
   → Multi-AZ not enabled (availability concern)

6. Lambda
   → Functions with no resource policy (public)
   → Functions with overprivileged roles
```

## Prowler — Compliance and Security Scanning

Prowler by Toni de la Fuente focuses on compliance frameworks (CIS, PCI DSS, SOC2, GDPR, HIPAA) and generates structured reports suitable for audit evidence.

### Install

```bash
pip3 install prowler
# Or from source:
git clone https://github.com/prowler-cloud/prowler && cd prowler
pip3 install -r requirements.txt
```

### AWS Scanning

```bash
# Run all checks (takes 30-60 min for large accounts)
prowler aws

# Specific services
prowler aws --services iam s3 ec2 cloudtrail lambda rds

# Specific compliance framework
prowler aws --compliance cis_1.5_aws
prowler aws --compliance pci_3.2.1_aws
prowler aws --compliance soc2_aws
prowler aws --compliance gdpr_aws
prowler aws --compliance aws_foundational_security_best_practices

# Specific profile and region
prowler aws --profile pentest-profile --region us-east-1

# Output formats
prowler aws -M html -o ./report/
prowler aws -M json -o ./report/
prowler aws -M csv -o ./report/

# Only show failures
prowler aws --status FAIL

# Specific check IDs
prowler aws --check iam_root_hardware_mfa_enabled
prowler aws --check s3_bucket_public_access
```

### Azure Scanning

```bash
# Azure CLI auth
prowler azure --az-cli-auth

# Service principal
prowler azure \
  --sp-env-auth \
  --tenant-id TENANT_ID

# Specific subscription
prowler azure --az-cli-auth --subscription-ids SUB_ID

# Compliance
prowler azure --compliance cis_2.0_azure
prowler azure --compliance ens_rd2022_azure
```

### GCP Scanning

```bash
# Application default credentials
prowler gcp --project-id PROJECT_ID

# Service account key
prowler gcp \
  --credentials-file /path/to/key.json \
  --project-id PROJECT_ID
```

### Key Prowler Check Categories

```
IAM:
  iam_root_hardware_mfa_enabled          → Root MFA
  iam_user_mfa_enabled_console_access    → All console users have MFA
  iam_access_key_rotation_check          → Keys rotated within 90 days
  iam_policy_no_administrative_privileges → No admin wildcards
  iam_support_role_created               → Support role exists for billing issues

S3:
  s3_bucket_public_access_block_account  → Account-level block public access
  s3_bucket_default_encryption           → Server-side encryption
  s3_bucket_versioning_mfa_delete        → MFA delete for versioned buckets

CloudTrail:
  cloudtrail_multi_region_enabled        → Multi-region trail
  cloudtrail_log_file_validation_enabled → Log integrity validation
  cloudtrail_cloudwatch_logging_enabled  → Logs to CloudWatch

EC2:
  ec2_securitygroup_allow_ingress_from_internet_to_ssh  → SSH from internet
  ec2_instance_imdsv2_enabled            → IMDSv2 required
  ec2_ebs_volume_encryption              → Volume encryption
```

## CloudMapper — AWS Network Topology Visualisation

CloudMapper creates an interactive network diagram of your AWS VPC topology — essential for understanding network architecture during assessments.

```bash
git clone https://github.com/duo-labs/cloudmapper
cd cloudmapper
pip3 install -r requirements.txt

# Collect account data
python3 cloudmapper.py collect --account ACCOUNT_ALIAS

# Generate network diagram
python3 cloudmapper.py prepare --account ACCOUNT_ALIAS

# Serve the web UI
python3 cloudmapper.py webserver
# Open: http://localhost:8000

# The diagram shows:
# - VPCs and their CIDR ranges
# - Subnets (public vs private)
# - EC2 instances with their security group connections
# - Security group rules as connection lines
# - Internet gateways and NAT gateways
```

**What to look for in CloudMapper:**
- EC2 instances in public subnets with direct internet access
- Unusual cross-VPC peering connections
- Security groups with very broad access (thick connection lines)
- Database servers in public subnets

## Steampipe — SQL Queries Against Cloud APIs

Steampipe turns cloud APIs into SQL tables. You can write SQL queries to enumerate and analyse cloud configuration.

```bash
# Install
brew tap turbot/tap && brew install steampipe  # macOS
# Linux: curl -s -L https://github.com/turbot/steampipe/releases/latest/download/steampipe_linux_amd64.tar.gz | tar xz

# Install cloud plugins
steampipe plugin install aws
steampipe plugin install azure
steampipe plugin install gcp

# Authenticate via existing CLI credentials
# AWS: uses ~/.aws/credentials
# Azure: uses az cli session
# GCP: uses gcloud auth

# Start interactive SQL shell
steampipe query

# Example queries:
-- Find S3 buckets with public access
SELECT name, region, bucket_policy_is_public
FROM aws_s3_bucket
WHERE bucket_policy_is_public = true;

-- Find IAM users with no MFA
SELECT name, create_date, mfa_enabled
FROM aws_iam_user
WHERE mfa_enabled = false AND password_enabled = true;

-- Find Lambda functions with admin roles
SELECT f.name, f.role, p.policy_std
FROM aws_lambda_function f
JOIN aws_iam_role_policy_attachment rpa ON rpa.role_name = split_part(f.role, '/', 2)
JOIN aws_iam_policy p ON p.arn = rpa.policy_arn
WHERE p.policy_std LIKE '%"Action":"*"%' AND p.policy_std LIKE '%"Resource":"*"%';

-- Find EC2 security groups allowing SSH from internet
SELECT group_id, group_name, vpc_id
FROM aws_vpc_security_group_rule
WHERE type = 'ingress'
  AND (cidr_ipv4 = '0.0.0.0/0' OR cidr_ipv6 = '::/0')
  AND from_port <= 22 AND to_port >= 22;

-- Find RDS instances publicly accessible
SELECT db_instance_identifier, engine, publicly_accessible, endpoint_address
FROM aws_rds_db_instance
WHERE publicly_accessible = true;
```

### Steampipe Mods — Pre-Built Query Packs

```bash
# Install the AWS Compliance mod (pre-built CIS/PCI queries)
git clone https://github.com/turbot/steampipe-mod-aws-compliance
```

## Tool Combination Strategy

```
ASSESSMENT WORKFLOW:

1. Run ScoutSuite first (broadest coverage, best HTML report)
   → Provides visual overview of ALL misconfigurations
   → 20-40 minutes

2. Run Prowler for compliance framework output
   → Required if client needs CIS/PCI/SOC2 evidence
   → 30-60 minutes

3. Use Steampipe for custom queries
   → Answer specific questions not covered by ScoutSuite/Prowler
   → Query-driven, on demand

4. Use CloudMapper for network topology
   → Run once, use throughout engagement
   → 5-10 minutes

5. Manual testing for exploitability
   → Take ScoutSuite/Prowler findings and test which are actually exploitable
   → Remainder of engagement time
```
