# 06.1 — Cloud Privilege Escalation Mindset and Methodology

## What This Note Covers

The mental model for cloud privilege escalation — how it differs from OS-level privesc, the decision framework for every step, and how to systematically map escalation paths before acting. This is the strategic layer that all the technique notes sit beneath.

---

## Cloud Privesc vs OS Privesc

In traditional penetration testing, privilege escalation means moving from a low-privilege OS user to root or SYSTEM. The tools are kernel exploits, SUID binaries, writable cron jobs, weak service permissions, and password reuse.

Cloud privilege escalation is fundamentally different:

```
OS Privesc:
  Low OS user → exploit kernel/SUID/service → root
  The escalation IS the exploit

Cloud Privesc:
  Low IAM identity → find permission that lets you affect IAM → admin identity
  The escalation IS a chain of legitimate API calls
  No exploit needed — just misconfigured permissions
```

> [!tip] Cloud privesc is always legal API calls
> You are not exploiting a software vulnerability. You are using permissions you legitimately have to reach permissions you shouldn't have. This is why it's harder to detect — it looks like normal administrative activity.

---

## The Core Question at Every Step

At every point in a cloud engagement, ask:

> **"Does any permission I currently hold allow me to, directly or indirectly, modify IAM — or execute code as an identity with more permissions?"**

If yes — that's your escalation path. If no — enumerate more, look for new credentials, find new services.

---

## The IAM Modify Hierarchy

Not all IAM permissions are equal. Rank them by impact:

```
Tier 1 — Direct IAM Modification (instant admin)
  iam:AttachUserPolicy / iam:AttachRolePolicy
  iam:PutUserPolicy / iam:PutRolePolicy
  iam:CreatePolicyVersion (→ set as default)
  resourcemanager.projects.setIamPolicy (GCP)
  Microsoft.Authorization/roleAssignments/write (Azure)

Tier 2 — Indirect IAM via Code Execution
  lambda:UpdateFunctionCode + invocation
  iam:PassRole + lambda:CreateFunction
  iam:PassRole + ec2:RunInstances (→ metadata theft)
  cloudfunctions.functions.create + actAs (GCP)

Tier 3 — Identity Hopping (assume higher-privilege identity)
  sts:AssumeRole (AWS)
  iam.serviceAccounts.actAs (GCP)
  Managed Identity token via IMDS (Azure)
  iam.serviceAccounts.getAccessToken (GCP)

Tier 4 — Credential Access (find existing admin credentials)
  secretsmanager:GetSecretValue
  ssm:GetParameter (--with-decryption)
  lambda:GetFunctionConfiguration (env vars)
  az keyvault secret show (Azure)
  gsutil cat gs://bucket/sa-key.json (GCP)
```

When you have permissions in multiple tiers, **always try the highest tier first**.

---

## The Escalation Decision Framework

```
START: You have a cloud identity (user, role, SA, managed identity)

STEP 1: Who am I?
  aws sts get-caller-identity
  az account show && az ad signed-in-user show
  gcloud auth list && gcloud config list

STEP 2: What can I do? (permission enumeration)
  aws: enumerate-iam, pacu iam__enum_permissions
  azure: az role assignment list --all, roadrecon gather
  gcp: gcloud projects get-iam-policy PROJECT_ID

STEP 3: Do I have Tier 1 permissions?
  YES → directly modify IAM → instant admin
  NO  → check Tier 2

STEP 4: Do I have Tier 2 permissions?
  YES → find a high-privilege role I can pass/create functions with
        → write malicious code → invoke → acts as that role → admin
  NO  → check Tier 3

STEP 5: Do I have Tier 3 permissions (identity hopping)?
  YES → find roles/SAs I can impersonate
        → check those identities' permissions
        → if they have Tier 1/2 → escalate through them
  NO  → check Tier 4

STEP 6: Do I have Tier 4 permissions (credential access)?
  YES → retrieve secrets → validate any credentials found
        → if new identity found → restart from STEP 1 with new identity
  NO  → need new credentials

STEP 7: No direct escalation path from current identity
  → enumerate all accessible resources for lateral movement
  → look for new credentials in: S3/GCS/Blob, secrets managers, instance metadata
  → find other identities you can reach and restart
```

---

## Blast Radius Assessment — Before You Act

Before exploiting any escalation path, mentally model the blast radius:

```
1. What resources does the target identity have access to?
2. Can it read/write production data?
3. Can it modify other identities (IAM admin)?
4. Can it reach other accounts/projects/subscriptions?
5. Will my actions be immediately visible? (CloudTrail alerts, Defender alerts)
6. Is this a production environment? (extra caution)
7. What's the minimum action to demonstrate impact? (don't do more than needed)
```

> [!warning] Minimum necessary action
> In a real pentest, demonstrate the escalation path clearly but don't exfiltrate real data, don't create resources that cost money, don't modify production IAM permanently. Confirm the path is exploitable, document it, and report it. Clean up any test resources created.

---

## Mapping the Attack Graph Before Acting

Before firing any escalation commands, build a mental (or written) map:

```
CURRENT IDENTITY: jenkins-deploy (IAM user)
  ↓ has permission: lambda:UpdateFunctionCode on ProcessingFunction
  ↓ ProcessingFunction's execution role: LambdaProcessingRole
  ↓ LambdaProcessingRole has: secretsmanager:GetSecretValue on *
  ↓ One of those secrets is: "prod/rds/master-credentials"
  ↓ That secret contains: DB admin credentials for production RDS

PATH: jenkins-deploy → update Lambda code → invoke → steal DB creds
IMPACT: Full access to production database
EVIDENCE: Screenshot of GetSecretValue response

ALTERNATIVE PATH:
  ↓ jenkins-deploy also has: iam:PassRole on LambdaProcessingRole
  ↓ LambdaProcessingRole has: iam:CreateAccessKey on *
  PATH: jenkins-deploy → create new Lambda → pass LambdaProcessingRole
        → invoke → create permanent access key for admin user
  IMPACT: Persistent admin access to AWS account
```

---

## Cross-Cloud and Cross-Account Paths

Escalation doesn't have to stay within one cloud or one account. Look for:

```
AWS → Azure:
  Lambda function reads Azure service principal creds from S3
  → use SP to access Azure resources

GCP → AWS:
  GCP Cloud Function with GCS read access
  → GCS bucket contains Terraform state
  → Terraform state has AWS provider credentials
  → use credentials in AWS

Cross-account (AWS):
  Assume a role in Account A
  → that role has sts:AssumeRole into Account B
  → Account B has admin

Cross-project (GCP):
  SA in Project A can actAs SA in Project B
  → Project B SA has roles/owner on Project B

Cross-subscription (Azure):
  Managed identity in Sub A has Reader on Sub B
  → Sub B Key Vault contains SP secret for Global Admin
  → use Global Admin to access all subs
```

---

## Operational Security During Escalation

Your actions are logged. Minimise noise:

```
LOW NOISE operations (do these first, more often):
  - Read-only API calls: list*, describe*, get*
  - Enumeration via IAM policy reading (not brute-force)
  - Reading existing secrets (not creating new ones)
  - Assuming roles that already trust you

HIGH NOISE operations (use sparingly, accept they'll be logged):
  - Creating new IAM users, roles, or policies
  - Attaching policies (especially AdministratorAccess)
  - Modifying trust policies
  - Creating access keys
  - Modifying function code

IMMEDIATELY ALERTING in mature environments:
  - Attaching AdministratorAccess to anything
  - Disabling CloudTrail
  - Creating new IAM users
  - Assuming roles cross-account
  - Root account activity
```

## Quick Reference — First 15 Minutes on Any Cloud Account

```bash
# AWS
aws sts get-caller-identity                    # who am I?
aws iam list-attached-user-policies --user-name USERNAME
aws iam list-user-policies --user-name USERNAME
aws iam list-groups-for-user --user-name USERNAME
python3 enumerate-iam.py --access-key KEY --secret-key SECRET
python3 pacu.py → run iam__privesc_scan

# Azure
az account show                                # who am I? what subscription?
az account list                                # other subscriptions?
az role assignment list --all --assignee ME
roadrecon auth --device-code && roadrecon gather

# GCP
gcloud auth list                               # who am I?
gcloud projects list                           # what projects?
gcloud projects get-iam-policy PROJECT_ID
python3 enumerate_member_permissions.py --project-id PROJECT_ID
python3 check_for_privesc.py --project-id PROJECT_ID
```
