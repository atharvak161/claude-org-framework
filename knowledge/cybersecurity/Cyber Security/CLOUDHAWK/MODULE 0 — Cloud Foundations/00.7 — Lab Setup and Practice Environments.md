# 00.7 — Lab Setup and Practice Environments

## What This Note Covers

How to build a free cloud security lab, and where to practice cloud pentesting skills legally. This covers free tier account setup, intentionally vulnerable cloud environments, and curated practice platforms.

---

## Free Tier Account Setup

Before practising against intentionally vulnerable labs, you need cloud provider accounts. All three offer free tiers.

### AWS Free Tier

```
URL: https://aws.amazon.com/free/
Duration: 12 months for most services (some always-free)
Key free services:
  - EC2: 750 hrs/month t2.micro (or t3.micro in some regions)
  - S3: 5GB storage
  - Lambda: 1M requests/month (always free)
  - IAM: Always free
  - CloudTrail: 1 trail, management events free

Setup:
1. Create account at aws.amazon.com
2. Set up MFA on root account immediately
3. Create an IAM admin user — NEVER use root for daily work
4. Set up billing alerts (SNS + CloudWatch) to avoid surprise charges
5. Enable CloudTrail in all regions
```

> [!danger] Protect your root account and API keys
> AWS root account = full unrestricted access, including billing. If compromised, you can face thousands in fraudulent charges. Enable MFA on root immediately. Never generate root access keys.

### Azure Free Account

```
URL: https://azure.microsoft.com/en-us/free/
Includes:
  - $200 credit for 30 days
  - 12 months of popular services free
  - Always-free services: Azure Functions (1M requests), App Service (10 apps), etc.

Setup:
1. Create account at azure.microsoft.com/free
2. An Azure AD tenant is automatically created
3. Set up MFA for your account
4. Create a resource group for lab work
5. Enable Microsoft Defender for Cloud (free tier)
```

### GCP Free Tier

```
URL: https://cloud.google.com/free
Includes:
  - $300 credit for 90 days
  - Always-free products: Compute Engine f1-micro (1/month), Cloud Storage (5GB), etc.

Setup:
1. Create account at cloud.google.com
2. Create a project for lab work
3. Enable billing alerts
4. Enable Cloud Audit Logs
```

## Intentionally Vulnerable Cloud Labs

These are deliberately misconfigured cloud environments designed for learning. Legal, safe, and exactly what you need to build practical skills.

---

### flaws.cloud — AWS (Attacker Path)

**What it is:** A series of AWS security challenges created by Scott Piper. Each level is a misconfigured AWS resource that you must compromise to find the secret that unlocks the next level.

**Topics covered:**
- Public S3 buckets and enumeration
- EC2 instance profile credential theft
- IMDSv1 metadata service exploitation
- AWS snapshot exposure
- IAM role enumeration and assumption

**URL:** http://flaws.cloud
**Cost:** Free
**Access:** No account needed — you interact from your own machine using AWS CLI

```bash
# Getting started
# Level 1 starts at: http://flaws.cloud
# You'll need: aws cli, nslookup, basic enumeration skills

# Example: enumarating a public S3 bucket (level 1 concept)
aws s3 ls s3://flaws.cloud --no-sign-request
```

---

### flaws2.cloud — AWS (Attacker + Defender Paths)

**What it is:** Sequel to flaws.cloud, by the same author. Two parallel tracks:
- **Attacker path:** Compromise cloud services to progress
- **Defender path:** You're given CloudTrail logs and must answer forensic questions about what the attacker did

**Why it's valuable:** The defender track teaches you what cloud attacks look like from a detection and IR perspective — essential for both red and blue team work.

**URL:** http://flaws2.cloud
**Cost:** Free

---

### CloudGoat — AWS (Scenario-Based Exploitation)

**What it is:** "Vulnerable by Design" AWS deployment tool by Rhino Security Labs. Creates real AWS infrastructure (in your own AWS account) with specific vulnerable scenarios to practice against.

**Topics covered (scenarios):**
- `iam_privesc_by_rollback` — IAM policy version privilege escalation
- `lambda_privesc` — Lambda execution role abuse
- `cloud_breach_s3` — S3 bucket enumeration and data exfiltration
- `ec2_ssrf` — SSRF to metadata service to IAM credential theft
- `codebuild_secrets` — Secrets leaked in CI/CD environment variables
- `sqs_flag_shop` — SQS injection and business logic abuse

**URL:** https://github.com/RhinoSecurityLabs/cloudgoat
**Cost:** Free (you pay AWS costs, usually minimal — a few dollars per session)

```bash
# Install CloudGoat
git clone https://github.com/RhinoSecurityLabs/cloudgoat.git
cd cloudgoat
pip3 install -r requirements.txt
chmod +x cloudgoat.py

# Configure with your AWS credentials
./cloudgoat.py config profile default

# Deploy a scenario
./cloudgoat.py create vulnerable_lambda

# Destroy when done (to stop AWS charges)
./cloudgoat.py destroy vulnerable_lambda
```

> [!warning] CloudGoat deploys real AWS resources
> These are real resources in your AWS account. They cost real money (usually $1-5 per session). Always destroy scenarios when done. Set up a billing alert.

### AzureGoat — Azure

**What it is:** INE's intentionally vulnerable Azure environment. Deploys a series of misconfigurations covering Azure services.

**Topics covered:**
- Azure AD misconfigurations
- Managed identity abuse
- Storage account exposure
- App Service vulnerabilities
- Function App privilege escalation

**URL:** https://github.com/ine-labs/AzureGoat
**Cost:** Free (Azure costs apply, use free credit)

```bash
git clone https://github.com/ine-labs/AzureGoat
cd AzureGoat
# Follow deployment guide in README — uses Terraform
terraform init
terraform apply
```

---

### GCPGoat — GCP

**What it is:** INE's intentionally vulnerable GCP environment, equivalent to AzureGoat for Google Cloud.

**Topics covered:**
- Service account key exposure
- GCS bucket misconfigurations
- Compute Engine metadata abuse
- Cloud Function privilege escalation
- IAM binding weaknesses

**URL:** https://github.com/ine-labs/GCPGoat
**Cost:** Free (GCP costs apply, use $300 credit)

```bash
git clone https://github.com/ine-labs/GCPGoat
cd GCPGoat
# Deployment via Terraform — see README
gcloud auth application-default login
terraform init && terraform apply
```

### HackTheBox — Cloud Machines

HackTheBox (HTB) has cloud-themed machines and challenges. As of 2024, they have a dedicated Cloud category. These are real Linux/Windows machines with cloud misconfigurations or cloud credential-based attack paths.

**Access:** https://hackthebox.com
**Cost:** Free tier + Pro subscription (£14/month) for active machines

---

### TryHackMe — Cloud Rooms

TryHackMe has guided cloud security rooms, good for beginners before tackling CloudGoat.

Notable rooms:
- AWS — IAM misconfigurations
- Cloud Storage Attacks
- Intro to Cloud Security

**Access:** https://tryhackme.com
**Cost:** Free tier + Premium (£10/month) for all rooms

## Recommended Learning Path

Follow this sequence to build from zero to capable:

```
PHASE 1 — Foundations (do this first)
  1. Set up AWS free tier account
  2. Install and configure AWS CLI, Azure CLI, gcloud
  3. Read MODULE 0 notes thoroughly
  4. Play flaws.cloud (levels 1-6)

PHASE 2 — Guided Exploitation
  5. Complete flaws2.cloud (attacker path)
  6. Deploy and complete CloudGoat: cloud_breach_s3 scenario
  7. Deploy and complete CloudGoat: ec2_ssrf scenario

PHASE 3 — Intermediate Practice
  8. Deploy and complete CloudGoat: iam_privesc_by_rollback
  9. Deploy AzureGoat and work through Azure scenarios
  10. Complete flaws2.cloud defender path

PHASE 4 — Advanced
  11. Deploy remaining CloudGoat scenarios
  12. Deploy GCPGoat
  13. HTB cloud machines
  14. Start bug bounty on cloud-heavy targets
```

## Quick Reference — Lab Setup Commands

```bash
# CloudGoat quick start
pip3 install -r requirements.txt
./cloudgoat.py config profile default
./cloudgoat.py create ec2_ssrf
./cloudgoat.py destroy ec2_ssrf

# ScoutSuite against your lab AWS account
source scoutsuite-venv/bin/activate
python3 scout.py aws --profile labprofile

# Prowler against your lab
prowler aws --profile labprofile -M html

# Set temporary lab credentials
export AWS_ACCESS_KEY_ID=AKIA...
export AWS_SECRET_ACCESS_KEY=...
export AWS_DEFAULT_REGION=us-east-1
aws sts get-caller-identity  # confirm
```
