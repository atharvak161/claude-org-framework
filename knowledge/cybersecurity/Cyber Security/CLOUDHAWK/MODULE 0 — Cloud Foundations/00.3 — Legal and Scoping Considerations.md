# 00.3 — Legal and Scoping Considerations

## What This Note Covers

Cloud pentesting has a legal and contractual dimension that doesn't exist in on-prem testing. Testing cloud infrastructure without understanding the boundaries can get you — and your client — banned from the platform, or worse, prosecuted. This note covers exactly what you need to know before touching anything.

---

## The Fundamental Rule

You need **two layers of authorisation** to legally pentest cloud infrastructure:

1. **Written permission from the customer** — the organisation that owns the cloud account
2. **Compliance with the cloud provider's penetration testing policy** — AWS, Azure, and GCP each have their own rules about what testing is and isn't allowed

> [!danger] No exceptions
> Having permission from the customer is not enough. If AWS's terms say you need to notify them before load testing, and you don't, you may have your account suspended regardless of what the customer agreed to.

---

## AWS Penetration Testing Policy

### Current Status (as of 2024)
AWS has a **customer service policy for penetration testing**. The key points:

- AWS **no longer requires prior approval** for most standard penetration testing activities on services you own
- You can test without prior approval against: EC2, RDS, CloudFront, Aurora, API Gateway, Lambda, Lightsail, Elastic Beanstalk
- **Prohibited activities** (require explicit AWS approval or are never permitted):
  - DNS zone walking via Amazon Route 53 Hosted Zones
  - DoS, DDoS, simulated DoS, simulated DDoS
  - Port flooding
  - Protocol flooding
  - Request flooding (login request flooding, API request flooding)

### What Always Requires Customer Written Permission (not just AWS policy)
- Any testing of the customer's AWS account
- Any data access, enumeration, or exfiltration testing
- Social engineering simulations
- Testing in production environments

> [!warning] Always check the latest AWS policy
> Policies change. Before any engagement, verify at: https://aws.amazon.com/security/penetration-testing/

### AWS Rules of Engagement for Pentests

```
Pre-engagement checklist — AWS:
[ ] Written Statement of Work / authorisation letter from customer
[ ] Customer confirms they own the AWS account being tested
[ ] Identify target account IDs, regions, and services in scope
[ ] Confirm CloudTrail is enabled (you want your activity logged for the client's benefit)
[ ] Agree on testing window (avoid peak business hours for production)
[ ] Confirm out-of-scope: multi-tenant services, AWS infrastructure itself
[ ] No DoS, no load testing without explicit separate agreement
```

## Azure Penetration Testing Policy

### Current Status
Microsoft has a **Penetration Testing Rules of Engagement** document. Key points:

- Microsoft **does require** advance notification for certain types of testing
- You must submit a **Microsoft Cloud Penetration Testing Notification** form before testing
- Testing is allowed against resources in your own Azure subscription
- Multi-tenant services (where other customers' data could be affected) are out of scope
- DoS attacks against Azure infrastructure are prohibited

### Microsoft's Specific Requirements
- Notification must be submitted **at least 1 week in advance** for most testing
- Notification form available at: https://portal.msrc.microsoft.com/en-us/engage/pentest
- Physical security testing of Microsoft datacentres is **never permitted**
- Social engineering against Microsoft employees is prohibited

> [!warning] Azure notification is mandatory
> Unlike AWS (which dropped the prior notification requirement for most services), Microsoft still requires you to submit a notification form before testing. Skipping this can result in account suspension.

---

## GCP Penetration Testing Policy

### Current Status
Google Cloud's policy:

- GCP **does not require pre-approval** for security testing of your own GCP resources
- Testing must be conducted only against infrastructure and services owned by the customer (their project/org)
- DoS and DDoS testing against Google's infrastructure is prohibited
- Attacks that could affect other GCP customers are prohibited

> [!tip] GCP is the most permissive
> GCP has the fewest restrictions of the three major providers for standard security testing. That said, always document your authorisation and stay within your project boundaries.

## Scoping a Cloud Penetration Test

### What Goes Into a Cloud Pentest Scope

A proper scope document for a cloud pentest should define:

**Account/Subscription Boundaries:**
- AWS Account ID(s) in scope
- Azure Subscription ID(s) and Tenant ID(s) in scope
- GCP Project ID(s) and Org ID(s) in scope

**Service Boundaries:**
- Which services are in scope: e.g., "All services in us-east-1 except production RDS"
- Explicitly out-of-scope services or environments (prod vs dev vs staging)

**Access Level Granted:**
- Black box: no credentials provided (external attacker simulation)
- Grey box: limited credentials provided (e.g., a low-privilege IAM user)
- White box: full read-only access, IAM policy exports, architecture diagrams

**Testing Activities Permitted:**
- Enumeration and reconnaissance: yes/no
- Privilege escalation testing: yes/no
- Data exfiltration simulation: yes/no (usually a read-only proof of access, not actual exfil)
- Destructive testing: almost never (no deleting resources, no modifying IAM)
- Social engineering: yes/no (separate agreement usually)

**Prohibited Actions:**
- No modification of production data
- No deletion of resources
- No changes to billing or account settings
- No adding IAM users or roles (unless specifically testing persistence)
- No DoS

### Black Box vs White Box vs Grey Box

| Type | Access Given | What You're Simulating |
|---|---|---|
| Black box | Nothing | External attacker with no insider info |
| Grey box | Low-privilege IAM credentials | Compromised employee, leaked API key |
| White box | Full read access + architecture docs | Security audit, assume-breach scenario |

> [!tip] Grey box is most common
> Most cloud assessments are grey box. You're given a low-privilege user to simulate what an attacker could do after finding a leaked API key. Pure black box cloud testing often yields little beyond public resource discovery.

## Key Legal Frameworks to Know

### Computer Fraud and Abuse Act (CFAA) — US
Unauthorised access to computer systems is a federal crime. "Authorised" in cloud means both customer permission AND provider policy compliance. A signed SOW with the customer is essential.

### Computer Misuse Act 1990 — UK
UK equivalent. Unauthorised access to computer material is a criminal offence. Same principle: you need explicit written authorisation.

### GDPR — EU/UK
If you access or handle personal data during the test (which is likely in any real cloud environment), GDPR applies. Your engagement must address:
- How discovered PII will be handled
- Whether test data should be anonymised
- Reporting obligations if you discover a breach

### Cloud Provider Terms of Service
Violating AWS/Azure/GCP ToS is grounds for immediate account suspension, which could take down the client's production environment. This is an engagement-ending risk. Always read the provider's ToS on penetration testing.

## Rules of Engagement Template (Quick Reference)

```
CLOUD PENTEST RULES OF ENGAGEMENT — CHECKLIST

AUTHORISATION:
[ ] Signed SOW or authorisation letter from account owner
[ ] Account/subscription IDs explicitly listed
[ ] Test window agreed (dates, hours, timezone)
[ ] Emergency contact for client (to pause testing)

PROVIDER COMPLIANCE:
[ ] AWS: reviewed current AWS pentest policy (no prior approval needed for standard services)
[ ] Azure: submitted Microsoft notification form (≥1 week in advance)
[ ] GCP: reviewed GCP policy (no prior approval needed for own resources)

SCOPE:
[ ] Regions / locations in scope defined
[ ] Services explicitly in scope listed
[ ] Environments (prod/dev/staging) clarified
[ ] Data handling procedure agreed (no exfil of real PII)

PROHIBITED ACTIONS CONFIRMED:
[ ] No DoS/DDoS
[ ] No deletion of resources
[ ] No modification of production data
[ ] No adding accounts/backdoors without explicit permission
[ ] No testing of other tenants / AWS infrastructure itself
```
