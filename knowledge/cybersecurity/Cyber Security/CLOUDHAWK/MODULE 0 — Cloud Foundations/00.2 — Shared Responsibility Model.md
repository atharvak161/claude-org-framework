# 00.2 — Shared Responsibility Model

## What This Note Covers

The shared responsibility model defines who is responsible for securing what in a cloud environment. It's the foundational legal and operational concept that determines the scope of every cloud penetration test. Getting this wrong means either testing things outside your authority or missing the actual attack surface.

---

## The Core Concept

When an organisation uses a cloud provider, security responsibility is **split** between:

1. **The Cloud Provider** — responsible for securing the physical infrastructure, hypervisors, networking hardware, and the services themselves
2. **The Customer** — responsible for securing everything they build, configure, and deploy on top of that infrastructure

> [!tip] Simple mental model
> The provider locks the building. The customer is responsible for what they put inside it, how they configure the rooms, and who has the keys.

The exact split depends on the **service model** being used — IaaS, PaaS, or SaaS. (Service models are covered in detail in [[00.4 — Cloud Service Models and Attack Surface]].)

---

## AWS Shared Responsibility Model

AWS describes this as: **"Security OF the cloud vs Security IN the cloud"**

### AWS is responsible for (Security OF the cloud):
- Physical data centres, power, cooling, physical access
- Hardware: servers, networking equipment, storage hardware
- Hypervisor and virtualisation layer
- Managed service infrastructure: the underlying compute that runs Lambda, RDS, etc.
- Global network backbone

### The Customer is responsible for (Security IN the cloud):
- **IAM**: who has access, what permissions they have, MFA enforcement
- **Operating systems** on EC2 instances: patching, hardening, endpoint security
- **Application code** running on cloud infrastructure
- **Data**: encryption at rest and in transit, classification, backup
- **Network configuration**: VPC design, security groups, NACLs, routing
- **S3 bucket policies and ACLs**: whether data is publicly accessible
- **CloudTrail and logging**: whether audit logging is enabled and protected
- **Secrets management**: not hardcoding credentials, using Secrets Manager

> [!warning] The most exploitable gap
> The customer is responsible for IAM, data exposure, and network config. These are exactly where 90% of real-world cloud breaches originate. AWS will never stop you from making your S3 bucket public or giving a role `AdministratorAccess` — that's your responsibility.

### AWS Shared Responsibility by Service Type

| Service | Provider Secures | Customer Secures |
|---|---|---|
| EC2 (IaaS) | Hypervisor, hardware | OS, patches, apps, networking, IAM |
| RDS (PaaS) | OS, DB engine patching | DB config, access control, encryption, parameter groups |
| S3 (Object Storage) | Durability, availability | Bucket policies, ACLs, encryption, access logging |
| Lambda (Serverless) | Runtime environment, scaling | Function code, execution role permissions, env vars |
| DynamoDB (Managed DB) | Infrastructure, availability | Table access policies, encryption, data classification |

## Azure Shared Responsibility Model

Azure's model follows the same principle but is articulated across their service tiers.

### Microsoft Azure is responsible for:
- Physical infrastructure: datacentres, hardware, power, cooling
- Network controls at the physical/infrastructure layer
- Host infrastructure (hypervisor)
- Physical security and access to facilities

### The Customer is responsible for:
- **Azure Active Directory / Entra ID**: user accounts, MFA, Conditional Access policies
- **Data governance**: classification, encryption, retention
- **Applications**: code security, authentication logic
- **Network configuration**: NSGs, Azure Firewall, Virtual Networks
- **Identity and access management**: RBAC roles, service principal permissions
- **OS-level security** on VMs: patching, endpoint protection

### Azure Shared Responsibility by Service Type

| Service | Provider Secures | Customer Secures |
|---|---|---|
| Azure VM (IaaS) | Hypervisor, hardware | OS, network config, apps |
| Azure App Service (PaaS) | Runtime, scaling infra | App code, identity, CORS, TLS config |
| Azure SQL Managed Instance | Infrastructure, HA | DB config, auth, firewall rules, encryption |
| Azure Blob Storage | Infrastructure | Access policies, SAS tokens, firewall, encryption |
| Azure Functions (Serverless) | Execution environment | Function code, managed identity permissions |
| Microsoft 365 (SaaS) | Entire application stack | Data, accounts, access policies, DLP |

## GCP Shared Responsibility Model

Google Cloud Platform uses similar language: **"Google manages security of the cloud; you manage security in the cloud."**

### Google is responsible for:
- Physical datacentre security
- Hardware, network infrastructure, storage systems
- Hypervisor and virtualisation layer
- Core GCP service infrastructure (the underlying systems running Compute Engine, Cloud Storage, etc.)

### The Customer is responsible for:
- **IAM**: service accounts, roles, bindings, org policies
- **Network design**: VPC, firewall rules, private Google access
- **Data**: encryption keys (if using CMEK), classification, DLP
- **Application security**: code running on GCP
- **Container security**: images, Kubernetes RBAC, node pool permissions
- **Audit logging**: Cloud Audit Logs configuration and retention
- **Org policies**: constraints on what can be deployed

## Why This Matters for Penetration Testing

### Scope Definition

Before starting any cloud pentest, you must establish which layer of the model you're testing:

- **Testing customer-managed resources** (IAM, storage, compute configs) — almost always in scope, this is where misconfigs live
- **Testing the cloud provider's infrastructure** (hypervisor, physical layer) — this is **out of scope** and may violate provider terms
- **Testing shared-responsibility edge cases** (e.g., managed service configurations) — clarify with the client

### What You're Actually Authorised to Test

When a client engages you for a cloud pentest, you are authorised to test:
- Their IAM configurations
- Their deployed services (EC2, S3, Lambda, etc.)
- Their network configurations (VPCs, security groups)
- Their application code and pipelines
- Their secrets management practices
- Their logging and monitoring setup

You are **not** authorised to:
- Attack AWS/Azure/GCP infrastructure itself
- Attempt to break out of the hypervisor
- Test other customers sharing the same cloud infrastructure
- Perform DDoS or availability testing without explicit written permission

> [!danger] Provider permission requirements
> AWS, Azure, and GCP all have policies on penetration testing. AWS requires advance notification for certain activities (load testing, simulated DDoS). Azure has a similar policy. GCP requires a formal request. Always read the provider's current penetration testing policy before engagement. This is covered in [[00.3 — Legal and Scoping Considerations]].

## Shared Responsibility Failure Patterns (Real-World)

These are the most common ways customers fail their side of the shared responsibility model:

1. **S3 bucket left public** — customer failed to configure bucket ACL/policy (not AWS's fault)
2. **IAM user with `AdministratorAccess` and no MFA** — customer failed to enforce MFA and least privilege
3. **RDS database publicly accessible** — customer checked "publicly accessible" during setup without understanding the implication
4. **EC2 instance with IMDSv1 enabled** — customer didn't enforce IMDSv2, leaving SSRF-to-credential-theft open
5. **Secrets in environment variables** — customer stored database passwords in Lambda env vars instead of Secrets Manager
6. **CloudTrail disabled in a region** — customer didn't enable audit logging, so attacker activity goes undetected
7. **Over-permissive service roles** — Lambda function role has `s3:*` on `*` instead of just the bucket it needs

## Quick Reference

```
AWS Responsibility Split (simplified):
- AWS owns: hardware → hypervisor → managed service runtime
- Customer owns: IAM → OS → apps → data → network config → logging

Azure Responsibility Split:
- Microsoft owns: physical → hypervisor → (in SaaS, everything)
- Customer owns: identities → data → devices → access config

GCP Responsibility Split:
- Google owns: physical → hypervisor → service infrastructure  
- Customer owns: IAM → network → data → apps → audit config
```
