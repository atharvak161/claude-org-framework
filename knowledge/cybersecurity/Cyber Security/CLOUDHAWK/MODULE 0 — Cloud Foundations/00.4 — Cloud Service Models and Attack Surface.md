# 00.4 — Cloud Service Models and Attack Surface

## What This Note Covers

IaaS, PaaS, and SaaS are the three cloud service delivery models. Each one gives the customer a different level of control — and correspondingly, a different attack surface. Understanding this is essential for knowing what to test and where to look in any cloud engagement.

---

## The Three Service Models — Overview

Think of it as a spectrum of control:

```
More customer control ←————————————————→ More provider control
        IaaS                PaaS                  SaaS
(you manage the OS)    (you manage the app)   (you manage the data)
```

As you move right, the provider takes on more responsibility and the customer has fewer moving parts to misconfigure — but also fewer levers to pull for hardening.

---

## IaaS — Infrastructure as a Service

### What It Is
The cloud provider gives you virtualised compute, storage, and networking. You get a virtual machine (or bare metal), and everything above the hypervisor is your problem.

**Examples:** AWS EC2, Azure Virtual Machines, Google Compute Engine, AWS EBS (storage), Azure Blob Storage (when used raw)

### What the Customer Controls
- Operating system (choice, patching, configuration)
- Runtime environment (language runtimes, web servers)
- Application code
- Middleware
- Data
- Network configuration (security groups, firewall rules, VPC/VNET routing)

### IaaS Attack Surface

| Attack Vector | Description |
|---|---|
| Exposed services | SSH, RDP, web servers, databases exposed to internet via misconfigured security groups |
| Unpatched OS | Customer is responsible for OS patching — missed patches = CVEs |
| IMDSv1 SSRF | Instance metadata service accessible from instance — SSRF can steal IAM credentials |
| Instance profile abuse | EC2 instance attached to an over-permissive IAM role — compromise the instance, get the role |
| EBS snapshot exposure | Snapshots shared publicly — attacker can mount and read the disk |
| Userdata scripts | EC2 userdata (startup scripts) often contain credentials or config that's readable |
| Public AMIs | Custom AMIs (disk images) shared publicly may contain secrets or sensitive config |
| Default credentials | Out-of-box services running with default passwords |

> [!warning] IaaS is the broadest attack surface
> You're responsible for the OS and everything up. One missed patch, one misconfigured security group, one IMDSv1 SSRF — and the whole instance, and its IAM role, is compromised.

---

## PaaS — Platform as a Service

### What It Is
The provider manages the OS, runtime, middleware, and scaling infrastructure. You deploy your application code and configure the service. You don't manage servers.

**Examples:** AWS Lambda, AWS RDS, Azure App Service, Azure SQL Database, Google App Engine, Google Cloud Functions, Heroku, AWS Elastic Beanstalk

### What the Customer Controls
- Application code
- Data
- Service configuration (connection strings, environment variables, access policies)
- Identity and access (who can invoke the function, who can connect to the database)

### PaaS Attack Surface

| Attack Vector | Description |
|---|---|
| Execution role abuse | Lambda/Function App has an over-permissive IAM/managed identity role — compromise the function, get the role |
| Environment variable exposure | Database passwords, API keys stored in function environment variables (visible in console, sometimes in logs) |
| Publicly accessible databases | RDS/Azure SQL with "public access" enabled and weak auth |
| Event injection | Lambda triggered by SQS/S3/API Gateway — malicious payloads in the event source affect function behaviour |
| Dependency vulnerabilities | Application code using outdated libraries with known CVEs (provider doesn't patch your code) |
| Function enumeration | Listing all Lambda functions, their triggers, their roles — maps the application architecture |
| Misconfigured API Gateway | No authentication on API endpoints, resource policy too permissive |
| Container image vulnerabilities | If PaaS uses container-based deployment — vulnerable base images |

> [!tip] PaaS narrows OS-level risk, opens IAM risk
> You don't worry about OS patches with Lambda. But the IAM execution role attached to that Lambda, and what's in its environment variables, is entirely your problem.

---

## SaaS — Software as a Service

### What It Is
The provider delivers a fully managed application. The customer uses it — they don't manage the infrastructure, OS, runtime, or (usually) the application code.

**Examples:** Microsoft 365 (Exchange, SharePoint, Teams), Google Workspace, Salesforce, Slack, Dropbox, ServiceNow, GitHub (the SaaS product)

### What the Customer Controls
- Data and content
- User accounts and access control (who can access what)
- Configuration of the SaaS product (sharing settings, external access, integrations)
- Authentication settings (SSO, MFA enforcement)

### SaaS Attack Surface

| Attack Vector | Description |
|---|---|
| Misconfigured sharing settings | Files/docs/repos shared publicly or with entire organisation when should be restricted |
| Weak authentication | No MFA enforced, weak password policies, no Conditional Access |
| OAuth app abuse | Third-party apps granted excessive permissions to the SaaS (e.g., app with mail read access to all inboxes) |
| Account takeover | Phishing, credential stuffing against SaaS login — if no MFA, game over |
| Data exfiltration via SaaS features | Legitimate sync features used to exfiltrate data (SharePoint sync, Google Drive sharing) |
| API key exposure | SaaS platforms issue API tokens that can be leaked (GitHub tokens, Slack tokens in code) |
| Insider threat via SaaS | Over-permissioned admin accounts in the SaaS product |
| Supply chain via integrations | Third-party integrations/plugins to the SaaS with excessive access or vulnerabilities |

> [!warning] In SaaS, identity is everything
> The provider handles almost all technical security. What remains is identity, access configuration, and data governance. A compromised admin account in Microsoft 365 can be catastrophic without MFA.

---

## Service Model Comparison — Pentester's View

| Factor | IaaS | PaaS | SaaS |
|---|---|---|---|
| Pentest focus | OS, network, IAM, CVEs | IAM, env vars, code, event injection | Auth, config, API tokens, data access |
| Primary risk | Unpatched OS + over-permissive IAM | Over-permissive execution roles + exposed secrets | Account takeover + misconfigured sharing |
| OS-level access | Yes (you own the OS) | No (provider owns runtime) | No |
| Most impactful finding | IMDSv1 SSRF to IAM credential theft | Execution role with wildcard permissions | Admin account with no MFA |
| Logging | CloudTrail + OS logs | CloudTrail + function logs | SaaS audit log (if enabled) |

## Serverless — A Special Case

Serverless is a subset of PaaS where you write functions that execute in response to events. No servers to manage at all — not even conceptually. The compute is completely ephemeral.

**Why it matters for security:**
- No OS to harden — that's the provider's problem
- No persistent environment — can't install a rootkit or maintain a reverse shell the traditional way
- All persistence must be via IAM backdoors, data store modifications, or function code modification
- Attack surface is concentrated on: the function's **IAM execution role**, **environment variables**, **trigger configuration**, and **function code quality**

This is covered in depth in [[MODULE 5 — Serverless & PaaS Attack Surface]].

## Quick Reference — Attack Surface by Service Model

```
IaaS:
  → Security groups / NSGs (open ports?)
  → OS patch level (CVEs?)
  → IMDSv1 enabled? (SSRF → IAM cred theft)
  → Instance profile permissions (over-privileged?)
  → EBS snapshots public?
  → Userdata scripts contain creds?

PaaS:
  → Execution role permissions (wildcard policies?)
  → Environment variables contain secrets?
  → Database publicly accessible?
  → API endpoints authenticated?
  → Event triggers exploitable?

SaaS:
  → MFA enforced on all accounts?
  → Admin accounts audited?
  → External sharing disabled or scoped?
  → Third-party OAuth apps audited?
  → API tokens rotated and scoped?
```
