# 01.4 — Technology Fingerprinting

## What It Is

Technology fingerprinting is the process of **identifying what software, frameworks, and services a target is running** before you start testing.

Why it matters:
- Different technologies have different vulnerabilities (WordPress ≠ Django ≠ .NET)
- Knowing the version lets you look up known CVEs directly
- Identifies what tools and attack techniques are relevant
- Reveals mismatches (e.g., server claims nginx but responds like Apache — WAF in front)

You're building a **technology profile** that guides every subsequent attack decision.

---

## What to Fingerprint

| Layer | What to Look For |
|---|---|
| Web server | Apache, nginx, IIS, Caddy — and version |
| Backend language | PHP, Python, Ruby, Java, ASP.NET |
| Framework | WordPress, Laravel, Django, Rails, Spring |
| CMS | WordPress, Drupal, Joomla, Magento |
| JavaScript frameworks | React, Angular, Vue, jQuery |
| CDN / WAF | Cloudflare, Akamai, Imperva, AWS CloudFront |
| Cloud provider | AWS, Azure, GCP |
| Analytics / third-party | Google Analytics, Hotjar, HubSpot |
| Programming language version | PHP 7.4 vs 8.1, Python 2 vs 3 |

## HTTP Headers — Manual Analysis

HTTP response headers are the single most information-rich passive source for technology fingerprinting. They often directly reveal the server software and version.

```bash
# Use curl to fetch just the headers
curl -I https://example.com

# Follow redirects
curl -IL https://example.com

# Include response body (useful for meta tags)
curl -s https://example.com | head -100

# Custom User-Agent (some servers respond differently to known scanners)
curl -A "Mozilla/5.0 (Windows NT 10.0; Win64; x64)" -I https://example.com
```

### Key Headers to Analyse

| Header | What It Reveals |
|---|---|
| `Server` | Web server software and version — e.g., `Apache/2.4.51 (Ubuntu)` |
| `X-Powered-By` | Backend technology — e.g., `PHP/7.4.3`, `ASP.NET` |
| `X-Generator` | CMS — e.g., `WordPress 6.2`, `Drupal 9` |
| `X-AspNet-Version` | .NET version |
| `X-Frame-Options` | Security headers (or lack thereof) |
| `Content-Security-Policy` | CSP presence and configuration |
| `Strict-Transport-Security` | HSTS configuration |
| `Set-Cookie` | Cookie names reveal tech: `PHPSESSID` = PHP, `JSESSIONID` = Java/Tomcat, `ASP.NET_SessionId` = ASP.NET, `wordpress_` = WordPress |
| `Via` | Reverse proxy / CDN in use |
| `CF-Ray` | Cloudflare is in the path |
| `X-Varnish` | Varnish cache |

### Example Output Analysis

```
HTTP/1.1 200 OK
Server: Apache/2.4.51 (Ubuntu)              ← Apache 2.4.51 on Ubuntu
X-Powered-By: PHP/7.4.3                     ← PHP 7.4.3 (check CVEs — EOL version)
Set-Cookie: wordpress_test_cookie=WP+Cookie+check; path=/   ← WordPress
X-Frame-Options: SAMEORIGIN                 ← Some security headers present
Content-Type: text/html; charset=UTF-8
```

From these headers alone: Ubuntu Linux, Apache 2.4.51, PHP 7.4.3 (end-of-life, multiple CVEs), WordPress.

Next step: `searchsploit php 7.4`, `searchsploit apache 2.4.51`, look up WordPress version.

## WhatWeb

WhatWeb identifies technologies used by websites. It queries the site and analyses responses, HTML, JavaScript, cookies, and headers.

```bash
# Already on Kali
whatweb example.com

# Verbose output (shows what triggered each detection)
whatweb -v example.com

# Aggressive mode (makes more HTTP requests for deeper detection)
whatweb -a 3 example.com

# Multiple targets
whatweb -i urls.txt --log-brief=results.txt

# JSON output
whatweb --log-json=results.json example.com

# Follow redirects
whatweb --follow-redirect=always example.com
```

### Output Interpretation

```
http://example.com [200 OK] 
  Apache[2.4.51], 
  Country[UNITED STATES][US], 
  HTML5, 
  HTTPServer[Ubuntu Linux][Apache/2.4.51 (Ubuntu)], 
  IP[93.184.216.34], 
  JQuery[3.6.0], 
  PHP[7.4.3], 
  WordPress[6.2], 
  X-Powered-By[PHP/7.4.3]
```

Each detected technology is a lead. WhatWeb knows 1800+ technologies — it's identifying them from signatures in the response.

## Wappalyzer

Wappalyzer is a browser extension (Chrome/Firefox) that identifies technologies on any page you visit. It's the passive, point-and-click version of WhatWeb.

Install: https://www.wappalyzer.com/

Also available as a CLI tool:

```bash
# Install
npm install -g wappalyzer

# Analyse a URL
wappalyzer https://example.com --pretty
```

Good for quick checks when browsing a target's site manually.

## Nikto — Web Server Scanner

Nikto is a web server scanner that checks for:
- Outdated server software
- Dangerous files and directories (backup files, readme files, install files)
- Server misconfigurations
- Default credentials
- Known vulnerable files (e.g., `/phpinfo.php`, `/server-status`)

> [!warning]
> Nikto is **noisy** — it makes hundreds of HTTP requests in a short period. It will appear in the target's web logs and may trigger IDS/WAF alerts. Use it only when active scanning is authorised.

```bash
# Basic scan
nikto -h https://example.com

# Specify port
nikto -h example.com -p 8080

# SSL
nikto -h example.com -ssl

# Save output
nikto -h example.com -o nikto_results.txt -Format txt
nikto -h example.com -o nikto_results.html -Format html

# Tune scan (focus on specific checks)
nikto -h example.com -Tuning 1    # Interesting files
nikto -h example.com -Tuning 2    # Misconfiguration
nikto -h example.com -Tuning 4    # Injection
nikto -h example.com -Tuning 9    # SQL injection
```

### What Nikto Finds

```
+ /phpinfo.php: Output from the phpinfo() function was found.
+ /admin/: Admin directory found.
+ /backup/: Backup directory found.
+ Apache/2.4.51 appears to be outdated (current is at least Apache/2.4.54).
+ /server-status: Apache server-status found. Leaks server information.
+ OSVDB-3233: /icons/README: Apache default file found.
```

Each finding is actionable: `/phpinfo.php` reveals the full PHP configuration including environment variables that may contain credentials.

## Identifying WAF (Web Application Firewall)

Before you start testing, determine if a WAF is present. If it is, your payloads may need to be obfuscated, and aggressive scanning may trigger blocking.

### wafw00f

```bash
# Install
pip3 install wafw00f
# Or:
sudo apt install wafw00f

# Detect WAF
wafw00f https://example.com

# Test multiple URLs
wafw00f -i urls.txt

# Find all WAFs (not just the first match)
wafw00f -a https://example.com
```

### Output Examples

```
[*] Checking https://example.com
[+] The site https://example.com is behind Cloudflare (Cloudflare Inc.)
```

```
[*] Checking https://example.com
[+] The site https://example.com is behind Imperva (Imperva Inc.)
```

```
[~] The site https://example.com does not seem to be behind a WAF or IDS.
```

### Manual WAF Detection

Send a clearly malicious request and observe the response:

```bash
# Send XSS payload and observe response
curl -s "https://example.com/?q=<script>alert(1)</script>" -v

# Send SQLi payload
curl -s "https://example.com/?id=1' OR '1'='1" -v
```

WAF indicators:
- Response code 403 or 406 for obviously malicious input
- Unusual redirect (to `blocked.cloudflare.com`, etc.)
- Response body contains "Access Denied", "Blocked", "Security Policy"
- `CF-Ray` header → Cloudflare
- `X-Sucuri-ID` → Sucuri
- `X-Powered-By: AWS WAF` → AWS WAF

## CMS-Specific Fingerprinting

### WordPress

```bash
# wpscan — purpose-built WordPress scanner
wpscan --url https://example.com

# Enumerate users
wpscan --url https://example.com --enumerate u

# Enumerate plugins (with vulnerability check)
wpscan --url https://example.com --enumerate p --plugins-detection aggressive

# Enumerate themes
wpscan --url https://example.com --enumerate t

# With WPScan API token (for vulnerability data)
wpscan --url https://example.com --api-token YOUR_TOKEN --enumerate vp,vt,u
```

WPScan finds: WordPress version, installed plugins/themes with known CVEs, usernames (from `/wp-json/wp/v2/users`), exposed sensitive files.

Manual WordPress fingerprinting:
```bash
# Version in readme
curl -s https://example.com/readme.html | grep -i "version"

# Version in generator meta tag
curl -s https://example.com | grep -i "generator"

# Login page
curl -I https://example.com/wp-login.php

# REST API user enumeration
curl -s https://example.com/wp-json/wp/v2/users | jq '.[].name'
```

### Drupal

```bash
# droopescan — CMS scanner for Drupal, WordPress, Joomla, SilverStripe
pip3 install droopescan
droopescan scan drupal -u https://example.com
```

Manual Drupal fingerprinting:
```bash
# Version in CHANGELOG.txt (often exposed)
curl -s https://example.com/CHANGELOG.txt | head -5

# Version in core/modules
curl -s https://example.com/core/install.php
```

### Joomla

```bash
droopescan scan joomla -u https://example.com

# Manual — version often in XML files
curl -s https://example.com/administrator/manifests/files/joomla.xml
```

## OS and Server Fingerprinting via TTL

You can guess the operating system from the TTL (Time to Live) value in ping responses:

```bash
ping -c 1 example.com
```

| TTL Value | Likely OS |
|---|---|
| 64 | Linux / Unix |
| 128 | Windows |
| 255 | Cisco IOS / network device |

> [!tip]
> TTL decrements by 1 at each router hop, so a received TTL of 120 suggests Windows (128 - 8 hops = 120). Use this as a rough indicator, not a definitive identification.

## Building Your Technology Profile

After fingerprinting, document everything in your notes:

```
TARGET: example.com
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TECHNOLOGY STACK
  OS:          Ubuntu Linux (from Apache header)
  Web Server:  Apache 2.4.51 (known CVEs: CVE-2021-41773, CVE-2021-42013)
  Language:    PHP 7.4.3 (EOL — no security patches since Nov 2022)
  CMS:         WordPress 6.2 (check WPScan for plugin vulns)
  Database:    MySQL (assumed from WP stack)
  JS:          jQuery 3.6.0

WAF/CDN
  Cloudflare detected (real IP hidden — try historical DNS lookup, Shodan)

SECURITY HEADERS
  X-Frame-Options: SAMEORIGIN ✓
  HSTS: Not present ✗
  CSP: Not present ✗
  X-Content-Type-Options: Not present ✗

CMS FINDINGS (WPScan)
  Plugins: Contact Form 7 5.7 (no known CVEs)
  Plugins: Elementor 3.12 (no known CVEs)
  Users: admin (ID 1), john.smith (ID 2)

NEXT STEPS
  1. PHP 7.4 EOL — research public CVEs
  2. WordPress user enumeration complete — attempt password spray
  3. Cloudflare WAF present — start with non-intrusive payloads
  4. No CSP — XSS likely to succeed without restrictions
```

## Quick Reference

```bash
# Manual headers
curl -IL https://example.com

# WhatWeb
whatweb -v -a 3 example.com

# Nikto
nikto -h https://example.com -o nikto.html -Format html

# WAF detection
wafw00f https://example.com

# WordPress scan
wpscan --url https://example.com --enumerate vp,vt,u --api-token TOKEN

# CMS detection
droopescan scan -u https://example.com
```

| Tool | Purpose | Noise Level |
|---|---|---|
| curl -I | HTTP header analysis | Very low |
| WhatWeb | Tech stack identification | Low |
| Wappalyzer | Browser-based passive | Passive |
| Nikto | Server misconfigs + files | High |
| wafw00f | WAF detection | Low |
| WPScan | WordPress full audit | Medium |
| droopescan | Multi-CMS scan | Medium |
