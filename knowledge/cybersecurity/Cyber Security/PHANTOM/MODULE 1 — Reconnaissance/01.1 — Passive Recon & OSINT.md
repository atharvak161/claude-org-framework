# 01.1 — Passive Recon & OSINT

## What It Is

Passive reconnaissance is the process of **gathering information about a target without directly interacting with their systems**.

You are not sending packets to the target. You are querying public sources — search engines, DNS records, certificate transparency logs, social media, job postings — and building a picture of the target's attack surface.

Why this matters: passive recon is **completely undetectable** by the target. No firewall log, no IDS alert, no connection. You learn as much as possible before you touch anything.

---

## What You're Looking For

Before you start, define your intelligence goals:

| Goal | What to Find |
|---|---|
| Infrastructure | IP ranges, ASNs, hosting providers, CDN presence |
| Domains | Subdomains, related domains, old/forgotten domains |
| Technology | Web frameworks, CMS, server software, cloud provider |
| People | Employee names, email addresses, job roles |
| Credentials | Leaked passwords, API keys in public repos |
| Documents | PDFs, spreadsheets, config files accidentally exposed |
| Network | Open ports visible from internet, exposed admin panels |

## Google Dorks

Google dorks use advanced search operators to find information that normal searches miss. This is free, powerful, and entirely passive.

### Core Operators

| Operator | What It Does | Example |
|---|---|---|
| `site:` | Restrict results to a domain | `site:example.com` |
| `filetype:` | Find specific file types | `filetype:pdf site:example.com` |
| `inurl:` | Match text in the URL | `inurl:admin site:example.com` |
| `intitle:` | Match text in the page title | `intitle:"index of" site:example.com` |
| `intext:` | Match text in page body | `intext:"password" site:example.com` |
| `ext:` | Alias for filetype | `ext:sql site:example.com` |
| `cache:` | Google's cached version | `cache:example.com` |
| `-` | Exclude results | `site:example.com -www` |
| `"..."` | Exact phrase match | `"internal use only" site:example.com` |

### High-Value Dorks

```
# Find exposed config files
site:example.com ext:xml OR ext:conf OR ext:cnf OR ext:ini OR ext:env OR ext:log

# Find backup files
site:example.com ext:bak OR ext:backup OR ext:old OR ext:orig OR ext:sql

# Find login pages
site:example.com inurl:login OR inurl:signin OR inurl:admin OR inurl:dashboard

# Find directory listings (web server misconfiguration)
intitle:"index of" site:example.com

# Find documents with sensitive content
site:example.com filetype:pdf OR filetype:xlsx OR filetype:docx "confidential"

# Find API keys or credentials in public files
site:example.com intext:"api_key" OR intext:"api-key" OR intext:"Authorization: Bearer"

# Find error messages revealing stack traces
site:example.com intext:"stack trace" OR intext:"SQLException" OR intext:"Warning: mysql"

# Find subdomains (complement to dedicated tools)
site:*.example.com -www

# GitHub dorks (search on github.com)
"example.com" password OR secret OR key OR token language:python
```

> [!tip]
> Google Hacking Database (GHDB) at https://www.exploit-db.com/google-hacking-database contains thousands of pre-built dorks categorised by type (files, login pages, sensitive info, etc.). Always check it before building dorks from scratch.

## WHOIS Lookup

WHOIS reveals domain registration data — who registered a domain, when, and through which registrar. Useful for:
- Identifying the organisation behind a domain
- Finding related domains registered by the same person
- Historical ownership changes
- Abuse contact details

```bash
whois example.com
```

Key fields to note:
- **Registrant Organisation** — company name (often redacted via privacy protection)
- **Registrant Email** — can be used to find other domains registered by the same person
- **Name Servers** — tells you who manages DNS (important for later)
- **Creation Date** — old domains may have more legacy exposure
- **Expiry Date** — near-expiry domains are sometimes hijackable

Online alternatives (when CLI output is minimal):
- https://who.is
- https://lookup.icann.org
- https://whois.domaintools.com (historical records, paid)

## DNS Enumeration

DNS records reveal the structure of a target's infrastructure.

### Record Types to Enumerate

| Record Type | What It Contains | Why It Matters |
|---|---|---|
| A | IPv4 address for a hostname | Reveals server IPs |
| AAAA | IPv6 address | IPv6-exposed services |
| MX | Mail server hostnames | Email infrastructure, phishing setup |
| NS | Authoritative name servers | DNS provider |
| TXT | Arbitrary text | SPF/DKIM/DMARC (email security posture), sometimes API keys |
| CNAME | Alias to another hostname | Often reveals internal names or third-party services |
| SOA | Start of Authority | Zone info, admin contact |
| PTR | Reverse DNS (IP → hostname) | Reveals hostnames for known IPs |
| SRV | Service locator records | Internal services (SIP, XMPP, LDAP) |

### Manual DNS Queries with `dig`

```bash
# A record (IPv4)
dig example.com A

# All records
dig example.com ANY

# MX records
dig example.com MX

# NS records
dig example.com NS

# TXT records (SPF, DKIM, verification tokens)
dig example.com TXT

# Reverse DNS (IP to hostname)
dig -x 93.184.216.34

# Query a specific DNS server (e.g., Google's 8.8.8.8)
dig @8.8.8.8 example.com A

# Short output only (just the answer)
dig example.com A +short
```

### DNS Zone Transfer (AXFR)

A zone transfer is a mechanism for DNS servers to replicate records to each other. If a nameserver is misconfigured to allow transfers from any IP, you can dump all DNS records for a domain at once.

```bash
# Find the authoritative name servers first
dig example.com NS +short

# Attempt zone transfer from each NS
dig @ns1.example.com example.com AXFR
dig @ns2.example.com example.com AXFR
```

Expected output if misconfigured:
```
; <<>> DiG 9.x <<>> @ns1.example.com example.com AXFR
example.com.    3600  IN  SOA   ns1.example.com. admin.example.com. ...
www             3600  IN  A     93.184.216.34
mail            3600  IN  A     93.184.216.50
vpn             3600  IN  A     10.0.0.1
internal-db     3600  IN  A     10.0.0.50
...
```

> [!warning]
> Zone transfers are increasingly rare on modern infrastructure. But when they work, they're gold — you get the entire DNS map of the target in one query.

## theHarvester

theHarvester is a tool for gathering **emails, hostnames, IPs, and employee names** from public sources including search engines, DNS, and breach databases.

```bash
# Already installed on Kali
theHarvester --help
```

### Core Usage

```bash
# Search using multiple sources
theHarvester -d example.com -b google,bing,duckduckgo,crtsh,dnsdumpster

# Search with a higher result limit
theHarvester -d example.com -b google -l 500

# Save results to HTML and XML
theHarvester -d example.com -b all -f results_example
```

### Key Flags

| Flag | Meaning |
|---|---|
| `-d` | Target domain |
| `-b` | Data source(s) — comma-separated |
| `-l` | Limit results (default 500) |
| `-f` | Output filename (saves .html and .xml) |
| `-s` | Start result offset |

### Available Sources

```bash
# View all available sources
theHarvester -d test.com -b all
```

Most useful: `google`, `bing`, `duckduckgo`, `crtsh` (certificate transparency), `dnsdumpster`, `urlscan`, `otx` (AlienVault), `shodan`

### What to Do with Output

- **Email addresses** → Use for phishing simulation (if in scope), or to check breach databases
- **Hostnames** → Feed into subdomain enumeration tools, scan for open ports
- **IPs** → Add to scope map, scan with Nmap

## Shodan

Shodan is a search engine for internet-connected devices. Unlike Google which indexes web pages, Shodan indexes **banners** — the responses that services give when you connect to them (HTTP headers, SSH banners, FTP welcome messages, etc.).

Website: https://www.shodan.io  
CLI: `shodan` (requires free API key)

### What Shodan Reveals

- Open ports and services on every IP Shodan has scanned
- Software versions (often revealing CVEs)
- SSL certificate details (hostnames, expiry, weak ciphers)
- Geographic location, ASN, ISP
- Industrial control systems (SCADA/ICS) exposed to internet
- Default credentials on devices (cameras, routers, printers)

### Shodan Web Search

```
# Find all hosts for a domain
hostname:example.com

# Find specific service on a domain
hostname:example.com port:8080

# Find hosts with a specific banner text
hostname:example.com "Apache/2.2"

# Find hosts by organisation name
org:"Example Corp"

# Find hosts by IP range (CIDR)
net:93.184.216.0/24

# Find hosts running a specific product
product:"nginx" hostname:example.com

# Find hosts with SSL cert for a domain (finds subdomains)
ssl.cert.subject.cn:example.com

# Find open RDP (port 3389) for an org
org:"Example Corp" port:3389

# Find exposed Elasticsearch instances (no auth)
product:"Elastic" port:9200
```

### Shodan CLI

```bash
# Install
pip3 install shodan

# Initialise with API key (free at shodan.io/account)
shodan init YOUR_API_KEY

# Search
shodan search hostname:example.com

# Get info on a specific IP
shodan host 93.184.216.34

# Count results without showing them
shodan count hostname:example.com
```

> [!tip]
> Shodan's free tier limits searches. The CLI `shodan host <IP>` command is extremely useful — run it on every IP you discover to see what services are exposed and what versions they're running, before you even touch Nmap.

## Censys

Similar to Shodan but with better certificate search and structured JSON output. Particularly powerful for:
- Finding all IPs behind a domain's SSL cert
- Discovering assets the client doesn't know about
- Tracking infrastructure changes over time

Website: https://search.censys.io

```
# Find hosts matching a domain's certificate
services.tls.certificates.leaf_data.subject.common_name: example.com

# Find specific ports and services
services.port: 8443 AND services.tls.certificates.leaf_data.subject.cn: example.com
```

## Certificate Transparency Logs

When a CA (Certificate Authority) issues an SSL/TLS certificate, it's logged publicly in Certificate Transparency (CT) logs. This means **every subdomain that has ever had an SSL cert issued is publicly visible**.

### crt.sh

```bash
# Browser
https://crt.sh/?q=%.example.com

# API (curl)
curl -s "https://crt.sh/?q=%.example.com&output=json" | jq '.[].name_value' | sort -u
```

Output gives you every subdomain that has ever had a certificate — including:
- Staging environments (`staging.example.com`)
- Dev environments (`dev.api.example.com`)
- Internal tools (`jira.example.com`, `gitlab.example.com`)
- Historical subdomains no longer in DNS but potentially still live

## Maltego

Maltego is a graphical OSINT tool that maps relationships between entities (domains, IPs, emails, people, organisations) visually as a graph.

Comes pre-installed on Kali. Free Community Edition available.

```bash
maltego
```

Transforms (queries you run on an entity):
- Domain → IP addresses
- IP → Hostnames (reverse DNS)
- Email → Social media accounts
- Person → Email addresses

> [!tip]
> Maltego is powerful but slow for large targets. Better used for targeted investigation of specific entities (a person, a domain) than broad recon of an entire org.

## Breach Data & Credential Exposure

Check if target email addresses appear in known data breaches. This is intelligence for phishing simulations and password spray attacks (if in scope).

### haveibeenpwned.com API

```bash
# Check if an email is in a breach (API key required for bulk)
curl "https://haveibeenpwned.com/api/v3/breachedaccount/user@example.com" \
  -H "hibp-api-key: YOUR_KEY"
```

### DeHashed (paid)
https://dehashed.com — search by email, username, password, IP, domain

### GitHub Credential Leaks

Developers often accidentally commit API keys, passwords, and tokens to public GitHub repos.

```bash
# Search GitHub directly
https://github.com/search?q=%22example.com%22+password&type=code
https://github.com/search?q=%22example.com%22+api_key&type=code

# Tools for automated GitHub searching
# truffleHog — scans repos for high-entropy strings and known secret patterns
pip3 install truffleHog
trufflehog github --repo=https://github.com/target/repo

# gitleaks
apt install gitleaks
gitleaks detect --source /path/to/cloned/repo
```

## Quick Reference

```bash
whois example.com                                          # Domain registration
dig example.com ANY                                        # All DNS records
dig @ns1.example.com example.com AXFR                     # Zone transfer attempt
theHarvester -d example.com -b google,crtsh,dnsdumpster   # Email/host harvest
curl "https://crt.sh/?q=%.example.com&output=json" | jq '.[].name_value' | sort -u  # CT logs
shodan host <IP>                                           # Shodan IP lookup
```

| Source | Best For |
|---|---|
| Google Dorks | Exposed files, login pages, error messages |
| WHOIS | Registrant info, related domains |
| DNS / dig | Infrastructure mapping, zone transfer |
| theHarvester | Emails, hostnames, combined OSINT |
| Shodan | Open ports, software versions, exposed devices |
| crt.sh | Subdomain discovery via SSL certs |
| GitHub search | Leaked credentials, API keys |
| HaveIBeenPwned | Breach exposure of email addresses |
