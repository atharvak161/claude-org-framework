# 00.3 — Lab Setup

## Why You Need a Lab

You cannot practice pentesting on systems you don't own or have permission to test. A local lab gives you:

- A **safe, isolated environment** to practice exploits without breaking real systems or committing crimes
- **Full control** over the target — you can reset, snapshot, and experiment freely
- The ability to run **both attacker and victim** machines on the same laptop

A home lab is the single most important investment in your pentesting journey. Everything else is theory until you build one.

---

## Architecture Overview

```
┌─────────────────────────────────────────────┐
│              Your Physical Machine           │
│                                             │
│  ┌──────────────┐    ┌──────────────────┐  │
│  │  Kali Linux  │    │  Target Machine  │  │
│  │  (Attacker)  │◄──►│  (Metasploitable │  │
│  │              │    │   / DVWA / etc.) │  │
│  └──────────────┘    └──────────────────┘  │
│         Isolated Internal Network Only      │
└─────────────────────────────────────────────┘
```

Both VMs sit on an **internal/host-only network** — no connection to the internet from the target. This prevents your exploits from accidentally hitting real systems.

## Step 1 — Install a Hypervisor

A hypervisor lets you run multiple operating systems (virtual machines) on your physical machine simultaneously.

### Option A: VirtualBox (Free, Recommended for Beginners)

Download: https://www.virtualbox.org/wiki/Downloads

- Works on Windows, macOS, Linux
- Free and open source
- Slightly less performant than VMware but excellent for labs
- Install VirtualBox Extension Pack as well (USB 3.0, better networking)

### Option B: VMware Workstation Pro (Free since 2024)

Download: https://www.vmware.com/products/workstation-pro.html

- Better performance, better snapshot management
- Now free for personal use
- Better USB passthrough if you're testing physical hardware (Wi-Fi adapters, etc.)

> [!tip]
> If you have a decent machine (8GB+ RAM, SSD), either works. VMware is faster. VirtualBox is simpler to start with.

## Step 3 — Network Modes Explained

This is critical to understand. The wrong network mode can expose your attack traffic to the internet or prevent you from reaching your target.

### NAT (Network Address Translation)
- VM shares your host machine's IP address
- VM can reach the internet (to install tools, update packages)
- **Host cannot initiate connections to VM** (VM is behind NAT)
- Other VMs on Host-Only cannot see NAT VMs
- **Use for: Kali's internet adapter**

### Host-Only
- Creates an isolated network between your VMs and host machine
- VMs can talk to each other and to the host
- **No internet access**
- Perfect for attacker ↔ target communication
- **Use for: Lab network between Kali and target VMs**

### Bridged
- VM gets its own IP on your real physical network
- Visible to your router and other devices on your network
- **Do not use for targets** — your exploits could hit real network devices

### Internal Network
- Completely isolated — only VMs on the same internal network name can communicate
- Not even the host can see these VMs
- More isolated than Host-Only
- Good for multi-machine scenarios

> [!danger]
> **Never put a target VM (Metasploitable, DVWA, etc.) on a Bridged adapter.** It will be accessible from your entire home network and potentially the internet if your router is misconfigured.

## Step 4 — Set Up Target Machines

You need something to attack. These are the best starting targets:

### Metasploitable 2 (Beginner Essential)

A deliberately vulnerable Ubuntu VM. Contains dozens of intentionally broken services — old Samba, FTP with backdoors, PHP shells, vulnerable web apps.

Download: https://sourceforge.net/projects/metasploitable/

Import into VirtualBox:
1. Create a new Linux VM, skip the disk creation step
2. Under Storage, attach the `.vmdk` file as the disk
3. Set network to Host-Only only (no NAT)

Credentials: `msfadmin` / `msfadmin`

### DVWA — Damn Vulnerable Web Application

A PHP/MySQL web app with configurable vulnerability levels for SQLi, XSS, CSRF, File Upload, Command Injection, etc.

Run it inside Metasploitable 2 (it's included) or install it separately:

```bash
# On Kali, using Docker:
docker run --rm -it -p 80:80 vulnerables/web-dvwa
```

Access at: http://localhost/

### VulnHub Machines

https://www.vulnhub.com — hundreds of downloadable CTF-style vulnerable VMs, from beginner to advanced. All downloadable as `.ova` or `.vmdk`.

Good beginner machines:
- **Kioptrix Level 1** — classic SMB exploitation
- **Mr. Robot** — web + Linux privesc
- **Basic Pentesting 1** — beginner-friendly intro

### HackTheBox Starting Point

https://www.hackthebox.com — online platform, no local setup needed for most machines. Connect via their VPN. Starting Point machines are free and walk you through concepts step by step.

## Step 5 — Essential Post-Install Config for Kali

Once Kali is running, do this before anything else:

### Update Everything

```bash
sudo apt update && sudo apt full-upgrade -y
```

This updates the package list and upgrades all installed packages. Run this regularly — security tools update frequently.

### Install Common Tools (if not present)

```bash
# Confirm core tools are present
which nmap burpsuite sqlmap hydra metasploit-framework ffuf gobuster
```

If any are missing:

```bash
sudo apt install -y nmap burpsuite sqlmap hydra metasploit-framework ffuf gobuster
```

### Set Up Metasploit Database

Metasploit uses PostgreSQL to store scan results. Set it up:

```bash
sudo msfdb init
msfconsole
```

Inside msfconsole, verify the DB is connected:

```
msf6 > db_status
[*] Connected to msf. Connection type: postgresql.
```

### Set Up Burp Suite

Burp Suite Community Edition comes pre-installed on Kali. Launch it:

```bash
burpsuite &
```

Or find it under Applications → Web Application Analysis.

First-time setup:
1. Proxy → Options → confirm listener is on `127.0.0.1:8080`
2. Configure your browser (or use Burp's built-in browser) to proxy through `127.0.0.1:8080`
3. Install Burp's CA certificate in your browser (Proxy → Intercept → Open Browser → navigate to `http://burp`)

### Install tmux (Essential for Terminal Management)

```bash
sudo apt install -y tmux
```

tmux lets you split your terminal into panes and keep sessions alive. You'll use this on every engagement.

Basic tmux commands:
```bash
tmux                    # Start new session
Ctrl+B, %               # Split vertically
Ctrl+B, "               # Split horizontally
Ctrl+B, arrow keys      # Navigate panes
Ctrl+B, d               # Detach session
tmux attach             # Reattach
```

### Install SecLists (Wordlists)

SecLists is a collection of wordlists used for fuzzing, brute-forcing, and directory enumeration. Essential.

```bash
sudo apt install -y seclists
# Installed to: /usr/share/seclists/
ls /usr/share/seclists/
```

Key directories:
```
/usr/share/seclists/Discovery/Web-Content/     # Directory wordlists
/usr/share/seclists/Passwords/                 # Password lists
/usr/share/seclists/Usernames/                 # Username lists
/usr/share/seclists/Fuzzing/                   # Fuzzing payloads
```

Also already on Kali:
```
/usr/share/wordlists/rockyou.txt.gz            # 14 million passwords
gunzip /usr/share/wordlists/rockyou.txt.gz     # Decompress it
```

## Step 6 — Snapshots

Take a snapshot of your Kali VM **after setup is complete and before any engagement**. If you break something or get infected by a sample, you can roll back instantly.

**VirtualBox:**
```
Machine → Take Snapshot → Name it "Clean Base"
```

**VMware:**
```
VM → Snapshot → Take Snapshot → Name it "Clean Base"
```

Take snapshots of target VMs too — before and after exploitation.

### Step 4 — Create Lab Users and Groups (Intentionally Vulnerable)

This script creates a realistic but deliberately vulnerable AD environment for practising attacks:

```powershell
# Run as CORP\Administrator on DC01

Import-Module ActiveDirectory

# --- Create Organisational Units ---
New-ADOrganizationalUnit -Name "LabUsers"    -Path "DC=corp,DC=local"
New-ADOrganizationalUnit -Name "LabComputers" -Path "DC=corp,DC=local"
New-ADOrganizationalUnit -Name "ServiceAccounts" -Path "DC=corp,DC=local"

# --- Create regular domain users ---
$users = @(
    @{Name="John Smith";   SamAccountName="jsmith";   Password="Password123"},
    @{Name="Alice Brown";  SamAccountName="abrown";   Password="Summer2024!"},
    @{Name="Bob Wilson";   SamAccountName="bwilson";  Password="Welcome1"},
    @{Name="Carol Davis";  SamAccountName="cdavis";   Password="Password1!"}
)

foreach ($u in $users) {
    New-ADUser `
        -Name $u.Name `
        -SamAccountName $u.SamAccountName `
        -UserPrincipalName "$($u.SamAccountName)@corp.local" `
        -Path "OU=LabUsers,DC=corp,DC=local" `
        -AccountPassword (ConvertTo-SecureString $u.Password -AsPlainText -Force) `
        -Enabled $true `
        -PasswordNeverExpires $true
}

# --- Create a local admin on workstations (for lateral movement practice) ---
New-ADUser `
    -Name "Local Admin" `
    -SamAccountName "localadmin" `
    -UserPrincipalName "localadmin@corp.local" `
    -Path "OU=LabUsers,DC=corp,DC=local" `
    -AccountPassword (ConvertTo-SecureString "LocalAdmin123!" -AsPlainText -Force) `
    -Enabled $true `
    -PasswordNeverExpires $true

# Add localadmin to Domain Admins for lab purposes
Add-ADGroupMember -Identity "Domain Admins" -Members "localadmin"

# --- Create service accounts (Kerberoasting targets) ---
$svcs = @(
    @{Name="SQL Service";    SamAccountName="svc_sql";    SPN="MSSQLSvc/SQL01.corp.local:1433"; Password="Service123!"},
    @{Name="Web Service";    SamAccountName="svc_web";    SPN="HTTP/web01.corp.local";          Password="WebPass99"},
    @{Name="Backup Service"; SamAccountName="svc_backup"; SPN="BackupService/dc01.corp.local";  Password="Backup2024"}
)

foreach ($s in $svcs) {
    New-ADUser `
        -Name $s.Name `
        -SamAccountName $s.SamAccountName `
        -UserPrincipalName "$($s.SamAccountName)@corp.local" `
        -Path "OU=ServiceAccounts,DC=corp,DC=local" `
        -AccountPassword (ConvertTo-SecureString $s.Password -AsPlainText -Force) `
        -Enabled $true `
        -PasswordNeverExpires $true

    # Set SPN — makes account Kerberoastable
    Set-ADUser -Identity $s.SamAccountName -ServicePrincipalNames @{Add=$s.SPN}
}

# --- Create an AS-REP Roastable account (no pre-auth required) ---
New-ADUser `
    -Name "Legacy User" `
    -SamAccountName "legacyuser" `
    -UserPrincipalName "legacyuser@corp.local" `
    -Path "OU=LabUsers,DC=corp,DC=local" `
    -AccountPassword (ConvertTo-SecureString "Legacy2020" -AsPlainText -Force) `
    -Enabled $true `
    -PasswordNeverExpires $true

# Disable pre-auth on legacyuser → makes it AS-REP Roastable
Set-ADAccountControl -Identity "legacyuser" -DoesNotRequirePreAuth $true

# --- Nested group for ACL abuse practice ---
New-ADGroup -Name "HelpDesk"   -SamAccountName "HelpDesk"   -GroupScope Global -Path "OU=LabUsers,DC=corp,DC=local"
New-ADGroup -Name "IT Admins"  -SamAccountName "ITAdmins"   -GroupScope Global -Path "OU=LabUsers,DC=corp,DC=local"

Add-ADGroupMember -Identity "HelpDesk"  -Members "jsmith"
Add-ADGroupMember -Identity "ITAdmins"  -Members "HelpDesk"   # nested — jsmith inherits ITAdmins membership

# Give HelpDesk GenericAll over legacyuser (ACL abuse target)
$acl = Get-Acl "AD:CN=Legacy User,OU=LabUsers,DC=corp,DC=local"
$identity = [System.Security.Principal.NTAccount]"CORP\HelpDesk"
$adRights = [System.DirectoryServices.ActiveDirectoryRights]"GenericAll"
$type = [System.Security.AccessControl.AccessControlType]"Allow"
$ace = New-Object System.DirectoryServices.ActiveDirectoryAccessRule($identity, $adRights, $type)
$acl.AddAccessRule($ace)
Set-Acl "AD:CN=Legacy User,OU=LabUsers,DC=corp,DC=local" $acl

Write-Host "[+] Lab users, service accounts, and ACLs created."
Write-Host ""
Write-Host "Attack targets summary:"
Write-Host "  Kerberoastable: svc_sql, svc_web, svc_backup"
Write-Host "  AS-REP Roastable: legacyuser"
Write-Host "  ACL abuse (GenericAll via HelpDesk group): legacyuser"
Write-Host "  Domain Admin path: localadmin"
```

### Step 5 — Disable Windows Defender on DC01 (Lab Only)

Defender will block many offensive tools. Disable it for the lab:

```powershell
# Disable all Defender protection
Set-MpPreference -DisableRealtimeMonitoring $true
Set-MpPreference -DisableIOAVProtection $true
Set-MpPreference -DisableScriptScanning $true
Set-MpPreference -DisableBehaviorMonitoring $true

# Permanently disable via registry (survives reboot)
reg add "HKLM\SOFTWARE\Policies\Microsoft\Windows Defender" /v DisableAntiSpyware /t REG_DWORD /d 1 /f

# Turn off Windows Firewall (all profiles)
netsh advfirewall set allprofiles state off
```

> [!warning] Only do this in an isolated host-only network lab. Never disable Defender on a machine with internet access or connected to your real network.

### Step 10 — Configure Kali to Reach the Lab

On Kali, the Host-Only adapter should already be on 192.168.56.x. Verify:

```bash
ip a | grep 192.168.56
# Should show: 192.168.56.10 (or similar)
```

If Kali doesn't have a Host-Only IP:
```bash
# Check which interface is on the host-only network
ip a
# Look for an interface without an IP on 192.168.56.x

# Assign manually (replace eth1 with your interface name)
sudo ip addr add 192.168.56.10/24 dev eth1
sudo ip link set eth1 up

# Make permanent (add to /etc/network/interfaces or use nmcli)
sudo nmcli connection modify "Wired connection 2" ipv4.addresses 192.168.56.10/24 ipv4.method manual
sudo nmcli connection up "Wired connection 2"
```

Configure `/etc/hosts` on Kali for convenience:
```bash
echo "192.168.56.20  dc01.corp.local dc01 corp.local" | sudo tee -a /etc/hosts
echo "192.168.56.30  ws01.corp.local ws01" | sudo tee -a /etc/hosts
```

Configure `/etc/resolv.conf` to use DC01 as DNS:
```bash
echo "nameserver 192.168.56.20" | sudo tee /etc/resolv.conf
```

Test connectivity:
```bash
ping -c 1 192.168.56.20    # DC01
ping -c 1 192.168.56.30    # WS01
nslookup corp.local 192.168.56.20
```

### Optional — Install ADCS on DC01 (for ESC Practice)

To practise the attacks in `04.5 — ADCS Certificate Abuse`, install ADCS on DC01:

```powershell
# Run on DC01 as CORP\Administrator

# Install ADCS role
Install-WindowsFeature -Name AD-Certificate -IncludeManagementTools

# Configure the CA
Install-AdcsCertificationAuthority `
    -CAType EnterpriseRootCA `
    -CACommonName "corp-CA" `
    -KeyLength 2048 `
    -HashAlgorithmName SHA256 `
    -ValidityPeriod Years `
    -ValidityPeriodUnits 5 `
    -Force

# Install Web Enrollment (required for ESC8)
Install-AdcsWebEnrollment -Force

# Restart Certificate Services
Restart-Service CertSvc
```

After installation, Certipy enumeration will show the CA and its templates:
```bash
certipy find -u jsmith@corp.local -p 'Password123' -dc-ip 192.168.56.20 -vulnerable -stdout
```

To create an ESC1-vulnerable template for practice:


### Troubleshooting Common Lab Issues

```
Problem: Kali can't reach DC01
Fix: Check both VMs are on the same Host-Only network (vboxnet0)
     Check IPs: ip a on Kali, ipconfig on DC01
     Ping in both directions

Problem: Domain join fails from WS01
Fix: WS01's DNS must point to DC01 (192.168.56.20)
     Confirm with: nslookup corp.local — should resolve from DC01

Problem: BloodHound.py fails with "LDAP error"
Fix: Ensure DC01 is reachable and LDAP (389) is open
     Test: nc -zv 192.168.56.20 389

Problem: Kerberoasting returns no results
Fix: Confirm SPNs are set: Get-ADUser svc_sql -Properties ServicePrincipalNames
     Re-run the user creation script if accounts are missing

Problem: VM is slow / out of RAM
Fix: Reduce DC01 RAM to 2 GB if host has <16 GB
     Disable Windows Search and Superfetch on DC01/WS01:
     sc stop WSearch && sc config WSearch start= disabled
     sc stop SysMain && sc config SysMain start= disabled
```
