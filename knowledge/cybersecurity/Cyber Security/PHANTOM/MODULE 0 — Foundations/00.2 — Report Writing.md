# 00.2 — Report Writing

## Why Reports Matter

The report is the **only deliverable the client keeps**. Your 40 hours of hacking means nothing if the report doesn't communicate what you found, why it matters, and what to do about it.

A bad report gets filed and forgotten. A good report gets vulnerabilities fixed.

Senior pentesters are judged as much on report quality as on technical skill.

---

## Report Structure

A professional pentest report has two audiences:
- **The executive** — doesn't care about CVE IDs, cares about business risk
- **The technical team** — needs exact reproduction steps to fix the issue

Both need to be in the same document.

### Standard Sections

#### 1. Cover Page
- Client name, engagement type, date range
- Pentest firm name, lead tester, classification (CONFIDENTIAL)
- Report version (v1.0, v1.1 after client review)

#### 2. Executive Summary
- Written in plain English — no jargon
- 1–2 pages maximum
- Answers: What did we test? What is the overall risk posture? What are the 3 most critical things to fix immediately?
- Include a risk rating chart (Critical/High/Medium/Low/Info counts)

#### 3. Scope and Methodology
- Exact scope (IPs, domains, applications)
- Dates and times of testing
- Methodology used (PTES, OWASP, etc.)
- Tools used (at a high level — detailed tool list goes in appendix)
- Testing type (black/grey/white box)

#### 4. Findings
This is the core of the report. Each finding gets its own subsection.

**Finding Template:**

```
Finding ID:       VULN-001
Title:            SQL Injection in /login endpoint
Severity:         Critical
CVSS Score:       9.8 (CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H)
Affected Asset:   https://app.example.com/login
```

**Description:**
What the vulnerability is, in plain terms. What is the root cause?

**Evidence:**
Screenshots, HTTP requests/responses, tool output. Everything must be reproducible.

**Reproduction Steps:**
```
1. Navigate to https://app.example.com/login
2. Enter the following in the username field: ' OR 1=1--
3. Observe that authentication is bypassed
```

**Impact:**
What can an attacker actually do with this? Be specific. "Attacker can extract all customer records from the database" is better than "data breach possible."

**Remediation:**
Specific, actionable fix. Reference the developer. "Use parameterised queries / prepared statements. Do not construct SQL strings from user input. Apply input validation as a secondary control."

**References:**
- OWASP: https://owasp.org/www-community/attacks/SQL_Injection
- CWE-89

---

#### 5. Remediation Summary
A table of all findings with severity, status, and owner. Used by the client to track fixes.

| ID | Title | Severity | Owner | Status |
|---|---|---|---|---|
| VULN-001 | SQL Injection | Critical | Dev Team | Open |
| VULN-002 | Missing HSTS | Low | DevOps | Open |

#### 6. Appendix
- Full tool list with versions
- Raw scan output (Nmap, Nikto, etc.)
- Detailed technical notes that don't fit the main findings
- Methodology reference

## CVSS Scoring — How to Rate Severity

CVSS (Common Vulnerability Scoring System) v3.1 is the industry standard for rating vulnerability severity. Every finding must have a CVSS score.

### Base Score Metrics

| Metric | Options |
|---|---|
| **Attack Vector (AV)** | Network (N), Adjacent (A), Local (L), Physical (P) |
| **Attack Complexity (AC)** | Low (L), High (H) |
| **Privileges Required (PR)** | None (N), Low (L), High (H) |
| **User Interaction (UI)** | None (N), Required (R) |
| **Scope (S)** | Unchanged (U), Changed (C) |
| **Confidentiality (C)** | None (N), Low (L), High (H) |
| **Integrity (I)** | None (N), Low (L), High (H) |
| **Availability (A)** | None (N), Low (L), High (H) |

### Severity Ratings

| Score | Rating |
|---|---|
| 9.0–10.0 | **Critical** |
| 7.0–8.9 | **High** |
| 4.0–6.9 | **Medium** |
| 0.1–3.9 | **Low** |
| 0.0 | **Informational** |

Use the NVD CVSS calculator: https://nvd.nist.gov/vuln-metrics/cvss/v3-calculator

### Practical Examples

| Finding | CVSS Vector | Score |
|---|---|---|
| Unauthenticated RCE via network | AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H | 9.8 |
| Stored XSS requiring login | AV:N/AC:L/PR:L/UI:R/S:C/C:L/I:L/A:N | 5.4 |
| Self-XSS (affects only yourself) | AV:N/AC:L/PR:N/UI:R/S:U/C:N/I:N/A:N | 0.0 (Info) |
| Local privesc via misconfigured sudo | AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H | 7.8 |

## Evidence Standards

> [!tip]
> **Screenshot everything before you exploit.** Once you pop a shell, capture proof immediately — the session may die. A finding without evidence is a finding the client can dispute.

Good evidence includes:
- HTTP requests and responses (from Burp Suite, use "Copy as curl command")
- Terminal output with the hostname and timestamp visible
- Screenshots of the vulnerable parameter and the impact
- Before and after — show the normal behaviour, then show the exploit working

Always redact sensitive data that isn't needed to prove the finding (real passwords, PII, other users' data).

---

## Writing Style

- **Passive voice for descriptions**: "The application was found to be vulnerable to SQL injection."
- **Active voice for impact**: "An attacker can extract the entire customer database."
- **Remediation is prescriptive, not vague**: "Implement parameterised queries" not "fix the SQL injection."
- No CVE names without explanation — always describe what the CVE actually is.
- Severity levels must be justified. "Critical because unauthenticated attacker achieves RCE over the internet" is justified. "Critical" alone is not.

## Quick Reference

| Section | Purpose |
|---|---|
| Executive Summary | Business risk, plain English, 1–2 pages |
| Scope | What was tested, when, how |
| Findings | Individual vulns with evidence + CVSS |
| Remediation Summary | Tracking table for fixes |
| Appendix | Raw output, tool list |
| CVSS 9.0–10.0 | Critical |
| CVSS 7.0–8.9 | High |
| CVSS 4.0–6.9 | Medium |
| CVSS 0.1–3.9 | Low |
