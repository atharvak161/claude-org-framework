# 00.1 — What is Penetration Testing

## What It Is

Penetration testing (pentest) is the practice of **legally and deliberately attacking a system, network, or application** to find security vulnerabilities before a real attacker does.

You are hired to think and act like a malicious hacker — but with written permission, a defined scope, and a professional report at the end.

It is **not** vulnerability scanning. A scanner finds known issues automatically. A pentest involves human creativity, chaining vulnerabilities together, and proving real-world impact.

---

## Why It Matters

- Organisations cannot defend what they don't know is broken
- Compliance frameworks (PCI-DSS, ISO 27001, SOC 2) require regular pentesting
- A real attacker doesn't care about your patch schedule — a pentester simulates that threat before the damage is done
- Insurance and regulatory bodies increasingly demand evidence of testing

---

## The Three Box Types

### Black Box
- You receive **nothing** — just a target (IP range, domain)
- Simulates an **external, unknown attacker**
- Most realistic, most time-consuming, highest chance of missing coverage
- You must do full recon from scratch

### Grey Box
- You receive **partial information** — maybe a low-privilege account, some network diagrams, or application documentation
- Simulates an **insider threat** or a compromised user
- Most common in commercial engagements — balances realism with efficiency

### White Box
- You receive **everything** — source code, architecture diagrams, credentials, config files
- Simulates a **code review + security audit**
- Maximum coverage, minimum wasted time
- Common for secure development lifecycle (SDL) testing and compliance audits

---

## Legal and Ethical Framework

### The Golden Rule
**Written permission before you touch anything.** No exceptions.

Unauthorised access to computer systems is a criminal offence in virtually every jurisdiction:
- UK: Computer Misuse Act 1990 (Section 1, 2, 3 — up to 10 years imprisonment)
- US: Computer Fraud and Abuse Act (CFAA)
- EU: Directive on Attacks Against Information Systems

> [!danger]
> **Scanning a system you don't have written permission to test is illegal.** Even if you "just wanted to see". Even on your employer's network without explicit authorisation. Even on a system you discovered via Google.

### What Protects You
- A signed **Statement of Work (SoW)** or **Rules of Engagement (RoE)** document
- Written scope defining exactly what you can test, when, and how
- Point-of-contact details for the client (to call if something breaks)
- Emergency stop clause — conditions under which testing must halt immediately

---

## Rules of Engagement (RoE)

This is the contract that governs the engagement. Before any testing begins, you and the client must agree on:

| Item | Example |
|---|---|
| Scope — in scope | `10.0.0.0/24`, `*.example.com` |
| Scope — out of scope | `payroll.example.com`, third-party SaaS |
| Testing window | Mon–Fri 09:00–17:00 UTC only |
| Allowed techniques | Port scanning, web app testing, phishing |
| Prohibited techniques | DoS attacks, physical access, social engineering staff |
| Emergency contact | CISO mobile number |
| Data handling | No exfiltration of real customer PII |
| Reporting deadline | 5 business days after test ends |

> [!warning]
> If you discover a critical vulnerability (e.g., unauthenticated RCE, active breach), **stop and call the emergency contact immediately**. Do not continue exploiting. Document and escalate.

## Scoping

Scoping is the process of defining what will be tested. Bad scoping = missed vulnerabilities or accidental damage to production systems.

### What Gets Scoped
- IP ranges / subnets
- Domain names and subdomains (wildcards vs explicit)
- Specific applications (e.g., "only the customer portal, not the admin portal")
- Cloud accounts (AWS account IDs, Azure subscriptions)
- Physical locations (if physical testing is included)
- People (if social engineering is included)

### Scoping Questions to Ask the Client
1. What are your crown jewels — what data/systems would cause the most damage if compromised?
2. Are there any systems we must avoid (life-critical, production databases with no backups)?
3. Are third-party systems in scope? (CDN providers, SaaS tools, payment processors)
4. Do you want us to test during business hours or outside?
5. What is your incident response capability — will our alerts trigger your SOC?

## Penetration Testing Methodologies

Methodologies are structured frameworks that ensure you don't miss anything. A good pentester follows a methodology even when improvising.

### PTES — Penetration Testing Execution Standard
The most widely used framework for commercial pentests. Seven phases:

1. **Pre-engagement** — Scope, RoE, legal sign-off
2. **Intelligence Gathering** — Recon (passive and active)
3. **Threat Modelling** — Who would attack this, and how?
4. **Vulnerability Analysis** — Identify weaknesses
5. **Exploitation** — Prove the vulnerability is real
6. **Post-Exploitation** — What can you do after access? Lateral movement, data access
7. **Reporting** — Document everything with evidence and remediation

### OWASP Testing Guide (OTG)
Specifically for web application testing. Provides a checklist of hundreds of test cases across categories like authentication, session management, input validation, and cryptography. Version 4.2 is current.

Use this for every web app engagement. It's the industry standard.

### OSSTMM — Open Source Security Testing Methodology Manual
More academic/compliance-oriented. Covers networks, communications, wireless, human factors. Less commonly used in commercial red teaming but appears in compliance-heavy industries.

### NIST SP 800-115
US government framework for technical security testing. Defines four phases: Planning, Discovery, Attack, Reporting. Used heavily in US federal contracts.

### MITRE ATT&CK
A knowledge base of real-world attacker **Tactics, Techniques, and Procedures (TTPs)**. Not a methodology for running tests — it's a framework for **mapping what you did** to how real threat actors operate.

- **Tactics** = the goal (e.g., Lateral Movement, Credential Access)
- **Techniques** = how the goal is achieved (e.g., Pass-the-Hash, Kerberoasting)
- **Sub-techniques** = specific implementations

Used heavily in red team reporting and purple team exercises. Learn to map your findings to ATT&CK IDs.

Website: https://attack.mitre.org

### Cyber Kill Chain (Lockheed Martin)
A linear 7-stage model of how attacks unfold:

1. **Reconnaissance** — Gather info on the target
2. **Weaponisation** — Create a payload/exploit
3. **Delivery** — Send the payload (phishing, USB, web)
4. **Exploitation** — Trigger the vulnerability
5. **Installation** — Establish persistence (malware, backdoor)
6. **Command & Control (C2)** — Maintain communication with compromised system
7. **Actions on Objectives** — Steal data, cause damage, move laterally

Useful for explaining attack chains to clients in reports.

## Types of Pentest Engagements

| Type | Description |
|---|---|
| **External Network** | Attack from the internet against public-facing infrastructure |
| **Internal Network** | Attack from inside the network (simulates rogue employee or post-phishing) |
| **Web Application** | Full OWASP-based assessment of a web app |
| **Mobile Application** | Android/iOS app assessment |
| **API Testing** | REST/GraphQL/SOAP API security assessment |
| **Wireless** | Wi-Fi security, rogue APs, WPA2 cracking |
| **Social Engineering** | Phishing, vishing, physical intrusion |
| **Red Team** | Full-scope, goal-based, long-duration adversary simulation |
| **Purple Team** | Red + Blue working together — attacker runs TTPs while defender tunes detection |
| **Cloud** | AWS/Azure/GCP misconfigurations, IAM abuse |
| **Active Directory** | Domain compromise path analysis |
