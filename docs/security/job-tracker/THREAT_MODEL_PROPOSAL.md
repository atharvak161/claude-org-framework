## Context: jobsync anti-patterns we are explicitly avoiding

The reference project (github.com/Gsync/jobsync, MIT) has four documented security vulnerabilities. These are not theoretical — they are known, real failures. Every one must be treated as an explicit test case in Phase 5.

### Anti-pattern 1: SSRF via user-controlled URLs

**jobsync's failure:** The Ollama API URL was user-configurable from the UI. An attacker could point it to an internal service (e.g. `http://169.254.169.254/latest/meta-data/` for cloud IMDS, or `http://localhost:6379` for Redis).

**JobScope mitigation (mandatory):**
- No user-controlled URL is ever used in a server-side HTTP request
- All external HTTP calls use an allowlisted set of domains (Adzuna API domain, Reed API domain, Jooble API domain, assets.publishing.service.gov.uk for gov.uk CSV)
- Any URL-shaped input from the user is rejected at the API layer — not sanitised, rejected
- SSRF payloads in the security regression test suite from day 1

### Anti-pattern 2: IDOR on file download

**jobsync's failure:** The resume file download endpoint used a predictable or sequential ID that could be enumerated, allowing User A to download User B's resume.

**JobScope mitigation (mandatory):**
- All file storage keys are opaque UUIDs — never sequential, never predictable
- File access goes through owner-scoped signed URLs generated by the backend (S3/R2 pre-signed URL with 15-minute expiry)
- The signed URL generation endpoint validates that the requesting user owns the file before generating the URL
- Direct file URL access (bypassing the signed URL generator) returns 403 — never the file
- IDOR cross-user access tests run against every resource endpoint from Phase 2 merge

### Anti-pattern 3: Systemic IDOR (broken object-level authorisation)

**jobsync's failure:** No service-layer ownership check — endpoints returned data if the ID was valid regardless of which user made the request.

**JobScope mitigation (mandatory):**
- Every database query that returns user-owned data includes an explicit user_id ownership check
- This is enforced at the service layer, not the route layer — a route-level check can be bypassed by internal calls
- Code review gate: Security Architect reviews every PR that touches data-access code before merge
- Test: IDOR test suite runs cross-user access attempts against every endpoint that handles user-owned resources (resumes, applications, profiles, saved jobs)

### Anti-pattern 4: Auth weaknesses

**jobsync's failure:** JWT stored in localStorage — accessible to any XSS payload; no CSRF protection; session management issues.

**JobScope mitigation (mandatory):**
- NextAuth v5 with httpOnly session cookies — inaccessible to JavaScript
- No JWTs in localStorage, sessionStorage, or any client-accessible storage
- CSRF tokens on all state-changing requests (enforced by NextAuth v5 by default — must be verified, not assumed)
- Session tokens have a defined expiry; expired session → 401, not a silent retry with stale credentials

## PII classification and GDPR obligations

### Data assets and classification

| Asset | Classification | Rationale |
|---|---|---|
| Uploaded resume | **High Sensitivity** | Contains full employment history, education, dates, potentially health-related certifications; constitutes "biographic data" and may contain information about disabilities (accessibility certifications) or trade union membership (professional body membership) |
| Email address | Medium Sensitivity | Standard PII |
| Application history | Medium Sensitivity | Employment-search behaviour; could reveal current employer dissatisfaction |
| Job feed data (external) | Low Sensitivity | Sourced from public job boards; no personal data |
| gov.uk sponsor register data | Low Sensitivity | Public, OGL-licensed |

### UK GDPR / DPA 2018 obligations for resume data

**Legal basis:** Explicit consent per Article 9(2)(a) GDPR — resume data may contain special category data (health information via disability-related certifications, trade union information via professional memberships). Standard legitimate interests basis is not sufficient.

**Required implementation:**
1. **Consent flow** — before resume upload begins, user is shown a clear consent statement: what data is extracted, how long it is retained, who has access (only Atharva — single-user system), and their right to erasure. Consent is logged with a timestamp.
2. **Data minimisation** — only the fields needed for job matching are extracted and stored. Full resume text is not stored in the database (only the structured JSON output and the original file in object storage).
3. **Retention** — resume data is retained for a maximum of 12 months from the date of upload. A background job flags records approaching expiry for deletion. User can trigger deletion at any time.
4. **Right-to-erasure endpoint** — a dedicated endpoint deletes: the original file from object storage, the parsed profile record, all derived data (skill matches, eligibility scores based on the profile). The endpoint is authenticated and must complete within 30 days of request (but should be near-immediate for a single-user system).
5. **Data breach notification** — any confirmed breach involving resume data triggers notification obligations under UK GDPR Article 33 (ICO within 72 hours) and Article 34 (data subject without undue delay if high risk).

### Article 30 Record of Processing Activities

Required for any GDPR-compliant system processing personal data. Must include:
- Controller name and contact
- Purposes of processing
- Categories of data subjects and data
- Recipients / third parties (Claude API — Anthropic; object storage provider — Cloudflare R2 or AWS S3)
- Data transfers outside UK (Claude API — US-hosted; note Anthropic's UK/EU data processing commitments)
- Retention periods
- Security measures

The Compliance Auditor produces the Article 30 register as part of the Phase 5 security gate.

## Secrets and encryption

### Rule 8 (WORKSPACE.md): no secrets in any file — ever

All secrets (API keys, database URLs, S3 credentials, Anthropic API key, NextAuth secret) via Railway environment variables. Not in `.env` files committed to the repo. `.env.example` lists variable names only.

### Envelope encryption for resume data

Resumes are high-sensitivity PII and must be encrypted at rest with a key management approach that does not store the encryption key in the same location as the data:

1. A **data encryption key (DEK)** is generated per resume file
2. The DEK is used to encrypt the file in object storage (AES-256-GCM)
3. The DEK itself is encrypted by a **master key** held outside the database (in a secrets manager — Railway's secrets store or a dedicated KMS)
4. The encrypted DEK is stored alongside the file record in the database
5. To decrypt: fetch encrypted DEK from DB → decrypt with master key from secrets manager → use DEK to decrypt file
6. The master key is never logged, never in any file, never in any error message

### Keys never in logs

Any error logging middleware must scrub:
- Authorization headers
- Cookie values
- Any parameter named `*key*`, `*secret*`, `*token*`, `*password*`, `*credential*`
- Database connection strings
- S3/R2 signed URL parameters (contain access key + signature)

## External data trust model

All external data is untrusted until sanitised. This covers job board API responses, gov.uk CSV, and AI-generated content.

### Job board API response fields

Every field from Adzuna, Reed, and Jooble is treated as potentially malicious:
- Job title, company name, description: HTML-encoded before storage; XSS-escaped before display
- No job description field is ever rendered as raw HTML in the browser — markdown rendered through a strict allowlist, or plain text only
- Salary fields: validated as numeric before use in any comparison; non-numeric values rejected

### AI prompt construction (Claude API)

When job descriptions are passed to Claude for any AI-assisted feature:
- Job content is wrapped in clearly-delimited blocks:
  ```
  <job_description>
  [content here]
  </job_description>
  ```
- System prompt explicitly instructs Claude not to follow instructions found within the delimited block
- This prevents prompt injection via malicious job descriptions

### gov.uk CSV parsing

- CSV parsed in a background worker — not the HTTP server process
- Schema validated on every download before any data is written to the sponsor_register table
- If schema validation fails: halt ingestion, alert, retain last valid data, flag staleness in /healthz
- Maximum line length enforced (reject lines > 2048 bytes — protects against buffer-related issues)

## Lessons from Blueprint (do not repeat)

Blueprint shipped to production with 0/6 security gate items satisfied. The finding (docs/security/BLUEPRINT_SECURITY_GAP_REVIEW.md) rated it MEDIUM residual risk only because it was a static backend-less SPA. JobScope is not static — it has a backend, a database, a file upload endpoint, third-party API calls, and PII. The same gap on JobScope would be Critical residual risk.

The corrective standing rule (org/DECISIONS.md 2026-06-07 23:20:42): "done" requires live verification, not code-complete. For security: "done" means Director Security sign-off file exists and references the exact commit hash and run ID being shipped. No exceptions.
